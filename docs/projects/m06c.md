# Embedding Evaluation Benchmark Suite — Thinking Guide
> **Module:** 06 — GenAI Core · **Difficulty:** Advanced · **Type:** THINKING · **Time:** ~8 hours

## The Problem

**Input:** Four embedding models — `all-MiniLM-L6-v2` (22M params, D=384), `all-mpnet-base-v2` (110M params, D=768), `BGE-small-en` (33M params, D=384), `GTE-base` (110M params, D=768). Three evaluation tasks: (1) semantic similarity on STS-B (Spearman $\rho$ between predicted cosine similarities and human scores), (2) document retrieval on a BEIR subset — SciFact NDCG@10, (3) topic clustering on 20 Newsgroups (k-means NMI).

**Output:** A model × task matrix of scores, overall rank, inference time per 1K samples, and a Pareto frontier of model size vs mean performance.

**Constraints:** Evaluation must be reproducible — fixed random seeds, deterministic k-means initialization. Each task uses the model's recommended pooling strategy. Wall-clock time per model reported alongside quality metrics.

---

## Before You Write Any Code: Build the Mental Model

No single task is sufficient to characterize an embedding model because different fine-tuning objectives create different inductive biases. A model trained entirely on STS data learns to produce similar embeddings for paraphrases but may collapse semantically distinct documents that share surface vocabulary — hurting retrieval. A model trained for retrieval produces embeddings that rank correctly but whose absolute cosine similarity values may not correlate with human similarity judgments — hurting STS evaluation.

The three task types are orthogonal in what they measure. STS-B asks: are distances in embedding space monotonically consistent with human similarity judgments? Tests pairwise geometry. Retrieval asks: given a query, are the relevant documents ranked above non-relevant ones? Tests query-collection discrimination. Clustering asks: do embeddings of topically similar documents cluster together regardless of absolute position? Tests macro-level geometric structure.

A model can rank #1 on STS-B while ranking #3 on retrieval and #2 on clustering. Understanding why requires knowing the training loss function and data. BGE-small uses hard-negative contrastive fine-tuning on MS-MARCO (retrieval data) — this makes it excellent at separating relevant from non-relevant documents but does not optimize pairwise similarity to match human judgments. `all-mpnet-base-v2` uses NLI and paraphrase data — good at similarity, weaker on discriminative retrieval. Model size does not predict which task wins.

## Decompose It

### Phase 1 — STS-B: Spearman vs Pearson and Why the Difference Matters

**The question this phase answers:** How do you measure whether cosine similarities correlate with human similarity judgments, and why is rank correlation the correct choice?

**Key decisions:**
- STS-B dataset: 8,628 sentence pairs with human similarity scores 0–5. Embed both sentences, compute cosine similarity, measure how well the ordering matches human scores.
- Spearman $\rho$: converts both variables to ranks, then computes Pearson on the ranks. Measures monotonic correlation — whether cosine similarity ordering matches human score ordering, regardless of scale. $\rho = 0.8$ means "80% of the variance in human rank-order is explained by the rank-order of cosine similarities."
- Pearson $r$: measures linear correlation. Assumes both variables are on ratio scales where absolute differences are meaningful. Cosine similarity values are model-dependent — two models can produce similarities of 0.3 and 0.9 for the same pair, with the 0.3 model being more accurate in ranking. Pearson penalizes the 0.3 model for having the wrong absolute scale.

**Why Spearman is correct:** Human similarity judgments are on an ordinal scale (0–5) where the gap between 3.0 and 4.0 may not equal the gap between 4.0 and 5.0 in any objective sense. Spearman evaluates only the ordering, which is model-agnostic. This is why the SBERT paper and all major embedding benchmarks report Spearman $\rho$, not Pearson $r$.

**Trap:** Scaling human scores (dividing by 5 to normalize to [0,1]) before computing Spearman. Spearman only uses ranks — monotonic scaling changes nothing. But if you then compute Pearson on the scaled values thinking the scales now match, Pearson is still wrong for the same reason.

**Done when:** For each model, a single Spearman $\rho$ value on the full STS-B test set (1,379 pairs). Expected values: `all-mpnet-base-v2` ≈ 0.84, `all-MiniLM-L6-v2` ≈ 0.82. Values below 0.70 indicate wrong pooling or wrong similarity metric (using Euclidean instead of cosine).

### Phase 2 — BEIR Retrieval: NDCG@10 and Evaluation Correctness

**The question this phase answers:** How do you run a BEIR-style retrieval evaluation and avoid the common scoring bugs?

**Key decisions:**
- Use SciFact (300 queries, 5,183 docs) for tractability. TREC-COVID has 171,332 docs — acceptable but slow for a benchmark comparison.
- NDCG@10 computation — the correctness details: For each test query, retrieve top-10 document IDs. Look up each retrieved doc's relevance grade in qrels (0 if absent). $\text{DCG@10} = \sum_{i=1}^{10} rel_i / \log_2(i+1)$. IDCG@10 = DCG of the ideal ranking (sort all relevant docs by grade descending, take top-10). NDCG@10 = DCG@10 / IDCG@10. Average across all queries that have at least one relevant document (skip zero-qrel queries — undefined metric).
- Binary vs graded relevance: SciFact has binary relevance (0 or 1). The linear NDCG formula ($rel_i / \log_2(i+1)$) and the exponential formula ($(2^{rel_i} - 1) / \log_2(i+1)$) are equivalent for binary relevance ($2^1 - 1 = 1$, $2^0 - 1 = 0$). Report which formula you use; inconsistency across papers is a major source of non-reproducibility.

**Trap:** Computing IDCG over all relevant documents in the corpus rather than capping at K. Standard BEIR computes IDCG over the full relevance list: the denominator assumes perfect retrieval of every relevant document. This gives lower NDCG than capping at K, because IDCG can only be achieved if all relevant documents happen to be in top-10. Check the benchmark's definition and be consistent.

**Done when:** NDCG@10 for each model on SciFact. Expected values: `BGE-small-en` ≈ 0.66, `all-mpnet-base-v2` ≈ 0.64, `all-MiniLM-L6-v2` ≈ 0.58. BGE-small beating a 3× larger model reflects retrieval-specific training, not superior architecture.

### Phase 3 — 20 Newsgroups Clustering: NMI and the Anisotropy Problem

**The question this phase answers:** How do you measure whether embeddings cluster by topic, and why does the geometry of high-dimensional embedding spaces make this harder than it sounds?

**Key decisions:**
- Task: embed all documents from the 20 Newsgroups test set (~7,532 docs, 20 categories), run k-means with k=20, measure cluster-to-label alignment.
- NMI (Normalized Mutual Information): $\text{NMI} = 2I(C; K) / (H(C) + H(K))$ where $C$ is the true class distribution, $K$ is the cluster assignment, $I$ is mutual information, $H$ is entropy. NMI = 0.0 means cluster assignments are independent of true labels. NMI = 1.0 means perfect recovery up to permutation. NMI is invariant to label permutation — cluster "3" doesn't have to align with category "sci.med" for NMI to be high.
- Why NMI over accuracy: k-means produces unlabeled cluster assignments. Accuracy requires finding the optimal mapping between cluster IDs and category labels (the Hungarian algorithm, $O(k^3)$). NMI avoids this by measuring information content directly.

**Anisotropy:** Embedding models produce vectors that are not uniformly distributed on the unit hypersphere — they cluster in a narrow cone. The average cosine similarity between two random (non-paired) embeddings can be 0.3–0.6 instead of the theoretical ~0.0 for uniform distribution. All embeddings share a common direction imposed by the model's training on the mean token embedding.

**Why anisotropy hurts clustering:** If all embeddings are geometrically close (cosine sim > 0.3 for random pairs), within-cluster and between-cluster distances are compressed into a small range. k-means partitions based on tiny differences in a crowded space — clustering is unstable and sensitive to initialization. Whitening helps: transform embeddings so each dimension has unit variance and zero covariance. `sklearn.decomposition.PCA(whiten=True)` implements this. Measure anisotropy first (average pairwise cosine sim on a random 1K-sample); apply whitening only if average > 0.2.

**Trap:** Running k-means on high-dimensional embeddings (D=768) once and reporting that NMI. k-means variance is high — different random seeds produce NMI differences up to ±0.03, which can exceed the difference between models. Run 10 restarts with `KMeans(n_init=10, random_state=42)` and report mean ± std over 5 independent runs.

**Done when:** NMI measured for both raw and whitened embeddings. Whitening improvement is often 0.03–0.08 NMI for anisotropic models. Runtime for KMeans(D=768, k=20, N=7500) is 30–120 seconds on CPU; reduce D with PCA to 64 dimensions first to speed up by 12×.

### Phase 4 — Pareto Frontier and Model Size vs Performance Trade-Off

**The question this phase answers:** Do bigger models always win, and how do you visualize the quality-efficiency trade-off?

**Key decisions:**
- Score normalization for aggregation: raw scores across tasks are on the same 0–1 scale but have different magnitudes (STS-B range 0.80–0.84, NDCG range 0.58–0.66, NMI range 0.30–0.45). Normalize each task score to [0,1] using min-max across the 4 models: $s_{norm} = (s - \min_m) / (\max_m - \min_m)$. Then average. This prevents any single task from dominating the aggregate ranking.
- A model is Pareto-optimal if no other model is both smaller AND better on all tasks. With 4 models and 3 tasks, a model dominates another only if it beats it on all 3 tasks simultaneously — common for models trained on the same type of data at different scales.
- The empirical finding: `BGE-small-en` is typically Pareto-optimal vs `all-MiniLM-L6-v2` — slightly larger (33M vs 22M) but better on retrieval due to hard-negative contrastive training. `all-mpnet-base-v2` and `GTE-base` are both 110M params; which is Pareto-superior depends on task weighting.

**Trap:** Using rank-based aggregation (average rank across tasks) without accounting for score gap. If STS-B separates models by 0.02 Spearman and NDCG separates them by 0.12, raw rank aggregation treats both as equally important 1-rank differences. Normalized score aggregation captures that the NDCG differences are 6× larger and should carry more weight in the overall ranking.

**Done when:** Printed comparison table: model | STS-B Spearman | BEIR NDCG@10 | 20NG NMI | throughput (docs/sec) | overall rank. Clear statement of which model is Pareto-optimal for retrieval-focused vs similarity-focused use cases.

### Phase 5 — Reproducibility and Fair Evaluation Protocol

**The question this phase answers:** What must be fixed for evaluation results to be comparable across runs and across papers?

**Key decisions:**
- Random seeds: fix k-means with `KMeans(n_init=10, random_state=42)`. Report mean NMI and standard deviation over 5 runs with different random states — the std characterizes the variance, not just the mean.
- Tokenizer truncation: `all-MiniLM-L6-v2` has max 256 tokens; `all-mpnet-base-v2` has max 512. Truncate at `min(model_max_length, 256)` for all models to ensure equal information access per document. Document this explicitly — it affects models differently.
- Hardware parity: all throughput measurements on the same device with the same batch size (batch_size=32). GPU vs CPU throughput comparison is meaningless — always compare on identical hardware.
- Library version: pin `sentence-transformers` version in requirements.txt. Pooling or normalization logic changes across versions can shift scores by 1–2 Spearman points.

**Trap:** Loading different models with different `device` settings (one on GPU, one on CPU) and comparing inference times as equivalent. Always evaluate all models on the same hardware with the same device setting.

**Done when:** Results are fully reproducible from a single script run with fixed seeds. A `--model` CLI argument selects which model to evaluate; no hardcoded assumptions in evaluation functions.

## Edge Cases to Handle Before You Code

1. **STS-B similarity scores for pairs with no shared tokens:** Some STS-B pairs have human similarity = 0.0 (completely unrelated sentences). Anisotropic models may still produce cosine similarities of 0.3–0.5 due to the common embedding cone direction — this inflates Pearson (creates an artificial floor) but doesn't affect Spearman as severely because it shifts all scores uniformly.

2. **BEIR queries with zero relevant documents in qrels:** Some queries have no annotated relevant documents. NDCG@10 = undefined for these. Skip these queries when averaging — divide by the count of queries with at least one qrel entry, not by total query count. Off-by-one here underestimates NDCG.

3. **k-means on D=768 embeddings with N=7,500 samples:** `KMeans(algorithm='lloyd')` in sklearn is $O(k \cdot N \cdot D \cdot \text{iter})$. With k=20, N=7500, D=768, 300 iterations: ~34B multiply-adds — takes 30–120 seconds on CPU. Use PCA to 64 dimensions before clustering: PCA(n_components=64, whiten=True).fit_transform() reduces runtime by 12× with < 5% NMI loss for sentence embeddings.

4. **Float16 vs float32 rounding differences across models:** Models auto-cast to float16 on CUDA. Float16 rounding changes cosine similarities by ~0.001, which is negligible for ranking but affects reproducibility. Use `model.float()` consistently across all models for deterministic results.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|----------|----------|----------------|
| Similarity metric for STS | Cosine similarity | Dot product | Always cosine — dot product conflates magnitude with direction; cosine isolates semantic direction which is what human similarity ratings measure |
| Aggregation across tasks | Normalized score mean | Mean rank | Normalized score mean — rank aggregation loses information about score gaps; 0.01 NDCG difference and 0.20 NDCG difference both contribute 1 rank unit |
| Whitening for clustering | Apply to all models | Apply selectively based on anisotropy | Selective — whitening an already-isotropic model can hurt NMI by distorting the metric space; measure anisotropy (average random cosine sim) first |
| BEIR subset | SciFact (5K docs, fast) | TREC-COVID (171K docs, slow) | SciFact for development and comparison; TREC-COVID only when you need a harder retrieval benchmark with graded relevance |

## What Makes This Non-Trivial

The difficulty is methodological, not algorithmic. Each task requires a different evaluation setup that can silently produce incorrect results: wrong pooling strategy produces embeddings that don't match model training targets, wrong NDCG formula produces scores 10–20% off, not fixing k-means seeds produces NMI variance larger than the difference between models, and anisotropy compresses clustering scores into a range that appears random. A practitioner who just runs `model.encode()` and plugs into sklearn metrics can produce a table where every model scores within noise of every other — not because the models are equal, but because the evaluation is broken.

The model size vs performance question has no clean answer. BGE-small beating all-mpnet-base-v2 on retrieval despite being 3× smaller is a consequence of training data and loss function, not architecture. This is only visible by knowing that BGE uses hard-negative contrastive fine-tuning on MS-MARCO while all-mpnet uses NLI data — a fact in the model cards, not the model sizes. Benchmark results without knowing the training data are uninterpretable.

## Interview Angles This Project Develops

1. "Why do you use Spearman correlation instead of Pearson for STS evaluation?" — Cosine similarity and human similarity scores are on arbitrary scales — absolute values are model-dependent and the scale is not linear. Spearman evaluates only the rank ordering, which is model-invariant and the only thing that matters for downstream ranking tasks.

2. "What is anisotropy in embedding models and why does it hurt clustering?" — Anisotropy means embeddings occupy a narrow cone of the vector space rather than distributing uniformly on the hypersphere. Average cosine similarity between random pairs is 0.3–0.6 instead of ~0. This compresses the effective range of cosine similarity, reducing discrimination between related and unrelated pairs, and makes k-means partitions unstable.

3. "How do you compute NMI for clustering evaluation?" — NMI = $2I(C;K) / (H(C) + H(K))$ where $C$ is true labels, $K$ is cluster assignments. Mutual information measures how much knowing one variable reduces uncertainty about the other. NMI normalizes by average entropy so score is in [0,1] and does not inflate with more clusters. NMI = 1.0 means perfect recovery up to label permutation.

4. "A smaller model outperforms a larger model on retrieval but underperforms on STS. What explains this?" — Training objective determines what the embedding space is optimized for. BGE-small fine-tuned on retrieval data (hard negatives from MS-MARCO) learns to separate relevant from irrelevant documents. all-mpnet fine-tuned on NLI and paraphrase data learns to encode semantic equivalence and entailment. These objectives produce different geometric structures.

5. "How would you extend this benchmark to evaluate domain-specific models?" — Add a domain-specific task to the suite: for biomedical, add BIOSSES (biomedical STS) and TREC-COVID retrieval; for legal, add LegalBENCH retrieval. The benchmark structure is identical; only the datasets change. General-purpose models often collapse relative to domain-specific fine-tuned models in specialized domains because their pretraining data under-represents domain vocabulary and reasoning patterns.

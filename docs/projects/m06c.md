## The Problem

**Input:** Four embedding models — `all-MiniLM-L6-v2` (22M params), `all-mpnet-base-v2` (110M params), `BGE-small-en` (33M params), `GTE-base` (110M params). Three evaluation tasks: (1) semantic similarity on STS-B (Spearman $\rho$ between predicted and human cosine similarity scores), (2) document retrieval on a BEIR subset — SciFact or TREC-COVID (NDCG@10), (3) topic clustering on 20 Newsgroups (k-means NMI score).

**Output:** A ranked comparison table: each model × task matrix of scores, with overall rank computed as mean rank across tasks. A Pareto frontier plot of model size vs mean performance.

**Constraints:** Evaluation must be reproducible — fixed random seeds, deterministic k-means initialization. Each task must use the model's recommended pooling strategy. Wall-clock time per model should be reported alongside quality metrics.

## Before You Write Any Code: Build The Mental Model

Embedding models are evaluated on multiple tasks because **no single task is sufficient**. The design choice in pretraining and fine-tuning creates inductive biases that make a model excellent on one task type and mediocre on another. A model trained entirely on STS data learns to produce similar embeddings for paraphrases but may collapse semantically distinct documents that share surface vocabulary — hurting retrieval. A model trained for retrieval may produce embeddings that rank correctly but whose absolute cosine similarity values don't correlate with human similarity judgments — hurting STS evaluation.

The three task types are orthogonal in what they measure:
- **STS-B:** Are the distances in embedding space monotonically consistent with human similarity judgments? Tests the geometry of the embedding space at the pair level.
- **Retrieval (BEIR):** Given a query, are the relevant documents ranked above non-relevant ones? Tests ranking at the query-collection level.
- **Clustering (20NG):** Do embeddings of topically similar documents cluster together, regardless of their absolute positions? Tests macro-level geometric structure.

A model can rank #1 on STS-B while ranking #3 on retrieval and #2 on clustering. Understanding *why* each model ranks differently requires knowing what loss function was used in fine-tuning and what data was seen.

## Decompose It

### Phase 1 — STS-B Evaluation: Spearman vs Pearson, and Why It Matters

**The question this phase answers:** How do you measure whether your model's cosine similarities correlate with human similarity judgments, and why is rank correlation the right choice?

**Key decisions:**

**STS-B dataset:** 8,628 sentence pairs with human similarity scores from 0 (unrelated) to 5 (equivalent). The task: embed both sentences, compute cosine similarity, measure how well cosine similarity ranks correlate with human scores.

**Pearson correlation ($r$):** Measures linear correlation between two variables. Assumes both variables are on ratio scales where the absolute values and differences are meaningful. Pearson $r = 0.8$ means "80% of the variance in human scores is explained by a linear function of cosine similarities."

**Spearman $\rho$:** Converts both variables to ranks, then computes Pearson on the ranks. Measures monotonic correlation — whether the cosine similarity ordering matches the human score ordering. Does not require the relationship to be linear. $\rho = 0.8$ means "80% of the variance in human rank-order is explained by the rank-order of cosine similarities."

**Why Spearman is correct for STS:** Human similarity judgments are on an arbitrary ordinal scale (0–5), not a ratio scale. The difference between 3.0 and 4.0 similarity may not equal the difference between 4.0 and 5.0 in any objective sense. Similarly, cosine similarity is bounded in $[-1, 1]$ and its absolute values are model-dependent — two models can produce cosine similarities of 0.3 and 0.9 for the same pair, with the 0.3 model actually being more accurate in ranking. Pearson would penalize the 0.3 model for having the wrong absolute scale; Spearman evaluates only the ordering, which is model-agnostic.

**Practical computation:**
```python
from scipy.stats import spearmanr
# cosine_sims: np.ndarray of shape (N,) — predicted similarities
# human_scores: np.ndarray of shape (N,) — gold scores 0–5
rho, pval = spearmanr(cosine_sims, human_scores)
```

**Trap:** Scaling the human scores (dividing by 5 to normalize to [0,1]) before computing Spearman. Spearman only uses ranks — scaling monotonically doesn't change ranks, so it has no effect. But if you then report Pearson after scaling, the absolute scale of cosine sim vs scaled human scores still doesn't match, and Pearson is still the wrong metric.

**Done when:** For each model, a single Spearman $\rho$ value computed on the full STS-B test set (1,379 pairs). Values for strong models: `all-mpnet-base-v2` ≈ 0.84, `all-MiniLM-L6-v2` ≈ 0.82. Values below 0.70 indicate either wrong pooling or wrong similarity metric.

### Phase 2 — BEIR Retrieval Evaluation: NDCG@10 and Evaluation Correctness

**The question this phase answers:** How do you run a BEIR-style retrieval evaluation and avoid the common scoring bugs?

**Key decisions:**

**BEIR dataset format:** Each BEIR dataset contains a `corpus` (doc_id → {"title", "text"}), `queries` (query_id → query_text), and `qrels` (query_id → {doc_id → relevance_grade}). The standard BEIR subsets for benchmarking embedding models: SciFact (300 queries, 5,183 docs), TREC-COVID (50 queries, 171,332 docs — large), FIQA (648 queries, 57,638 docs — medium). For this project, use SciFact for tractability.

**NDCG@10 computation — the correctness details:**
1. For each test query, retrieve the top-10 document IDs from your index.
2. Look up each retrieved doc's relevance grade in qrels. If not in qrels, grade = 0.
3. $DCG@10 = \sum_{i=1}^{10} \frac{rel_i}{\log_2(i+1)}$
4. $IDCG@10$ = DCG of the ideal ranking. Sort ALL relevant documents for this query by grade descending, take the top-10, compute their DCG. Crucially: IDCG is capped at 10 documents even if there are 100 relevant documents. You cannot retrieve more than 10.
5. $NDCG@10 = DCG@10 / IDCG@10$. If a query has no relevant documents in qrels, skip it (undefined).
6. Average NDCG@10 across all queries.

**Binary vs graded relevance:** SciFact has binary relevance (0 or 1). TREC-COVID has graded relevance (0, 1, 2). The NDCG formula handles both; for binary, $rel_i \in \{0, 1\}$.

**The `beir` Python library:** `pip install beir`. Provides `GenericDataLoader`, `EvaluateRetrieval`, and pre-built BEIR dataset downloads. Use it instead of re-implementing NDCG — the evaluation protocol has edge cases (how to handle queries with zero relevant docs, rounding) that the library handles correctly.

**Trap:** Using the alternative NDCG formula $\frac{2^{rel_i} - 1}{\log_2(i+1)}$ instead of $\frac{rel_i}{\log_2(i+1)}$. The exponential form amplifies graded relevance differences — it's preferred for graded relevance datasets (TREC). For binary relevance, both formulas are equivalent (since $2^1 - 1 = 1$ and $2^0 - 1 = 0$). Report which formula you use; BEIR's official leaderboard uses the linear form.

**Done when:** NDCG@10 for each model on SciFact. Expected values: `BGE-small-en` ≈ 0.66, `all-mpnet-base-v2` ≈ 0.64, `all-MiniLM-L6-v2` ≈ 0.58. Higher scores for larger models are not guaranteed — BGE-small is trained specifically for retrieval.

### Phase 3 — Clustering Evaluation: NMI and the Anisotropy Problem

**The question this phase answers:** How do you measure whether embeddings cluster by topic, and why does the geometry of high-dimensional embedding spaces make this harder than it sounds?

**Key decisions:**

**20 Newsgroups dataset:** 18,846 documents across 20 categories (e.g., sci.med, comp.graphics, alt.atheism). The clustering task: embed all documents, run k-means with k=20, measure how well the cluster assignments match the true category labels.

**NMI (Normalized Mutual Information):** $NMI = \frac{2 \cdot I(C; K)}{H(C) + H(K)}$ where $C$ is the true class distribution, $K$ is the cluster assignment distribution, $I$ is mutual information, and $H$ is entropy. NMI = 0.0 means cluster assignments are independent of true labels (random clustering). NMI = 1.0 means cluster assignments perfectly recover true labels (up to permutation). NMI is invariant to permutation of cluster labels — k-means cluster "3" doesn't have to align with category "sci.med" for NMI to be high.

**Why NMI over accuracy:** k-means doesn't produce labeled clusters — just group assignments. Accuracy would require finding the optimal mapping between cluster IDs and category labels (the Hungarian algorithm — $O(k^3)$). NMI avoids this by measuring information content directly.

**The anisotropy problem:** Embedding models produce vectors that are not uniformly distributed on the unit hypersphere — they cluster in a narrow cone. Quantitatively: the average cosine similarity between two random (non-paired) embeddings can be 0.3–0.6 instead of the theoretical 0.0 for uniformly distributed unit vectors. This is because all embeddings share a common "direction" imposed by the model's training on the mean token embedding.

**Why anisotropy hurts clustering:** If all embeddings are close to each other (cosine sim > 0.3 for random pairs), the within-cluster and between-cluster distances are compressed into a small range. k-means partitions based on small differences in a crowded space — clustering is less stable and more sensitive to initialization.

**Whitening as a fix:** Whitening transforms embeddings so that each dimension has unit variance and zero covariance (isotropic distribution). For a batch of embeddings $\mathbf{E} \in \mathbb{R}^{N \times D}$:
1. Compute mean $\mu$ and covariance $\Sigma = \frac{1}{N}\mathbf{E}^T\mathbf{E} - \mu\mu^T$
2. $\mathbf{E}_{white} = (\mathbf{E} - \mu) \mathbf{\Sigma}^{-1/2}$

In practice, use `sklearn.preprocessing.StandardScaler` (whitens per-dimension but not cross-dimensions) or full PCA whitening (`sklearn.decomposition.PCA(whiten=True)`).

**When whitening helps:** Anisotropic models (those trained on diverse text without explicit isotropy regularization) benefit more from whitening. You can measure anisotropy as the average pairwise cosine similarity of a random sample of embeddings: high average (>0.3) = high anisotropy = whitening likely helps NMI.

**Trap:** Running k-means on cosine similarities by converting to L2 distances after normalization. k-means minimizes squared Euclidean distances. For L2-normalized vectors, minimizing squared L2 = minimizing negative cosine similarity — so k-means on normalized vectors IS k-means in cosine space. Do not also apply whitening AND normalize — whitened vectors are not unit-norm.

**Done when:** k-means with k=20 run 10 times with different random seeds (average NMI to reduce variance). NMI measured for both raw embeddings and whitened embeddings. Report both; whitening improvement is often 0.03–0.08 NMI for anisotropic models.

### Phase 4 — Pareto Frontier and Model Size vs Performance Trade-Off

**The question this phase answers:** Do bigger models always win, and how do you visualize the quality-efficiency trade-off?

**Key decisions:**

**Model sizes:**
- `all-MiniLM-L6-v2`: 22M params, 6 layers, D=384
- `BGE-small-en`: 33M params, 6 layers, D=384
- `all-mpnet-base-v2`: 110M params, 12 layers, D=768
- `GTE-base`: 110M params, 12 layers, D=768

**Computing the Pareto frontier:** A model is Pareto-optimal if no other model is both smaller AND better on all tasks. Plot: x-axis = model parameters (or inference time), y-axis = mean performance score (average of normalized ranks or normalized scores across tasks). The Pareto frontier is the set of models such that moving to the left (smaller model) necessarily reduces quality.

**Score normalization for aggregation:** Raw scores across tasks are on different scales (Spearman ρ: 0–1, NDCG@10: 0–1, NMI: 0–1 — all happen to be in 0–1, but magnitudes differ). Normalize each task score to [0, 1] using min-max across the 4 models: $s_{norm} = (s - \min_m) / (\max_m - \min_m)$. Then average normalized scores. This prevents any single task from dominating the aggregate ranking.

**The empirical finding:** `BGE-small-en` is typically Pareto-optimal vs. `all-MiniLM-L6-v2` — slightly larger but better on retrieval tasks (BGE was fine-tuned for retrieval using contrastive learning on hard negatives). `all-mpnet-base-v2` and `GTE-base` are both 110M params; which is Pareto-superior depends on the task mix. Bigger is not always better: `all-mpnet-base-v2` was fine-tuned on NLI + STS data, so it may lag `BGE-small-en` on retrieval despite being 3× larger.

**Trap:** Using rank-based aggregation (average rank across tasks) without normalizing by variance. If STS-B separates models by 0.02 Spearman and NDCG separates them by 0.12, a raw rank aggregation treats them equally — but the NDCG differences are 6× larger and should matter more. Use normalized scores or weighted ranks.

**Done when:** A printed comparison table with columns: model, STS-B Spearman, BEIR NDCG@10, 20NG NMI, inference time/1K samples, overall rank.

### Phase 5 — Reproducibility and Fair Evaluation Protocol

**The question this phase answers:** What must be fixed for evaluation results to be comparable across runs and across projects?

**Key decisions:**

**Random seeds:** k-means is random — different initializations produce different NMI values with variance up to ±0.03. Fix the seed with `sklearn.cluster.KMeans(n_init=10, random_state=42)` and run 10 restarts (`n_init=10`) — KMeans picks the best. Report mean and std over 5 independent runs with different random states to characterize variance.

**Tokenizer truncation:** All four models have different max sequence lengths. `all-MiniLM-L6-v2` has max 256 tokens; `all-mpnet-base-v2` has max 512. If you truncate 20 Newsgroups posts at 512 tokens, `all-MiniLM-L6-v2` only sees 256 of those — it's at a disadvantage. Truncate at `min(model_max_length, 256)` for all models to ensure equal information access, and document the truncation length.

**Batch size effects on throughput measurement:** Embedding throughput (docs/second) depends on batch size. Measure throughput at batch_size=32 for all models — this is a common inference batch size for API deployments. Report throughput alongside quality so practitioners can make latency-aware model choices.

**The `sentence-transformers` library version:** Embedding quality can change across library versions if the pooling or normalization logic changes. Pin the library version in your requirements.txt and report it alongside results.

**Trap:** Loading different models with different `device` settings (one on GPU, one on CPU) and comparing inference times as if they're apples-to-apples. Always evaluate all models on the same hardware with the same device setting.

**Done when:** Results are fully reproducible from a single script run with fixed seeds. A `--model` CLI argument selects which model to evaluate; no hardcoded assumptions.

## Edge Cases to Handle Before You Code

1. **STS-B similarity scores for sentences with no shared tokens:** Some STS-B pairs have human similarity = 0.0 (completely unrelated sentences). Embedding models may still produce cosine similarities of 0.3–0.5 due to anisotropy — these systematically inflate the Pearson correlation (anisotropy creates a floor). This is a known issue; report it as a caveat, especially if comparing to older models without anisotropy.

2. **BEIR queries with zero relevant documents in the test qrels:** Some queries have no relevant documents annotated. NDCG@10 = undefined for these. Standard BEIR practice: skip these queries in the average. The `beir` library handles this automatically. If implementing manually, filter `qrels` to only include queries with at least one relevant document.

3. **k-means on high-dimensional embeddings (D=768) with N=7,500 samples:** KMeans in sklearn uses Lloyd's algorithm, which is $O(k \cdot N \cdot D \cdot \text{iter})$. With k=20, N=7500, D=768, 300 iterations: ~34B operations — takes 30–120 seconds on CPU. Use `KMeans(algorithm='elkan')` for faster convergence when cluster sizes are balanced, or reduce D with PCA before clustering (PCA to 64 dimensions loses <10% variance for sentence embeddings and speeds up KMeans by 12×).

4. **Embedding model produces different results on CPU vs GPU (float16 vs float32):** Some models auto-cast to float16 on CUDA. float16 rounding can change cosine similarities by ~0.001, which is negligible for ranking but affects reproducibility. Use `model.half()` consistently or `model.float()` consistently.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|----------|----------|----------------|
| Similarity metric for STS | Cosine similarity | Dot product | Pick A — dot product conflates magnitude with direction; cosine isolates semantic direction which is what human similarity ratings measure |
| Aggregation across tasks | Normalized score mean | Mean rank | Pick A — rank aggregation loses information about score gaps; two models separated by 0.01 NDCG and two separated by 0.20 NDCG both contribute 1 rank unit to the mean rank |
| Whitening for clustering | Apply to all models | Apply selectively based on anisotropy measure | Pick B — applying whitening to an already-isotropic model can hurt NMI by distorting the metric space; measure anisotropy first and apply only if average random cosine sim > 0.2 |
| BEIR subset | SciFact (small, fast) | TREC-COVID (large, slow) | Pick SciFact for development and comparison; use TREC-COVID only if you need a harder retrieval benchmark with graded relevance |

## What Makes This Non-Trivial

The difficulty is methodological, not algorithmic. Each task requires a different evaluation setup that interacts with the model in ways that can silently produce incorrect results: wrong pooling strategy → embeddings that don't match model training, wrong NDCG formula → scores 10–20% off, not fixing k-means seeds → NMI variance larger than the differences between models, anisotropy → clustering scores compressed into a narrow range that appears random. A practitioner who just runs `model.encode()` and plugs into sklearn metrics may produce a table where every model scores within noise of every other — not because the models are equal, but because the evaluation is broken.

The model size vs performance question also rarely has a clean answer. BGE-small beating all-mpnet-base-v2 on retrieval despite being 3× smaller is a consequence of training data and loss function design, not architecture. Understanding this requires knowing that BGE uses hard-negative contrastive fine-tuning on MS-MARCO while all-mpnet uses NLI data — a fact visible only in the model cards, not the model sizes.

## Interview Angles This Project Develops

1. **"Why do you use Spearman correlation instead of Pearson for STS evaluation?"** Strong answer: cosine similarity and human similarity scores are on arbitrary scales — the absolute values are model-dependent and the scale is not linear. Spearman evaluates only the rank ordering, which is model-invariant. Pearson would penalize a model for having the right ordering but the wrong absolute scale, which is not a meaningful distinction for retrieval or downstream tasks.

2. **"What is anisotropy in embedding models and why does it hurt?"** Strong answer: anisotropy means embeddings occupy a narrow cone of the vector space rather than being distributed uniformly on the hypersphere. The average cosine similarity between random (non-paired) embeddings is 0.3–0.6 instead of ~0. This compresses the effective range of cosine similarity, reducing discrimination between related and unrelated pairs. It hurts clustering (within-cluster and between-cluster distances are both small) and can inflate STS Pearson correlation by creating a false floor.

3. **"How do you compute NMI for clustering evaluation?"** Strong answer: NMI = $2I(C;K) / (H(C) + H(K))$ where $C$ is the true label distribution and $K$ is the cluster assignment. Mutual information $I$ measures how much knowing one variable reduces uncertainty about the other. NMI normalizes by the average entropy so that the score is in [0, 1] and doesn't inflate with more clusters. NMI = 1.0 means perfect recovery up to label permutation; 0.0 means independent.

4. **"A smaller model outperforms a larger model on retrieval but underperforms on STS. What explains this?"** Strong answer: the training objective determines what the model's embedding space is optimized for. A small model fine-tuned on retrieval-specific data (hard negatives from MS-MARCO) learns to separate relevant from irrelevant documents in the embedding space. A large model fine-tuned on NLI and paraphrase data learns to encode semantic equivalence and entailment relationships. These objectives produce different geometric structures — retrieval needs separation of distinct-but-related documents; STS needs correlation with human similarity judgments.

5. **"How would you extend this benchmark to evaluate domain-specific models?"** Strong answer: add a domain-specific task to the suite — for biomedical, add BIOSSES (biomedical STS) and TREC-COVID retrieval; for legal, add LegalBENCH. The relative rankings of general-purpose models collapse in specialized domains because their pretraining data under-represents domain vocabulary. Domain-specific fine-tuned models (PubMedBERT, LegalBERT) often outperform larger general models. The benchmark suite structure remains identical; only the datasets change.

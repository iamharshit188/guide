# ONNX Model Export & Optimization Pipeline — Thinking Guide

> **Module:** 05 — Deep Learning & MLOps · **Difficulty:** Advanced · **Type:** THINKING · **Time:** ~5 hours

---

## The Problem

**Input:** A trained sklearn `Pipeline` (e.g., `StandardScaler → LogisticRegression`) and a NumPy-based 2-layer MLP (weight matrices only, no framework).
**Output:** Both models exported to `.onnx`, run via `onnxruntime`, latency benchmarked at p50/p95/p99 over 1000 inference calls, INT8 dynamic quantization applied to the MLP, model sizes (bytes) compared before/after quantization, all results printed as a benchmark report.
**Constraints:** No PyTorch or TensorFlow in the export path. Use `skl2onnx` for the sklearn pipeline and `onnx` + manual graph construction for the MLP.

---

## Before You Write Any Code: Build The Mental Model

ONNX is a serialisation format — a computation graph where each node is a named operator (MatMul, Relu, Softmax) from a versioned operator set (opset), and edges are named tensors with declared shapes and dtypes. It is not a runtime. `onnxruntime` is a separate execution engine that loads an ONNX graph, applies graph-level optimisations (constant folding, operator fusion), selects an execution provider (CPU, CUDA, TensorRT), and runs the result. The separation matters: the same `.onnx` file runs on any ONNX-compatible runtime without recompilation.

When you export an sklearn `Pipeline`, `skl2onnx` introspects each step and emits the equivalent ONNX subgraph. The `StandardScaler` becomes a `Sub + Div` on the scale_ and mean_ attributes. `LogisticRegression` becomes a `MatMul + Add + Softmax`. These are composed into a single graph with a single input/output interface.

Manually constructing an MLP graph requires using `onnx.helper.make_node`, `make_graph`, and `make_model` — the operations map directly to the forward pass: `MatMul`, `Add`, `Relu`, `Softmax`. You declare input and output tensor shapes as `TensorProto` value_info, declare weight tensors as graph initializers (constants embedded in the file), and connect nodes by naming tensors consistently.

Quantization converts float32 weights and activations to int8. Dynamic quantization does this at runtime for each inference call using the observed min/max of the activation tensor — no calibration dataset needed, but you give up some accuracy because the scale is computed per-call. Static quantization computes scales offline from a representative calibration dataset and embeds them in the model, giving higher throughput at the cost of a calibration step.

## Decompose It

### Phase 1 — Train and Verify Baseline Models

**The question this phase answers:** Do both models produce correct float32 predictions before any export?

**Key decisions:**
- Use a tabular dataset (e.g., `sklearn.datasets.make_classification`) so the sklearn pipeline is straightforward.
- The NumPy MLP should use the same architecture as in other projects (2 layers, ReLU, softmax) — don't invent a new one; reuse weights you already trust.
- Measure baseline Python inference latency here. Time `model.predict(X)` for 1000 calls using `timeit`, record raw times, compute p50/p95/p99 with `np.percentile`.

**Trap:** Don't time the first call alone — it includes import overhead, JIT compilation (in sklearn), and OS scheduling jitter. Use a warm-up loop (10 calls) before the timed loop.

**Done when:** Both models produce predictions you can compare against ONNX output for correctness verification.

---

### Phase 2 — Export sklearn Pipeline to ONNX

**The question this phase answers:** What ONNX graph does `skl2onnx` generate, and does it match sklearn's output?

**Key decisions:**
- `skl2onnx.convert_sklearn` requires an `initial_type` argument specifying the input tensor shape and dtype. Use `FloatTensorType([None, n_features])` — `None` for batch dimension means any batch size is accepted.
- Choose opset version ≥ 12 (needed for `Softmax` with `axis=-1` semantics).
- After export, verify: load with `onnxruntime.InferenceSession`, run `session.run(None, {"X": X_test.astype(np.float32)})`, compare output probabilities to `pipeline.predict_proba(X_test)`. Max absolute difference should be < 1e-5.

**Trap:** sklearn uses `float64` internally; ONNX sessions default to `float32`. If you pass `float64` inputs, onnxruntime raises a type mismatch error. Cast all inputs to `float32` before inference.

**Done when:** ONNX predictions match sklearn predictions within 1e-5 absolute error on the full test set.

---

### Phase 3 — Manually Construct MLP ONNX Graph

**The question this phase answers:** How do you express arbitrary NumPy operations as an ONNX graph?

**Key decisions:**
- Each layer is: `MatMul(X, W) + b → Relu → MatMul → Add → Softmax`.
- Weights are stored as `onnx.numpy_helper.from_array` initializers embedded directly in the model. This makes the file self-contained — no separate weight file.
- Node names must be unique. Intermediate tensor names (e.g., `"z1"`, `"a1"`, `"z2"`, `"logits"`) must be consistent between the output of one node and the input of the next.
- Input and output shapes must be declared explicitly in `make_graph`'s `inputs` and `outputs` arguments. Use `onnx.helper.make_tensor_value_info` with `TensorProto.FLOAT` and shape `[None, n_units]`.

**Trap:** The `Gemm` operator (General Matrix Multiplication) is preferred over `MatMul + Add` for linear layers because it folds the bias addition and supports transposing one operand — the runtime can fuse this into a single BLAS call. If you use separate `MatMul` and `Add` nodes, some runtimes won't fuse them, leaving latency on the table.

**Done when:** `onnxruntime` loads the manually constructed graph without errors, and predictions match NumPy forward-pass outputs within 1e-5.

---

### Phase 4 — Benchmark: Python vs ONNX

**The question this phase answers:** Where does onnxruntime's speed advantage actually come from?

**Key decisions:**
- Benchmark protocol: 10 warm-up calls, then 1000 timed calls. Record each call's elapsed time. `np.percentile(times, [50, 95, 99])` gives p50/p95/p99.
- Use `time.perf_counter()` around each individual call, not around the whole loop — this gives per-call latencies, not amortised average.
- Test with batch size 1 and batch size 128 separately. The latency advantage of ONNX is most visible at small batch sizes because Python overhead dominates.

**Where onnxruntime wins:**
1. **Graph-level constant folding:** nodes with only constant inputs (e.g., the `Div` in `StandardScaler`) are computed once at session initialisation, not every inference call.
2. **Operator fusion:** consecutive `MatMul + Add + Relu` nodes may be fused into a single kernel call, eliminating intermediate tensor allocations.
3. **Memory layout optimisation:** onnxruntime may transpose weight matrices at session load time to match the CPU's preferred BLAS layout, amortising that cost.
4. **Zero Python overhead in the hot path:** once the session is loaded, inference is a single C++ function call — no Python loops over layers.

**Trap:** If your MLP is tiny (< 1000 parameters), Python overhead may dominate and ONNX won't look faster for batch size 1. Use batch size ≥ 32 to see a meaningful difference.

**Done when:** You have a table with 6 rows (Python/ONNX × batch1/batch32/batch128) and p50/p95/p99 columns.

---

### Phase 5 — INT8 Dynamic Quantization and Report

**The question this phase answers:** What does quantization buy in file size and latency, and what does it cost in accuracy?

**Key decisions:**
- `onnxruntime.quantization.quantize_dynamic` takes a float32 ONNX path and produces an INT8 ONNX path. It quantizes `MatMul` (and optionally `Gemm`) weight matrices to int8 using per-channel scaling.
- **Per-channel vs per-tensor:** Per-channel assigns a separate scale factor $s_i$ to each output channel of a weight matrix. This preserves more precision when weight ranges differ across channels (common in trained networks). Per-tensor uses a single scale for the entire matrix — faster, less accurate.
- The dynamic part means activations are quantized at runtime per call (not pre-calibrated). This is why no calibration data is needed.
- Report: file sizes with `os.path.getsize`, accuracy with `session.run` on test set, latency benchmark repeated for the quantized model.

**Trap:** Dynamic quantization only quantizes weights to int8; activations are quantized dynamically at inference time but still computed in float32 on CPU (int8 arithmetic requires a supporting kernel — on x86, this requires AVX2/AVX-512 with VNNI instructions). On a standard CPU without VNNI, INT8 may be *slower* than float32 because the quantize/dequantize overhead exceeds the savings. Check `onnxruntime.get_device()` and CPU flags.

**Done when:** Final benchmark report printed: model sizes, p50/p95/p99 for float32 and INT8, accuracy before/after. Report includes the compression ratio (float32_size / int8_size ≈ 4x for weight-only quantization).

## Edge Cases to Handle Before You Code

1. **Dynamic shapes:** ONNX models built with fixed batch size (e.g., `[32, 16]`) will reject batch size 1 at inference. Always declare batch dimension as `None` to support variable batch sizes.
2. **dtype mismatch:** onnxruntime is strict about input dtypes. If your model expects `float32` and you pass `float64` (NumPy default), the session raises `InvalidGraph`. Cast all inputs with `.astype(np.float32)`.
3. **Weight endianness in ONNX:** `onnx.numpy_helper.from_array` serialises in row-major (C) order. If your NumPy weights were transposed for a different convention, the graph will produce wrong outputs silently. Verify predictions numerically.
4. **Quantization of unsupported ops:** `quantize_dynamic` only quantizes ops in its supported set (`MatMul`, `Gemm`, `LSTM`, etc.). If you used custom ops or less common ONNX operators, they pass through unchanged. Always inspect which nodes were actually quantized by diffing the float32 and INT8 graphs.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|----------|----------|---------------|
| sklearn export | `skl2onnx` | Reimplement in ONNX manually | Always use `skl2onnx` — it handles edge cases (multi-output, sparse, etc.) you'd miss manually |
| MLP export | `Gemm` nodes | `MatMul + Add` nodes | `Gemm` when using onnxruntime ≥ 1.14 — guaranteed fusion; `MatMul + Add` if targeting other runtimes |
| Quantization type | Dynamic (no calibration) | Static (calibration data) | Dynamic when you cannot afford to collect or store a calibration dataset; static when throughput is the primary metric |
| Per-channel vs per-tensor | Per-channel | Per-tensor | Per-channel for classification heads with diverse weight ranges; per-tensor for embedding layers where channels are semantically uniform |
| Benchmark batch size | 1 | 128 | Batch 1 reveals latency (interactive serving); batch 128 reveals throughput (batch inference jobs) |

## What Makes This Non-Trivial

The manual ONNX graph construction forces you to understand the exact tensor contract of every operator: which inputs are weights (initializers) vs activations (runtime tensors), how shapes propagate through `Gemm` with optional transpose flags, and why the opset version affects operator semantics. Getting this wrong produces a model that loads but computes garbage.

The quantization section is deceptively complex: INT8 inference on CPU is not always faster than float32 because it depends on hardware ISA support (AVX-512 VNNI on Intel, SDOT on ARM Neon). The benchmark may show *higher* latency for INT8 on older hardware — which is the correct answer, not a bug. Understanding this requires knowing that quantized integer arithmetic needs different CPU instructions than FP32 SIMD, and those instructions aren't universally available.

Comparing Python inference to onnxruntime correctly requires controlling for all confounders: warm-up, batch size, output allocation, and Python function call overhead. A naive timing loop that includes session creation or input allocation in the hot path will show inflated speedups.

## Interview Angles This Project Develops

1. **"What is ONNX and why would you use it?"** — Strong answer explains the graph + operator set abstraction, the runtime-independence guarantee, and the practical workflow (train in any framework, deploy via onnxruntime to avoid framework dependencies in production).
2. **"What graph optimisations does onnxruntime apply?"** — Name at least: constant folding (compute constant subgraphs once at session load), op fusion (merge adjacent ops into single kernel), memory planning (pre-allocate output buffers). Weak answers say "it makes it faster."
3. **"When would you use dynamic vs static quantization?"** — Dynamic: no calibration data available, latency is acceptable to increase slightly, model is weight-heavy (transformers). Static: throughput is critical, you have a representative dataset, willing to run calibration offline.
4. **"Why might INT8 not be faster on your machine?"** — AVX-512 VNNI is required for vectorised int8 matrix multiply on Intel. Without it, the CPU emulates int8 with int32 ops or falls back to scalar code. Always benchmark; never assume.
5. **"How do you verify export correctness?"** — Numerical comparison of float32 outputs between framework and onnxruntime on a fixed input. Assert max absolute error < 1e-5. If the error is larger, check dtype casting, weight layout (row-major vs column-major), and opset mismatch.

## Extensions

1. **ONNX graph inspection** — use `onnx.helper.printable_graph(model.graph)` to print human-readable node list. Count how many nodes exist before vs after quantization.
2. **Execution provider benchmark** — if CUDA is available, create `InferenceSession(path, providers=["CUDAExecutionProvider"])` and repeat the benchmark. Compare CPU vs GPU latency at batch sizes 1, 32, 512.
3. **Mixed-precision export** — quantize only the first linear layer, leave the second in float32. Compare accuracy vs full INT8 to demonstrate per-layer granularity.

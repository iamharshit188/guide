# Custom BPE Tokenizer — Thinking Guide

> **Module:** 07 — Transformers from Scratch · **Difficulty:** Advanced · **Type:** THINKING · **Time:** ~3 hours

## The Problem

**Input:** A raw text corpus (e.g., the Shakespeare dataset, ~1.1M characters, or a web-crawled corpus in UTF-8).

**Output:** A tokenizer with three capabilities:
1. A learned vocabulary of $V$ tokens (typically 1,000–50,000 depending on corpus and use case).
2. An `encode(text) → list[int]` function that splits any string into token IDs.
3. A `decode(ids) → str` function that perfectly reconstructs the original string.

**Constraints:** The tokenizer must handle every byte of valid UTF-8 (no UNK token needed if designed correctly). Encode/decode must be a perfect round-trip: `decode(encode(s)) == s` for all valid input strings. The vocabulary must be deterministic given the same corpus and number of merge steps.

**Evaluation target:** After training on the Shakespeare corpus with 500 merge steps, encoding the full corpus should produce token counts within 5% of what HuggingFace's GPT-2 tokenizer produces on the same text (adjusted for vocabulary size differences).

## Before You Write Any Code: Build The Mental Model

A tokenizer's job is to map from the continuous space of characters to a discrete vocabulary that balances two opposing forces: smaller tokens (characters) give a small vocabulary and handle any text, but produce extremely long sequences; larger tokens (words) give short sequences but explode vocabulary size and fail on unseen words, inflected forms, and subwords from other languages.

BPE resolves this tension by learning the compression from data. It starts with the smallest possible atomic units (characters or bytes) and repeatedly merges the most frequent adjacent pair into a new symbol. After $k$ merge steps you have $|\text{base}| + k$ symbols in your vocabulary. The vocabulary learned this way is corpus-specific: on Shakespeare it learns theatre vocabulary; on Python source it learns common programming patterns.

The key insight is that BPE is a greedy frequency-based compression algorithm. It does not learn the "optimal" segmentation in any global sense — the greedy merge at each step is locally optimal but not globally. This is acceptable because the goal is not linguistic correctness but efficient compression that downstream models (transformers) can learn from. The attention mechanism can reconstruct meaning from whatever segmentation BPE produces.

Bytes (not characters) are the right base unit for a production tokenizer. Characters fail on emoji, non-Latin scripts, and unusual Unicode because a single code point can be up to 4 bytes; if you haven't seen that code point, you have no token for it. Bytes always have a fixed 256-symbol base vocabulary, making OOV impossible.

## Decompose It

### Phase 1 — Pre-tokenization

**The question this phase answers:** What are the atomic units we merge, and how do we split the corpus before running BPE?

BPE is not run on the raw character stream. A pre-tokenizer first splits the text into rough word-like units using a regex pattern. GPT-2's pattern (Radford et al. 2019) is:

```
'(?i:'s|'t|'re|'ve|'m|'ll|'d)|[^\r\n\p{L}\p{N}]?\p{L}+|\p{N}{1,3}| ?[^\s\p{L}\p{N}]+[\r\n]*|\s*[\r\n]+|\s+(?!\S)|\s+'
```

This prevents merges from crossing word boundaries (e.g., merging the `e` at the end of "the" with the space before "next"). Without pre-tokenization, BPE would happily merge across spaces and produce tokens like `" the"` (space-the) which are awkward and inconsistent.

**Key decisions:**
- Simple whitespace split vs. regex pre-tokenizer: whitespace split is faster to implement but produces worse tokenizers because it doesn't handle punctuation-word boundaries or contractions correctly. The regex adds 10 lines and significantly improves consistency.
- Whether to include the leading space in the token: GPT-2 puts the space at the start of the next word (` cat` not `cat `). This makes the tokenizer position-invariant with respect to word-internal structure.

**Trap:** If you pre-tokenize with `.split()` (Python default), `"don't"` becomes `["don't"]` — a single unit — and the apostrophe-t contraction is never separated. The GPT-2 pattern splits `"don't"` into `["don", "'t"]`, which is linguistically correct and consistent.

**Done when:** `pre_tokenize("Hello world!\nNew line.")` returns a list of string segments that, when concatenated, reproduce the original text exactly. Merge statistics are computed within segments, never across them.

### Phase 2 — Base Vocabulary Construction

**The question this phase answers:** What is the starting vocabulary before any merges?

For a character-level BPE: scan all unique characters in the pre-tokenized corpus and assign them integer IDs. For a byte-level BPE (GPT-2 style): always start with all 256 byte values (IDs 0–255), regardless of what appears in the corpus.

**Key decisions:**
- Characters vs. bytes: characters give a smaller base vocabulary (~65 for Shakespeare, ~150 for general English) but fail to handle any character not in the training corpus. Bytes give exactly 256 always — every possible byte sequence is representable.
- Special tokens: should `<BOS>`, `<EOS>`, `<PAD>` be part of the vocabulary? Yes, but add them after learning the merge table so they don't interfere with frequency counts. Reserve IDs for them explicitly.

**Trap:** When using bytes, display requires a byte-to-Unicode mapping (GPT-2 uses a specific 256-entry mapping table). If you skip this and try to decode raw bytes as UTF-8, you will get `UnicodeDecodeError` on multi-byte characters because individual bytes of a multi-byte sequence are not valid UTF-8 on their own.

**Done when:** Every character (or byte) in the corpus maps to a single token ID. `encode_base("abc")` returns `[97, 98, 99]` (ASCII values) for byte-level, or `[0, 1, 2]` for a character-indexed tokenizer. `sum(freq.values())` over all base tokens equals the total number of characters (or bytes) in the corpus.

### Phase 3 — The BPE Merge Loop

**The question this phase answers:** Which pairs do we merge, in what order, and how do we update the vocabulary and word representations efficiently?

The algorithm:

```
for step in range(num_merges):
    1. Count all adjacent pairs across all pre-tokenized words,
       weighted by word frequency (how many times each word appears).
    2. Find the most frequent pair (a, b).
    3. Create a new token (a, b) with the next available ID.
    4. Add merge rule: (a, b) → new_id to the merge table.
    5. In every word's token sequence, replace all occurrences of
       (a, b) with new_id.
    6. Recompute affected pair counts (only pairs involving new_id
       or pairs that lost a neighbor due to the replacement).
```

**Key decisions:**
- Naive recount vs. incremental update: naive recount scans every word after every merge — $O(V \cdot k)$ for $k$ merges and $V$ vocabulary words. Incremental update tracks which pairs changed — $O(k \cdot \text{avg\_word\_length})$, typically 10–100x faster for large corpora.
- Tie-breaking when multiple pairs have the same frequency: break ties lexicographically (sort pairs alphabetically). This ensures determinism. If you break ties randomly, two runs on the same corpus produce different merge tables.
- When to stop: fix the number of merge steps as a hyperparameter. Common values: 500 (toy), 8000 (small-vocabulary models), 50000 (GPT-2).

**Trap:** Counting pairs without weighting by word frequency is a critical bug. If the word "the" appears 50,000 times and its token sequence is `[t, h, e]`, the pair `(t, h)` has a frequency contribution of 50,000 — not 1. If you count occurrences in the flat corpus without grouping by word, you accidentally double-count pairs that span pre-tokenization boundaries, which is exactly what pre-tokenization was designed to prevent.

**Done when:** After $k$ merge steps, the merge table has exactly $k$ entries. The vocabulary size is `base_vocab_size + k`. Applying all merge rules in order to any word in the training corpus produces the same segmentation that the merge loop produced.

### Phase 4 — Encoding Arbitrary Text

**The question this phase answers:** Given a new string (not in the training corpus), how do you apply the learned merge rules to tokenize it?

The encoding algorithm applies merge rules in the order they were learned:

```
for each word in pre_tokenize(text):
    token_seq = [char2id[c] for c in word]   # start at base level
    for (a, b) → new_id in merge_table:      # apply merges in order
        merge all (a, b) pairs in token_seq with new_id
return flatten(all token sequences)
```

The order of application is critical: merge rules must be applied in the same sequence they were learned. Rule 1 must be applied before rule 2. If you sort them differently (e.g., by merged token length), you produce different — and wrong — segmentations.

**Key decisions:**
- Linear scan vs. priority queue for applying merges: a linear scan applies all $k$ rules one at a time, $O(k \cdot T)$ per word. A priority queue (or precomputed merge rank lookup) can reduce this to $O(T \log T)$ by always merging the highest-priority adjacent pair first. The priority queue approach is what production tokenizers (tiktoken, HuggingFace fast tokenizers) use.
- Caching: pre-tokenized words repeat constantly in natural text ("the", "and", "of"). Cache the encoding of each unique word to avoid re-applying all $k$ rules repeatedly.

**Trap:** Encoding a word that contains characters not in the base vocabulary crashes with a `KeyError`. With byte-level BPE this cannot happen (all 256 bytes always exist). With character-level BPE you must either handle unknown characters by mapping them to a special `<UNK>` token or by UTF-8 encoding them and tokenizing at byte level before applying merges.

**Done when:** `decode(encode(s)) == s` for all test strings including strings with punctuation, numbers, newlines, and repeated characters. Encoding the full training corpus and summing token counts matches the count you'd compute by exhaustively applying the merge table.

### Phase 5 — Saving, Loading, and Validation

**The question this phase answers:** How do you serialize the tokenizer and verify it matches reference implementations?

A tokenizer consists of exactly two artifacts:
1. `vocab.json` — a dict from token string representation to integer ID
2. `merges.txt` — a newline-separated list of merge rules in the order they were learned, format: `a b` per line (the pair that was merged)

These two files are sufficient to reconstruct the full tokenizer state. The vocab file also contains special tokens. The merges file encodes priority implicitly through line order — line 0 is the first merge applied, line 499 is the last.

**Validation against HuggingFace:** Load `GPT2TokenizerFast` from `transformers` on the same text and compare:
- Total token count on the full corpus: should be within 5% for similar vocabulary sizes
- Tokenization of specific words: `"hello"`, `"tokenization"`, `"BPE"`, `"don't"` — your output vs. theirs
- Round-trip fidelity: `decode(encode(s)) == s` for 1000 random substrings

**Trap:** HuggingFace's GPT-2 tokenizer uses bytes, not characters, at the base level, and has a specific byte-to-Unicode rendering map. If your tokenizer uses characters, your segmentations will differ from HuggingFace's even with the same merge count — this is expected and acceptable as long as your token counts are within 5% on the corpus level.

**Done when:** `tokenizer.save("vocab.json", "merges.txt")` writes two files. `tokenizer2 = load("vocab.json", "merges.txt")` and `tokenizer2.encode(s) == tokenizer.encode(s)` for all test strings.

## Data Flow

```
Raw text corpus
       │
       ▼
Pre-tokenizer (regex split)
       │ word segments with frequencies
       ▼
Base vocabulary (chars or bytes)
       │ initial token sequences per word
       ▼
BPE merge loop (k iterations)
       │ most frequent pair → new token, update sequences
       ▼
Merge table (k rules in order) + Extended vocabulary (base + k tokens)
       │
  ┌────┴────┐
  │         │
encode     decode
  │         │
  ▼         ▼
List[int]  str
(token IDs) (reconstructed text)
       │
       ▼
Validation: round-trip + HuggingFace comparison
```

## Edge Cases to Handle Before You Code

1. **Empty string input:** `encode("")` should return `[]`. `decode([])` should return `""`. Pre-tokenizing an empty string returns an empty list. Failing to handle this crashes on the first line of the merge loop.

2. **Single-character words that equal their own token:** After merges, the token for `"a"` is still the base character `"a"` — it has ID 97 (if byte-level). Encoding `"a"` should return `[97]`, not fail because there are no pairs to merge. The merge loop must handle the trivial case where the token sequence has length 1.

3. **Very long repeated sequences:** `"aaaaaaa..."` repeated 10,000 times. Naive pair counting assigns the pair `(a, a)` a frequency of ~10,000. After merging, you get `(aa)`. Then `(aa, aa)` is the most frequent pair. After $\log_2(10000) \approx 13$ merges, you have a single token for the full sequence. This is correct behavior but can cause the first few merge steps to be dominated by repetitive content, skewing the vocabulary toward repeated-character tokens that are useless for other text.

4. **Multilingual text with non-ASCII characters:** If your base vocabulary is characters, a single Chinese character like `"的"` is represented as one token. But if you use bytes, `"的"` is 3 bytes (`\xe7\xb9\x9a`...) and requires 3 base tokens. The byte-level approach handles all text but produces longer sequences for non-Latin scripts unless the merge table was trained on multilingual data. A tokenizer trained only on Shakespeare will not have merged common Chinese byte sequences.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|---------|---------|----------------|
| Base unit | Characters (unique chars in corpus) | Bytes (all 256) | A: toy tokenizers where corpus is fixed; B: any production use |
| Pair counting | Recount from scratch each step | Incremental update with affected pairs only | B for corpora > 100K words; A is fine for Shakespeare-sized |
| Pre-tokenization | None (raw BPE on char stream) | Regex split respecting word/punctuation boundaries | B always — raw BPE merges across boundaries and produces inconsistent tokens |
| Tie-breaking | Lexicographic (deterministic) | First seen (insertion order) | A whenever reproducibility matters |
| Encode algorithm | Apply merges in order (linear) | Priority queue (merge highest-priority adjacent pair) | B for k > 5000; A is readable and correct for small vocabularies |

## What Makes This Non-Trivial

BPE is simple to describe but has hidden complexity in efficient implementation. The naive algorithm — rescan all words after each merge — is $O(k \cdot C)$ where $C$ is corpus size. For GPT-2's 50,000 merges on a 10GB corpus, this takes days. The incremental update, which only recomputes pair counts for words containing the newly merged pair, brings this to practical runtime, but implementing it correctly requires careful bookkeeping of which words contain which pairs.

The pre-tokenization design choice has lasting consequences on the model trained on top of it. A tokenizer that allows merges across word boundaries produces tokens like `"e the"` (end of word, space, start of next word) — these tokens appear in one context only and waste vocabulary slots. GPT-2's regex was carefully engineered to prevent this while still merging common prefixes and suffixes within words.

Decoding is not just concatenation. The merge table does not store string representations of merged tokens — it stores (ID, ID) → ID mappings. To decode, you need the reverse: ID → string. For byte-level tokenizers, individual byte IDs decode to single bytes, not necessarily valid UTF-8. You must accumulate all bytes and then decode the byte sequence as UTF-8, handling errors (replacing invalid sequences with the replacement character `�`).

The 5% match target with HuggingFace is achievable but not trivial. HuggingFace's GPT-2 tokenizer uses a specific byte-to-Unicode mapping table (not raw bytes), and its pre-tokenization regex handles edge cases in contractions, numbers, and punctuation differently from a naive whitespace split. Expect to spend significant time on the pre-tokenizer to hit the match target.

## Interview Angles This Project Develops

1. **"Why not word-level tokenization?"** — Strong answer explains OOV problem quantitatively: a word-level tokenizer on English needs 100K+ tokens to cover 95% of web text; on code or technical text coverage drops further; morphologically rich languages (Finnish, Turkish) would need millions of tokens. BPE gets 95%+ coverage with 8,000–50,000 tokens.

2. **"Why does BPE tokenization matter for attention complexity?"** — Transformer attention is $O(T^2)$ in sequence length $T$. Character-level tokenization of a 1000-word document gives $T \approx 5000$; BPE with 8K vocab gives $T \approx 200$. That's a 625x reduction in attention computation. The choice of vocabulary size is directly a choice about training and inference cost.

3. **"What's wrong with BPE for morphology?"** — BPE is purely frequency-driven: it will merge `"ing"` with `"play"` into `"playing"` if that string is common enough, but it will not recognize that `"ing"` is a morpheme and keep it separate for less frequent verbs. WordPiece (BERT's tokenizer) and Unigram (SentencePiece) both have better morphological properties by using likelihood-based objectives rather than raw frequency.

4. **"How would you handle a new language not in the training corpus?"** — With byte-level BPE, any UTF-8 text is encodable because all 256 bytes are in the base vocabulary. The segmentation will be byte-by-byte (no merges apply) so sequences will be long, but encoding won't fail. The model's performance will degrade because it's never seen those byte patterns, but the tokenizer is robust.

5. **"What's the difference between encode speed and train speed for BPE?"** — Training BPE is slow (one-time cost, proportional to corpus size and merge count). Encoding at inference time should be fast. Production tokenizers (tiktoken) achieve encode speeds of ~1M tokens/second by using a compiled regex, a word-frequency cache, and a priority queue for merge application — avoiding Python loops entirely.

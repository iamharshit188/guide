# Customer Churn XGBoost Pipeline — Thinking Guide

## The Problem

**Input:** IBM Telco Churn dataset — 7,043 customer records, 21 features. Features include contract type, monthly charges, tenure, internet service type, payment method, and various add-on service flags.

**Output:** A production-ready pipeline that outputs: (1) calibrated churn probability per customer, (2) SHAP feature attributions per customer, (3) a ranked list of the top-N customers to contact under a budget constraint.

**Constraints:** You can contact at most $B$ customers in a retention campaign (e.g., 500). The objective is to maximize retained revenue among those contacted, not to maximize overall accuracy. Churn rate in this dataset is ~26.5% — mild imbalance, but the business constraint makes it a precision-at-K problem, not a balanced classification problem.

---

## Before You Write Any Code: Build The Mental Model

XGBoost is not "decision trees but better." It is a **second-order optimization algorithm** applied to an additive function space. At each boosting round, it fits a tree to the **negative gradient of the loss** (like gradient boosting) but also uses the **second-order curvature (Hessian)** to set leaf weights optimally. This is Newton's method in function space. The practical consequence is that XGBoost converges in fewer trees and is more robust to learning rate choice than vanilla gradient boosting.

The business objective — contact top-500 customers — is fundamentally a **ranking problem with a cutoff**, not a classification problem. The right metric is recall among the top-500 ranked by predicted churn probability. This means you never need to tune a classification threshold in the usual sense; you just rank by $\hat{p}$ and take the top-B. However, you still need calibrated probabilities if the campaign ROI calculation uses $\hat{p}$ to estimate expected retained revenue.

SHAP values are not feature importance — they are a **fair allocation of the model's prediction** (relative to a baseline) across features, satisfying efficiency, symmetry, and dummy axioms from cooperative game theory. The TreeExplainer computes exact Shapley values for tree models in polynomial time by exploiting the tree structure. This is fundamentally different from KernelExplainer, which approximates Shapley values model-agnostically using weighted linear regression — much slower, and approximate.

Monotonic constraints encode domain knowledge as a hard architectural constraint on the tree structure, not as a regularization penalty. This changes what functions the model can represent, not just what it prefers.

## Decompose It

### Phase 1 — Data Audit and Leakage Detection

**The question this phase answers:** Which features in your dataset encode future information about whether the customer churns?

**Key decisions:**

**Time-conditioned features:** "Days since last call" sounds like a behavioral feature. But if the churn label is defined as "left within the next 30 days," and the feature is computed at the same snapshot time, then a customer who has not called in 200 days may have lower engagement — which is genuinely predictive. However, if the feature is computed *after* the churn event date for customers who did churn, it leaks. You need to know the exact timestamp semantics of every feature.

**Post-termination data in the IBM dataset:** The IBM Telco dataset is relatively clean because it's synthetic. But "number of tech support tickets in last 3 months" would leak if the churn date was mid-period — churned customers may have had zero tickets after leaving. Always ask: could this feature have a different value if computed at the time of prediction vs at the time of dataset construction?

**TotalCharges vs MonthlyCharges × tenure:** TotalCharges is derived from MonthlyCharges and tenure. Including all three introduces perfect multicollinearity, which doesn't break tree models (they're invariant to linear combinations) but it inflates SHAP values for TotalCharges while deflating the others — makes SHAP harder to interpret. Drop TotalCharges or create a normalized version.

**Trap:** "SeniorCitizen" is encoded as 0/1 in the IBM dataset, not as a string like other binary features. Running `df.dtypes` will mislead you into treating it as numeric when all other binary columns are "Yes"/"No". Standardize encoding before building the pipeline.

**Done when:** Feature audit table complete — each of 21 features tagged as safe, potentially leaky, or redundant. TotalCharges handled. SeniorCitizen encoding aligned with other binary features.

### Phase 2 — Why XGBoost Uses Second-Order Gradients

**The question this phase answers:** What does XGBoost minimize at each boosting step, and why does Hessian information matter?

**Key decisions:**

**Gradient boosting baseline:** At iteration $t$, gradient boosting fits a new tree $f_t$ to the negative gradient $-g_i = -\frac{\partial \mathcal{L}(y_i, \hat{y}_i^{(t-1)})}{\partial \hat{y}_i^{(t-1)}}$. This is first-order — it knows which direction to move but not how far.

**Newton boosting (XGBoost):** XGBoost uses a second-order Taylor expansion of the loss:
$$\mathcal{L}^{(t)} \approx \sum_i \left[ g_i f_t(\mathbf{x}_i) + \frac{1}{2} h_i f_t(\mathbf{x}_i)^2 \right] + \Omega(f_t)$$
where $g_i$ is the first-order gradient and $h_i = \frac{\partial^2 \mathcal{L}}{\partial \hat{y}_i^2}$ is the Hessian. For binary cross-entropy, $h_i = \hat{p}_i(1-\hat{p}_i)$. The optimal leaf weight for leaf $j$ is $w_j^* = -\frac{\sum_{i \in j} g_i}{\sum_{i \in j} h_i + \lambda}$, derived analytically. This is why XGBoost leaf values are not just mean residuals — they're curvature-weighted.

**Practical implication:** XGBoost's `learning_rate` (eta) has less effect on final performance than in vanilla gradient boosting because the Newton step already adapts step size to the local curvature. You can often get good results with `learning_rate=0.1` and `n_estimators=300` without extensive tuning.

**Trap:** Setting `max_depth` too high without corresponding `min_child_weight` regularization. `max_depth=6` with `min_child_weight=1` allows leaves with a single sample — the Hessian sum is tiny, leaf weights are huge, model memorizes training data. The interaction matters: high depth requires high min_child_weight.

**Done when:** You understand what XGBoost is solving at each step and can explain why `min_child_weight` (minimum Hessian sum per leaf) is the right regularizer for leaf splitting decisions.

### Phase 3 — Feature Engineering and Monotonic Constraints

**The question this phase answers:** What derived features improve discrimination, and how do you encode domain knowledge as hard constraints?

**Key decisions:**

**Tenure segments:** Raw tenure (months) is less discriminative than bucketed tenure — new customers (0–12 months) churn for different reasons than long-tenured customers (>48 months). But tree models can discover these breakpoints automatically; explicit binning is optional. The real question is whether the relationship is monotone: longer tenure → lower churn probability. If yes, encode it as a monotonic constraint.

**Charge-to-service ratio:** MonthlyCharges divided by number of active services creates a "price per service" feature. High price per service with month-to-month contract is a strong churn signal. This interaction feature can be important because trees can only build interactions implicitly through depth.

**Monotonic constraints in XGBoost:** `xgb.XGBClassifier(monotone_constraints={'tenure': -1, 'MonthlyCharges': 1})`. The value -1 means the output must be non-increasing in tenure (longer tenure → lower churn probability). Value 1 means non-decreasing. Internally, XGBoost enforces this by only allowing splits in the constrained direction during tree construction. This can hurt training AUC slightly but often improves generalization and always improves stakeholder trust.

**When constraints hurt:** If the true relationship is non-monotone (e.g., very new customers have low churn because of contracts, then mid-tenure spikes, then long-tenure drops again), a monotone constraint will force an incorrect shape. Validate the empirical marginal relationship before imposing constraints: plot `churn_rate` vs `tenure_decile`.

**Trap:** Applying monotonic constraints to one-hot encoded categoricals. XGBoost's monotone_constraints applies to the feature index in the input matrix. If you one-hot encode "ContractType" into 3 columns, you cannot enforce a joint constraint that "TwoYear contract → lower churn than Month-to-month." Use label encoding + ordinal constraint instead.

**Done when:** Feature matrix is constructed, monotonic constraints validated against empirical marginals, and a `sklearn.pipeline.Pipeline` object encapsulates preprocessing + XGBoost.

### Phase 4 — SHAP: TreeExplainer vs KernelExplainer

**The question this phase answers:** Which SHAP explainer to use and what do the outputs actually mean?

**Key decisions:**

**What a SHAP value is:** For feature $j$ and instance $i$, $\phi_j^{(i)}$ is the contribution of feature $j$ to the model's prediction $\hat{p}_i$ relative to the mean prediction $\bar{\hat{p}}$. The sum property: $\hat{p}_i = \bar{\hat{p}} + \sum_j \phi_j^{(i)}$. This is exact, not approximate, for TreeExplainer.

**TreeExplainer:** Exact Shapley values computed by traversing the tree structure. Time complexity $O(T \cdot L \cdot D)$ where $T$ = number of trees, $L$ = max leaves, $D$ = max depth. For XGBoost with 300 trees and max_depth=6, this runs in seconds for thousands of samples. Use `shap.TreeExplainer(model)`.

**KernelExplainer:** Model-agnostic approximation using weighted linear regression on coalitions of features. Requires $O(2^p)$ model evaluations per sample in the worst case (approximated with sampling). For $p=20$ features and 7,000 samples, this is prohibitively slow. Use KernelExplainer only when: (1) you need a consistent SHAP implementation across heterogeneous model types, or (2) the model is not tree-based (e.g., a neural network without a dedicated explainer).

**SHAP summary plot vs dependence plot:** The summary plot shows global feature importance (mean |SHAP| across all samples) plus direction (positive = increases churn probability). The dependence plot shows $\phi_j$ vs the actual feature value for feature $j$, with color encoding a second feature — revealing interactions. For churn, the dependence plot for MonthlyCharges colored by ContractType will show the interaction you expect: high charges matter more for month-to-month customers.

**Trap:** Interpreting SHAP values as causal effects. SHAP explains what the model computed, not the causal structure of reality. If tenure and TotalCharges are correlated, SHAP may arbitrarily split attribution between them depending on coalition ordering. This is a property of Shapley values (they handle correlated features via marginal expectations), not a bug, but it means SHAP is not causal attribution.

**Done when:** For each test customer, you have a SHAP vector $[\phi_1, ..., \phi_{20}]$ and can produce a waterfall chart showing why their predicted churn probability is high or low relative to the mean.

### Phase 5 — Threshold Optimization Under Budget Constraint

**The question this phase answers:** Given a budget to contact $B$ customers, which $B$ should you pick and what metric proves your selection was optimal?

**Key decisions:**

**The ranking formulation:** Sort all customers by $\hat{p}$ descending. Take the top $B$. This is the Bayes-optimal policy under the assumption that: (1) all customers have equal expected revenue if retained, (2) all respond to the intervention equally. In practice, you want to multiply by estimated intervention effect (uplift modeling), but rank-by-churn-probability is the standard baseline.

**Precision@K and Recall@K:** With $B=500$ and 26.5% churn rate, the expected number of churners in the dataset is ~1,866. Precision@500 = (churners in top-500) / 500. Recall@500 = (churners in top-500) / 1866. These are the business metrics, not overall F1.

**Expected revenue retained:** If average monthly revenue per customer is $R$ and retention probability given contact is $q$, then:
$$\text{Revenue Retained} = R \cdot q \cdot |\{i \in \text{top-}B : y_i = 1\}|$$
This is what you report to the business, not AUC.

**Precision-recall trade-off under constraint:** If you increase $B$ from 500 to 1000, precision drops (more non-churners contacted, wasted budget) but recall rises (more true churners reached). The optimal $B$ maximizes expected revenue minus contact cost: $\arg\max_B \; R \cdot q \cdot \text{TP}(B) - c \cdot B$ where $c$ is cost per contact. Compute this by sweeping $B$ from 1 to $N$.

**Trap:** Choosing threshold by maximizing F1 on the full test set, then reporting precision@500. These objectives are not equivalent. A threshold of 0.5 maximizing F1 does not imply the top-500 by score are the right 500 — though in practice, if the model is well-calibrated, ranking by score IS the right approach, and threshold is irrelevant to the ranking.

**Done when:** A ranked list of 500 customers is produced, with their predicted $\hat{p}$, their SHAP top-3 reasons, and an estimated expected revenue retained. Precision@500 and Recall@500 computed on test set.

## Edge Cases to Handle Before You Code

1. **TotalCharges is stored as string in the IBM dataset:** Rows where `tenure=0` have `TotalCharges=" "` (empty string). `pd.to_numeric(df['TotalCharges'], errors='coerce')` will produce NaN for those rows. Either impute with 0 (new customer, no charges yet) or drop — but document the decision.

2. **XGBoost with `use_label_encoder` deprecation warning:** XGBoost ≥ 1.6 removed the label encoder. Set `use_label_encoder=False` and `eval_metric='logloss'` to suppress warnings and set the correct metric for binary classification.

3. **SHAP values sum to prediction minus base value, not to prediction minus 0.5:** The base value is the mean model output on the training set, typically ~0.265 for this dataset. A SHAP waterfall that sums to 0.8 means the model predicts 0.265 + 0.535 = 0.8, not that 0.8 is all from features. Common misinterpretation when presenting to stakeholders.

4. **Budget constraint with ties in p_hat:** If multiple customers have identical $\hat{p}$ at the cutoff rank 500, the selection is arbitrary. Break ties using a secondary criterion (e.g., higher MonthlyCharges = more revenue at stake if retained) and document the tie-breaking rule.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|----------|----------|----------------|
| Encoding categoricals | OrdinalEncoder + monotone constraint | OneHotEncoder + no constraint | Pick A when you want to enforce ordering constraints on multi-category features; OHE loses ordinal structure |
| Explainer | TreeExplainer (exact) | KernelExplainer (approximate) | Pick A always for XGBoost — exact and fast; KernelExplainer only when model changes to non-tree type |
| Business target | Maximize Recall@500 | Maximize F1 overall | Pick A — matches the actual campaign budget constraint; F1 is a modeling artifact irrelevant to the business |
| Monotone constraint enforcement | XGBoost `monotone_constraints` | Post-hoc monotone isotonic calibration | Pick A — enforces constraint during training, not just at output; post-hoc isotonic calibration enforces monotonicity in calibration but not in the feature→prediction mapping |

## What Makes This Non-Trivial

The feature engineering and leakage detection are more important than the model choice — XGBoost on clean features trivially outperforms any model on leaky ones. The business-metric disconnect (model AUC vs campaign Precision@K) is where most practitioners go wrong: a model optimized for AUC can rank differently in the top-decile than a model optimized for precision at the campaign cutoff. Monotonic constraints require empirical validation before application — imposing an incorrect monotonicity degrades AUC without the practitioner noticing because the constraint is "obviously right." SHAP interpretation at scale (7,000 customers) is fast, but explaining why a specific customer's top-3 SHAP features make business sense requires domain knowledge that SHAP cannot supply.

## Interview Angles This Project Develops

1. **"Why does XGBoost outperform gradient boosting on tabular data?"** Strong answer: second-order optimization (Newton method in function space) means each tree's leaf weights are set analytically to minimize the curvature-adjusted loss, not just the gradient. This gives faster convergence and better regularization via the Hessian denominator. Also: XGBoost's regularization terms $\lambda$ (L2 on leaf weights) and $\gamma$ (minimum gain per split) are built into the objective, not post-hoc pruning.

2. **"How does SHAP handle correlated features?"** Strong answer: SHAP uses marginal expectations over all feature coalitions, which means correlated features share credit via the averaging across orderings in which they enter the coalition. The total attribution is correct (efficiency property) but split between correlated features can be arbitrary. This is why SHAP is not causal — it explains the model, not the data-generating process.

3. **"What does the monotone_constraints parameter do mechanically?"** Strong answer: during tree construction, when choosing a split on a constrained feature, XGBoost only allows directions consistent with the constraint. For a -1 constraint on tenure, only splits where the left (lower tenure) branch has higher predicted values are allowed. This prunes the split search space, not just the output.

4. **"You have a budget of 500 contacts. Walk me through your model evaluation."** Strong answer: I rank all test customers by predicted churn probability and take the top 500. I compute Precision@500 (fraction of those 500 who actually churn), Recall@500 (fraction of all churners captured), and expected revenue retained. I compare to a naive policy (contact 500 random customers) and to a perfect oracle (contact the 500 with true churn=1). The gap between my model and the oracle is the information value left on the table.

5. **"What's the difference between feature importance from XGBoost and SHAP importances?"** Strong answer: XGBoost's built-in importance (gain/coverage/frequency) is global and inconsistent — the same feature can have different importance depending on how many times it was used across trees. SHAP importance (mean |SHAP|) satisfies the consistency axiom: if a feature's effect increases in a new model, its SHAP importance cannot decrease. SHAP is the right choice for communication with stakeholders.

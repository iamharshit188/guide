# Recipe Recommendation System — Thinking Guide

> **Module:** 03 — Databases & Vector DBs · **Difficulty:** Intermediate · **Type:** THINKING · **Time:** ~3 hours

## The Problem

Build a hybrid recommender that returns the top-10 recipes per user by combining content-based similarity (recipe ingredient embeddings stored in FAISS) with collaborative filtering signal (user interaction history stored in SQLite). The system must handle cold-start users, cold-start recipes, and staleness — recommendations should reflect recent user behavior, not just their first five ratings.

**Input:** a user ID (may be new)  
**Output:** an ordered list of 10 recipe IDs with confidence scores  
**Constraints:** the content and collaborative components must be independently runnable; the fusion must be explainable ("you might like this because it's similar to X you rated 5 stars")

---

## Before You Write Any Code: Build The Mental Model

A recipe is a bag of ingredients: "garlic, olive oil, pasta, parmesan, black pepper". Treating the ingredient list as a sentence and passing it through a sentence-transformer produces an embedding that captures culinary semantics — garlic bread and pizza cluster together because their ingredient sets co-occur in similar recipe contexts in the pretraining corpus. This is content-based filtering: recommend items whose embeddings are close to the items a user has liked.

Pure content-based filtering has a ceiling: it can only recommend recipes that are ingredient-similar to what you already ate. It cannot discover that users who like pasta also tend to like risotto — a different dish in the same culinary space that shares no ingredient with any pasta dish the user has tried. That discovery requires collaborative filtering: find users with similar interaction histories and recommend items those users liked that you haven't tried.

The hybrid combines both signals. The design question is not "which is better" but "when does each dominate." For a user with 2 interactions, collaborative filtering has almost no signal (not enough history to find similar users); content-based dominates. For a user with 200 interactions spread across many cuisines, collaborative filtering can find non-obvious cross-cuisine recommendations that content-based would never surface.

FAISS handles the approximate nearest-neighbour search in embedding space; SQLite handles the relational data (users, recipes, ratings, timestamps). The combination is intentional — FAISS is not a relational store, and SQLite is not a vector store. Each tool does what it is designed for.

## Decompose It

### Phase 1 — Data Model and Schema Design

**The question this phase answers:** What is the minimum schema that supports content, collaborative, and hybrid retrieval without requiring a full ORM?

**Key decisions:**
Three tables in SQLite:

```
recipes(recipe_id INTEGER PK, name TEXT, ingredients TEXT, cuisine TEXT, created_at INTEGER)
users(user_id INTEGER PK, joined_at INTEGER)
interactions(interaction_id INTEGER PK, user_id INTEGER FK, recipe_id INTEGER FK,
             rating REAL, interacted_at INTEGER)
```

Index `interactions` on `(user_id, interacted_at DESC)` — the most frequent query is "give me this user's N most recent interactions sorted by recency".

The `ingredients` column stores a comma-separated string (or JSON array). It will be passed directly to the sentence-transformer to produce the FAISS vector. The recipe content representation is the ingredient string, not the name — names are too ambiguous ("carbonara" in Italian vs. "egg and bacon pasta" in English both mean the same dish).

**Trap:** Storing embeddings in SQLite as BLOBs. This works but forces you to deserialize every embedding for every query, losing all the indexing benefit of FAISS. The split is deliberate: FAISS owns vectors, SQLite owns everything else. The join key is `recipe_id`.

**Done when:** You can insert 500 recipes, create 50 users, and generate 5000 random interactions without schema errors. You can retrieve a user's interaction history sorted by recency in a single SQL query.

### Phase 2 — Content-Based Retrieval via FAISS

**The question this phase answers:** Given a set of recipes the user has liked, what other recipes are ingredient-similar?

**Key decisions:**

**FAISS index selection** — the most important decision in this phase:

- `IndexFlatIP` (exact inner product over L2-normalized vectors = exact cosine): $O(N)$ per query. Use for $N < 50K$. No training required, perfect recall.
- `IndexIVFFlat` (inverted file with flat quantizer): clusters vectors into $\sqrt{N}$ Voronoi cells, searches `nprobe` cells. $O(\text{nprobe} \cdot N/\sqrt{N}) = O(\text{nprobe} \cdot \sqrt{N})$ per query. Requires a training step with representative vectors. At $N = 500$, IVF adds overhead without speed benefit — don't use it here.
- `IndexHNSWFlat` (hierarchical navigable small world graph): best approximate retrieval quality at scale. $O(\log N)$ per query with high recall. No training required but memory-intensive (graph stored in RAM). At $N = 500$, slower to build than Flat with no query speed advantage.

**Decision:** `IndexFlatIP` for any corpus under 50K recipes. Document this threshold explicitly — when the corpus crosses 100K, rebuild with `IndexHNSWFlat`.

**User profile construction:** do not search using a single liked recipe's embedding — aggregate. Compute the user's profile vector as the weighted mean of embeddings for their $N$ most recently liked recipes (rating ≥ 4), with recency weighting $w_i = \frac{1}{1 + \text{days\_ago}_i}$. This ensures recent preferences dominate without completely discarding older tastes.

**Trap:** Including recipes the user has already interacted with in the search results. FAISS returns the $k$ nearest neighbours — some of those will be recipes the user rated. Filter these out post-search using the user's interaction history from SQLite.

**Done when:** Given `user_id=5` with 10 liked recipes, you can produce a ranked list of 20 candidate recipes (excluding already-seen ones) with cosine similarity scores.

### Phase 3 — Collaborative Filtering Signal

**The question this phase answers:** Which recipes do users similar to this user tend to rate highly?

**Key decisions:**

User-user collaborative filtering with cosine similarity on interaction vectors:

1. Build a user-recipe rating matrix $R \in \mathbb{R}^{U \times I}$ (sparse — most entries are 0).
2. Find the $K$ most similar users to the target user via cosine similarity on their rating rows.
3. Score each candidate recipe as the weighted average rating from similar users: $\hat{r}_{u,i} = \frac{\sum_{v \in \text{neighbors}} \text{sim}(u,v) \cdot r_{v,i}}{\sum_{v \in \text{neighbors}} \text{sim}(u,v)}$

**Sparsity problem:** at 5000 interactions over 50 users × 500 recipes, each user has rated on average 100 recipes — that is 20% density, which is unusually high. In production systems, density is < 0.1%. At low density, cosine similarity between user vectors is dominated by the few items they share in common, and most user pairs have cosine similarity exactly $0$ (no shared ratings). The fix is implicit feedback: treat a low rating (1-2 stars) as a weak negative signal rather than a missing entry, and use Bayesian Personalized Ranking (BPR) for learning rather than raw cosine on ratings. For this project, document the problem and use the simple cosine approach with the understanding that it breaks at scale.

**Trap:** Cold-start users have no rating history — their row in $R$ is all zeros. Cosine similarity of a zero vector with any other vector is undefined (division by zero). Handle this explicitly: if the user has fewer than 3 interactions, fall back to content-based only, weighted 100/0 instead of the default 60/40.

**Done when:** For a user with ≥ 10 interactions, you can find their 5 nearest neighbour users and produce a weighted score for each recipe those neighbours rated.

### Phase 4 — Hybrid Fusion and Cold-Start Handling

**The question this phase answers:** How do you merge content and collaborative scores into a single ranked list that degrades gracefully for cold-start users?

**Key decisions:**

Dynamic weighting based on interaction count:

```
n_interactions = len(user_history)
collab_weight = min(1.0, n_interactions / 20) * 0.6   # max 60% collaborative
content_weight = 1.0 - collab_weight                   # min 40% content-based
```

This ramp means: 0 interactions → pure content (random popular recipes as fallback), 10 interactions → 30% collaborative, 20+ interactions → 60% collaborative.

Score normalization before fusion: content scores are cosine similarities in $[0, 1]$; collaborative scores are weighted averages of ratings in $[1, 5]$. Normalize each to $[0, 1]$ via min-max over the candidate set before combining. Do not use global min-max — it's undefined for new items — use the range of scores within the current candidate set.

**New-item cold start** (a recipe just added to the corpus): FAISS can add its embedding with `index.add()` without rebuilding the index. But collaborative scores for new items are $0$ (no one has rated them yet). The system will never recommend them through collaborative filtering. The fix is a separate "explore" slot: always include 1-2 new (< 7 days old) recipes in the top-10, injected after fusion, to give new items visibility before they accumulate ratings.

**Diversity vs. relevance trade-off:** the top-10 returned by cosine similarity in embedding space are often very similar to each other — if you liked carbonara, your top-10 might be 8 pasta dishes. Maximal Marginal Relevance (MMR) re-ranks to balance relevance and diversity:

$$\text{MMR}_i = \lambda \cdot \text{sim}(q, d_i) - (1 - \lambda) \cdot \max_{d_j \in S} \text{sim}(d_i, d_j)$$

where $S$ is the already-selected set. $\lambda = 0.7$ gives 70% relevance, 30% diversity. Whether to implement MMR depends on user feedback — measure click-through rate on diverse vs. pure-relevance lists before committing to the complexity.

**Done when:** A single function `recommend(user_id, n=10)` handles new users (cold start), returning users, and new recipes correctly, with documented fallback behavior for each case.

### Phase 5 — Staleness and Freshness

**The question this phase answers:** How do you prevent the system from permanently recommending the same 10 recipes once a user's profile stabilizes?

**Key decisions:**

Three sources of staleness:

1. **FAISS index staleness:** new recipes added to SQLite are not in FAISS until a rebuild. Implement periodic `index.add()` for batches of new recipes. An online FAISS index (adding vectors without rebuilding) is supported for `IndexFlatIP` — safe to add new vectors at any time. For IVF-based indices, adding without rebuilding degrades recall.

2. **User profile staleness:** if you cache the user's profile vector, it becomes stale after new interactions. Cache with a TTL (e.g., 1 hour for frequent users, 24 hours for inactive users). When the cache misses, recompute the profile vector from the most recent interactions.

3. **Rating distribution shift:** user tastes change over time. A rating from 3 years ago should matter less than one from last week. The recency weighting in Phase 2 handles this for the profile vector, but the collaborative filtering matrix uses all historical ratings equally. Introduce a decay: $r'_{v,i} = r_{v,i} \cdot e^{-\lambda \cdot \text{days\_ago}}$ with $\lambda = 0.01$ (half-life ~70 days) when constructing the rating matrix.

**Trap:** Rebuilding the entire FAISS index on every new recipe. At 500 recipes, this takes < 1 second — acceptable. At 500K recipes, a full rebuild takes minutes and blocks recommendation serving. Design with the assumption that rebuilds happen offline (nightly batch job) while the live index serves from the previous version, then atomically swaps.

**Done when:** You can add a new recipe at runtime and have it appear in content-based results immediately; you can add a new user interaction and have it reflected in recommendations within the TTL window.

```
    |                      |

            collab  → [0,1]
                |

        w_c * content + w_collab * collab
                |

                |
        [Inject new-item slots]
                |
        [Optional MMR re-rank]
                |
        Top-10 recommendations
        (recipe_id, score, reason)
```

## Edge Cases to Handle Before You Code

1. **User with 0 interactions:** collaborative score is undefined, FAISS profile vector doesn't exist. Return a "popularity baseline" — the top-10 recipes by average rating across all users with ≥ 5 ratings. Document that this is not personalized.

2. **Recipe with 0 ratings but a valid embedding:** it can appear in content-based results but has no collaborative score. Assign `collab_score = 0` and let `content_weight = 1.0` handle it — the recipe still gets recommended if it's content-similar.

3. **All recipes in the user's top-50 have already been seen:** return the next-50 FAISS results (increase `n_results`). Always over-retrieve (100 candidates) and filter down — never retrieve exactly 10 and then discover 9 are already seen.

4. **Two users have identical interaction histories (duplicate accounts):** their collaborative profiles are identical, which is fine — they get the same recommendations. Not a bug.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|----------|----------|----------------|
| Content representation | Ingredient list as text → sentence-transformer | One-hot ingredient vector → cosine | Sentence-transformer when ingredient semantics matter (saffron near turmeric); one-hot when you want exact ingredient matching and interpretability |
| FAISS index type | `IndexFlatIP` (exact) | `IndexHNSWFlat` (approximate) | Flat for $N < 50K$ — no recall penalty and simpler rebuild logic |
| Collaborative approach | User-user with cosine similarity | Matrix factorization (ALS/SVD) | User-user for < 10K users and fast iteration; matrix factorization for production at scale (handles sparsity better, offline training required) |
| Cold-start fallback | Popularity baseline | Random sample | Popularity baseline if you have any interaction data; random sample only if the corpus is very new with no signal |
| Diversity mechanism | MMR re-ranking | No diversity constraint | MMR when click-through data shows users skip homogeneous lists; skip MMR if you don't have feedback data to validate the $\lambda$ parameter |

## What Makes This Non-Trivial

The cold-start problem has no clean solution — it is a fundamental tension between exploration (showing new content to gather signal) and exploitation (showing high-confidence recommendations to maximize engagement). Any threshold you set for "enough interactions to use collaborative filtering" is arbitrary; the right threshold varies by user and corpus density.

Embedding space semantics are surprising and brittle. The claim that "garlic bread is near pizza in embedding space" holds when the model was trained on recipe text and ingredient co-occurrence. But if the model was trained on general web text, "garlic bread" may cluster near "bread recipes" and "garlic supplements" rather than Italian food. Always visualize a 2D UMAP projection of your recipe embeddings before trusting the retrieval results.

Evaluation is harder than the system. A recipe recommender cannot be evaluated with NDCG alone — a system that always recommends the same 10 popular recipes achieves high NDCG on average because popular items are likely to be relevant. Serendipity (recommending something the user wouldn't have found themselves), novelty (recommending items not already known to the user), and coverage (fraction of the catalog ever recommended) are the metrics that distinguish a useful recommender from a popularity ranker.

## Interview Angles This Project Develops

1. "Explain the cold-start problem and how you handled it." — Strong answer: cold-start has two variants — new user (no interaction history, no collaborative signal) and new item (no ratings, but embedding exists immediately). For new users, fall back to content-based recommendations seeded from the user's stated preferences (onboarding survey) or from the popularity baseline. For new items, inject them into recommendations via an "explore" slot regardless of predicted relevance, then let collaborative signal accumulate.

2. "Why store embeddings in FAISS instead of SQLite?" — Strong answer: SQLite is a row-oriented relational store. Nearest-neighbour search over embeddings requires computing a dot product between the query vector and every stored vector, then sorting. In SQLite, this would require deserializing every BLOB and running the computation in Python — $O(N)$ deserialization cost plus Python loop overhead. FAISS stores vectors in contiguous memory, computes dot products with BLAS (AVX2/AVX-512 on modern CPUs), and achieves ~$10^9$ operations/second — three orders of magnitude faster.

3. "When would you use item-item collaborative filtering instead of user-user?" — Strong answer: item-item is more stable — items accumulate ratings over their lifetime while new users are constantly joining with no history. Computing item-item similarity offline (nightly batch) means the recommendation step is just a lookup into a precomputed similarity matrix. User-user requires finding similar users at query time, which is $O(U \cdot I)$ without indexing. Item-item is the better choice when users are high-churn and the catalog is relatively stable.

4. "What is the difference between implicit and explicit feedback?" — Strong answer: explicit feedback is a rating (1-5 stars) — the user's stated preference. Implicit feedback is a behavioral signal (click, view duration, add-to-cart) — the user's revealed preference. Implicit is higher volume but noisier (a user might click a recipe out of curiosity, not intent to cook it). Most production recommenders use implicit feedback because explicit ratings are rare (< 1% of interactions). The model must learn to distinguish positive signal (long view, save, cook) from noise (brief click, immediate back-navigation).

5. "How do you prevent the recommender from getting stuck in a filter bubble?" — Strong answer: a pure collaborative filter amplifies existing preferences — users who rate action movies highly get more action movies, never discovering drama. Three interventions: (1) MMR diversity penalty in re-ranking, (2) serendipity injection (one "wildcard" slot per page), (3) periodic exploration (Thompson sampling over item categories, balancing exploitation of known preferences with exploration of uncovered categories). The right mix depends on the product goal — a meal-prep app wants high relevance; a discovery-focused app wants high serendipity.

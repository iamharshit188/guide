# Real-Time ML Prediction WebSocket Server — Thinking Guide
> **Module:** 04 — Backend Engineering · **Difficulty:** Advanced · **Type:** THINKING · **Time:** ~6 hours

## The Problem

**Input:** Streaming financial tick data or sensor events from 10+ concurrent clients, each sending JSON events at variable rate (burst to 500 events/sec per client). Each event: `{client_id, timestamp, features: [f1..fn]}`.

**Output:** Per-event prediction pushed back to that client: `{prediction, confidence, latency_ms}`. Prediction runs a pre-trained sklearn pipeline (scaler + classifier or regressor).

**Constraints:** Sub-100ms round-trip including inference. No event loss under normal load. Per-client state isolation — each client carries its own rolling window for feature engineering. Graceful cleanup on disconnect.

---

## Before You Write Any Code: Build the Mental Model

WebSockets are full-duplex persistent TCP connections. Unlike HTTP, the server can push without a client request. This changes the concurrency model: Flask's default threaded WSGI handles one request per thread, but a SocketIO server maintains thousands of open connections simultaneously — a fundamentally different I/O pattern.

Flask-SocketIO requires an async-capable backend: `eventlet` or `gevent`. These monkey-patch the standard library's socket and threading primitives, making blocking calls cooperative. Without monkey-patching, `socketio.sleep()` blocks the entire process, all client handlers serialize, and you get a thundering-herd disconnect when inference takes 50ms.

sklearn inference is CPU-bound, not I/O-bound. The GIL means Python threads don't parallelize numpy operations. With 10 concurrent clients each triggering inference simultaneously, threading gives you serial CPU work with context-switch overhead. A `multiprocessing.Pool` with shared model state (via pickle on init) gets true parallelism but carries IPC serialization cost (~0.5ms for small payloads). The break-even is roughly: if inference > 10ms, multiprocessing wins; if < 5ms, threads suffice.

Per-client state in SocketIO lives in Python memory keyed by `request.sid` (session ID). Each connection gets a unique SID. The trap is storing mutable state in a module-level dict without cleanup — every disconnect that doesn't trigger your cleanup handler leaks one dict entry plus whatever rolling window data it holds.

Backpressure is a policy decision with no universally correct answer. If inference is slower than input rate, your queue grows unboundedly. Three options: drop oldest (freshest data is most relevant — correct for financial ticks), drop newest (reject overload, force client to back off — correct for safety-critical sensors), or block (TCP backpressure propagates to sender — correct when no data can be lost). Financial ticks: drop oldest.

## Decompose It

### Phase 1 — Environment Setup and Monkey-Patching

**The question this phase answers:** What async backend is running, and is it set up before any other import?

**Key decisions:**
- `eventlet` vs `gevent`: eventlet is easier to install and better documented for Flask-SocketIO; gevent has a larger ecosystem. Pick eventlet unless you have a specific gevent dependency.
- Monkey-patch position: the `eventlet.monkey_patch()` call must be the very first import in your entry point — before Flask, before numpy, before anything that touches `socket` or `threading`. Placing it after Flask import silently breaks SSL, DNS, and threading primitives in ways that manifest as intermittent hangs, not immediate errors.

**Trap:** Importing `numpy` or `sklearn` before monkey-patching causes `fork`-safety issues on some platforms. The safe pattern is `import eventlet; eventlet.monkey_patch()` as literal lines 1–2 of your server file. Verify correctness with `python -c "import eventlet; eventlet.monkey_patch(); import socket; print(type(socket.socket))"` — should show the eventlet green wrapper, not `_socket.socket`.

**Done when:** `socketio.run(app, async_mode='eventlet')` starts without warning about async_mode mismatch. `flask_socketio.test_client()` connects and disconnects cleanly in a unit test.

---

### Phase 2 — Per-Client State Architecture

**The question this phase answers:** Where does rolling window data live, and how does it stay isolated between concurrent clients?

**Key decisions:**
- Module-level dict `client_state: dict[str, ClientState]` keyed by SID is the simplest approach. Thread-safety with eventlet is cooperative — no true concurrent writes — but you should still use `eventlet.semaphore.Semaphore` if any state mutation crosses a `yield` point (e.g., an async DB write between read and write of the dict).
- Dataclass vs plain dict for `ClientState`: use a dataclass. It enforces schema, allows `__slots__` for memory efficiency, and makes `del client_state[sid]` leave no dangling references if no other object holds the instance.
- Rolling window size: fixed-length `collections.deque(maxlen=N)` — O(1) append and automatic oldest-drop. Never use a plain list with `list.pop(0)` — that's O(N) per operation, catastrophic at 500 events/sec.

**Trap:** SocketIO rooms and namespaces are for grouping and broadcasting, not state isolation. They are not a substitute for a per-SID state dict. A room is just a set of SIDs that receive the same emit; it carries no application data.

**Done when:** `connect` handler creates a `ClientState`, `disconnect` handler removes it, and a 60-second idle test with `psutil.Process().memory_info().rss` shows flat memory.

### Phase 3 — Backpressure Queue and Drop Policy

**The question this phase answers:** What happens when inference can't keep up with inbound event rate, and how do you prevent unbounded queue growth?

**Key decisions:**
- Per-client bounded queue (`eventlet.queue.Queue(maxsize=N)`) decouples the receive handler (fast — just enqueue) from the inference worker (slow — dequeue, infer, emit).
- Drop-oldest implementation: `queue.put_nowait(event)` in try/except `Full` — on `Full`, call `queue.get_nowait()` to discard the oldest item, then `queue.put_nowait(event)` again. This guarantees the queue always holds the most recent N events.
- Drop-newest alternative: catch `Full` and discard the new event. Correct when data ordering matters and you can't skip any past event (rare in finance, more common in time-series replay).

**Trap:** Using a single global queue for all clients serializes inference across all of them — one slow client blocks every other. One queue per client, one inference greenlet per client. With eventlet, greenlets are cheap (< 10KB each); 1000 clients = ~10MB overhead, acceptable.

**Done when:** Under simulated 600 events/sec with an artificial 50ms inference sleep, queue depth stabilizes at maxsize and drops are logged per SID. No OOM after 5 minutes.

---

### Phase 4 — Inference Layer and GIL Strategy

**The question this phase answers:** Does threading give sufficient parallelism for your model's inference time, or do you need multiprocessing?

**Key decisions:**
- Profile first. Measure pure `model.predict_proba(X)` wall time with `time.perf_counter()` for a single client. numpy releases the GIL during BLAS matrix operations, so multiple eventlet greenlets can overlap on the CPU during those calls.
- If inference < 5ms: eventlet + shared global model is fine. The GIL is only held for Python bytecode between BLAS calls; BLAS itself runs outside the GIL.
- If inference > 15ms or model is pure Python (decision tree with no vectorized BLAS): switch to `concurrent.futures.ProcessPoolExecutor(initializer=load_model)`. Each worker process loads the model once; feature arrays are sent via pickle over a pipe. Pickle overhead for a (20, N) float64 array is ~0.1ms.

**Trap:** Loading the model inside each inference greenlet means deserialization on every event — `joblib.load` on a 50MB sklearn pipeline takes ~200ms. Load once at startup into a module-level variable protected by an `MODEL_READY` flag.

**Done when:** 10 concurrent clients each at 100 events/sec produces p99 round-trip < 100ms measured end-to-end (SocketIO emit to client receipt).

---

### Phase 5 — Graceful Disconnect and Memory Cleanup

**The question this phase answers:** What happens to per-client resources when a client disconnects abruptly — TCP reset, not clean WebSocket close?

**Key decisions:**
- Flask-SocketIO fires the `disconnect` event for both clean and abrupt disconnects within the heartbeat timeout. Register `@socketio.on('disconnect')` to pop from `client_state`. The inference greenlet checks `while sid in client_state` and exits.
- Heartbeat tuning: default ping_interval=25s, ping_timeout=5s means a dead client holds a slot for up to 30 seconds. For financial apps, use `SocketIO(app, ping_interval=5, ping_timeout=2)` — 7-second max leak window.
- Future/subprocess cleanup: if using `ProcessPoolExecutor`, store the `Future` in `ClientState.pending_future`. During disconnect, call `state.pending_future.cancel()` before deleting the state entry. A dangling future holds a reference to the submitted feature array and its closure.

**Trap:** Deleting a dict entry does not cancel in-flight work. If the inference worker greenlet is mid-`predict_proba` when disconnect fires, it will try to `socketio.emit('prediction', ..., to=sid)` after cleanup. Emit to a disconnected SID is silently dropped by SocketIO, so this is safe — but confirm it doesn't re-create the client state entry.

**Done when:** 50 rapid connect/disconnect cycles under load shows stable RSS over 10 minutes. Zero entries remain in `client_state` 30 seconds after all clients disconnect.

## Data Flow

```
Client (WebSocket)
    |
    | tick event {features: [...]}
    v
handle_tick() [eventlet greenlet, non-blocking receive]
    |
    | put_nowait → if Full: pop oldest, then put
    v
Per-Client Queue (maxsize=50, one per SID)
    |
    | get(timeout=1.0) [inference_worker greenlet, one per SID]
    v
state.window.append(features)       [deque maxlen=20, O(1)]
    |
    v
MODEL.predict_proba(X_flat)         [sklearn pipeline, GIL released in BLAS]
    |
    v
socketio.emit('prediction', result, to=sid)
    |
    v
Client receives {prediction, confidence, latency_ms}
```

## Edge Cases to Handle Before You Code

1. **Malformed feature vector from client (wrong length, NaN, inf):** sklearn scalers propagate `NaN` silently through `StandardScaler.transform` into predictions — the model returns garbage without raising. Validate before queuing: check `len(features) == N_FEATURES` and `all(math.isfinite(f) for f in features)`. Return `{"error": "invalid_features"}` via emit and skip the queue.

2. **Model not loaded when first event arrives:** startup race if model loading is slow (joblib loading a large pickle). Use a module-level `MODEL_READY = threading.Event()` set after loading completes. Inference worker calls `MODEL_READY.wait(timeout=10)` before processing; returns `{"reason": "model_loading"}` if it times out.

3. **Client reconnects after network blip — new SID, same logical user:** SocketIO SIDs are ephemeral; reconnect = new SID = fresh `ClientState`. Rolling window resets, causing 20 `warming_up` responses. Fix: client sends a stable `user_id` in connect handshake query params. Server maintains a secondary `user_state: dict[user_id, deque]` with TTL=30s. On new connect, restore the deque if TTL hasn't expired.

4. **Monkey-patching breaks SSL in production HTTPS/WSS deployment:** eventlet's SSL green socket has different timeout semantics than CPython's `ssl` module. Symptoms: TLS handshake timeouts that only appear under load (not in development), because the green socket's `do_handshake` blocks the event loop differently. Test with `wrk -c 100` over HTTPS before declaring production-ready.

5. **High-frequency client starves low-frequency client under eventlet:** eventlet's scheduler is cooperative FIFO. A 500 events/sec client generates 500 pending greenlet wakeups per second. Per-client queues and per-client inference workers isolate this: each worker blocks on its own queue and yields when waiting, regardless of other clients' queue depth.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|----------|----------|----------------|
| Async backend | eventlet (monkey-patch stdlib) | gevent (monkey-patch stdlib) | Default choice for Flask-SocketIO; switch to gevent only for specific library compatibility |
| Inference parallelism | Eventlet greenlets + shared model | ProcessPoolExecutor (true parallelism) | Greenlets when inference < 10ms (BLAS releases GIL); multiprocessing when > 15ms or pure-Python model |
| Backpressure policy | Drop oldest (freshest data wins) | Drop newest (reject overload) | Drop oldest for financial ticks where recency > completeness; drop newest for replay-sensitive time series |
| Client state lifetime | In-memory dict (fast, single-process) | Redis with TTL (survives restart, multi-process) | In-memory for single-server; Redis when load-balancing multiple SocketIO workers with sticky sessions |
| Model per worker | One global shared model | One model copy per process | Shared with eventlet/threads; per-process required for `ProcessPoolExecutor` (models not fork-safe if stateful) |

## What Makes This Non-Trivial

The interaction between eventlet's cooperative scheduler and numpy's BLAS GIL release is subtle: under light load, greenlets appear to run concurrently because they yield at I/O points (`socketio.emit`). Under heavy load, a greenlet running a 30ms numpy matmul holds the GIL for the full duration — eventlet cannot preempt it — starving all other greenlets. This latency spike only appears above a client-count threshold, making it a production surprise invisible during single-client testing.

Memory leak pathways compound each other: per-client dict entries without disconnect cleanup, futures referencing closures over large feature arrays, `deque` entries holding numpy arrays (each 20×N float64 = 1.28KB minimum), and eventlet's internal socket buffers for heartbeat-timed-out connections. All four must be addressed; fixing three of four still produces slow leaks that OOM a 4GB server after 48 hours.

## Interview Angles This Project Develops

1. "Why does Flask-SocketIO require monkey-patching, and what exactly does it replace?" — Demonstrates knowledge that eventlet replaces `socket.socket`, `threading.Event`, `time.sleep`, and `ssl` with cooperative equivalents so thousands of connections can multiplex on one OS thread without blocking each other.

2. "How would you scale this to 10,000 concurrent clients?" — Strong answer covers: sticky-session load balancing (SocketIO state is per-process), Redis pub/sub adapter (`flask_socketio.RedisManager`) for cross-process emit, stateless inference service behind gRPC, client state in Redis with per-SID TTL so any worker can handle any client.

3. "Your inference p99 is 150ms under load. How do you determine if it's GIL contention, queue depth, or model speed?" — Instrument each layer: log queue depth on dequeue (measures backpressure), log pure `predict_proba` wall time (measures model speed), log greenlet count with `eventlet` tracing (measures scheduler saturation).

4. "What happens if you put `eventlet.monkey_patch()` after `from flask import Flask`?" — Flask's `g`, `request`, and `current_app` use `threading.local()`. If monkey-patched after import, those locals are bound to OS threads, not greenlets, causing request context bleed — one client's `request.sid` appears in another's handler.

5. "How do you authenticate WebSocket connections without re-sending credentials on every tick?" — JWT in the connect handshake (`request.args.get('token')` in the `connect` handler). Validate once; store user identity in `ClientState`. Reject with `return False` from the connect handler to refuse connection. Never re-validate on tick events — that's O(events) auth checks instead of O(connections).

# Zero-Shot Image Classifier — Thinking Guide

> **Module:** 13 · **Difficulty:** Advanced · **Type:** THINKING · **Time:** ~3 hours

## The Problem

**Input:** An image $x$ and a set of class names $\{c_1, c_2, \ldots, c_K\}$ provided at inference time as strings.
**Output:** A probability distribution over $K$ classes, with a predicted label $\hat{y} = \arg\max_k p(c_k \mid x)$.
**Constraints:** No gradient updates. No labeled training data. The model has never explicitly seen the class names during the fine-tuning phase of this project. $K$ can be 2 or 10,000 — the architecture must not change.

---

## Before You Write Any Code: Build The Mental Model

CLIP creates a shared metric space where semantically related images and text live near each other. The key insight for zero-shot classification is that classification is just nearest-neighbor retrieval in that space: given an image embedding $v \in \mathbb{R}^{512}$, you want to find which text embedding $t_k$ it is closest to. The "classifier" is never trained — it is assembled at inference time from the class names.

The cosine similarity between the image and each class text embedding forms a raw score. Applying softmax with temperature $\tau$ converts these scores to probabilities:

$$p(c_k \mid x) = \frac{\exp(\text{sim}(v, t_k) / \tau)}{\sum_{j=1}^{K} \exp(\text{sim}(v, t_j) / \tau)}$$

The model works because CLIP was pretrained to align image and text representations. The text "a photo of a cat" was co-trained with actual cat images 400M times. The embedding space is not random — it encodes semantic structure learned from the natural distribution of internet image-text pairs.

## Decompose It

### Phase 1 — Prompt Engineering for Vision

**The question this phase answers:** What text string should represent each class?

**Key decisions:**

The naive approach uses the class name as the text: `processor(text=["cat", "dog", ...]`. This produces mediocre results because single words are out-of-distribution for CLIP. During pretraining, text inputs were full captions ("A photo of a golden retriever playing in the park"), not single tokens. The text encoder's context window is 77 BPE tokens; feeding 1 token means the `[EOS]` token dominates context and the embedding is mostly positional noise.

The standard prompt `"a photo of a {class}"` works significantly better because:
- It matches the distributional prior of pretraining captions (most captions describe what is in a photo)
- The surrounding context helps the text encoder disambiguate polysemous class names: "crane" → bird vs construction equipment depends on sentence context
- The 77-token context is better utilized, giving attention more meaningful positions to attend over

Domain-specific prompts outperform generic ones. For medical imaging: `"a dermoscopic image of {class}"`. For satellite imagery: `"a satellite view of {class}"`. The prompt should describe the visual modality, not just the semantic category.

**Trap:** Prompt selection is a form of overfitting to your evaluation set. If you tune prompts on the ImageNet validation set and report accuracy on the same set, you are leaking test signal. Always tune on a held-out subset or a different dataset.

**Done when:** You have a `make_prompt(class_name: str) -> str` function and a documented rationale for the prompt template. You have tried at least three templates and measured top-1 accuracy differences.

### Phase 2 — Ensemble Prompting: Averaging Embeddings vs. Averaging Logits

**The question this phase answers:** Given multiple phrasings per class (e.g., `"a photo of a {c}"`, `"a close-up of a {c}"`, `"a picture of the {c}"`, `"an image of a {c}"`), how do you combine them?

**Key decisions:**

**Option A — Average embeddings (then normalize):**
$$t_k = \text{normalize}\!\left(\frac{1}{P}\sum_{p=1}^{P} e_p(c_k)\right)$$
Compute the text embedding for each prompt, average them, then L2-normalize. The single averaged-and-normalized vector represents the class. This costs $P$ text encoder forward passes per class, but only $K$ dot products per image query.

**Option B — Average logits:**
$$p(c_k \mid x) = \frac{1}{P}\sum_{p=1}^{P} \text{softmax}(\text{sim}(v, e_p(c_k)) / \tau)_k$$
Compute one similarity score per prompt per class, apply softmax independently, then average the $K$-dimensional probability vectors. This costs $P \times K$ dot products per image but does not collapse the prompt diversity into a single vector.

**Which is better:** Averaging embeddings in representation space is almost always better. The intuition is geometric: you are finding the centroid of a cluster of semantically similar text embeddings and re-projecting it onto the unit hypersphere. This centroid lives closer to the true "semantic center" of the class than any individual prompt. Averaging logits after softmax is a nonlinear aggregation that amplifies the variance introduced by poorly-phrased prompts. OpenAI's original CLIP paper uses embedding averaging and reports consistent gains over single-prompt evaluation.

**Practical note:** Cache the class text embeddings. For 1,000 ImageNet classes and 80 prompt templates (the ensemble used in the CLIP paper), embedding computation takes ~2 minutes on CPU but the results are static — recompute only when classes change.

**Trap:** After averaging, you must re-normalize. The average of unit vectors is not a unit vector. Skipping renormalization means the inner product no longer equals cosine similarity, and classes with more spread-out prompts get systematically lower scores.

**Done when:** You can construct a matrix $T \in \mathbb{R}^{K \times 512}$ of L2-normalized, ensemble-averaged class embeddings. Classification is then a single matrix multiply: $\text{logits} = v \cdot T^T \in \mathbb{R}^K$.

### Phase 3 — Temperature $\tau$ Calibration

**The question this phase answers:** What happens to prediction confidence when $\tau$ is wrong, and how do you set it correctly?

**Key decisions:**

CLIP exposes a learned temperature parameter (initialized at $\tau = 0.07$ during pretraining). When you extract logits for classification, you should divide by this temperature before softmax. Without it, raw cosine similarities in $[-1, 1]$ produce near-uniform softmax outputs — the model assigns roughly equal probability to every class because the range of cosine similarities across classes is small (e.g., 0.18 to 0.32 for a 1000-class problem).

The effect of temperature on calibration: low $\tau$ sharpens the distribution (confident, potentially overconfident), high $\tau$ flattens it (underconfident, well-calibrated entropy but bad ECE if accuracy is high). The CLIP-pretrained $\tau$ is calibrated to the contrastive objective during pretraining, but it is not necessarily calibrated for downstream classification. Use temperature scaling on a held-out calibration set: find $\hat{\tau}$ that minimizes negative log-likelihood (NLL) on calibration examples, then apply it at test time.

Calibration matters if your classifier's confidence scores are consumed downstream (e.g., rejection threshold, confidence-weighted ensemble). If you only care about top-1 accuracy, temperature is a monotonic scaling and does not change the ranking — calibration has zero effect on accuracy. This is a common confusion.

**Trap:** Calibrating $\tau$ on the test set is data leakage. Use a validation split. If the dataset is too small for a validation split, use cross-validation for calibration.

**Done when:** You have measured Expected Calibration Error (ECE) with and without calibration on a 500-image held-out set. ECE should drop by at least 30% with proper temperature scaling.

### Phase 4 — Evaluate on ImageNet-1k Subset and Compare to ResNet-50

**The question this phase answers:** Is zero-shot CLIP competitive with a supervised baseline?

**Key decisions:**

CLIP ViT-B/32 achieves ~63% top-1 accuracy on full ImageNet-1k zero-shot. A fine-tuned ResNet-50 achieves ~76% top-1. The 13-point gap reflects the fundamental trade-off: ResNet-50 has seen labeled ImageNet examples and can overfit to the label distribution, class boundary placement, and evaluation preprocessing artifacts. CLIP has not.

The comparison is not apples-to-apples in two ways:
1. CLIP uses prompts; ResNet-50 uses a linear classifier head. The prompt engineering for CLIP is a form of engineering that requires human effort and dataset knowledge.
2. CLIP was pretrained on 400M image-text pairs (WIT); ResNet-50 was pretrained on ImageNet-21k then fine-tuned. Parameter counts differ (151M for ViT-B/32 vs 25M for ResNet-50).

The correct comparison for your project: use a 10-class or 100-class ImageNet subset. Report top-1 accuracy for (a) zero-shot CLIP with single prompt, (b) zero-shot CLIP with 5-prompt ensemble, (c) fine-tuned ResNet-50 (pretrained weights + linear probe on the same 100 classes with 50 images per class). The linear probe result should beat zero-shot CLIP if data is sufficient; zero-shot CLIP should beat a randomly-initialized ResNet-50 trained from scratch on 50 images/class.

**Trap:** ImageNet class names are not always human-readable. Class 281 is "tabby cat," but the folder name is "n02123045." Build the name mapping from ImageNet's `LOC_synset_mapping.txt` before constructing prompts.

**Done when:** You have a results table showing accuracy for each condition and can articulate why zero-shot is worse on average but better on rare tail classes where fine-tuned models have insufficient training examples.

### Phase 5 — Know CLIP's Failure Modes

**The question this phase answers:** Where does the joint embedding space break down?

**Key decisions:**

CLIP embeds concepts, not relationships between concepts. "A red cube on top of a blue sphere" and "A blue cube on top of a red sphere" receive nearly identical embeddings because the contrastive pretraining loss does not require spatial composition to be encoded. The text encoder sees both as a bag of concepts (red, blue, cube, sphere, on top) with similar attention patterns.

Counting is unreliable. "Two dogs" vs "three dogs" produces similar text embeddings and CLIP cannot reliably distinguish images with different object counts because the visual transformer's global pooling via `[CLS]` token discards spatial count information.

Fine-grained recognition (distinguishing dog breeds, car models, bird species) underperforms compared to coarse-grained classification. The web image-text pairs in WIT rarely describe breed-level distinctions; captions say "a dog" not "a Vizsla."

Negation fails. CLIP does not reliably handle "a person NOT wearing a hat" — the embedding of this text sits closer to hat-wearing images than hat-absent images in most studies. The attention mechanism encodes presence of tokens, and "NOT" has no reliable semantic inversion effect in the learned space.

**Done when:** You can cite a specific failure example from your test set and explain which structural property of the CLIP pretraining objective causes it.

## Data Flow

```
Class names [c_1 ... c_K]
        │
        ▼
Prompt templates × P
        │
        ▼
CLIP Text Encoder  →  embeddings (K×P, 512)
        │
        ▼  average + L2-normalize
Class embedding matrix T  (K, 512)
        │
        │                Query image x
        │                    │
        │               CLIP Vision Encoder
        │                    │
        │               image embedding v  (512,)
        │                    │
        └──── dot product ───┘
                │
                ▼
         logits = v · T^T   (K,)
                │
                ▼  divide by τ, softmax
         p(c_k | x)   (K,)
                │
                ▼
         ŷ = argmax_k  →  class label
```

## Edge Cases to Handle Before You Code

1. **Multi-word class names with commas:** ImageNet classes like "bulletproof vest, body armor" — split on comma and use the first token, or construct a separate prompt per alias and ensemble.
2. **Unicode class names:** Some fine-grained datasets use accented characters. Ensure your tokenizer and the CLIP BPE vocab handle them; if not, transliterate to ASCII.
3. **$K > 77$ tokens total for a class name:** This is rare but possible for taxonomy names. The BPE tokenizer will silently truncate to 77 tokens. Log a warning if any class name tokenizes to more than 10 tokens.
4. **Single-class degenerate case ($K = 1$):** Softmax of a single logit is always 1.0. This is correct but useless. Guard with `assert K >= 2`.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|---|---|---|---|
| Prompt aggregation | Average embeddings (pre-softmax) | Average logits (post-softmax) | Almost always — more geometrically principled, fewer parameters, faster at query time |
| Vision encoder | CLIP ViT-B/32 (frozen) | CLIP ViT-B/32 (fine-tuned head) | Frozen when data < 500 labeled examples; fine-tuned head when you have 1k+ per class |
| Temperature | CLIP default $\tau=0.07$ | Calibrated $\hat{\tau}$ on validation set | Calibrated when downstream uses confidence scores (rejection, ensembling); default otherwise |
| Evaluation subset | Full ImageNet-1k | 100-class subset | 100-class for iteration speed; full only for final benchmark comparison |
| Prompt set size | 1 prompt | 80 prompts (OpenAI ensemble) | 5–10 prompts captures 80% of ensemble gain at 10% of the compute cost |

## What Makes This Non-Trivial

Zero-shot classification sounds simple but hides significant engineering. The gap between 50% and 63% top-1 accuracy on ImageNet-1k is almost entirely explained by prompt engineering and ensemble strategy — not model architecture. This means that evaluation results are not reproducible unless the exact prompt templates are reported, which most papers fail to do.

Temperature calibration introduces a second evaluation-time hyperparameter that must be reported alongside accuracy. A model can appear well-calibrated on one dataset and severely miscalibrated on another because cosine similarity magnitudes shift when the semantic cluster density changes between datasets.

The comparison to fine-tuned ResNet-50 reveals the core trade-off: zero-shot CLIP generalizes to new classes at inference time but pays a fixed accuracy penalty on any class that a fine-tuned model has seen sufficient labeled examples of. The break-even point — when zero-shot beats fine-tuning — occurs only when labeled data per class falls below roughly 10–20 examples. This is the practical regime for zero-shot: tail classes, novel categories, rapidly-changing taxonomies.

## Interview Angles This Project Develops

1. **"How does zero-shot classification work without training data?"** — Strong answer: joint embedding space trained contrastively; classification is nearest-neighbor retrieval in that space; class prototypes are constructed from text at inference time. Explain why this is fundamentally different from few-shot (which uses adaptation) and supervised (which uses SGD on labeled data).

2. **"Why does prompt engineering matter for CLIP?"** — Strong answer: text encoder was pretrained on natural language captions, not single-word labels; OOD inputs produce low-quality embeddings; prompt context shifts the embedding toward the distributional prior; ensemble prompting reduces variance from any single phrasing artifact.

3. **"What is temperature calibration and when does it matter?"** — Strong answer: temperature scales logits before softmax; wrong temperature produces overconfident or underconfident probability estimates; matters for rejection thresholds, cost-sensitive decisions, and ensembling; does not affect top-1 accuracy (monotonic scaling).

4. **"Where does CLIP fail?"** — Strong answer: spatial relationships and object composition (not encoded by global CLS pooling); counting (spatial information discarded in pooling); negation (attention mechanism has no reliable semantic inversion); fine-grained recognition (training data rarely contained subcategory-level captions).

5. **"When would you use zero-shot CLIP instead of fine-tuning?"** — Strong answer: when class taxonomy changes frequently (e-commerce, medical coding updates); when labeled data is under ~20 examples per class; when you need to support open-vocabulary queries at inference time without model retraining; when deployment constraints prevent maintaining multiple fine-tuned models per domain.

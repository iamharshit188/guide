# Multi-Source Research Assistant — Thinking Guide
> **Module:** 08 — RAG & Vector Search · **Difficulty:** Advanced · **Type:** THINKING · **Time:** ~6 hours

## The Problem

**Input:** Heterogeneous sources: PDFs (research papers), web pages (scraped HTML), structured CSVs (tabular data), SQLite tables (relational data). A free-form natural language query. A maximum context budget of 8 chunks for the generation step.

**Output:** A synthesized answer with per-claim citations identifying which source chunk supports each statement. The system must detect low-confidence retrieval and respond "I don't know" rather than hallucinating.

**Constraints:** Query-time routing selects sources. Results from multiple sources fused and cross-encoder re-ranked. Context budget enforced before generation.

---

## Before You Write Any Code: Build the Mental Model

Heterogeneous RAG is harder than single-source RAG because the chunking strategy must vary per source type, and the format in which you present structured data to an embedding model affects retrieval quality significantly. A CSV row that reads `42,John,Sales,2021-04-15` is meaningless to a sentence embedding model without column context. The same row as `employee_id: 42, name: John, department: Sales, hire_date: 2021-04-15` embeds into a semantically informative region of the space.

Query routing matters because not all queries need all sources. A query about "quarterly revenue trends" should route to CSVs and SQLite tables, not PDFs. A query about "recent developments in CRISPR" should route to PDFs and web pages. Routing incorrectly degrades retrieval recall and fills the context budget with irrelevant chunks. The routing decision is a classification problem: you can use keyword matching, a lightweight LLM call, or embeddings of source descriptions.

Two-stage retrieval is mandatory when you have more than a few hundred chunks. The cross-encoder cannot score 10,000+ (query, chunk) pairs at query time. The bi-encoder retrieves 100 candidates per source in milliseconds; the cross-encoder then re-ranks the combined 400 candidates (100 × 4 sources) to 10, spending roughly 800ms on CPU. Without the pre-filter, you'd need 10,000+ cross-encoder passes per query.

Context window budget means you must select 8 chunks from 10 cross-encoder-scored candidates, choosing the 8 that maximize coverage and minimize redundancy. This is a greedy set cover problem: iteratively pick the highest-scoring chunk that adds new information not already covered by previously selected chunks. MMR (Maximal Marginal Relevance) formalizes this: $\text{MMR} = \lambda \cdot \text{relevance}(c, q) - (1-\lambda) \cdot \max_{s \in S} \text{similarity}(c, s)$ where $S$ is the already-selected set.

## Decompose It

### Phase 1 — Source-Specific Chunking Strategy

**The question this phase answers:** How does chunking strategy differ per source type, and why does format matter for embedding quality?

**Key decisions:**
- PDFs by paragraph: use a PDF parser (PyMuPDF or pdfplumber) that preserves paragraph boundaries. Chunk at paragraph level (200–500 tokens). Add document title and section header as a prefix: `"Paper: Attention Is All You Need | Section: Experiments | {paragraph_text}"`. This gives the embedding model structural context.
- Web pages by `<p>` tags: scrape with BeautifulSoup, extract `<p>` elements, filter by minimum length (> 50 characters to skip boilerplate like "Click here"). Add page title and URL as prefix.
- CSV rows with column headers prepended: do not embed raw CSV rows. Convert each row to `"column1: value1, column2: value2, ..."` format. Group related rows into a single chunk if rows are short (< 50 tokens each) — embed batches of 5–10 rows together.
- SQL rows as key:value text: same as CSV. For relational queries where a join is needed (employee name + department + salary from three tables), materialize the join result before embedding — don't embed raw foreign keys.

**Trap:** Chunking with fixed token count (512 tokens) regardless of source type. A 512-token chunk from a PDF may span three unrelated paragraphs; a 512-token chunk from a CSV may contain 20 rows about 20 different entities. Fixed-size chunking destroys semantic coherence. Source-aware chunking is not optional.

**Done when:** Each source type has a `chunk(source) -> list[Chunk]` function where each `Chunk` has: `text` (embedding input), `source_type`, `source_path`, `metadata` (page number for PDFs, URL for web, row range for CSV, table name for SQL), `chunk_id`. Self-retrieval test: embed a chunk, search, top-1 is that exact chunk.

---

### Phase 2 — Index Construction and Query Routing

**The question this phase answers:** How do you organize multi-source chunks into an index, and how does query-time routing select which sources to search?

**Key decisions:**
- Single FAISS index vs per-source indexes: use per-source indexes. This allows routing to search only selected source indexes without filtering a large combined index. With 4 sources, you have 4 `IndexFlatIP` instances. Query routing selects 1–4 of them.
- Routing strategies ranked by accuracy vs cost:
  1. **All-sources-always**: highest recall, highest latency. Search all 4 indexes, combine 400 candidates. Correct when you don't trust your router.
  2. **Keyword classifier**: fast, no LLM call. Pattern match: query contains "paper", "study", "research" → route to PDF; contains "trend", "data", "statistic" → route to CSV/SQL. Low accuracy on ambiguous queries.
  3. **LLM-as-router**: send query + source descriptions to a small LLM, ask it to select sources. High accuracy, 200–400ms latency overhead. Correct for production systems.
- Source description for LLM-as-router: each source needs a one-sentence description of its contents. Store these as constants. The router prompt: `"Given this query: {query}\n\nAvailable sources:\n1. PDFs: {pdf_desc}\n2. Web: {web_desc}\n...\n\nWhich sources are most likely to contain the answer? Reply with source numbers only."`

**Trap:** Binary routing (route to source OR don't) vs soft routing (score each source 0–1 and include all sources above threshold). Binary routing fails on multi-faceted queries ("Compare what the research papers say vs what the industry reports show about protein folding"). Soft routing or all-sources-always is safer for research assistants where recall matters more than latency.

**Done when:** `route_query(query) -> list[str]` returns a list of source type names. Test with 10 diverse queries and verify that at least 80% route to the correct dominant source.

---

### Phase 3 — Two-Stage Retrieval and Cross-Encoder Re-Ranking

**The question this phase answers:** How do you combine results from multiple source indexes and re-rank them coherently?

**Key decisions:**
- Bi-encoder retrieval: for each selected source index, retrieve top-25 candidates. With 4 sources selected, you have up to 100 candidates.
- Cross-encoder fusion: score all 100 (query, chunk_text) pairs with the cross-encoder. Cross-encoder scores are comparable across sources — a web chunk scoring 8.3 and a PDF chunk scoring 8.1 can be directly ranked against each other. This is the key advantage of cross-encoding: it normalizes scores across heterogeneous sources.
- Final ranking: sort all candidates by cross-encoder score descending. Take top-10.

**Trap:** Using bi-encoder scores for cross-source comparison. Bi-encoder cosine similarities are not comparable across sources because the embedding space geometry differs slightly for different text distributions (PDFs use more formal language, web pages use more casual language, CSV text is structured differently). Cross-encoder scores are derived from the full text interaction and are calibrated by the fine-tuning data — they're more reliable for cross-source comparison.

**Done when:** For a query answerable from multiple sources, the re-ranked top-10 contains chunks from at least 2 different source types. For a query answerable only from PDFs, the top-3 are all PDF chunks (web and CSV chunks ranked below).

---

### Phase 4 — Context Budget Selection and MMR

**The question this phase answers:** How do you select 8 chunks from the top-10 re-ranked candidates when some are redundant?

**Key decisions:**
- Maximal Marginal Relevance (MMR): iteratively select chunks that are both relevant (high cross-encoder score) and non-redundant (low similarity to already-selected chunks). $\lambda = 0.7$ balances relevance vs diversity (higher $\lambda$ = more relevance-focused).
- Similarity between candidates: compute pairwise cosine similarity between candidate chunk embeddings. If two chunks have similarity > 0.85, they're near-duplicates from the same passage. MMR will naturally deprioritize the second one.
- Budget enforcement: stop at 8 chunks regardless of MMR score. The generation model's context window can fit ~8 × 300-token chunks = 2,400 tokens, leaving ~5,600 tokens for system prompt and generated answer (assuming 8K context window).

**Trap:** Always including all chunks from the cross-encoder top-10 without MMR. Two web pages scraped from the same site, or two paragraphs from the same PDF section, may contain the same information. Including both wastes context budget and may cause the LLM to over-count that evidence. MMR is not optional when sources can be redundant.

**Done when:** For a query where 3 of the top-5 chunks are near-duplicates from the same PDF section, MMR selects at most 1 of those 3 and fills the remaining budget with chunks from other sources.

---

### Phase 5 — Answer Synthesis with Citations and Hallucination Detection

**The question this phase answers:** How do you generate an answer with per-claim citations, and how do you detect when retrieval has failed?

**Key decisions:**
- Per-claim citation format: in the system prompt, instruct the LLM to format claims as `[claim text] [source_id]` where `source_id` maps to the chunk metadata. Each selected chunk gets an ID (`[1]`, `[2]`, ...). The prompt includes a legend: `"Context:\n[1] (PDF: paper.pdf, p.14) {text}\n[2] (Web: example.com) {text}\n..."`.
- Hallucination detection via retrieval confidence: compute the average cross-encoder score of the top-3 chunks. If average < threshold (empirically: < 2.0 for `ms-marco-MiniLM-L-6-v2`), retrieval has likely failed. Return a structured `{"answer": "I don't know", "reason": "no_relevant_context", "top_scores": [...]}` rather than passing low-quality context to the LLM.
- Source contradiction handling: if two chunks from different sources make contradictory claims (detectable by semantic opposition — cosine sim < -0.1 between their embeddings, or explicit contradiction keywords), include both in context and instruct the LLM to note the disagreement: "Sources [1] and [3] disagree on this point: [1] states X while [3] states Y."

**Trap:** Passing all 8 chunks to the LLM without telling it which claim each chunk supports. The LLM will either hallucinate citations or ignore them entirely. The context must explicitly enumerate chunks as numbered references so the LLM can cite `[3]` meaningfully.

**Done when:** A query answerable from context produces an answer with at least one inline citation. A query not answerable from context (all cross-encoder scores < threshold) produces "I don't know" rather than a hallucinated answer. Verify with 5 unanswerable queries manually.

## Data Flow

```
Raw sources: PDFs + Web pages + CSVs + SQLite tables
    |
    | source-specific chunking (paragraph / <p> / row+header / key:value)
    v
Chunks: list[Chunk] with text, source_type, metadata, chunk_id
    |
    | model.encode(chunk.text for chunk in chunks)
    | faiss.normalize_L2()
    | per-source IndexFlatIP.add()
    v
4 FAISS indexes (one per source type)

Query at request time
    |
    | route_query(query) → selected_sources
    v
Selected indexes (1–4)
    |
    | for each selected index: index.search(query_vec, k=25)
    v
Up to 100 bi-encoder candidates (across sources)
    |
    | cross_encoder.predict([(query, chunk.text)] × 100)
    v
100 cross-encoder scores → sort descending → top-10
    |
    | confidence check: avg(top_3_scores) < threshold?
    |   yes → return "I don't know"
    |   no  → continue
    v
MMR selection: 8 chunks from top-10 (λ=0.7)
    |
    | format as numbered context with source metadata
    v
LLM generation with citation instructions
    |
    v
Answer with inline citations [1], [2], ... + source legend
```

## Edge Cases to Handle Before You Code

1. **PDF with tables and figures:** PyMuPDF extracts table cells as fragmented text with whitespace. A table row like "Q1 | $2.3M | 12% growth" becomes "Q1\n$2.3M\n12% growth" — semantically meaningless as a standalone embedding. Detect tables by checking for short line lengths and high `|` density; convert to Markdown table format or skip and note that the table exists at that page for SQL-side retrieval.

2. **Web pages with anti-scraping protections:** Many sites return JavaScript-rendered content or 403 on headless requests. Use `requests` with a realistic User-Agent; fall back to `selenium` for JavaScript-heavy sites; skip and log if 403. Never retry more than 3 times — you'll get IP-blocked. Rate-limit scraping to 1 request/2 seconds.

3. **SQL queries that return 10,000+ rows:** You cannot embed every row. Use database-side filtering: if the query mentions specific entities (a company name, a date range), translate that to a SQL WHERE clause before embedding. Only embed the filtered result set. A `text2sql` approach with a lightweight LLM converts the natural language query to SQL for structured pre-filtering.

4. **Source contradiction between high-confidence chunks:** Cross-encoder scores both chunks highly (scores 9.1 and 8.9), but they make opposite claims. Semantic similarity between their embeddings is -0.3. The LLM must present both perspectives. Instruct it explicitly: "If sources disagree, present all perspectives with citations rather than resolving the conflict." Resolving conflicts requires domain authority heuristics (peer-reviewed paper > blog post > forum comment) that the LLM cannot evaluate reliably.

5. **Context budget overflow when chunks are long:** 8 chunks × 500 tokens = 4,000 tokens, which may exceed the budget after adding system prompt and few-shot examples. Implement chunk truncation: trim each chunk to `budget_tokens / n_chunks` before passing to the LLM. Truncate from the end of the chunk — most important information is typically in the first sentences (for PDFs and news; not always for CSVs).

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|----------|----------|----------------|
| Query routing | LLM-as-router (high accuracy, 300ms overhead) | All-sources-always (perfect recall, 4× retrieval time) | LLM-as-router when latency budget is tight and source descriptions are well-defined; all-sources-always for research assistants where recall is paramount |
| Cross-source score normalization | Use cross-encoder raw scores (calibrated) | Normalize bi-encoder cosine sims per source | Always use cross-encoder — bi-encoder cosine sims are not comparable across heterogeneous text distributions |
| Contradiction handling | Present both perspectives with citations | Resolve by source authority (paper > blog) | Present both — authority heuristics are domain-specific and often wrong; let the user judge |
| Hallucination prevention | Score-threshold "I don't know" | Always answer, flag low confidence | Score-threshold for production — always answering trains users to trust unverifiable outputs |

## What Makes This Non-Trivial

The hardest part is not building any individual component — it is the system integration: ensuring that chunk IDs survive the full pipeline (chunking → embedding → FAISS → cross-encoder → MMR → generation) without getting misaligned. A single off-by-one in index remapping means citations point to the wrong source. Testing requires ground truth: for each test query, you must know which specific chunks should be cited. Building this ground truth by hand for 20 queries takes significant time but is non-negotiable for correctness verification.

Source contradiction handling is underspecified in most RAG tutorials. When a Wikipedia article says X and a research paper says Y, a naive RAG system picks the chunk with the higher cross-encoder score and presents it as fact. The correct behavior — detecting the contradiction, presenting both, deferring to the user — requires explicit detection logic and prompt engineering that most implementations skip.

## Interview Angles This Project Develops

1. "How does chunking strategy affect retrieval quality for structured data?" — Demonstrates that embedding a CSV row without column headers is semantically meaningless; key:value format gives the embedding model the context needed to map the row to the correct region of the embedding space.

2. "Why do you need a two-stage bi-encoder + cross-encoder pipeline rather than just cross-encoding everything?" — O(N) cross-encoder passes at query time: 10,000 chunks × 2ms = 20 seconds per query. Bi-encoder retrieves 100 candidates in < 50ms; cross-encoding 100 pairs takes ~200ms. Total: < 300ms.

3. "How do you handle the case where retrieval fails — all retrieved chunks are irrelevant?" — Confidence threshold on cross-encoder scores. If average top-3 score < 2.0, retrieval has failed. Return "I don't know" with the top retrieved chunks listed for transparency, rather than passing low-quality context to the LLM which will hallucinate.

4. "What is MMR and why do you need it for context selection?" — Maximal Marginal Relevance selects chunks that are relevant to the query AND not redundant with already-selected chunks. Without MMR, two near-duplicate paragraphs from the same PDF section consume 2 of your 8 context budget slots, displacing a more informative chunk from another source.

5. "How would you handle a query that requires synthesizing information from both a CSV (tabular) and a PDF (narrative)?" — Both are embedded in the same semantic space, so the cross-encoder can compare them on equal footing. The synthesis prompt instructs the LLM to combine statistical evidence from [CSV_chunk] with contextual explanation from [PDF_chunk], citing both. The key requirement is that the context format clearly labels each chunk's source type so the LLM knows which is the quantitative source and which is the narrative.

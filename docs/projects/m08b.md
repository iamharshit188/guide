# Multi-Source Research Assistant — Thinking Guide

> **Module:** 08 — RAG & Vector Search · **Difficulty:** Advanced · **Type:** THINKING · **Time:** ~5 hours

## The Problem

**Input:** A user research query (free-form natural language). Four heterogeneous source types are pre-indexed: (1) PDF research papers, (2) scraped web pages, (3) structured CSV files (tabular data), (4) a SQLite database with schema. Sources may contradict each other, have different freshness, and cover different domain depths.

**Output:** A synthesized answer grounded in retrieved evidence, with per-claim citations identifying the source type, source name, and row/section where evidence was found. Sources that contradict each other must be flagged explicitly rather than silently averaged.

**Constraints:** End-to-end query latency must be under 10 seconds for a production-quality user experience. A cross-encoder re-ranker is used after initial retrieval but cannot be applied to more than a few hundred candidates. Context window for the final LLM call is limited to 8 chunks (approximately 4K tokens for GPT-4-class models).

## Before You Write Any Code: Build The Mental Model

The fundamental architectural challenge is that heterogeneous sources require heterogeneous preprocessing — the same chunk_size=512 strategy that works well for PDFs produces terrible results for CSVs. A CSV row might be 30 tokens; combining 17 rows into one chunk destroys the row-column semantic structure that makes the CSV queryable. A web page `<p>` tag might be 5 tokens or 500 tokens — paragraph-level splitting is right for web but wrong for the surrounding navigation and boilerplate. The preprocessing layer must be source-type-aware, and this decision cascades into every downstream stage.

Query routing addresses a subtler problem: not all queries benefit from all source types. "What was the GDP of France in 2022?" is best answered from the database or a CSV — a RAG pipeline that retrieves prose paragraphs from PDFs will produce a vague answer. "What are the theoretical foundations of MMR?" is best answered from papers — routing it to the database produces nothing useful. Routing can be a classifier (trained on labeled query-source pairs), an LLM-as-router (flexible but adds 500ms latency per query), or a keyword heuristic (fast but brittle on ambiguous queries). The right choice depends on your latency budget and query diversity.

Cross-encoder re-ranking costs are prohibitive at scale and manageable at small scale. A bi-encoder (sentence-transformer) runs query and document independently, producing embeddings that can be pre-computed — query embedding is $O(d)$, retrieval is $O(d \times N)$ via HNSW. A cross-encoder concatenates query and document and runs them through a transformer together: it cannot pre-compute document representations, so every (query, document) pair requires a full forward pass. For 100 candidate chunks with 256-token average length and a query of 20 tokens, this is 100 forward passes of ~276 tokens each — feasible in ~2 seconds on a GPU, infeasible for 10,000 candidates.

## Decompose It

### Phase 1 — Source-Type-Aware Preprocessing

**The question this phase answers:** How do you convert each source type into (text, metadata) pairs that the embedding model can retrieve meaningfully?

**PDFs:** Extract text page-by-page with PyMuPDF. Split on paragraph boundaries (`\n\n`). Each chunk carries `{source_type: "pdf", filename, page, section_heading}`. The section heading is recoverable via PyMuPDF's `get_toc()` call — include it as a prefix in the chunk text because the LLM uses it for attribution ("According to Section 3.2 of paper.pdf...").

**Web pages:** Strip HTML with BeautifulSoup, extract `<p>` and `<h1>`–`<h6>` tags, discard `<nav>`, `<footer>`, `<script>`. Each `<p>` tag is a natural chunk boundary. Prepend the page `<title>` and the immediate `<h>` ancestor to every chunk for context: a `<p>` that says "This approach has three limitations" means nothing without the section header "Proposed Method." Carry `{source_type: "web", url, scraped_at, section_heading}`.

**CSVs:** Each row becomes a chunk formatted as `"Column1: value1 | Column2: value2 | ..."`. Prepend the column headers to every row-chunk so the embedding model understands the schema context. For wide tables (>10 columns), select only semantically relevant columns per row (use schema inspection to identify numeric vs. categorical). Carry `{source_type: "csv", filename, row_index, column_names}`.

**SQLite:** Convert rows to key-value strings: `"table_name: row — col1: val1, col2: val2, ..."`. Include the table name as a prefix because the same column name (e.g., `name`) appears in multiple tables and without context the embedding is ambiguous. For complex schemas, pre-generate a natural language description of each table as a schema summary chunk that helps routing. Carry `{source_type: "sqlite", table_name, row_id, db_path}`.

**Key decisions:**
- Format-matching for embedding: embeddings are trained on natural language, not structured data. Formatting CSV rows as `"Country: France | GDP: 2.78T | Year: 2022"` is dramatically better than raw `"France,2.78T,2022"` because the embedding model has seen "Country: France" in training but not bare comma-separated values.
- Whether to embed schema descriptions as separate chunks: yes — a schema chunk "The `countries` table has columns: name (text), gdp_usd_trillion (real), year (int)" allows the router to identify this source as relevant to economic queries even if no individual row matches the query embedding closely.

**Trap:** Chunking CSV data the same way as prose (character-level splitting at 512 tokens). A single chunk might contain the middle of row 4 and the start of row 5, losing the row boundary. Row-level chunking is the correct unit for tabular data; if rows exceed 512 tokens (wide tables), truncate by selecting the most informative columns.

**Done when:** Each source type produces a list of `(text, metadata)` dicts where metadata is type-specific but consistently includes `source_type` and a unique `doc_id`. A single ChromaDB collection stores all source types, identified by the `source_type` metadata field.

### Phase 2 — Query Routing

**The question this phase answers:** Which source types are worth searching for a given query, and how do you make this decision efficiently?

The three routing approaches, compared:

**Keyword heuristic:** Maintain a lookup from query keywords to source types. "how many", "average", "total", "rate" → database/CSV; "according to", "study", "paper", "published" → PDF; "as of today", "recently", "current" → web. Fast (< 1ms), no model required, but fails on paraphrases: "what is the GDP per capita" triggers numeric routing, but "describe France's economic output" does not.

**Classifier:** Train a small text classifier (logistic regression or DistilBERT fine-tuned) on labeled (query, source_type) pairs. Produces a probability distribution over source types; route to all types above a threshold (e.g., P > 0.2). Requires labeled training data — collect this by logging user queries and their ultimately useful sources. Accuracy ~85–90% on well-defined domains; degrades on cross-domain queries.

**LLM-as-router:** Send the query plus source descriptions to a small LLM (GPT-4o-mini) with a structured output request: `{"source_types": ["pdf", "csv"]}`. Flexible, handles novel query types, but adds 500–800ms latency. Only viable if total latency budget is > 5 seconds.

**Key decisions:**
- Single source vs. multi-source: for most queries, 1–2 source types are relevant. Searching all 4 types doubles retrieval latency and increases noise in the candidate set. Route aggressively; only fall back to multi-source when the primary source returns zero results above threshold.
- Fallback strategy: if routing selects only PDF but retrieval confidence is low (all cosine scores < 0.4), automatically re-route to all sources. This requires a confidence check before invoking the cross-encoder.

**Trap:** Routing to one source and never checking retrieval quality. A query "What caused the 2008 financial crisis?" routes to PDF (research papers), but if your PDF collection is exclusively about machine learning, all retrieved chunks are irrelevant. Without a retrieval confidence gate, the LLM receives irrelevant context and hallucinates rather than saying "I don't know."

**Done when:** A routing function takes a query string and returns a subset of `{"pdf", "web", "csv", "sqlite"}`. Latency of the routing decision is logged and under 100ms for heuristic/classifier approaches.

### Phase 3 — Cross-Encoder Re-Ranking

**The question this phase answers:** After bi-encoder retrieval gives you N candidates from each source type, how do you select the best 8 for the LLM's context window?

The two-stage pipeline: (1) bi-encoder retrieves top-20 candidates per routed source type — up to 80 candidates total. (2) Cross-encoder scores each (query, candidate) pair and selects the global top-8.

A cross-encoder like `cross-encoder/ms-marco-MiniLM-L-6-v2` produces a scalar relevance score for each pair. The model is fine-tuned on the MS-MARCO passage ranking dataset, which represents information-seeking queries well. Load with `CrossEncoder` from `sentence-transformers`:

```python
from sentence_transformers import CrossEncoder
model = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2", max_length=512)
scores = model.predict([(query, doc) for doc in candidates])
```

Complexity is $O(k \times L_q \times L_d / 64)$ transformer forward passes where 64 is a typical sequence parallelism factor. For 80 candidates of 200 tokens each and a 20-token query, this is 80 passes of 220 tokens — approximately 1.5 seconds on CPU, 0.2 seconds on GPU with batching.

**Key decisions:**
- How many candidates to retrieve per source before cross-encoding: retrieve more (top-50 per source) for higher recall at the cost of more cross-encoder time; retrieve fewer (top-10) for faster re-ranking with risk of missing the best chunk. `top-20` per routed source is a practical default.
- Whether to cross-encode all candidates or only those above a bi-encoder threshold: filter to bi-encoder score > 0.3 before cross-encoding to reduce cross-encoder workload on obviously irrelevant chunks.

**Trap:** Cross-encoding raw structured data (CSV rows formatted as `"Country: France | GDP: 2.78T"`) against a prose query. The cross-encoder is trained on prose pairs; structured data formatting breaks its relevance model. Apply light verbalization: convert the row to `"France had a GDP of 2.78 trillion USD in 2022"` before cross-encoding (keep the original for citation, use verbalized for scoring).

**Done when:** `rerank(query, candidates_list)` returns a list sorted by cross-encoder score, with the top-8 selected. The re-ranking latency is logged. Running on 80 candidates takes under 3 seconds on CPU.

### Phase 4 — Contradiction Detection and Context Budget Management

**The question this phase answers:** When selected chunks disagree, how do you surface the contradiction rather than silently averaging it? And how do you fit 8 chunks into a finite context window?

Contradiction detection: after cross-encoder re-ranking, check for factual conflicts among the top-8 chunks. A lightweight approach: for any numeric claim in the retrieved chunks (years, percentages, named values), extract all instances of the same entity type and check if they differ. A heavier approach: use a Natural Language Inference (NLI) model to check if any two chunks are in `contradiction` relation. The `cross-encoder/nli-deberta-v3-small` model outputs `(entailment, neutral, contradiction)` for pairs — apply it to all $\binom{8}{2} = 28$ pairs.

When contradiction is detected, the synthesis prompt must explicitly instruct the LLM to report the disagreement: "Sources [2] and [5] disagree on X. Source [2] (PDF, 2023) states Y; Source [5] (web, 2024) states Z. Please report this discrepancy rather than choosing one."

Context budget management: each of the 8 chunks may vary in length (100–512 tokens). The context window budget is approximately 4000 tokens (leaving room for system prompt, citations, and LLM output). Strategy: sort the 8 chunks by re-ranking score, then greedily include chunks until the budget is exhausted. This ensures the most relevant chunks are always included.

**Key decisions:**
- How to handle inter-source disagreement without NLI: majority vote (whichever claim appears in more chunks wins) is simple but incorrect when one source is more authoritative. Recency-weighted vote (newer source wins) is a reasonable heuristic for factual claims. Domain authority (peer-reviewed PDF > scrapped web > CSV) is appropriate for technical claims.
- Whether to include contradictory sources or exclude one: always include both and flag the contradiction. Excluding either one hides information the user should have.

**Trap:** Including all retrieved chunks regardless of budget. A 7000-token context that exceeds GPT-4o's 8K window causes a truncation error or a silent context cutoff, losing the most recent (and typically most relevant) chunks at the end of the context block. Always count tokens before sending.

**Done when:** The synthesis prompt includes at most 8 chunks fitting within the token budget. Contradictions between chunks are explicitly flagged as `[CONFLICT: sources [2] and [5] disagree]` in the context block.

### Phase 5 — Synthesis and "I Don't Know" Detection

**The question this phase answers:** How does the system detect when retrieval has failed and the LLM should decline to answer?

"I don't know" detection has two layers: (1) retrieval confidence check before calling the LLM; (2) LLM instruction to explicitly decline when context is insufficient.

Retrieval confidence: after cross-encoder re-ranking, if the highest-scoring chunk has a score below a threshold (e.g., 0.2 on a 0–1 scale for ms-marco cross-encoder), retrieval has likely failed. Log this and either return "The available sources do not contain relevant information on this topic" without calling the LLM, or call the LLM with a warning: "The retrieved context may not be relevant. Only answer if confident the context addresses the question."

LLM instruction: explicitly include in the system prompt: "If the context does not contain sufficient information to answer the question, respond with exactly: 'INSUFFICIENT_CONTEXT: [brief description of what is missing]'." Parse this response to show a structured "not found" message to the user, rather than a hallucinated answer.

**Done when:** Querying for a topic not covered by any indexed document returns a clean "insufficient context" message, not a hallucinated answer. Querying for a covered topic returns an answer with citations. The system's refusal rate on out-of-domain queries is > 80% (test with 10 deliberately out-of-domain queries).

```
                   │  40 total candidates
                   ▼
    Bi-encoder threshold filter (score > 0.3)
                   │  N ≤ 40 candidates
                   ▼
    Cross-encoder re-ranking (ms-marco-MiniLM)
                   │  scored (query, candidate) pairs
                   ▼
    Select top-8 by score (token budget aware)
                   │
                   ▼
    NLI contradiction check (deberta-nli, 28 pairs)
                   │  flag contradictions
                   ▼
    Build context block (numbered chunks + conflict flags)
                   │
                   ▼

                   │
                   ▼

                   │
                   ▼
    Structured response: {answer, citations, conflicts, source_types_used}
```

## Edge Cases to Handle Before You Code

1. **Query returns zero results from all routed sources:** Cross-encoder receives an empty list. The system must detect this before calling cross-encoder (check if `candidates_list` is empty) and return the "insufficient context" response immediately. Passing an empty list to `CrossEncoder.predict()` raises a shape error.

2. **CSV file with numeric-only columns:** The column name "2019" is ambiguous — it could be a year, an index, or a count. When formatting rows, use the column name literally: `"2019: 4.2"` tells the embedding model nothing. Inspect column names programmatically: if a column name parses as an integer, prefix it with the table name: `"gdp_data.2019: 4.2"`. This is a small fix with large impact on retrieval quality for tabular data.

3. **Web page with boilerplate dominating chunk content:** Navigation menus, footers, and cookie consent banners can produce 200-token chunks that embed similarly to any web page, polluting retrieval. After BeautifulSoup extraction, compute the ratio of unique words to total words — boilerplate has low uniqueness (every page has the same navigation). Filter chunks with uniqueness ratio < 0.4.

4. **SQLite database with 1M rows:** Row-level chunking produces 1M chunks — too many to embed without significant time and cost. Apply a pre-filter: for retrieval, generate SQL queries from the user question using an LLM (Text-to-SQL) and execute them, returning only the matched rows as chunks. This is a hybrid retrieval mode: vector search for unstructured sources, SQL for structured sources.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|---------|---------|----------------|
| Query routing method | Keyword heuristic (fast, brittle) | LLM-as-router (accurate, slow) | Heuristic when latency < 3s and query domain is well-defined; LLM-router when queries are diverse and latency budget allows |
| Cross-encoder scope | Re-rank all retrieved candidates (up to 80) | Re-rank only those above bi-encoder threshold | Apply threshold when cross-encoder latency exceeds 2s; accept slight recall loss for the speed gain |
| Contradiction resolution | Surface both and let user decide | Apply authority heuristic (recency, domain) | Surface both always for research use cases; apply heuristic for production consumer applications where "I don't know who's right" is unhelpful |
| "I don't know" threshold | Fixed score threshold (0.2) | Calibrated per-query (percentile of scores) | Fixed threshold for simplicity; calibrated percentile when you have diverse query types with very different typical score distributions |

## What Makes This Non-Trivial

The routing and retrieval layers are tightly coupled in a way that's easy to miss: if routing incorrectly excludes a source type, no amount of re-ranking quality can recover the missing chunks. The system must have a confidence-aware fallback that widens the search when primary retrieval confidence is low — otherwise hard routing decisions compound retrieval failures.

Cross-encoder re-ranking on heterogeneous data exposes the model's implicit assumptions about text format. A cross-encoder trained on (query, passage) pairs from web text scores PDFs well, web chunks reasonably, and CSV/SQL chunks poorly because the format is unlike anything in its training set. The verbalization step (converting structured rows to prose) is not optional for cross-encoder quality; it is the mechanism that bridges the format gap.

The context budget decision — which 8 chunks to include when you have 40 scored candidates — is implicitly a decision about what the LLM reads. Including the 8 highest-scoring chunks maximizes expected relevance, but if chunks 1–5 are all from the same PDF section and chunks 6–8 are from other sources, MMR-style diversity on top of cross-encoder scores provides better coverage.

## Interview Angles This Project Develops

1. **"How do you handle the cold-start problem when you have no labeled data for the query router?"** — Strong answer: start with the keyword heuristic for day 1. Log every query and which source types were ultimately useful (inferred from which citations the user clicked or which chunks the LLM cited). After 500 queries you have labeled training data; train a classifier to replace the heuristic. The heuristic bootstraps the system until you have data.

2. **"Why use a cross-encoder for re-ranking rather than a better bi-encoder?"** — Strong answer: bi-encoders encode query and document independently, compressing each into a fixed-size vector. Cross-encoders see the query-document pair jointly and can model token-level interactions: "the 2022 GDP figure" in the query interacting with "GDP: 2.78T (2022)" in the document. Joint encoding captures this fine-grained matching; bi-encoders cannot because the embeddings are computed in isolation. The cost is that cross-encoders cannot pre-compute document representations — they're only viable as a re-ranker over a small candidate set.

3. **"How would you evaluate this system's quality without human annotations?"** — Strong answer: build a synthetic evaluation set by generating (question, expected_source_type) pairs from your indexed data using an LLM. For retrieval quality, check if the expected source appears in the top-8. For answer quality, check citation coverage (does the answer cite the source that actually contains the answer?) and use RAGAS faithfulness as a proxy for hallucination rate.

4. **"What happens when sources are updated — how do you keep the index fresh?"** — Strong answer: implement change detection per source type: for PDFs, compare file hash to the stored hash in metadata; for web pages, re-scrape on a schedule and compare content hash; for CSVs, compare row count and modification timestamp; for SQLite, compare max(rowid) or a change log table. Only re-embed chunks whose source content changed. This keeps re-indexing incremental and fast.

5. **"How does the two-stage retrieval pipeline affect overall system recall vs. precision?"** — Strong answer: the bi-encoder provides high recall at the cost of precision (retrieves many candidates, some irrelevant). The cross-encoder improves precision (selects only the genuinely relevant ones) at the cost of latency. The total recall of the system is bounded by the bi-encoder's recall — if the bi-encoder misses the right chunk at stage 1, no re-ranking recovers it. This is why `n_candidates` at stage 1 must be set conservatively high (top-20–50), accepting some cross-encoder cost to preserve recall.

# A/B Testing Deployment API — Step-by-Step Guide
> **Module:** 11 · **Difficulty:** Advanced · **Type:** GUIDED · **Time:** ~3 hours

## What You'll Build

A Flask API that routes ML inference requests between two model variants using weighted random sampling. An in-memory, thread-safe metrics store tracks per-variant request counts, latency distributions, and error rates. A `/metrics` endpoint runs a Mann-Whitney U test to determine whether latency differences between variants are statistically significant. A `/adjust` endpoint changes traffic weights at runtime without a restart.

## The Mental Model

A/B testing in production is not about randomly splitting traffic — it's about maintaining two invariants simultaneously: **enough requests per variant to achieve statistical power**, and **stable enough traffic that model behavior is observable and not confounded by request-mix drift**. The weighted router is stateless on a per-request basis but the metrics store accumulates state over time. Mann-Whitney U is chosen over a t-test because it makes no normality assumption on latency distributions, which are always right-skewed in practice. Statistical significance ($p < 0.05$) is a necessary but not sufficient condition to act — sample size and effect size (difference in medians) both matter. The `/adjust` endpoint is how an ops engineer responds to a signal without redeploying.

## Phase 1 — Model Variant Loading with Version Metadata

**Why this first:** The router needs to know which models exist before it can route anything. Defining the model interface up front forces all downstream code to treat both variants uniformly.

```python
# app.py
import time
import random
import threading
from flask import Flask, request, jsonify
from scipy import stats
import numpy as np

app = Flask(__name__)

# ---------------------------------------------------------------------------
# Simulated model variants
# In production these would be loaded from disk (ONNX, TorchScript, etc.)
# ---------------------------------------------------------------------------

class ModelVariant:
    """Wraps an inference callable with version metadata."""

    def __init__(self, name: str, version: str, base_latency_ms: float, error_rate: float):
        self.name = name
        self.version = version
        self._base_latency_ms = base_latency_ms
        self._error_rate = error_rate

    def predict(self, payload: dict) -> dict:
        """
        Simulates inference. Returns prediction or raises RuntimeError.
        In a real system: run self.session.run(...) or model(input_tensor).
        """
        # Simulate variable latency: base + exponential jitter (realistic tail)
        latency_ms = self._base_latency_ms + random.expovariate(1.0 / 20.0)
        time.sleep(latency_ms / 1000.0)

        # Simulate random errors
        if random.random() < self._error_rate:
            raise RuntimeError(f"{self.name} inference failed (simulated)")

        return {
            "model": self.name,
            "version": self.version,
            "score": round(random.uniform(0.0, 1.0), 4),
            "latency_ms": round(latency_ms, 2),
        }

    def metadata(self) -> dict:
        return {"name": self.name, "version": self.version}


# Instantiate both variants
MODEL_V1 = ModelVariant(
    name="model_v1",
    version="1.0.3",
    base_latency_ms=45.0,   # stable baseline
    error_rate=0.02,
)

MODEL_V2 = ModelVariant(
    name="model_v2",
    version="2.0.0-beta",
    base_latency_ms=38.0,   # faster but under test
    error_rate=0.05,
)

VARIANTS = {
    "model_v1": MODEL_V1,
    "model_v2": MODEL_V2,
}
```

**Checkpoint:** `VARIANTS` contains two `ModelVariant` objects. Calling `MODEL_V1.predict({"text": "hello"})` returns a dict with `model`, `version`, `score`, `latency_ms`. `MODEL_V2.predict({})` may raise `RuntimeError` ~5% of the time.

## Phase 2 — Weighted Router Function

**Why this first:** The router must be pure and side-effect-free. Decoupling it from the metrics store means you can unit-test routing distribution in isolation.

```python
# ---------------------------------------------------------------------------
# Traffic weights — mutable at runtime via /adjust
# Format: {"model_v1": 80, "model_v2": 20}  (must sum to 100)
# ---------------------------------------------------------------------------
traffic_weights = {"model_v1": 80, "model_v2": 20}
weights_lock = threading.Lock()   # guards traffic_weights reads/writes


def select_variant() -> ModelVariant:
    """
    Weighted random selection over VARIANTS.
    Uses reservoir-style sampling: draw a uniform in [0, total_weight),
    then walk the weight table to find which bucket the draw falls in.

    This is equivalent to random.choices() but works with mutable weights
    and avoids rebuilding the population list on every call.
    """
    with weights_lock:
        names = list(traffic_weights.keys())
        weights = [traffic_weights[n] for n in names]

    total = sum(weights)
    draw = random.uniform(0, total)
    cumulative = 0.0
    for name, w in zip(names, weights):
        cumulative += w
        if draw < cumulative:
            return VARIANTS[name]
    # Fallback (floating-point edge case)
    return VARIANTS[names[-1]]
```

**Checkpoint:** Run `from collections import Counter; Counter(select_variant().name for _ in range(10000))` — should show ~80% `model_v1`, ~20% `model_v2` (within ±2% at 10k samples).

## Phase 3 — Thread-Safe Metrics Collector

**Why this first:** Metrics must be collected before the inference endpoint goes live, because the inference endpoint will write to the store on every request.

```python
# ---------------------------------------------------------------------------
# Metrics store — one entry per variant
# ---------------------------------------------------------------------------

class VariantMetrics:
    """All mutable state for one model variant. Protected by a single lock."""

    def __init__(self, name: str):
        self.name = name
        self._lock = threading.Lock()
        self.request_count: int = 0
        self.error_count: int = 0
        self.latencies_ms: list[float] = []   # full distribution for U-test

    def record_success(self, latency_ms: float) -> None:
        with self._lock:
            self.request_count += 1
            self.latencies_ms.append(latency_ms)

    def record_error(self) -> None:
        with self._lock:
            self.request_count += 1
            self.error_count += 1

    def snapshot(self) -> dict:
        """Returns a consistent read of all metrics."""
        with self._lock:
            n = self.request_count
            e = self.error_count
            lats = list(self.latencies_ms)   # copy before releasing lock

        if lats:
            mean_lat = float(np.mean(lats))
            p99_lat = float(np.percentile(lats, 99))
        else:
            mean_lat = 0.0
            p99_lat = 0.0

        return {
            "name": self.name,
            "request_count": n,
            "error_count": e,
            "error_rate": round(e / n, 4) if n > 0 else 0.0,
            "mean_latency_ms": round(mean_lat, 2),
            "p99_latency_ms": round(p99_lat, 2),
            "sample_size": len(lats),
        }

    def latencies(self) -> list[float]:
        with self._lock:
            return list(self.latencies_ms)


metrics_store = {
    "model_v1": VariantMetrics("model_v1"),
    "model_v2": VariantMetrics("model_v2"),
}
```

**Checkpoint:** `metrics_store["model_v1"].record_success(42.1)` followed by `metrics_store["model_v1"].snapshot()` returns `request_count=1, mean_latency_ms=42.1`. The lock prevents data races when Flask's threaded server handles concurrent requests.

## Phase 4 — Inference Endpoint with Per-Variant Timing

**Why this order:** The POST `/predict` endpoint ties together Phase 2 (routing) and Phase 3 (metrics). Timing wraps the model's own reported latency with wall-clock time, catching overhead the model doesn't see (serialisation, queue wait).

```python
@app.route("/predict", methods=["POST"])
def predict():
    """
    Selects a variant, runs inference, records metrics.
    Returns the prediction or an error dict with HTTP 500.

    Thread safety: select_variant() holds weights_lock briefly;
    record_success/record_error hold the per-variant lock briefly.
    The inference call itself is outside any lock — it can block
    for tens of milliseconds without starving other threads.
    """
    payload = request.get_json(force=True, silent=True) or {}
    variant = select_variant()
    m = metrics_store[variant.name]

    wall_start = time.perf_counter()
    try:
        result = variant.predict(payload)
        wall_latency_ms = (time.perf_counter() - wall_start) * 1000.0
        m.record_success(wall_latency_ms)
        return jsonify({
            "status": "ok",
            "prediction": result,
            "wall_latency_ms": round(wall_latency_ms, 2),
        }), 200

    except RuntimeError as exc:
        m.record_error()
        return jsonify({
            "status": "error",
            "variant": variant.name,
            "detail": str(exc),
        }), 500
```

**Checkpoint:** Start the Flask app with `flask run --threaded`. Send `curl -X POST http://localhost:5000/predict -H "Content-Type: application/json" -d '{}'`. Response contains `status: ok`, `prediction.model` is either `model_v1` or `model_v2`. After 100 requests, ~80 should have come from `model_v1`.

## Phase 5 — /metrics with Mann-Whitney U Test

**Why Mann-Whitney U:** The null hypothesis is that the two latency distributions are identical. U is valid for any continuous distribution (no normality required), and latency is always skewed right due to GC pauses and tail events.

$$U = \sum_{i \in A} \sum_{j \in B} \mathbf{1}[x_i > x_j]$$

where $\mathbf{1}[\cdot]$ is the indicator function. $p < 0.05$ means the observed difference is unlikely under the null.

```python
MIN_SAMPLES_FOR_TEST = 30   # below this, U-test p-value is unreliable

@app.route("/metrics", methods=["GET"])
def get_metrics():
    """
    Returns per-variant stats and, if both variants have >= MIN_SAMPLES_FOR_TEST
    latency observations, runs Mann-Whitney U to test whether their latency
    distributions differ significantly.
    """
    snapshots = {name: m.snapshot() for name, m in metrics_store.items()}
    lats_v1 = metrics_store["model_v1"].latencies()
    lats_v2 = metrics_store["model_v2"].latencies()

    stat_test = None
    if len(lats_v1) >= MIN_SAMPLES_FOR_TEST and len(lats_v2) >= MIN_SAMPLES_FOR_TEST:
        u_stat, p_value = stats.mannwhitneyu(lats_v1, lats_v2, alternative="two-sided")
        significant = bool(p_value < 0.05)

        median_v1 = float(np.median(lats_v1))
        median_v2 = float(np.median(lats_v2))

        stat_test = {
            "test": "Mann-Whitney U",
            "u_statistic": round(float(u_stat), 2),
            "p_value": round(float(p_value), 6),
            "significant_at_0.05": significant,
            "median_latency_ms": {
                "model_v1": round(median_v1, 2),
                "model_v2": round(median_v2, 2),
            },
            "interpretation": (
                "Latency distributions differ significantly — consider acting on this signal."
                if significant
                else "No significant latency difference detected at current sample size."
            ),
        }
    else:
        n1, n2 = len(lats_v1), len(lats_v2)
        stat_test = {
            "test": "Mann-Whitney U",
            "status": "insufficient_data",
            "required_per_variant": MIN_SAMPLES_FOR_TEST,
            "current": {"model_v1": n1, "model_v2": n2},
            "remaining": {
                "model_v1": max(0, MIN_SAMPLES_FOR_TEST - n1),
                "model_v2": max(0, MIN_SAMPLES_FOR_TEST - n2),
            },
        }

    with weights_lock:
        current_weights = dict(traffic_weights)

    return jsonify({
        "variants": snapshots,
        "traffic_weights": current_weights,
        "statistical_test": stat_test,
    }), 200
```

**Checkpoint:** After 50+ requests, `GET /metrics` returns `statistical_test.significant_at_0.05: true` (because v2 has a genuinely lower base latency in the simulation). The `p_value` should be well below 0.05 at 100+ samples.

## Phase 6 — /adjust for Live Traffic Shift

**Why this last:** Runtime adjustment is a control-plane operation — it changes how the inference plane (Phase 4) behaves. It must validate the new weights before applying them to avoid routing all traffic to a non-existent variant.

```python
@app.route("/adjust", methods=["POST"])
def adjust_traffic():
    """
    Changes traffic weights at runtime.
    Body: {"model_v1": 50, "model_v2": 50}
    Rules:
      - All keys must correspond to known variants.
      - All values must be non-negative integers.
      - Values need not sum to 100 (the router normalises them),
        but they must not all be zero.
    """
    body = request.get_json(force=True, silent=True) or {}

    # Validate keys
    unknown = set(body.keys()) - set(VARIANTS.keys())
    if unknown:
        return jsonify({"error": f"Unknown variants: {unknown}"}), 400

    # Validate values
    try:
        new_weights = {k: int(v) for k, v in body.items()}
    except (TypeError, ValueError):
        return jsonify({"error": "All weight values must be integers"}), 400

    if any(v < 0 for v in new_weights.values()):
        return jsonify({"error": "All weights must be >= 0"}), 400

    if sum(new_weights.values()) == 0:
        return jsonify({"error": "At least one weight must be > 0"}), 400

    # Fill missing variants with 0 (not in body → no traffic)
    for name in VARIANTS:
        new_weights.setdefault(name, 0)

    with weights_lock:
        traffic_weights.update(new_weights)

    return jsonify({
        "status": "updated",
        "traffic_weights": new_weights,
        "note": "Router normalises weights — no requirement to sum to 100.",
    }), 200


# ---------------------------------------------------------------------------
# Entry point
# ---------------------------------------------------------------------------

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000, threaded=True, debug=False)
```

**Checkpoint:** `curl -X POST http://localhost:5000/adjust -H "Content-Type: application/json" -d '{"model_v1": 0, "model_v2": 100}'` — subsequent `/predict` calls should return only `model_v2`. Then `GET /metrics` shows `traffic_weights: {model_v1: 0, model_v2: 100}`.

## Debugging Guide

| Symptom | Likely Cause | Fix |
|---|---|---|
| `p_value` is always 1.0 | Both variants have identical latency | Widen the `base_latency_ms` gap between variants in the simulator |
| `significant_at_0.05: false` with 50 samples | Low effect size — distributions overlap heavily | Increase sample size; U-test power is proportional to $\sqrt{N}$ |
| `500` errors even at `error_rate=0.0` | Payload parsing failure | Ensure `Content-Type: application/json` header is set on POST |
| Metrics show only one variant | `select_variant()` always picks same bucket | Verify `traffic_weights` are not both zero; add logging to `select_variant` |
| Race condition: `error_rate` appears negative | `error_count` or `request_count` read outside lock | Ensure all reads use `snapshot()` — never access fields directly |

## What This Project Teaches You to Say in an Interview

1. "I use Mann-Whitney U over Student's t because production latency distributions are right-skewed — t-test requires normality, which latency violates almost universally."
2. "The metrics store uses a per-variant `threading.Lock` so concurrent Flask threads can record observations without blocking each other across variants."
3. "The inference call is outside any lock — a 50ms blocking call inside a lock would serialize all requests to that variant."
4. "Statistical significance at $p < 0.05$ is insufficient to ship v2 — you also need effect size (median difference) and enough traffic to rule out novelty effects."
5. "The `/adjust` endpoint enables gradual rollout: start at 1%, observe error rate, bump to 10% — exactly the canary ramp pattern, implemented as a control-plane API call rather than a redeployment."

# Real-Time Voice Assistant — Thinking Guide

> **Module:** 14 — Frontend & Full-Stack · **Difficulty:** Expert · **Type:** THINKING · **Time:** ~8 hours

## The Problem

**Input:** Browser microphone audio captured via `MediaRecorder`. User speaks a natural language request. The system must transcribe it, generate a response, and speak the response back — with full-duplex behavior (user can interrupt while assistant is speaking).

**Output:** Spoken audio response played in the browser. The pipeline: audio capture → backend transcription (Whisper) → LLM streaming (sentence-by-sentence) → TTS per sentence → audio playback in sequence. Visual state: speaking indicator shows when the assistant is talking; recording indicator shows when mic is active.

**Constraints:** Must run in a browser (no native app). Mic access requires user permission (one-time) and HTTPS (except localhost). Maximum perceived latency target: first audio from TTS within 2 seconds of the user finishing speaking. Interruption must be clean — no audio artifacts, no orphaned LLM requests.

## Before You Write Any Code: Build The Mental Model

The pipeline has five sequential stages, each with its own latency budget and failure mode. Visualize the path of a single user utterance:

1. `MediaRecorder` captures WebM/Opus chunks until silence is detected (VAD)
2. Browser sends audio blob to Flask via HTTP POST
3. Flask converts to WAV, runs Whisper, returns transcript (~800ms)
4. Flask receives transcript, calls LLM, streams response token by token
5. Browser accumulates tokens until a sentence boundary, sends to TTS
6. TTS returns audio, browser plays it, then plays next sentence

The perceived latency the user experiences is dominated by step 3 (Whisper) and the first chunk of step 5 (TTS for the first sentence). Everything after that is pipelined — while the user hears sentence 1, the LLM is generating sentence 2, and TTS is processing sentence 2. The architecture's job is to eliminate any waiting at step 4: do not wait for the full LLM response before starting TTS.

Sentence-boundary streaming is the core latency optimization. A typical LLM response is 3–5 sentences. If you wait for the complete response before TTS, perceived latency = Whisper + full LLM generation + TTS. If you TTS sentence-by-sentence, perceived latency = Whisper + first-sentence LLM generation + first-sentence TTS — roughly 3× lower. The sentence boundary detection is simple: accumulate tokens until you see `. `, `? `, `! `, or `\n`. The risk: if the LLM generates a very long sentence, the first TTS call is delayed.

Full-duplex is harder than it sounds. If the user speaks while the assistant is talking, you must simultaneously: cancel the current `Audio` object's playback, abort any pending LLM SSE stream, cancel any pending TTS requests, restart the recording pipeline. This requires clean resource management — not just `audio.pause()` but `audio.src = ''` to release the audio object, and `abortController.abort()` on the active fetch streams.

## Decompose It

### Phase 1 — Audio Capture with VAD

**The question this phase answers:** When does the user finish speaking, and how do you reliably detect silence without cutting them off?

**Key decisions:**

- **`MediaRecorder` API basics.** `navigator.mediaDevices.getUserMedia({ audio: true })` returns a `MediaStream`. Wrap it in `new MediaRecorder(stream, { mimeType: 'audio/webm;codecs=opus' })`. Call `recorder.start(100)` — the argument is the timeslice in ms, which fires `ondataavailable` every 100ms with a chunk. Accumulate chunks in an array. When silence is detected, call `recorder.stop()`, which fires `onstop`, at which point you construct a `Blob` from all chunks.

- **Voice Activity Detection (VAD) via AudioWorklet.** Connect the stream to an `AudioContext` → `AnalyserNode`. On each animation frame (or at a polling interval), call `analyser.getByteFrequencyData(dataArray)` and compute RMS: $\text{RMS} = \sqrt{\frac{1}{N}\sum_i x_i^2}$. If RMS drops below a threshold (e.g., -40dB) for more than 1500ms continuously, fire the "silence detected" event and stop recording.

- **Threshold calibration.** The -40dB threshold is environment-dependent. A quiet room might have background RMS at -60dB; a noisy office at -30dB. Implement adaptive thresholding: measure ambient RMS for the first 500ms of each recording session (before the user speaks), then set threshold = ambient + 15dB. This adapts to the user's environment rather than requiring manual calibration.

- **Minimum utterance length.** If the user accidentally triggers recording (a cough, background noise), don't send 200ms of audio to Whisper. Gate on minimum duration: only process if total recording > 400ms. Below that, discard and restart listening.

- **Maximum utterance length.** If the user talks continuously for > 60 seconds without a silence event, force-stop the recording and process what you have. Prevents runaway recording sessions and avoids Whisper's context window limit.

**Trap:** Using a fixed silence threshold in dB without accounting for the browser's automatic gain control (AGC). Most browsers apply AGC to microphone input, which means the raw audio level is normalized — a quiet room and a loud room may produce the same output amplitude. VAD based on absolute dB level will fail with AGC. Disable AGC: `getUserMedia({ audio: { noiseSuppression: false, autoGainControl: false, echoCancellation: false } })` for more predictable behavior, but note that this gives the user a noisier experience.

**Done when:** A user saying "hello" followed by 1.5 seconds of silence produces a `Blob` of type `audio/webm` with duration > 400ms. A cough or background noise burst < 400ms is discarded. The UI shows a recording indicator (animated microphone icon) while the `MediaRecorder` is active.

### Phase 2 — Audio Format Conversion and Whisper Transcription

**The question this phase answers:** Why can't Whisper receive WebM directly, and what is the most reliable conversion strategy?

**Key decisions:**

- **The format mismatch.** `MediaRecorder` in Chrome produces `audio/webm;codecs=opus`. Firefox produces `audio/ogg;codecs=opus`. Safari produces `audio/mp4`. Whisper (OpenAI API) accepts: `mp3`, `mp4`, `mpeg`, `mpga`, `m4a`, `wav`, `webm`. WebM is technically on the list, but Whisper's WebM support is inconsistent across versions — it fails silently on some codec configurations. WAV is universally supported, unambiguous, and easy to produce with ffmpeg.

- **Server-side conversion.** Convert on the Flask server using `pydub` (wraps ffmpeg): `AudioSegment.from_file(io.BytesIO(audio_bytes), format="webm").export(wav_buffer, format="wav")`. This requires ffmpeg installed on the server (`apt install ffmpeg` or `brew install ffmpeg`). Pydub adds ~50ms latency for a 5-second clip — negligible.

- **Why not convert on the client.** The Web Audio API can decode audio (via `AudioContext.decodeAudioData`), but cannot re-encode to WAV natively — you'd need a WASM-compiled ffmpeg, adding 3–5MB to the page bundle and significant client-side complexity. Server conversion is the pragmatic choice.

- **Whisper model selection.** OpenAI's Whisper API uses `whisper-1` (medium English model). For local Whisper: `whisper-tiny` runs in ~200ms; `whisper-base` in ~600ms; `whisper-medium` in ~3s on CPU. For a real-time application, `whisper-tiny` or `whisper-base` is the maximum feasible choice on CPU. On GPU: `whisper-large-v3` in ~1s. Report transcription latency in the response so the frontend can display it.

- **Language detection.** Whisper auto-detects language. If your application is English-only, pass `language="en"` to skip the language detection step (~100ms faster). If multilingual, omit it and display the detected language in the UI.

**Trap:** Sending the raw WebM blob as `multipart/form-data` and forgetting the filename extension. Whisper's API identifies format from the filename extension, not the MIME type. If you send the blob without a filename, Whisper may reject it or misidentify the format. Always set the filename: `files={"file": ("audio.webm", audio_bytes, "audio/webm")}` — or after server-side conversion, `("audio.wav", wav_bytes, "audio/wav")`.

**Done when:** A 3-second WebM recording arrives at Flask, is converted to WAV, transcribed by Whisper, and the transcript is returned in < 1.5s. Verify with a known utterance ("What is the capital of France?") and check Whisper's exact output — it should be clean, no hallucinations for clear speech.

### Phase 3 — Streaming LLM → Sentence-by-Sentence TTS

**The question this phase answers:** How do you minimize time-to-first-audio without waiting for the full LLM response?

**Key decisions:**

- **SSE (Server-Sent Events) for LLM streaming.** Flask endpoint `GET /stream?transcript=...` uses `response_class=Response` with `mimetype='text/event-stream'`. The generator function calls `openai.chat.completions.create(..., stream=True)` and yields each chunk as `data: {token}\n\n`. Browser receives tokens as they arrive — no waiting for the full response.

- **Sentence accumulation in the browser.** The browser's `EventSource` handler accumulates tokens in a buffer. On each token, check if the buffer ends with a sentence boundary: `/[.!?]\s$/`. When a boundary is detected: (1) extract the complete sentence, (2) clear the sentence from the buffer (keep any partial next sentence), (3) enqueue the sentence for TTS. This is a streaming state machine — the buffer is the state.

- **TTS request queuing.** Do not send all sentence-TTS requests simultaneously. Maintain a queue: send sentence 1 to TTS, await its audio, play it, then send sentence 2 to TTS. This ensures: (a) audio plays in order, (b) only one TTS request is in-flight at a time (reducing API rate limit risk), (c) if the user interrupts during sentence 1 playback, sentence 2's TTS request has not been sent yet (easy to cancel).

- **TTS provider choice.** OpenAI TTS (`tts-1`): ~500ms latency, $0.015/1K chars. Google Cloud TTS: ~300ms, more voices. Browser's built-in `SpeechSynthesisUtterance` (`window.speechSynthesis`): zero latency, zero cost, but voice quality varies by OS and has a 200-char bug in some browsers (long utterances silently stop). For a demo: use `window.speechSynthesis`. For production: OpenAI TTS.

- **WebSocket vs. SSE for streaming.** SSE is unidirectional (server → client), HTTP/1.1 compatible, and auto-reconnects. WebSocket is bidirectional, more setup, requires a different server. For this pipeline, the streaming direction is always server → client (LLM tokens). SSE is the correct choice. Use WebSocket only if the client needs to send mid-stream interrupts as structured messages — but in this design, interrupts go via a separate HTTP DELETE/PATCH request, keeping SSE unidirectional.

**Trap:** Treating the sentence-boundary detector as infallible. LLMs produce abbreviations ("Dr. Smith", "e.g.", "U.S."), numbers ("1.5 billion"), and ellipses ("...") that contain periods but are not sentence boundaries. A regex that fires on any period-space will split "The model was trained by Dr. Smith." into two bogus TTS calls. Acceptable mitigations: require the sentence to be > 30 characters before triggering, or use a more sophisticated boundary detector that checks if the word before the period is a common abbreviation.

**Done when:** For a 4-sentence LLM response, the first sentence's audio begins playing within 1.5 seconds of the transcript being sent. The second sentence's audio begins within 200ms of the first ending (not after a 1.5s TTS call for the second sentence, because TTS for sentence 2 was pre-fetched during sentence 1 playback). The pipeline's total latency for 4 sentences is not 4 × TTS_latency — it approaches 1 × TTS_latency after the first sentence.

### Phase 4 — Interrupt Handling and Full-Duplex

**The question this phase answers:** If the user speaks while the assistant is talking, what must be cancelled, in what order, and how?

**Key decisions:**

- **What "interrupt" means at each layer.** The user starts speaking (VAD fires "voice detected"). At this moment: (a) current `Audio` object playing TTS output must stop, (b) the TTS queue must be cleared (pending sentences should not be sent to TTS), (c) the active LLM SSE connection must be closed, (d) the new recording must begin. These four actions must happen atomically before the VAD event returns.

- **Stopping audio cleanly.** `audio.pause()` stops playback but leaves the `src` loaded, which can cause issues with garbage collection and the `ended` event. Clean stop: `audio.pause(); audio.src = ''; audio.load()`. This releases the audio resource and resets the element state. If using the Web Audio API with decoded buffers, call `bufferSource.stop()` (not `pause()` — `AudioBufferSourceNode` has no `pause`).

- **Aborting the SSE stream.** The `EventSource` API has no abort method — once opened, it auto-reconnects on close. Instead, use `fetch` with `ReadableStream` for SSE: `const response = await fetch('/stream?...', { signal: abortController.signal })`. Call `abortController.abort()` on interrupt. The fetch is cancelled server-side via `GeneratorExit` exception in Flask's generator — wrap the `yield` in a try/finally and close the OpenAI connection in the finally block.

- **Preventing double-recordings.** VAD must not fire while the assistant is speaking (the microphone would pick up the speaker output). Two solutions: (a) mute the microphone during TTS playback (call `stream.getAudioTracks()[0].enabled = false` while audio is playing, re-enable after); (b) use echo cancellation (`getUserMedia({ audio: { echoCancellation: true } })`) which filters the speaker output from the microphone signal. Echo cancellation is more user-friendly (the mic stays active so the user can interrupt) but requires acoustic echo cancellation to work at the speaker volume.

- **State machine for the conversation.** Define states: `IDLE → LISTENING → PROCESSING → SPEAKING → IDLE`. Transitions: (IDLE + button press) → LISTENING; (LISTENING + silence) → PROCESSING; (PROCESSING + transcript) → SPEAKING; (SPEAKING + audio end) → IDLE; (LISTENING or SPEAKING + voice detected) → interrupt → LISTENING. Every UI element's state should derive from this machine — not ad-hoc flags.

**Trap:** Not aborting the OpenAI LLM request when the user interrupts. If you only stop playback on the client but the LLM request continues on the server, you pay for the full LLM completion even though the user interrupted it. Additionally, Flask's generator keeps the SSE connection alive and continues yielding tokens — these tokens accumulate in the browser's SSE buffer and will be sent to TTS as soon as the connection is reopened. Always abort the server-side generator on interrupt.

**Done when:** User says something → assistant starts speaking → user speaks again mid-response → first audio stops immediately (within 50ms of VAD trigger) → new transcription begins. No "ghost" audio plays after the interrupt. The SSE connection shows as "cancelled" in browser DevTools Network tab.

### Phase 5 — HTTPS, Permissions, and Deployment

**The question this phase answers:** Why does `getUserMedia` fail in production but not in local dev, and how do you deploy this securely?

**Key decisions:**

- **Secure context requirement.** `navigator.mediaDevices.getUserMedia` requires a secure context: HTTPS or `localhost`. An IP address (e.g., `192.168.1.5`) is not a secure context even on a local network — microphone access will be denied with `DOMException: NotAllowedError`. This is enforced by browsers since Chrome 47 and is non-negotiable. Local dev: use `localhost` explicitly, not `127.0.0.1` (though both work in most browsers). Production: HTTPS is required, which means a TLS certificate.

- **Permission persistence.** Browser microphone permission is per-origin. Once granted, it persists until explicitly revoked. If the user denies permission, the page cannot retry without a browser settings change. Handle `NotAllowedError` specifically: show a clear message ("Microphone access denied — enable in browser settings") rather than a generic error.

- **HTTPS for Flask in production.** Flask's built-in server is HTTP-only. Options: (a) Reverse proxy (nginx + certbot/Let's Encrypt) — add TLS termination in front of Flask; (b) Deploy to a platform that handles TLS automatically (Render, Railway, Fly.io); (c) Use `gunicorn` + `gevent` for production WSGI and TLS from the platform layer. Never expose the Flask dev server to the internet.

- **WebRTC for peer-to-peer alternatives.** If the TTS audio is being played from the browser and the microphone is also in the browser, WebRTC's echo cancellation is more effective than browser-level echo cancellation — because WebRTC knows exactly what audio is being output and can cancel it from the input signal. For a production system, consider using WebRTC's `RTCPeerConnection` to handle the audio pipeline rather than `MediaRecorder` + `Audio` objects.

- **Rate limiting and cost control.** Every user utterance costs: Whisper ($0.006/min), LLM tokens (~$0.001–0.01 per exchange), TTS ($0.015/1K chars). A 5-minute conversation: ~$0.10–0.30. Implement per-user rate limiting (e.g., 20 requests/hour via Flask-Limiter) and a cost dashboard that logs token usage per session.

**Trap:** Assuming `window.isSecureContext` is True in all production environments. Behind certain corporate proxies or CDN configurations, the browser may not receive the HTTPS signal correctly and `isSecureContext` will be `false`. Always check `window.isSecureContext` before calling `getUserMedia` and display a clear error if `false`: "Microphone access requires HTTPS — check that the site URL begins with https://".

**Done when:** The app deploys to a public URL with HTTPS. `getUserMedia` prompt appears on first visit. Microphone works after permission grant. Denial is handled gracefully. The site shows a certificate warning-free padlock. Rate limiting returns 429 after 20 requests in one hour.

## Data Flow

```
User speaks
     │
     ▼
MediaRecorder (WebM/Opus chunks, every 100ms)
     │
     ▼ VAD: silence > 1500ms
Stop recording → Blob (audio/webm)
     │
     ▼ HTTP POST /transcribe (multipart/form-data)
Flask:
  pydub: WebM → WAV
  Whisper API (or local): WAV → transcript
  Return: {transcript: "What is gradient descent?"}
     │
     ▼ Browser receives transcript
Display transcript in UI
     │
     ▼ HTTP GET /stream?transcript=...  (SSE)
Flask (generator):
  OpenAI chat.completions.create(stream=True)
  Yield tokens one by one as SSE events
     │
     ▼ Browser accumulates tokens
Sentence buffer: "Gradient descent minimizes..."
  → detect ". " boundary
  → extract complete sentence
     │
     ▼ TTS call (sentence 1)
OpenAI TTS / window.speechSynthesis
     │
     ▼
Audio.play() → sentence 1 plays
     │ (while playing)
     ▼
TTS prefetch: sentence 2 ready in queue
     │
     ▼
Audio.play() → sentence 2 plays
     ....
     │
     ▼
All sentences played → state → IDLE
```

## Edge Cases to Handle Before You Code

1. **User speaks before the recording started (button press latency).** There is a 100–200ms gap between button press and `MediaRecorder.start()` completing. Pre-allocate the `MediaStream` via `getUserMedia` on page load (before any recording) — this eliminates the device-open latency from the recording path. The stream can idle silently until needed.

2. **Whisper transcribes silence as "[Music]" or "(clears throat)".** Whisper fills short silence clips with these hallucinations rather than returning an empty string. Check: if transcript contains only `[` brackets or is < 3 words, treat it as empty and restart listening. Do not send it to the LLM.

3. **The LLM response is a single very long sentence (no boundary for 10+ seconds).** The sentence-boundary approach fails — TTS waits for a period that never comes within the budget. Fallback: set a maximum token count before force-flushing the buffer to TTS regardless of sentence boundary (e.g., 100 tokens ~ 70 words). This may produce a TTS call mid-sentence, but that is preferable to 10-second silence.

4. **`SpeechSynthesisUtterance` silently stops after ~200 characters on some iOS Safari versions.** This is a known browser bug. Workaround: chunk utterances to < 200 characters at natural word boundaries before passing to `speechSynthesis.speak()`. Alternatively, use an external TTS API and an `Audio` element, which is immune to this bug.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|---|---|---|---|
| VAD method | RMS-based energy threshold | WebRTC VAD (via WASM) | RMS for simplicity and zero dependencies; WebRTC VAD when false-positive rate is too high in noisy environments |
| LLM streaming transport | SSE via HTTP GET | WebSocket | SSE when streaming is unidirectional (server → client); WebSocket when bidirectional control messages are needed at high frequency |
| TTS provider | `window.speechSynthesis` | External API (OpenAI TTS) | speechSynthesis for zero cost/latency in dev; external API for production quality and consistent cross-device behavior |
| Echo cancellation | Browser `echoCancellation: true` | Mute mic during playback | Browser echo cancellation for seamless interruption support; mic mute for simpler implementation when interruption is not required |
| Abort strategy on interrupt | `AbortController` on fetch streams | Close `EventSource` | `AbortController` (with `fetch` SSE) — gives programmatic control and clean abort; `EventSource` has no abort, only auto-reconnect |

## What Makes This Non-Trivial

The pipeline has five asynchronous stages, each with its own error modes, and the state machine must remain coherent across all of them. A bug in interrupt handling can leave the state machine in a half-open state — recording is stopped but the LLM SSE is still streaming, the audio queue has stale sentences queued, and the UI shows "listening" while the backend is still generating. Defensive programming means every state transition must be idempotent and must explicitly clean up all resources of the previous state.

Audio format is a browser compatibility minefield. WebM/Opus works in Chrome and Edge. Ogg/Opus in Firefox. MP4 in Safari. The server must handle all three. Using `MediaRecorder.isTypeSupported('audio/webm;codecs=opus')` before creating the recorder and falling back to `audio/mp4` for Safari is necessary for cross-browser support. The server-side ffmpeg conversion handles all three container formats transparently.

The sentence-boundary streaming requires choosing between two failure modes: cut too early (mid-sentence, awkward TTS output) or cut too late (long silence waiting for a boundary). Neither is perfect. The production solution is a smarter sentence boundary detector that understands abbreviations and lists — not a simple regex. For a demo, the simple regex is acceptable with a fallback flush at 100 tokens.

## Interview Angles This Project Develops

1. **"Why does `getUserMedia` require HTTPS?"** Strong answer: browsers enforce the secure context requirement because microphone access is a sensitive permission. An HTTP page can be intercepted by a network attacker who could inject JavaScript that silently records audio. HTTPS ensures the page code is exactly what the server sent. `localhost` is exempt because loopback traffic cannot be intercepted.

2. **"How would you reduce perceived latency in the LLM → TTS pipeline?"** Strong answer explains sentence-boundary streaming: don't wait for the full response — accumulate until a sentence boundary and TTS each sentence immediately. The first audio begins after one sentence of LLM generation + one TTS call, not after the full response. Contrast with the naive approach: full LLM wait → full TTS wait.

3. **"What is the difference between SSE and WebSocket, and when would you choose each?"** Strong answer: SSE is HTTP/1.1, unidirectional (server → client), auto-reconnects, simple to implement in Flask (a generator + text/event-stream). WebSocket is bidirectional, requires a separate protocol upgrade, more complex to implement but necessary for bidirectional high-frequency communication. For LLM token streaming: SSE is correct. For a game where client and server exchange messages at 60fps: WebSocket.

4. **"How do you handle a user interrupting the assistant mid-sentence?"** Strong answer covers all four cleanup steps: stop audio playback and release the Audio object, abort the LLM SSE fetch using AbortController, clear the TTS sentence queue, enable the microphone VAD and begin new recording. Emphasizes that these must happen atomically and that the LLM server-side generator must also terminate (not just client-side).

5. **"What is echo cancellation and why does it matter here?"** Strong answer: echo cancellation removes the speaker's output from the microphone signal. Without it, when the assistant speaks, its audio is captured by the microphone, transcribed by Whisper ("What is gradient descent?"), and fed back into the LLM — causing an infinite loop. Browser-level echo cancellation (`echoCancellation: true` in `getUserMedia`) uses digital signal processing to subtract the known speaker output from the mic input.

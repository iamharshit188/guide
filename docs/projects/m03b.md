# Hybrid BM25 + Dense Search Engine — Thinking Guide

> **Module:** 03 — Databases & Vector DBs · **Difficulty:** Intermediate · **Type:** THINKING · **Time:** ~3 hours

## The Problem

Build a search system over a document corpus (≥500 documents) that combines BM25 sparse scores and dense embedding cosine scores via Reciprocal Rank Fusion (RRF). No off-the-shelf hybrid library — implement BM25 from the formula, implement RRF from the formula, and combine the two ranked lists manually. Evaluate retrieval quality using NDCG@10 against a labeled query set.

**Input:** a natural-language query string  
**Output:** a ranked list of document IDs with fused relevance scores  
**Constraints:** BM25 and embedding scoring must be independently testable; fusion must be a pure function of two ranked lists

---

## Before You Write Any Code: Build The Mental Model

Dense retrieval maps both query and document into $\mathbb{R}^d$ and finds the nearest neighbours by cosine distance. This works well when the query and document use different words to express the same concept — "heart attack" and "myocardial infarction" land near each other in embedding space because the model was trained on co-occurrence patterns. But dense models compress semantics into a fixed-size vector, so very rare or domain-specific tokens (e.g., `CVSS-3.1`, `errno 28`, a product SKU) may map to a region of the space that is poorly calibrated — the model has seen these tokens too rarely to learn their discriminative meaning.

BM25 is the complement: it is a bag-of-words TF-IDF variant that rewards exact lexical match. A document containing the query term many times, in a short document, gets a high score for that term — regardless of whether the model has ever seen that term. The weakness is symmetrical: BM25 gives zero score to a document that uses synonyms but not the exact query token.

Reciprocal Rank Fusion sidesteps the score incompatibility problem (BM25 scores are unbounded; cosine similarities are in $[-1, 1]$) by operating only on ranks. Each document's contribution from each retriever is $\frac{1}{k + r_i}$ where $r_i$ is its rank in that retriever's list. The constant $k$ (typically 60) dampens the advantage of top-ranked results and prevents one retriever from completely dominating when its top-1 is extremely confident.

The goal is not to tune a perfect system — it is to understand exactly why each component is necessary and what breaks when you remove it.

## Decompose It

### Phase 1 — Corpus and Vocabulary

**The question this phase answers:** What is the token universe over which BM25 will compute statistics?

**Key decisions:**
- Tokenization granularity: whitespace-split is fast but misses morphological variants ("running" ≠ "run"). Porter stemming or lemmatization normalizes these but adds a dependency and slightly increases vocabulary collision. For a thinking exercise, whitespace + lowercase + strip punctuation is sufficient — document the trade-off explicitly.
- Stopword removal: BM25 IDF heavily downweights high-frequency terms anyway ($\text{IDF}(t) \approx 0$ when $df_t \approx N$), so removing stopwords is an optimization, not a correctness requirement.
- Vocabulary size: a vocabulary of 50K tokens is typical for a 500-document corpus. Terms seen only once globally are noise — consider a minimum document frequency of 2.

**Trap:** Building vocabulary from the query set as well as the corpus. Vocabulary must be built from the corpus only. At query time, OOV query tokens simply contribute nothing to the BM25 score — that is correct behavior, not a bug.

**Done when:** You have a mapping `{term: df}` (document frequency for each term), the average document length `avgdl`, and a per-document `{term: tf}` counter.

---

### Phase 2 — BM25 Scoring

**The question this phase answers:** Given a query, how do you produce a score for every document that has at least one query term in it?

**Key decisions:**
BM25 score for query $q$ against document $d$:

$$\text{BM25}(q, d) = \sum_{t \in q} \text{IDF}(t) \cdot \frac{f(t,d) \cdot (k_1 + 1)}{f(t,d) + k_1 \cdot \left(1 - b + b \cdot \frac{|d|}{\text{avgdl}}\right)}$$

where $\text{IDF}(t) = \log\frac{N - df_t + 0.5}{df_t + 0.5}$ (Robertson IDF variant, which can be negative for very frequent terms — clamp to 0).

Parameters: $k_1 \in [1.2, 2.0]$, $b \in [0.5, 0.75]$. The standard defaults are $k_1 = 1.5$, $b = 0.75$.

- $k_1$ controls term frequency saturation: as $f(t,d) \to \infty$, the TF component approaches $k_1 + 1$. Higher $k_1$ = less saturation = more reward for repeated terms.
- $b$ controls length normalization: $b = 0$ means no length penalty; $b = 1$ means full normalization relative to `avgdl`. For short documents (tweets, code snippets), lower $b$ is better; for long articles, higher $b$ prevents long documents from winning by sheer term count.

**Efficiency decision:** Do not score every document for every query. Maintain an inverted index: `{term: [doc_id, ...]}`. For a query, union the posting lists for all query terms — only these documents have non-zero BM25 scores. This takes scoring from $O(N \cdot |q|)$ to $O(df_t \cdot |q|)$.

**Trap:** Computing IDF once at index time (correct) versus recomputing it per query (wasteful but equivalent — the trap is doing it wrong by using query-corpus statistics). The more insidious trap: forgetting to clamp negative IDF values. Extremely common terms produce negative IDF under the Robertson formula; adding a negative contribution for a query term that appears in a document would penalize relevant documents. Clamp to `max(idf, 0)`.

**Done when:** Given the query "machine learning model evaluation", you can produce a sorted list `[(doc_id, score), ...]` for all documents with at least one match, and the scores are positive and bounded by approximately $(k_1 + 1) \cdot |q|$.

### Phase 3 — Dense Retrieval

**The question this phase answers:** Which documents are semantically nearest to the query in embedding space, regardless of lexical overlap?

**Key decisions:**
- Model choice: `all-MiniLM-L6-v2` (384-dim, fast) for development; `all-mpnet-base-v2` (768-dim, higher quality) if you have GPU. The choice affects recall — a stronger encoder finds more relevant documents in the top-20, which matters more than re-ranking quality at this stage.
- Index type: for ≤100K documents, an exact flat index (brute-force cosine) is perfectly fast (< 100ms per query on CPU). FAISS `IndexFlatIP` over L2-normalized vectors gives exact cosine similarity. Approximate indices (IVF, HNSW) become necessary above ~1M documents.
- Normalization: L2-normalize all document embeddings at index time, then normalize the query embedding at search time. Inner product then equals cosine similarity. If you skip normalization, inner product rewards long vectors (numerically large embeddings), which introduces a document-length bias unrelated to semantic relevance.

**Trap:** Using the same embedding for both retrieval and re-ranking. Dense retrieval is a bi-encoder (query and document encoded separately). If you want a re-ranker, that is a separate cross-encoder that takes the concatenated (query, document) pair — much slower but more accurate. Do not conflate the two.

**Done when:** Given the same query, you can produce a ranked list of `(doc_id, cosine_score)` tuples, with scores in $[0, 1]$ after normalization.

---

### Phase 4 — Reciprocal Rank Fusion

**The question this phase answers:** How do you merge two ranked lists whose scores are on incompatible scales into a single ranked list?

**Key decisions:**

RRF score for document $d$:

$$\text{RRF}(d) = \sum_{r \in \text{retrievers}} \frac{1}{k + \text{rank}_r(d)}$$

where $\text{rank}_r(d)$ is the 1-indexed position of $d$ in retriever $r$'s list, and $k = 60$ is the standard default. Documents not present in a retriever's list contribute $0$ from that retriever.

- Why $k = 60$: this was empirically derived in Cormack et al. (2009). A higher $k$ makes the fusion more democratic (all documents contribute similarly); a lower $k$ amplifies the advantage of top-ranked documents. For most retrieval tasks, $k \in [30, 100]$ produces nearly identical NDCG results — do not over-tune this.
- Weighted RRF: you can add retriever-specific weights: $\frac{w_r}{k + \text{rank}_r(d)}$. Use this when you have evidence (from NDCG measurements) that one retriever is significantly better calibrated for your corpus. Default to equal weights first.
- What RRF does NOT do: it does not re-score documents based on the content of the query. It purely aggregates ranks. This means a document that is rank-1 in BM25 and rank-1 in dense will win, but a document that is rank-3 in both will usually beat one that is rank-1 in only one retriever.

**Trap:** Using only the top-K results from each retriever before fusion. If you retrieve top-10 from BM25 and top-10 from dense, documents that rank 11th in one retriever but 2nd in the other are invisible to fusion — you lose recall. Retrieve a larger candidate set (top-100 per retriever) before fusing, then truncate the fused list to top-10 for the final output.

**Done when:** The fusion function is a pure function `rrf(bm25_ranked, dense_ranked, k=60) -> [(doc_id, score)]` that handles documents appearing in only one list correctly.

---

### Phase 5 — Evaluation with NDCG

**The question this phase answers:** Is the hybrid system actually better than either retriever alone?

**Key decisions:**
NDCG@K (Normalized Discounted Cumulative Gain):

$$\text{DCG@K} = \sum_{i=1}^{K} \frac{2^{rel_i} - 1}{\log_2(i+1)}, \quad \text{NDCG@K} = \frac{\text{DCG@K}}{\text{IDCG@K}}$$

where $rel_i$ is the relevance label of the document at rank $i$ (binary: 0/1, or graded: 0/1/2), and IDCG is the DCG of the ideal (perfectly sorted) ranking.

- You need a labeled query set: 20-50 queries with ground-truth relevant document IDs. For a public corpus (MS MARCO, BEIR), labels exist. For a custom corpus, you must label manually or use a strong model (GPT-4) to generate silver labels.
- Compare three conditions: BM25 only, Dense only, Hybrid. Expect Hybrid to match or exceed both on average. On queries with rare technical terms, BM25 will often outperform Dense alone; on paraphrase queries, Dense will outperform BM25 alone.
- Statistical significance: with only 20 queries, differences < 5% NDCG points are likely noise. You need ≥100 queries for reliable conclusions.

**Trap:** Evaluating only on queries you used during system development. This is test-set leakage. If you tuned $k_1$, $b$, or the fusion weight $k$ on the same queries you measure NDCG on, your numbers are optimistic.

**Done when:** You have a table of NDCG@10 scores for BM25, Dense, and Hybrid, and Hybrid is at least as good as the better of the two individual retrievers on your eval set.

## Data Flow

```
Query string
     |
     +---------------------------+
     |                           |
  [Tokenize]               [Embed query]
  query terms              (384-dim vec)
     |                           |
[Inverted index]          [FAISS flat index]
  lookup posting             brute-force
  lists per term             inner product
     |                           |
[BM25 score]              [Cosine score]
  per doc                   per doc
     |                           |
[Sort → BM25 ranked list]  [Sort → Dense ranked list]
  (doc_id, score)×N         (doc_id, score)×N
     |                           |
     +----------[RRF]------------+
                  |
          [Fused ranked list]
          (doc_id, rrf_score)
                  |
           [Truncate top-K]
                  |
          Display to user
```

## Edge Cases to Handle Before You Code

1. **Query term not in vocabulary (OOV for BM25):** The inverted index lookup returns an empty posting list. The term contributes $0$ to the BM25 score for all documents. This is correct — do not raise an error. A query made entirely of OOV terms returns an empty BM25 list; RRF then falls back to the dense ranking entirely.

2. **Document appears in dense top-100 but not in BM25 top-100:** Its RRF contribution from BM25 is $0$. Its total RRF score is $\frac{1}{k + r_{\text{dense}}}$. This is correct behavior — the system does not assume the document is irrelevant just because BM25 missed it.

3. **Empty corpus or single-document corpus:** `avgdl` is well-defined (just that one document's length). IDF for every term is $\log\frac{1.5}{1.5} = 0$, meaning BM25 gives zero scores — all queries return empty BM25 results. Dense retrieval still works. Handle this gracefully by documenting the degenerate case rather than crashing.

4. **Very short query (single token):** BM25 produces a ranking based on one IDF-TF product — perfectly valid. Dense embedding of a single word is less informative than a phrase; recall may suffer. This is the honest weakness of bi-encoders on keyword queries — worth noting in evaluation.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|----------|----------|----------------|
| Retrieval candidate size before RRF | Top-100 per retriever | Top-10 per retriever | Always pick A — fusion recall is bounded by candidate set size |
| BM25 IDF formula | Robertson: $\log\frac{N - df + 0.5}{df + 0.5}$ | Classic: $\log\frac{N}{df}$ | Robertson when $df > N/2$ (common terms) — classic IDF goes to $0$ but Robertson goes negative; clamp either way |
| Fusion method | RRF (rank-based) | Linear score combination | RRF when retrievers are incompatibly scaled (almost always); linear when you can normalize scores and have validation data to tune weights |
| Dense index type | Exact flat index | HNSW approximate | Flat for $N < 500K$ — HNSW adds engineering complexity with no speed benefit at small scale |
| Evaluation labels | Manual annotation | LLM-generated silver labels | Manual for final benchmark; silver labels for rapid iteration during development — acknowledge the quality gap |

## What Makes This Non-Trivial

The surface implementation (call BM25 library + call FAISS + sort two lists) takes two hours. The non-trivial parts are:

**Score calibration:** BM25 and cosine scores are not on the same scale, and they are not independent — documents with high BM25 scores are often the ones with exact term matches that the dense model also ranks highly, so naive linear combination double-counts these. RRF avoids this by discarding score magnitude, but then you lose the ability to say "this document is rank-1 with a cosine similarity of 0.97 vs. 0.63 for rank-2" — both contribute $\frac{1}{k+1}$ to the fused score.

**The sparse-dense gap is query-dependent:** for queries that use common language, dense almost always wins; for queries with technical jargon, sparse wins. Building a query classifier that routes queries to the appropriate retriever (or weights the fusion) is a research-level problem. For this project, the insight is to measure and document when each retriever wins, not to solve the routing problem.

**Inverted index efficiency:** a naive implementation loops over all documents for every query term. A real inverted index is a compressed sorted list of document IDs per term with skip pointers. The difference is 100ms vs. 10ms at 10K documents — not critical here, but understanding why the gap exists (cache misses, branch prediction, sorted merge vs. linear scan) is the kind of detail that distinguishes a strong answer from a textbook answer.

## Interview Angles This Project Develops

1. "Why does dense retrieval fail on rare terms?" — Strong answer: the model's embedding for an OOV or low-frequency token is poorly calibrated because it appears in few training examples. The attention mechanism distributes weight to neighboring context tokens instead, so the resulting embedding encodes the context more than the rare term itself. BM25's IDF naturally handles this — a rare term has high IDF, making an exact match highly discriminating.

2. "What does the $b$ parameter in BM25 control?" — Strong answer: $b$ interpolates between no length normalization ($b=0$, where $|d|$ does not affect the score) and full length normalization ($b=1$, where the TF component is divided by $|d|/\text{avgdl}$). High $b$ prevents long documents from winning by term repetition alone; low $b$ is appropriate when document length is not a proxy for topic breadth (e.g., all documents are the same length).

3. "Why use RRF instead of normalizing BM25 scores to $[0,1]$ and averaging?" — Strong answer: BM25 score distribution is highly query-dependent — for a single-term query over a large corpus, most documents score near $0$ and one document scores very high. Min-max normalization of this distribution produces a degenerate result where nearly every document maps to $\approx 0$ and one maps to $1.0$. RRF's rank-based aggregation is robust to this distribution shift.

4. "How would you improve recall without degrading precision?" — Strong answer: expand the candidate set per retriever before fusion, then re-rank with a cross-encoder on the top-50 fused results. The cross-encoder sees the full (query, document) pair and can catch relevance that bi-encoders miss. The cost is $O(50)$ cross-encoder forward passes per query — acceptable if query latency budget is > 200ms.

5. "How do you set $k$ in RRF without overfitting?" — Strong answer: $k = 60$ is a literature-derived default that is robust across retrieval tasks. You can sweep $k \in \{10, 30, 60, 100\}$ on a held-out validation set, but if NDCG differences are < 2%, the choice does not matter — stop tuning and ship.

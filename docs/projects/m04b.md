# Rate-Limited API Gateway — Thinking Guide

> **Module:** 04 — Backend Flask · **Difficulty:** Intermediate · **Type:** THINKING · **Time:** ~5 hours

---

## The Problem

**Input:** HTTP requests from multiple users hitting multiple ML model endpoints. Each user has a per-endpoint rate limit (e.g., user A can call `/predict/sentiment` at most 100 times per minute).

**Output:** Requests within the limit pass through to the downstream ML endpoint and return its response. Requests exceeding the limit receive HTTP 429 with a `Retry-After` header specifying seconds until the next available token. All rate limit events are logged with user ID, endpoint, timestamp, and current token count.

**Constraints:** The gateway must be correct under concurrent load — multiple workers serving the same user must not each allow 100 requests/minute (adding up to 1,000 if 10 workers). Rate limit state must survive worker restarts. The gateway must not add more than 5ms of latency to passing requests.

---

## Before You Write Any Code: Build The Mental Model

Rate limiting is a **distributed state problem**, not a web development problem. The algorithm itself is simple; the hard part is that state must be shared, mutations must be atomic, and the system must remain correct under partial failure (Redis down, worker crash mid-decrement).

The token bucket model is a **leaky container metaphor**: tokens refill at rate $r$ tokens/second up to a maximum capacity $C$. Each request consumes 1 token. If the bucket has tokens, the request passes; otherwise it waits or is rejected. This naturally handles burst traffic — a user who has been idle accumulates tokens up to $C$ and can send a burst. The alternative (leaky bucket) enforces a strictly constant output rate, rejecting bursts rather than absorbing them. For ML APIs where a user might batch several requests after thinking, token bucket is the correct model.

Redis is the right choice for rate limit state because it is a **single-threaded command executor** — each Redis command is atomic by definition. Two workers simultaneously sending `DECR key` cannot race: Redis queues them and executes sequentially. An in-process Python dict cannot guarantee this under gunicorn's multi-process model (each process has a separate memory space) or threading (GIL does not protect compound operations across multiple Python statements).

**Atomicity is the core requirement.** Read-check-decrement must be a single atomic operation. If you split it into three commands (`GET key`, check value, `DECR key`), two concurrent workers can both GET a value of 1, both check that it's > 0, and both DECR, producing -1. The user was allowed two requests when they should have been blocked after one.

## Decompose It

### Phase 1 — Choose the Rate Limiting Algorithm

**The question this phase answers:** Which algorithm fits the burst + average rate requirements of ML API usage?

**Key decisions:**

**Token bucket:** State = (current_tokens, last_refill_time). On each request: compute elapsed time since last refill, add $r \times \Delta t$ tokens (capped at $C$), if tokens ≥ 1 then decrement and allow, else reject. Natural for APIs — absorbs bursts, enforces average rate. State size: 2 values per (user, endpoint) pair.

**Leaky bucket:** State = (queue_size). Requests enter a queue; they drain at a constant rate $r$. Excess requests are rejected when queue is full. Guarantees constant output rate regardless of burst. Use this when downstream ML service is sensitive to burst load and cannot handle irregular request patterns. For a gateway that's just enforcing quotas (not shaping traffic), token bucket is more user-friendly.

**Sliding window counter:** Divide time into sub-windows (e.g., 6 × 10-second windows for a 1-minute limit). Count requests per sub-window; total = sum of last $W/w$ windows. More accurate than fixed-window (which has a 2× burst problem at window boundaries) but requires storing $W/w$ counters per (user, endpoint). Redis sorted sets work well: member = timestamp, score = timestamp; count members in range [now - window, now].

**Fixed window counter:** Count resets at the start of each window. Simplest to implement. Has the boundary bug: a user can send $C$ requests at 11:59:59 and $C$ more at 12:00:00 — 2× the intended rate in 2 seconds. Acceptable only when boundary bursts don't matter (not true for ML APIs).

**For this project, use token bucket.** It is the most appropriate for ML API usage patterns (bursty) and is the industry standard (Stripe, GitHub, OpenAI all use token bucket or variants).

**Trap:** Implementing token bucket with floating-point arithmetic for token count. Redis stores integers natively; storing tokens as a float requires serialization (string or hash) and adds complexity. Use integer tokens (1000 tokens = 1 second of quota) — multiply refill rate by a fixed precision multiplier.

**Done when:** You have a clear specification: bucket capacity $C$, refill rate $r$ (tokens/second), Redis key schema `ratelimit:{user_id}:{endpoint}`, and a decision on integer vs float token representation.

### Phase 2 — Redis Atomicity: Why Lua Beats MULTI/EXEC

**The question this phase answers:** How do you make read-check-decrement atomic in Redis?

**Key decisions:**

**Redis MULTI/EXEC (optimistic transactions):** Wraps multiple commands in a transaction. All commands are queued and executed atomically. However, MULTI/EXEC uses optimistic locking — if WATCH detects a concurrent modification between WATCH and EXEC, the transaction aborts and returns nil. You must retry on abort. Under high concurrency, this creates a retry storm. MULTI/EXEC is appropriate for low-contention scenarios.

**Lua scripts:** Redis executes Lua scripts atomically as a single command. The script runs within the single-threaded Redis event loop — no interleaving possible. Syntax: `redis.call('GET', KEYS[1])`. Scripts are registered with `SCRIPT LOAD` and called with `EVALSHA`. Under high concurrency, Lua scripts win over MULTI/EXEC because there's no optimistic abort — the script always succeeds atomically.

```lua
-- token_bucket.lua
local key = KEYS[1]
local capacity = tonumber(ARGV[1])
local refill_rate = tonumber(ARGV[2])  -- tokens per second
local now = tonumber(ARGV[3])           -- unix timestamp in ms

local data = redis.call('HMGET', key, 'tokens', 'last_refill')
local tokens = tonumber(data[1]) or capacity
local last_refill = tonumber(data[2]) or now

-- compute refill
local elapsed = (now - last_refill) / 1000.0
local refilled = math.floor(elapsed * refill_rate)
tokens = math.min(capacity, tokens + refilled)
last_refill = last_refill + math.floor(refilled / refill_rate * 1000)

if tokens >= 1 then
    tokens = tokens - 1
    redis.call('HMSET', key, 'tokens', tokens, 'last_refill', last_refill)
    redis.call('EXPIRE', key, 3600)  -- TTL cleanup
    return {1, tokens}               -- allowed, remaining
else
    -- compute time until next token available
    local wait_ms = math.ceil((1 - tokens) / refill_rate * 1000)
    return {0, wait_ms}             -- rejected, ms to wait
end
```

**Python side:** Load the script once at startup with `r.script_load(script)`, store the SHA, call with `r.evalsha(sha, 1, key, capacity, refill_rate, now_ms)`.

**Trap:** Using separate `GET` + `SET` or `INCR` + `EXPIRE` calls. If the process crashes between INCR and EXPIRE, the key has no TTL and the rate limit bucket never resets — the user is permanently blocked until manual intervention.

**Done when:** Lua script implemented, loaded on startup, called on every request. Verified with two concurrent Python threads hitting the same key — total allowed requests should not exceed capacity.

### Phase 3 — Flask Gateway Architecture and Multi-Worker Correctness

**The question this phase answers:** How do you structure the Flask app so that Redis-based rate limiting is correct under gunicorn multi-process deployment?

**Key decisions:**

**In-process dict (wrong):** Each gunicorn worker has its own process with its own memory. A dict in worker 1 is invisible to worker 2. With 10 workers and a 100 req/min limit, each worker enforces 100 independently — effectively allowing 1,000 req/min. This is the most common mistake.

**Connection pooling:** Each gunicorn worker needs its own Redis connection or a pool. Use `redis.ConnectionPool` with `max_connections` set to 1–3 per worker (not a global pool shared across processes, which requires serialization). The pool is created at Flask app startup, before `fork()`, but this can cause socket inheritance issues — create the pool after fork with gunicorn's `post_fork` hook, or create it lazily on first request.

**Rate limit middleware as a Flask `before_request` hook:**
```python
@app.before_request
def check_rate_limit():
    user_id = extract_user_id(request)  # from JWT or API key header
    endpoint = request.endpoint
    key = f"ratelimit:{user_id}:{endpoint}"
    allowed, value = lua_rate_limit(key, CAPACITY, REFILL_RATE, now_ms())
    if not allowed:
        retry_after = math.ceil(value / 1000)
        response = jsonify({"error": "rate limit exceeded"})
        response.status_code = 429
        response.headers['Retry-After'] = str(retry_after)
        response.headers['X-RateLimit-Limit'] = str(CAPACITY)
        response.headers['X-RateLimit-Remaining'] = '0'
        return response
    g.rate_limit_remaining = value  # pass to response headers
```

**Retry-After header:** RFC 7231 specifies `Retry-After` as either an HTTP-date or a delta-seconds integer. Use delta-seconds: `math.ceil(wait_ms / 1000)`. Round up to avoid telling the client to retry too early.

**Trap:** Extracting user identity from an unauthenticated header (`X-User-Id: anything`). Rate limits only work if the user ID is from a trusted, tamper-proof source: a JWT validated with your secret key, or an API key looked up in your database. Never rate limit on an untrusted header.

**Done when:** Flask app runs under gunicorn with 4 workers, load test with `locust` or `ab` confirms that total requests allowed ≤ capacity across all workers.

### Phase 4 — Circuit Breaker for Downstream Model Endpoints

**The question this phase answers:** When a downstream ML model endpoint is overloaded or down, how does the gateway fail fast instead of queueing requests?

**Key decisions:**

**Without a circuit breaker:** Each gateway request that reaches a slow downstream endpoint blocks a gunicorn worker thread for the full timeout (e.g., 30 seconds). With 4 workers and 30-second timeouts, 4 concurrent slow requests exhaust all workers — the gateway is completely unresponsive to all users, even those under the rate limit. This is a cascading failure.

**Circuit breaker states:** CLOSED (pass requests through), OPEN (fail immediately without calling downstream), HALF-OPEN (allow one probe request; if it succeeds, return to CLOSED; if it fails, stay OPEN). Transitions: CLOSED→OPEN when failure rate exceeds threshold over a window. OPEN→HALF-OPEN after a reset timeout (e.g., 30 seconds).

**Redis-backed circuit breaker state:** Store circuit state per endpoint in Redis: `circuit:{endpoint}:state` (CLOSED/OPEN/HALF-OPEN), `circuit:{endpoint}:failures` (count), `circuit:{endpoint}:last_failure` (timestamp). State transitions must be atomic — use Lua or MULTI/EXEC.

**Graceful degradation options:**
- **Reject with 503:** Immediately return "Service Unavailable" with `Retry-After`. Clean, honest, forces clients to implement retry logic.
- **Serve cached last response:** Store the last successful response per endpoint in Redis. Return it with a `Stale-If-Error` header. Acceptable for ML predictions where slightly stale output is better than no output (e.g., product recommendations).
- **Queue request:** Add request to a Redis list; a worker processes it asynchronously; client polls or receives a webhook. Correct for long-running inference; adds complexity.

**Trap:** Setting the circuit breaker failure threshold too low (e.g., 1 failure → OPEN). A single timeout from a temporarily overloaded model opens the circuit and blocks all traffic. Use a window-based threshold: OPEN only if >50% of requests in the last 10 seconds fail.

**Done when:** Simulate downstream model failure by killing the model server. Gateway returns 503 immediately within 1 second (not 30-second timeout), with Retry-After set to the circuit breaker reset time.

### Phase 5 — Logging and Observability

**The question this phase answers:** What do you log for rate limit events, and how do you make it actionable?

**Key decisions:**

**Structured logging:** Log as JSON: `{"event": "rate_limit_exceeded", "user_id": "u123", "endpoint": "/predict/sentiment", "limit": 100, "window": "1m", "timestamp": "2024-01-15T10:23:45Z"}`. Structured logs are queryable — you can count violations per user, per endpoint, per time period without parsing.

**What to log at minimum:** Every 429 response (user, endpoint, timestamp, remaining wait). Every circuit breaker state transition. Every request that reaches the downstream endpoint (latency, status code). Do NOT log every passing request at INFO level — at high throughput this floods the log sink.

**Metrics vs logs:** Rate limit violations should also emit a metric counter (`rate_limit_exceeded_total{user_id, endpoint}`) to a time-series database (Prometheus). Log individual events for debugging specific users; metrics for dashboards and alerting. These serve different purposes; don't use logs as a substitute for metrics at scale.

**Trap:** Logging user PII (email, name) in log lines that go to a shared log aggregator. Log user_id (opaque identifier), not the email or name. Apply data minimization by design.

**Done when:** A rate limit event produces a JSON log line readable by `jq`. A simple log analysis script can identify the top-5 users by violation count in the last hour.

## Edge Cases to Handle Before You Code

1. **Redis is down:** If the Redis connection fails, you cannot enforce the rate limit. Choose: fail open (allow all requests — availability > correctness) or fail closed (reject all requests — safety > availability). For ML APIs, fail open is usually correct: losing rate limit enforcement for 30 seconds during a Redis blip is better than blocking all paying customers. Implement a try/except around the Redis call with a flag.

2. **Clock skew between workers:** Token bucket refill uses `now_ms()` from the calling worker. If two workers have different system clocks (possible in a distributed deployment), the same key may be refilled differently. Pass server time to the Lua script from each worker independently; Redis will serialize the calls. Clock skew of <1 second is acceptable for a 1-minute rate limit window.

3. **Key expiry race:** Redis key TTL is reset on each request. If a user is inactive for exactly the TTL period and then resumes, their key has expired. On the next request, the Lua script initializes a new bucket at full capacity — correct behavior (token bucket should allow a burst after a long idle period up to capacity $C$).

4. **JWT expiry mid-session:** A request arrives with a valid JWT that expires 100ms later, but the same user sends a second request 200ms later with an expired JWT. Your auth middleware rejects the second request with 401. The rate limit has already decremented for the first request. This is correct — don't refund tokens for authenticated requests that subsequently fail auth on a different reason.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|----------|----------|----------------|
| Rate limit algorithm | Token bucket | Sliding window counter | Pick A for APIs where users have bursty patterns and absorbing short bursts is desirable; sliding window is more accurate but more complex to implement atomically |
| Atomicity mechanism | Lua script | MULTI/EXEC with WATCH | Pick A under high concurrency — Lua never aborts; MULTI/EXEC with WATCH retries on conflict, creating thundering herd under load |
| Circuit breaker state storage | Redis | In-process (each worker) | Pick A — in-process state means different workers have different views of downstream health; one worker may be OPEN while another is CLOSED |
| Redis down behavior | Fail open (allow requests) | Fail closed (block requests) | Pick A for commercial APIs — a short rate limit outage is tolerable; blocking all customers during a Redis blip is not |

---

## What Makes This Non-Trivial

Rate limiting is deceptively simple to implement incorrectly. The common mistake is implementing it correctly in a single-process test environment, shipping it, and discovering the multi-process bug under production load. The atomicity requirement cannot be discovered by unit testing — it requires load testing with concurrent clients and careful instrumentation. The circuit breaker adds another layer: its state is also distributed and must be consistent across workers, creating the same class of problem at the downstream boundary. The interaction between rate limiting, circuit breaking, and graceful degradation requires explicit sequencing decisions (what order do you check?) with correctness implications.

## Interview Angles This Project Develops

1. **"Why can't you use an in-process dict for rate limiting with gunicorn?"** Strong answer: gunicorn uses multiple worker processes (not threads by default), each with separate memory. A dict in process 1 is invisible to process 2. With 10 workers and a 100 req/min limit, each worker enforces its own 100 independently, effectively allowing 1,000 req/min. Rate limit state must live in a shared external store — Redis is the standard choice because of its atomic single-threaded command model.

2. **"Why use a Lua script instead of MULTI/EXEC for atomicity?"** Strong answer: MULTI/EXEC uses optimistic locking — if WATCH detects a concurrent modification between WATCH and EXEC, the transaction aborts and the caller must retry. Under high concurrency, this creates a retry storm as all workers compete to complete their transactions. Lua scripts are executed atomically as a single Redis command within the single-threaded event loop — no conflicts, no retries, always completes in one round trip.

3. **"Explain token bucket vs leaky bucket. When would you use leaky bucket for an ML API?"** Strong answer: token bucket absorbs bursts up to capacity $C$; leaky bucket enforces a strictly constant output rate. Leaky bucket is appropriate when the downstream ML service cannot handle burst load and needs a constant request rate — for example, an expensive inference endpoint where you want to guarantee exactly 10 req/sec regardless of client behavior. Token bucket is more user-friendly for quota enforcement (allows a burst after idle periods).

4. **"What does your circuit breaker do when the model endpoint is down?"** Strong answer: after a threshold of failures (e.g., >50% of requests in a 10-second window), the circuit transitions CLOSED→OPEN and immediately returns 503 with a Retry-After header equal to the reset timeout. This prevents blocking gunicorn workers on long timeouts. After the reset period, the circuit transitions to HALF-OPEN, allows one probe request, and returns to CLOSED if it succeeds.

5. **"How do you calculate the Retry-After value for a 429 response?"** Strong answer: the Lua script returns the milliseconds until the next token will be available, computed from the refill rate: `wait_ms = ceil((1 - current_tokens) / refill_rate * 1000)`. Divide by 1000, ceiling to seconds, and return as the `Retry-After` header integer value per RFC 7231. Always round up — rounding down tells the client to retry too early, which will get them another 429.

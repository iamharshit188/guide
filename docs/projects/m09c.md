# PEFT Method Comparison Benchmark — Thinking Guide

> **Module:** 09 — Fine-Tuning LLMs · **Difficulty:** Expert · **Type:** THINKING · **Time:** ~7 hours

## The Problem

**Input:** A domain-specific instruction dataset (medical or legal Q&A, ~5K–20K examples). A base model: Llama-3-8B (8 billion parameters). Four PEFT methods to compare: LoRA, QLoRA, IA³, and Prompt Tuning.

**Output:** A ranked comparison table with 6 measurements per method: (1) trainable parameter count, (2) peak GPU memory during training, (3) training throughput in tokens/sec, (4) final evaluation loss, (5) inference latency per token (ms), (6) ROUGE-L on a held-out evaluation set. A written diagnosis identifying which method is optimal given specific deployment constraints.

**Constraints:** Single GPU (A100 40GB or RTX 4090 24GB). Evaluation must be reproducible: fix random seed, use the same train/eval split for all methods, use the same number of training steps (not epochs, since throughput varies). Methods must use identical base model weights — no cherry-picking model revisions.

## Before You Write Any Code: Build The Mental Model

The central question of PEFT is: what is the minimum number of parameters you need to adapt in order to achieve task-specific performance close to full fine-tuning? Each method answers this differently, and the answer reveals a different hypothesis about where task-specific knowledge lives in a transformer.

Full fine-tuning shifts all $W$ parameters (8B for Llama-3-8B). Every gradient update changes the model's entire knowledge representation — this is why it achieves the best per-task accuracy and also why it causes catastrophic forgetting of other knowledge. The entire weight space is in play during gradient descent, so the model can overfit deeply to domain-specific patterns, destroying its general-purpose representations.

LoRA's hypothesis: task-specific adaptations lie in a low-rank subspace. For a weight matrix $W \in \mathbb{R}^{d \times k}$, LoRA parameterizes the update as $\Delta W = BA$ where $B \in \mathbb{R}^{d \times r}$ and $A \in \mathbb{R}^{r \times k}$, with $r \ll \min(d, k)$. Trainable parameters: $r(d + k)$ per adapted weight, versus $dk$ for full fine-tuning. For Llama-3-8B with attention weight matrices $d = k = 4096$ and $r = 16$: 16 × (4096 + 4096) = 131K parameters per weight, versus 16.8M for full. LoRA is typically applied to the query and value projections in each attention layer, leaving feedforward layers unchanged.

IA³'s hypothesis: task-specific adaptations are even simpler — just rescaling certain activations, not adding to weight matrices. IA³ learns three learned vectors per transformer layer: $l_k$ (scales keys in attention), $l_v$ (scales values in attention), $l_{ff}$ (scales feedforward activations). The modified attention is $\text{Attn}(Q, l_k \odot K, l_v \odot V)$ where $\odot$ is element-wise multiplication. Trainable parameters per layer: $d_k + d_v + d_{ff}$ — typically 3 × 4096 = 12,288 for Llama-3-8B, versus LoRA's ~131K per layer. IA³ has roughly 10× fewer trainable parameters than LoRA at rank 16.

Prompt tuning's hypothesis: task-specific behavior can be encoded in the input representation, not the weights. Soft prompt tuning prepends $m$ learnable tokens (each of dimension $d_{model}$) to every input sequence. Trainable parameters: $m \times d_{model}$ — e.g., 100 tokens × 4096 dim = 409,600 parameters total, regardless of model depth. This is remarkably few. The catch: soft prompts are low-dimensional and cannot express complex domain adaptations at small model scales.

## Decompose It

### Phase 1 — Dataset Preparation and Evaluation Protocol

**The question this phase answers:** What is the train/eval/test split, and how do you ensure fair comparison across methods?

For a domain dataset of 10K instruction examples: 8K train, 1K eval (used during training for early stopping), 1K test (held out, used only for final ROUGE-L computation). The test set is never seen during hyperparameter selection.

Instruction format: use the standard Alpaca format with `<|begin_of_text|>`, `<|user|>`, `<|assistant|>` tokens for Llama-3. The input mask must mark only the response tokens for loss computation — the instruction (user message) is not supervised. Setting `ignore_index=-100` on instruction tokens prevents the model from learning to predict the instruction, which would inflate loss metrics and distort gradient signals.

**Key decisions:**
- Whether to truncate long examples or filter them: truncation at a fixed length (512 tokens) loses information from long legal or medical documents. Filtering (removing examples longer than 512 tokens) reduces dataset size. Prefer truncation with a warning log for examples > 2× the target length.
- How to ensure identical tokenization across methods: all methods use the same base tokenizer (Llama-3 BPE, vocabulary size 128,256). The tokenizer does not change with PEFT method — only the model weights change.
- Number of training steps: fix at 2000 steps with batch size 4 and gradient accumulation of 8 (effective batch size 32). This is identical for all methods. Training loss at step 2000 is directly comparable — no epoch-count differences.

**Trap:** Computing ROUGE-L on the training split to check for overfitting. ROUGE measures n-gram overlap with reference answers — it is not a generation-quality metric and should only be computed on the held-out test set. Using training ROUGE as an early stopping criterion leads to stopping when the model has memorized training examples.

**Done when:** Train/eval/test JSONLs are written. A `load_dataset` function returns tokenized and masked tensors. A `compute_rouge_l` function takes model + test_dataloader and returns a ROUGE-L F1 score. All functions are seeded with `torch.manual_seed(42)`.

### Phase 2 — LoRA Implementation and Rank Selection

**The question this phase answers:** How does rank $r$ control the expressivity of LoRA updates, and what is the right rank for domain adaptation vs. style transfer?

Rank $r$ is the single most important LoRA hyperparameter. It controls the dimension of the update subspace: $r = 1$ allows only rank-1 updates (a single direction in weight space), which can capture global bias shifts (e.g., "always respond formally") but not complex domain knowledge. $r = 64$ allows 64 orthogonal directions — sufficient to represent complex domain-specific patterns.

Practical guidelines:
- Style transfer (e.g., make all responses more concise): $r = 4$–$8$. The stylistic adaptation is a low-dimensional transformation of the output distribution.
- Domain adaptation (e.g., medical terminology, legal reasoning): $r = 16$–$64$. The model must learn a substantial new vocabulary and reasoning pattern.
- Task-specific fine-tuning (e.g., instruction following for a specific format): $r = 16$ is a strong default.

Which weight matrices to apply LoRA to: the original LoRA paper applies to $W_q$ and $W_v$ (query and value projections). Empirical work shows that adding $W_k$ (key), $W_o$ (output projection), and feedforward matrices $W_1$ and $W_2$ improves task performance at the cost of more trainable parameters. For Llama-3-8B, applying to all 6 matrices at $r = 16$ gives approximately 40M trainable parameters (0.5% of 8B).

The LoRA $\alpha$ parameter scales the updates: the effective update is $\Delta W = \frac{\alpha}{r} BA$. Setting $\alpha = 2r$ (e.g., $\alpha = 32$ for $r = 16$) is a common default that normalizes the update magnitude relative to rank.

**Key decisions:**
- LoRA dropout on $A$ and $B$ matrices: yes, use 0.05–0.1 to prevent the low-rank matrices from overfitting, especially on small datasets (< 10K examples).
- Whether to merge LoRA weights into base model after training: merging ($W' = W + \frac{\alpha}{r}BA$) eliminates inference overhead (no additional forward pass through $B$ and $A$). Do not merge during hyperparameter search — you need to reload the base model multiple times. Merge only for final inference latency measurement.

**Trap:** Not initializing $A$ with random Gaussian and $B$ with zeros. The convention is $B = 0$ at initialization so $\Delta W = BA = 0$ — the model starts training from the exact base model weights. If you initialize $B$ randomly, the model starts training from a corrupted set of weights, which increases training variance and can increase loss at early steps to well above the random initialization baseline.

**Done when:** `LoRALinear` layer wraps a frozen `nn.Linear`, adds B and A matrices, and outputs `frozen_linear(x) + (x @ A.T @ B.T) * alpha / r`. Parameter count is correct: `model.trainable_parameters()` returns `r * (d_in + d_out)` per LoRA layer.

### Phase 3 — QLoRA: Double Quantization and NF4

**The question this phase answers:** What does "quantizing the quantization constants" mean, and how does NF4 achieve better quantization than INT8 for normally distributed weights?

QLoRA combines three techniques: 4-bit NormalFloat (NF4) quantization of base model weights, double quantization of the quantization constants, and LoRA adapters trained in full precision (bfloat16) on top of the frozen NF4 base.

NF4 quantization: transformer weight values are approximately normally distributed $\mathcal{N}(0, \sigma^2)$. Standard INT4 uses uniform quantization levels $\{-8, -7, \ldots, 7\}$ regardless of the distribution. NF4 uses quantization levels placed at the quantiles of $\mathcal{N}(0, 1)$ — the levels are denser where the weight distribution is dense (near zero) and sparser at the tails. For the same 4-bit budget, NF4 achieves lower quantization error on normally distributed weights than INT4.

Quantization of a block: weights are quantized in blocks of size 64. Each block has a scale factor $c_1$ (the absmax of the block) that maps the block to $[-1, 1]$ for quantization. Each scale factor $c_1$ is itself stored in bfloat16 — 2 bytes per 64 weights. Double quantization: the set of scale factors $\{c_1\}$ across the model is itself quantized (as a second level) using INT8, with its own scale factor $c_2$. This saves approximately 0.5 bits per parameter — on an 8B model, that is about 0.5 GB of memory savings.

Memory calculation: NF4 stores 4 bits per weight. An 8B parameter model requires 8B × 4 bits / 8 = 4 GB for weights alone. Add LoRA adapters in bfloat16: 40M params × 2 bytes = 80 MB. Add activations, gradients on adapters, optimizer states on adapters: roughly 2–4 GB total overhead. QLoRA of Llama-3-8B runs comfortably in 10–12 GB VRAM — accessible on RTX 3090/4090.

**Key decisions:**
- Block size for quantization: 64 is the QLoRA default. Smaller blocks (32) give better quantization accuracy but require more memory for scale factors. Larger blocks (128) save memory but increase quantization error. The default 64 is well-characterized.
- Paged optimizer: if VRAM is insufficient for optimizer states, use `paged_adamw_8bit` from `bitsandbytes`. This pages optimizer states to CPU RAM during backward passes, with minimal performance cost.

**Trap:** Expecting QLoRA inference to be faster than standard LoRA inference. It is not — 4-bit weights must be dequantized to bfloat16 before matrix multiplication, adding a dequantization step per layer. QLoRA saves training-time memory at the cost of inference throughput (roughly 1.2–1.5× slower than bfloat16 inference for the same token). Report inference latency separately for QLoRA vs. LoRA (with merged weights).

**Done when:** `peak_memory_gb` during QLoRA training of Llama-3-8B is under 14 GB. Weight quantization is verified: `model.model.layers[0].self_attn.q_proj.weight.dtype == torch.uint8` (NF4 stored as uint8 with a special quant_type flag in bitsandbytes). LoRA adapters on top of the quantized model have `dtype == torch.bfloat16`.

### Phase 4 — IA³ Implementation and Why It Cannot Merge

**The question this phase answers:** Why are IA³ learned vectors not mergeable into the base model weights in the same way LoRA matrices are, and what are the practical consequences?

IA³ (Infused Adapter by Inhibiting and Amplifying Inner Activations) modifies the computation path by element-wise scaling, not weight additive updates. Consider the key projection: $K = XW_k$. IA³ produces $\tilde{K} = l_k \odot K = l_k \odot (XW_k)$. This is equivalent to $\tilde{K} = X(W_k \odot l_k^T)$ — you can absorb the element-wise scaling into the weight matrix by scaling each column of $W_k$ by the corresponding $l_k$ element. So IA³ CAN be merged into $W_k$ and $W_v$, but not into $W_{ff}$.

The feedforward activation scaling: $h_{ff} = l_{ff} \odot \text{GELU}(XW_1)$. This cannot be merged because the scaling is applied after the nonlinearity. $l_{ff} \odot \text{GELU}(XW_1) \neq \text{GELU}(X(W_1 \odot l_{ff}^T))$ — the nonlinearity breaks the distributive property. This means IA³ adapters must remain as separate vectors during inference for the feedforward layers, adding a small but non-zero inference overhead.

The consequence for inference latency: IA³ has essentially zero inference overhead for the attention scaling (mergeable) but a small overhead for feedforward scaling (~3 element-wise multiplications per layer, negligible in practice). The practical reason IA³ is competitive in efficiency is not zero overhead — it's that the adapter parameter count is so small (12K params per layer vs. 130K for LoRA) that loading, saving, and swapping adapters is nearly instantaneous. In multi-tenant serving where you need to switch between many task-specific adapters, IA³ minimizes adapter swap latency.

**Key decisions:**
- Which layers to apply IA³ to: all transformer layers is standard. You can selectively apply to top layers only (final N/2 layers) for an extreme parameter reduction, but empirical results show per-layer IA³ is necessary for domain adaptation tasks.
- Learning rate for IA³: IA³ vectors benefit from a higher learning rate than LoRA (LoRA: 3e-4, IA³: 3e-3) because updates are simpler (element-wise not low-rank) and the optimization landscape is smoother.

**Trap:** Initializing IA³ vectors to zero. IA³ vectors must be initialized to all-ones, not zeros. Initializing to ones means $l_k \odot K = K$ at initialization — the model starts from the exact base model computation. Initializing to zeros produces $l_k \odot K = \mathbf{0}$, destroying all information in the keys at step 0 and causing training to diverge.

**Done when:** An `IA3Layer` module wraps a frozen linear layer and applies learned scale vectors to the output. `model.trainable_parameters()` for IA³ on Llama-3-8B returns approximately 600K–1M parameters (depends on architecture specifics). Training loss at step 100 is comparable to LoRA training loss at step 100 (within 10%) — if it is much higher, check the initialization.

### Phase 5 — Prompt Tuning and Why It Fails at Small Scale

**The question this phase answers:** What is the theoretical explanation for why soft prompt tuning underperforms at model sizes below 10B parameters?

Soft prompt tuning prepends $m$ learnable embedding vectors to the input. The model processes `[soft_prompt_1, ..., soft_prompt_m, token_1, ..., token_T]`. Only the $m \times d_{model}$ soft prompt parameters are updated during training; all model weights are frozen.

The theoretical limitation: soft prompts operate in the model's input embedding space, which is $d_{model}$-dimensional. For Llama-3-8B, $d_{model} = 4096$. The soft prompt occupies a subspace of $\mathbb{R}^{m \times 4096}$ in the first layer's attention. However, this subspace is constrained by what the frozen model's attention mechanism can express: the model can only attend to and process the soft prompt tokens through the same mechanisms it uses for all tokens. If the model has not been trained to understand abstract, uninterpretable "concepts" encoded in arbitrary 4096-dimensional vectors, the soft prompt cannot steer behavior in complex ways.

At 10B+ parameters, the model's attention mechanism is rich enough that soft prompts can consistently activate specific behavior patterns. At 7–8B parameters (Llama-3-8B), soft prompts can capture simple formatting or simple style changes but cannot reliably induce complex domain knowledge. The empirical evidence: prompt tuning matches fine-tuning at 11B+ parameters but trails by 10–20 ROUGE points at 7–8B.

Number of soft prompt tokens $m$: larger $m$ gives the soft prompt more expressivity but increases input sequence length (reducing effective context window). $m = 100$ tokens is the recommendation from the original paper; $m = 10$ is too small for complex tasks; $m = 200$ provides minimal additional benefit over $m = 100$ while consuming 100 extra context tokens per inference.

**Key decisions:**
- Whether to initialize soft prompts from existing vocabulary embeddings vs. random: initializing from the embeddings of a task-relevant vocabulary (e.g., medical terms for a medical QA task) provides a better starting point than random initialization, reducing training time by 20–30%.
- Length of training for prompt tuning: prompt tuning requires more training steps than LoRA to converge (3000–5000 steps vs. 1000–2000 for LoRA) because the optimization landscape for soft prompts is harder — the gradients must propagate through the entire frozen model to update just the soft prompt parameters.

**Trap:** Using prompt tuning for a small dataset (< 2K examples). Soft prompts have few parameters and tend to overfit to small datasets in a degenerate way: the model learns to ignore the soft prompt after training because the loss gradient through the soft prompt is too noisy on small datasets. With < 2K examples, LoRA at r=4 or IA³ will significantly outperform prompt tuning.

**Done when:** Soft prompt parameters are registered as `nn.Parameter` and are the only parameters with `requires_grad=True`. Inference with soft prompts prepends the learned vectors to the input embeddings before the first attention layer. `model.trainable_parameters()` returns exactly $m \times d_{model}$ parameters.

## Edge Cases to Handle Before You Code

1. **Gradient checkpointing interaction with PEFT:** Gradient checkpointing recomputes activations during backward pass instead of storing them, reducing activation memory at the cost of ~30% more compute. For QLoRA (which already compresses weights), gradient checkpointing is almost always necessary to fit training in 24GB. However, gradient checkpointing interacts differently with each PEFT method: for prompt tuning, the soft prompt must not be checkpointed (it is always small), while for LoRA the adapter matrices should not be checkpointed (they are also small). Apply gradient checkpointing only to the frozen base model layers.

2. **Double-counting parameters in IA³ when scales are transposed:** IA³ scales keys and values in attention. If your implementation applies the scale to $W_k$ output (shape $B \times T \times d_k$) via a learned vector of shape $d_k$, and separately to $W_v$ output, the parameter count is $2 \times d_k$ per attention head multiplied by $n_{heads}$. For Llama-3-8B with 32 layers and 32 attention heads of dimension 128: $32 \times 2 \times 128 = 8192$ params per layer × 32 layers = 262K params for attention scaling alone. Verify this matches your implementation's `trainable_parameters()` output.

3. **Prompt tuning with variable-length inputs and max_length truncation:** The soft prompt adds $m$ tokens to every input. If you set `max_length=512` in the tokenizer, the effective input capacity is $512 - m = 412$ tokens for $m = 100$. Long examples will be truncated more aggressively than with other PEFT methods. This creates an unfair comparison: prompt tuning sees shorter inputs. Either reduce max_length for all methods (to match prompt tuning's effective length) or account for this in the evaluation by reporting average input length per method.

4. **Seed instability in multi-GPU evaluation:** When computing ROUGE-L across the 1K test set, the generation order may differ if distributed evaluation distributes examples across GPUs differently. Use `dataloader_drop_last=False` and sort the test set by index before computing ROUGE to ensure reproducible generation order regardless of the number of evaluation GPUs used.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|---------|---------|----------------|
| LoRA rank $r$ | r=16 (standard) | r=64 (high capacity) | r=16 for style/format adaptation; r=64 for complex domain adaptation with > 20K training examples |
| QLoRA precision | NF4 4-bit base + bfloat16 adapters | INT8 8-bit base + bfloat16 adapters | NF4 for maximum memory savings (24GB GPU); INT8 when quality degradation from 4-bit is unacceptable and 40GB GPU is available |
| IA³ vs LoRA for production | IA³ (smaller, faster swap) | LoRA (higher quality) | IA³ when serving 100+ task-specific adapters simultaneously (adapter swap latency matters); LoRA when quality delta is unacceptable |
| Prompt tuning token count $m$ | m=100 (standard) | m=20 (minimal) | m=100 for task complexity; m=20 only when input length budget is extremely tight (e.g., long-document tasks where 100 extra tokens significantly reduces context) |

## What Makes This Non-Trivial

The comparison is not just about ROUGE-L scores — it is about characterizing the Pareto frontier of quality, memory, and inference speed. A method that achieves 95% of full fine-tuning quality with 0.5% of the trainable parameters and 2× faster inference is strictly better for deployment, even if its ROUGE-L is 1 point lower in absolute terms. The benchmark must capture this multi-dimensional trade-off to be useful.

The double quantization in QLoRA is subtle in its memory savings calculation. It is not 0.5 bits/parameter applied uniformly — it is 0.5 bits/parameter only on the scale factors, which themselves occupy a small fraction of total memory. The practical saving is approximately 0.37 bits/parameter on the total model, not 0.5. Anyone who reports "QLoRA saves 0.5 bits/param" without this nuance has not implemented double quantization and checked the actual numbers.

The IA³ mergeability question has no simple "yes/no" answer — the attention scaling vectors can be merged, the feedforward scaling cannot. This matters for deployment: if you're building a multi-tenant adapter system, IA³ gives you a partially-merged model (attention layers are merged for free, feedforward layers require adapter vectors at runtime). Understanding this hybrid nature distinguishes a deep understanding of IA³ from surface-level familiarity.

## Interview Angles This Project Develops

1. **"What is the intuition behind LoRA's low-rank assumption?"** — Strong answer: empirical evidence (Aghajanyan et al. 2020) shows that task-specific weight updates during full fine-tuning have low intrinsic rank — the updates cluster in a low-dimensional subspace even when the full weight matrix is updated. LoRA makes this explicit and constrains updates to rank-$r$ from the start, reducing parameters while preserving most of the task-relevant update direction. The intrinsic dimensionality of the task (not the model size) determines the optimal $r$.

2. **"Why does QLoRA achieve the same quality as LoRA despite 4-bit quantization?"** — Strong answer: NF4 quantization error on normally distributed weights is approximately $\sigma^2 / 3 \cdot 2^{-2b}$ where $b$ is the bit width — about 0.4% relative error for 4-bit on normally distributed weights. The LoRA adapters, trained in bfloat16, exactly compensate for this quantization error by learning to produce outputs that account for the base model's slight shift in values. The key insight: the LoRA adapters learn on top of the quantized model, not the full-precision model, so they implicitly learn to correct quantization noise.

3. **"When would you choose prompt tuning over LoRA?"** — Strong answer: only when (1) the base model is large enough (≥ 11B parameters) for soft prompts to be expressive, (2) you need to support a very large number of tasks simultaneously (prompt tuning has trivially small adapter size — just 409K parameters — so you can store 1000 task adapters in the RAM occupied by a single LoRA adapter set), and (3) training data is very limited (< 500 examples) where LoRA's slightly larger parameter count causes more overfitting. For any other scenario, LoRA or IA³ is preferred.

4. **"How does catastrophic forgetting manifest differently in LoRA vs. full fine-tuning, and why?"** — Strong answer: full fine-tuning updates all 8B weights, pulling the model's representations toward the domain distribution and away from the general pretraining distribution. Catastrophic forgetting is severe and measurable: MMLU performance (general knowledge benchmark) drops 5–15 points after aggressive full fine-tuning. LoRA constrains updates to a rank-16 subspace of weight space, leaving 99.99% of each weight matrix's structure unchanged. The base model's general knowledge is preserved in the null space of the low-rank update. This is why LoRA models maintain near-baseline performance on general benchmarks while achieving strong domain adaptation.

5. **"How would you serve 500 different LoRA adapters for 500 different customer tenants?"** — Strong answer: use a batched serving architecture where the base model weights are loaded once in shared GPU memory. Each request includes a tenant ID that maps to an adapter ID. At inference time, load the relevant LoRA matrices (B and A for each layer) from a fast storage layer (NVMe SSD), add them to the base weight during the forward pass. For high-throughput serving, pre-load the most frequently used adapters in GPU memory and cache less-used ones on CPU. Tools like vLLM with LoRA support and Punica (multi-LoRA inference) implement this batching at the CUDA kernel level, enabling near-zero overhead per-request adapter switching.

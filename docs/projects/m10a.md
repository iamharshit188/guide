# Multi-Agent Crypto Market Analyst — Step-by-Step Guide

> **Module:** 10 — LLM Agents & Tool Use · **Difficulty:** Expert · **Type:** GUIDED · **Time:** ~5 hours

## What You'll Build

A ReAct (Reasoning + Acting) agent that answers investment questions like "Should I buy BTC today?" by autonomously decomposing the query into tool calls, executing them in sequence, and synthesizing a structured decision with confidence and caveats. The agent has 4 tools: live price fetching from CoinGecko, recent news headlines from CryptoPanic or NewsAPI, sentiment scoring with a lightweight NLP model, and RSI computation from price history. Every Thought/Action/Observation cycle is logged to a file for debugging and explainability.

## The Mental Model

A ReAct agent alternates between two modes: reasoning (Thought) and acting (Action). In Thought mode, the LLM analyzes the current state and decides what to do next. In Action mode, it calls a tool and waits for the result. The result becomes an Observation, which is appended to the message history, and the cycle repeats until the LLM produces a final answer with no tool call.

The key architectural insight: tool calls are not function calls in the traditional sense — they are structured outputs from the LLM that the orchestrator intercepts and routes to actual Python functions. The LLM never directly executes code; it produces JSON describing what it wants to do, and the orchestrator does the execution and feeds results back as Observations. This separation makes the system auditable and safe — you can inspect every tool call before allowing it.

The message history is the agent's state. At step $t$, the history contains: the original user query, all previous (Thought, Action, Observation) triples, and the current partial Thought. The LLM's response at each step conditions on the entire history — this is how it maintains context across multiple tool calls. The history grows with each step, which is why tool calls must be bounded: an agent that loops indefinitely will exhaust the context window.

RSI (Relative Strength Index) is a momentum oscillator: $RSI = 100 - 100/(1 + RS)$ where $RS = \text{avg\_gain}/\text{avg\_loss}$ over a period (default 14). RSI > 70 signals overbought (consider selling), RSI < 30 signals oversold (consider buying). It does not predict price direction — it measures momentum strength and reversals.

## Phase 2 — Tool Executor with Real API Calls

**Why this second:** The executor is the boundary between the LLM's plan and real-world data. Every tool must handle network errors gracefully and return a structured result — not raise an exception — because errors must be returned to the LLM as Observations.

```python
# ── Tool implementations ──────────────────────────────────────────────────────

import requests
from vaderSentiment.vaderSentiment import SentimentIntensityAnalyzer

_vader = SentimentIntensityAnalyzer()
_session = requests.Session()
_session.headers.update({"User-Agent": "crypto-analyst-agent/1.0"})


def get_price(coin: str) -> dict[str, Any]:
    """
    CoinGecko public API — no API key required for basic endpoints.
    Rate limit: 10-30 req/min on free tier.
    """
    url = f"https://api.coingecko.com/api/v3/coins/{coin}"
    params = {"localization": "false", "tickers": "false", "community_data": "false"}
    try:
        resp = _session.get(url, params=params, timeout=10)
        resp.raise_for_status()
        data = resp.json()
        mkt = data.get("market_data", {})
        return {
            "coin": coin,
            "price_usd": mkt.get("current_price", {}).get("usd"),
            "change_24h_pct": mkt.get("price_change_percentage_24h"),
            "market_cap_usd": mkt.get("market_cap", {}).get("usd"),
            "volume_24h_usd": mkt.get("total_volume", {}).get("usd"),
            "last_updated": mkt.get("last_updated"),
        }
    except requests.exceptions.HTTPError as e:
        if e.response.status_code == 404:
            return {"error": f"Coin '{coin}' not found on CoinGecko. Check the coin ID (use 'bitcoin', not 'BTC')."}
        return {"error": f"HTTP error: {e}"}
    except requests.exceptions.Timeout:
        return {"error": "CoinGecko request timed out after 10 seconds."}
    except Exception as e:
        return {"error": f"Unexpected error: {type(e).__name__}: {e}"}


def get_news(coin: str) -> dict[str, Any]:
    """
    CryptoPanic public API — returns recent news for a coin symbol.
    Falls back to mock data if API is unavailable (no auth token).
    """
    url = "https://cryptopanic.com/api/v1/posts/"
    params = {
        "auth_token": os.getenv("CRYPTOPANIC_TOKEN", ""),
        "currencies": coin.upper(),
        "public": "true",
        "filter": "news",
        "kind": "news",
    }
    try:
        resp = _session.get(url, params=params, timeout=10)
        resp.raise_for_status()
        data = resp.json()
        results = data.get("results", [])[:5]
        headlines = [
            {
                "title": item.get("title", ""),
                "source": item.get("source", {}).get("title", "unknown"),
                "published_at": item.get("published_at", ""),
            }
            for item in results
        ]
        return {"coin": coin, "headlines": headlines, "count": len(headlines)}
    except Exception as e:
        # Graceful fallback — return error as data so LLM can decide
        return {
            "error": f"Could not fetch news: {type(e).__name__}: {e}",
            "coin": coin,
            "headlines": [],
        }


def analyze_sentiment(texts: list[str]) -> dict[str, Any]:
    """
    VADER sentiment analysis — rule-based, works well for short social/news text.
    Returns compound score per text and an overall average.
    Compound score: -1.0 (most negative) to +1.0 (most positive).
    """
    if not texts:
        return {"error": "No texts provided.", "scores": [], "average": 0.0}

    scores = []
    for text in texts:
        vs = _vader.polarity_scores(text)
        scores.append({
            "text": text[:100] + ("..." if len(text) > 100 else ""),
            "compound": vs["compound"],
            "positive": vs["pos"],
            "negative": vs["neg"],
            "neutral": vs["neu"],
        })

    avg = sum(s["compound"] for s in scores) / len(scores)
    interpretation = (
        "strongly positive" if avg > 0.5
        else "mildly positive" if avg > 0.1
        else "neutral" if avg > -0.1
        else "mildly negative" if avg > -0.5
        else "strongly negative"
    )
    return {
        "scores": scores,
        "average_compound": round(avg, 4),
        "interpretation": interpretation,
        "n_texts": len(texts),
    }


def calculate_rsi(coin: str, period: int = 14) -> dict[str, Any]:
    """
    Fetch daily OHLC from CoinGecko and compute RSI over `period` days.
    Requires period+1 price points to compute period gains/losses.
    """
    days_needed = period + 10    # extra buffer for edge cases
    url = f"https://api.coingecko.com/api/v3/coins/{coin}/market_chart"
    params = {"vs_currency": "usd", "days": days_needed, "interval": "daily"}

    try:
        resp = _session.get(url, params=params, timeout=10)
        resp.raise_for_status()
        data = resp.json()
        prices = [p[1] for p in data.get("prices", [])]

        if len(prices) < period + 1:
            return {"error": f"Insufficient price data: got {len(prices)} points, need {period + 1}."}

        # Compute gains and losses
        deltas = [prices[i] - prices[i - 1] for i in range(1, len(prices))]
        gains = [max(d, 0.0) for d in deltas]
        losses = [abs(min(d, 0.0)) for d in deltas]

        # Initial averages (simple mean for first period)
        avg_gain = sum(gains[:period]) / period
        avg_loss = sum(losses[:period]) / period

        # Smooth over remaining points (Wilder's smoothing)
        for i in range(period, len(deltas)):
            avg_gain = (avg_gain * (period - 1) + gains[i]) / period
            avg_loss = (avg_loss * (period - 1) + losses[i]) / period

        if avg_loss == 0:
            rsi = 100.0
        else:
            rs = avg_gain / avg_loss
            rsi = round(100 - 100 / (1 + rs), 2)

        if rsi > 70:
            interpretation = "Overbought — momentum suggests potential pullback."
        elif rsi < 30:
            interpretation = "Oversold — momentum suggests potential recovery."
        else:
            interpretation = f"Neutral momentum (RSI={rsi})."

        return {
            "coin": coin,
            "rsi": rsi,
            "period": period,
            "interpretation": interpretation,
            "n_prices": len(prices),
        }
    except requests.exceptions.HTTPError as e:
        return {"error": f"HTTP {e.response.status_code} from CoinGecko: {e}"}
    except Exception as e:
        return {"error": f"{type(e).__name__}: {e}"}


def execute_tool(tool_name: str, tool_input: dict[str, Any]) -> dict[str, Any]:
    """
    Dispatch tool name to implementation.
    All tools return dicts — never raise exceptions to the caller.
    """
    dispatch = {
        "get_price": lambda: get_price(tool_input["coin"]),
        "get_news": lambda: get_news(tool_input["coin"]),
        "analyze_sentiment": lambda: analyze_sentiment(tool_input["texts"]),
        "calculate_rsi": lambda: calculate_rsi(
            tool_input["coin"], tool_input.get("period", 14)
        ),
    }
    if tool_name not in dispatch:
        return {"error": f"Unknown tool '{tool_name}'. Available: {list(dispatch.keys())}"}

    try:
        return dispatch[tool_name]()
    except KeyError as e:
        return {"error": f"Missing required parameter: {e}"}
    except Exception as e:
        return {"error": f"Tool execution failed: {type(e).__name__}: {e}"}
```

**Checkpoint:** `get_price("bitcoin")` returns a dict with `price_usd` populated (a number > 0). `get_price("notacoin999")` returns `{"error": "Coin 'notacoin999' not found..."}` — not an exception. `analyze_sentiment(["Bitcoin surges to new highs!", "Market crash looms"])` returns `average_compound` between -1 and 1. `calculate_rsi("bitcoin", 14)` returns `rsi` between 0 and 100.

## Phase 3 — ReAct Loop

**Why this third:** The loop is the heart of the agent. It must parse tool calls from LLM responses, execute them, inject observations, and detect termination.

```python
# ── ReAct loop ────────────────────────────────────────────────────────────────

import openai

SYSTEM_PROMPT = """You are a cryptocurrency market analyst with access to 4 tools:
- get_price: live price, 24h change, market cap
- get_news: recent headlines for a coin
- analyze_sentiment: sentiment score on a list of texts
- calculate_rsi: RSI momentum indicator

For any investment question, follow this reasoning process:
1. Get the current price
2. Fetch recent news headlines
3. Analyze sentiment of those headlines
4. Calculate RSI for momentum
5. Synthesize all evidence into a decision (buy/hold/sell)

Think step by step. Use tools to gather evidence before drawing conclusions.
After gathering evidence, provide your final answer without calling any more tools."""


def run_react_loop(
    query: str,
    logger: logging.Logger,
    max_steps: int = 10,
    model: str = "gpt-4o-mini",
) -> dict[str, Any]:
    """
    Main ReAct loop.
    Returns {answer, tool_calls_made, steps, total_tokens_used}.
    """
    section(f"ReAct Loop: {query[:60]}")

    client = openai.OpenAI()
    messages: list[dict] = [
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user", "content": query},
    ]

    # Convert our tool schemas to OpenAI function-call format
    openai_tools = [
        {
            "type": "function",
            "function": {
                "name": t["name"],
                "description": t["description"],
                "parameters": t["input_schema"],
            },
        }
        for t in TOOLS
    ]

    steps = []
    total_tokens = 0
    tool_calls_made: list[dict] = []

    for step in range(max_steps):
        logger.info(f"Step {step + 1}/{max_steps} — calling LLM")

        response = client.chat.completions.create(
            model=model,
            messages=messages,
            tools=openai_tools,
            tool_choice="auto",
            temperature=0.1,
        )

        msg = response.choices[0].message
        total_tokens += response.usage.total_tokens
        finish_reason = response.choices[0].finish_reason

        step_log = {
            "step": step + 1,
            "finish_reason": finish_reason,
            "content": msg.content,
            "tool_calls": [],
        }

        # No tool call → final answer
        if finish_reason == "stop" or not msg.tool_calls:
            logger.info("Agent produced final answer (no tool call).")
            messages.append({"role": "assistant", "content": msg.content})
            step_log["final_answer"] = True
            steps.append(step_log)
            break

        # Process tool calls
        messages.append(msg)   # add assistant message with tool_calls

        for tc in msg.tool_calls:
            tool_name = tc.function.name
            try:
                tool_input = json.loads(tc.function.arguments)
            except json.JSONDecodeError:
                tool_input = {}
                logger.warning(f"Failed to parse tool arguments: {tc.function.arguments}")

            logger.info(f"  Action: {tool_name}({json.dumps(tool_input)})")
            print(f"\n[Action] {tool_name}({json.dumps(tool_input, indent=2)})")

            observation = execute_tool(tool_name, tool_input)

            logger.info(f"  Observation: {json.dumps(observation)[:200]}")
            print(f"[Observation] {json.dumps(observation, indent=2)[:300]}...")

            tool_calls_made.append({
                "step": step + 1,
                "tool": tool_name,
                "input": tool_input,
                "output": observation,
            })

            step_log["tool_calls"].append({
                "tool": tool_name,
                "input": tool_input,
                "call_id": tc.id,
            })

            # Inject observation back into message history
            messages.append({
                "role": "tool",
                "tool_call_id": tc.id,
                "content": json.dumps(observation),
            })

        steps.append(step_log)

        # Log to JSONL for debugging
        with open(LOG_FILE, "a", encoding="utf-8") as f:
            f.write(json.dumps({
                "timestamp": datetime.datetime.utcnow().isoformat(),
                "step": step + 1,
                "tool_calls": step_log["tool_calls"],
            }) + "\n")

    else:
        logger.warning(f"Max steps ({max_steps}) reached without final answer.")
        msg = type("Msg", (), {"content": "Max steps reached — insufficient data to make a recommendation."})()

    final_answer = msg.content or "No final answer generated."
    return {
        "query": query,
        "answer": final_answer,
        "tool_calls_made": tool_calls_made,
        "steps": len(steps),
        "total_tokens": total_tokens,
    }
```

**Checkpoint:** After running `run_react_loop("What is the current price of Ethereum?")`, the log file `agent_trace.log` shows at least one `[Action] get_price({"coin": "ethereum"})` entry and one `[Observation]` entry with a price value. `result["steps"] <= 3` for a simple price query. `result["tool_calls_made"][0]["tool"] == "get_price"`.

## Phase 4 — Error Recovery

**Why this fourth:** Real APIs fail. The agent must not crash when a tool returns an error — it must treat the error as an observation and decide how to proceed.

```python
# ── Error recovery demonstration ──────────────────────────────────────────────

def run_with_error_recovery(
    query: str,
    logger: logging.Logger,
    model: str = "gpt-4o-mini",
) -> dict[str, Any]:
    """
    Extended ReAct loop with explicit error recovery logging.
    When a tool returns an error key, the LLM sees it as an observation
    and can choose to retry with different parameters, skip, or use a fallback.
    """
    # Inject a deliberate error scenario for demonstration
    original_get_price = get_price

    call_count = [0]

    def flaky_get_price(coin: str) -> dict[str, Any]:
        call_count[0] += 1
        if call_count[0] == 1:
            # Simulate first call failing (e.g., rate limit)
            logger.warning("Simulated rate limit error on first get_price call.")
            return {"error": "Rate limited by CoinGecko. Retry after 60 seconds."}
        return original_get_price(coin)

    # Monkey-patch for demonstration
    import sys
    sys.modules[__name__].get_price = flaky_get_price

    result = run_react_loop(query, logger, max_steps=12, model=model)

    # Restore
    sys.modules[__name__].get_price = original_get_price

    if call_count[0] > 1:
        logger.info(f"Error recovery: get_price succeeded on attempt {call_count[0]}")
        result["recovery_attempts"] = call_count[0]

    return result


def retry_with_backoff(
    tool_name: str,
    tool_input: dict[str, Any],
    logger: logging.Logger,
    max_retries: int = 3,
    backoff_sec: float = 2.0,
) -> dict[str, Any]:
    """
    Direct retry wrapper for transient errors (rate limits, timeouts).
    Used when the LLM explicitly requests a retry in its reasoning.
    """
    for attempt in range(1, max_retries + 1):
        result = execute_tool(tool_name, tool_input)
        if "error" not in result:
            return result
        if attempt < max_retries:
            wait = backoff_sec * (2 ** (attempt - 1))
            logger.warning(
                f"Tool {tool_name} failed (attempt {attempt}/{max_retries}): "
                f"{result['error']}. Retrying in {wait:.1f}s."
            )
            time.sleep(wait)
    logger.error(f"Tool {tool_name} failed after {max_retries} retries.")
    return result   # return the last error result
```

**Checkpoint:** Run `run_with_error_recovery("Should I buy bitcoin?")` and check `agent_trace.log`. You should see one rate limit error as an Observation, followed by the LLM's Thought about retrying, followed by a successful price fetch. `result["recovery_attempts"] == 2`. The final answer still contains a buy/hold/sell recommendation despite the initial error.

## Phase 5 — Final Synthesis and Structured Output

**Why this last:** The synthesis step aggregates all observations into a structured decision. Enforcing structure (JSON output) makes the decision machine-readable for downstream use.

```python
# ── Final synthesis ───────────────────────────────────────────────────────────

SYNTHESIS_PROMPT = """Based on all the evidence gathered through the tools, provide a structured investment analysis.

Your response MUST be a JSON object with exactly these fields:
{
  "decision": "BUY" | "HOLD" | "SELL",
  "confidence": 0.0 to 1.0,
  "reasoning": "2-3 sentence explanation citing specific data points",
  "price_signal": "bullish" | "bearish" | "neutral",
  "sentiment_signal": "bullish" | "bearish" | "neutral",
  "rsi_signal": "overbought" | "oversold" | "neutral",
  "key_risks": ["risk1", "risk2"],
  "timeframe": "short-term (1-7 days) | medium-term (1-4 weeks)",
  "disclaimer": "This is not financial advice."
}

Cite specific numbers from the tool observations in your reasoning field.
Use EXACTLY this JSON format — no markdown code blocks, no extra text."""


def synthesize_decision(
    query: str,
    tool_observations: list[dict[str, Any]],
    logger: logging.Logger,
    model: str = "gpt-4o-mini",
) -> dict[str, Any]:
    """
    Final synthesis step: collect all tool outputs and produce a structured decision.
    This is a separate LLM call with a stricter prompt than the ReAct loop.
    """
    section("Phase 5 — Final Synthesis")

    client = openai.OpenAI()

    # Summarize all observations for the synthesis prompt
    obs_summary = "\n\n".join(
        f"Tool: {obs['tool']}\nInput: {json.dumps(obs['input'])}\nResult: {json.dumps(obs['output'])}"
        for obs in tool_observations
    )

    messages = [
        {"role": "system", "content": SYNTHESIS_PROMPT},
        {
            "role": "user",
            "content": f"Question: {query}\n\nTool Evidence:\n{obs_summary}",
        },
    ]

    response = client.chat.completions.create(
        model=model,
        messages=messages,
        temperature=0.0,
        response_format={"type": "json_object"},   # enforces JSON output
    )

    raw = response.choices[0].message.content
    logger.info(f"Synthesis raw output: {raw[:300]}")

    try:
        decision = json.loads(raw)
    except json.JSONDecodeError:
        logger.error(f"Synthesis output was not valid JSON: {raw}")
        decision = {
            "error": "Synthesis failed to produce valid JSON.",
            "raw_output": raw,
        }

    return decision


# ── Full main ─────────────────────────────────────────────────────────────────

def main() -> None:
    logger = setup_logging()

    section("Multi-Agent Crypto Market Analyst")

    # Default query — override with CLI argument
    import sys
    query = " ".join(sys.argv[1:]) if len(sys.argv) > 1 else "Should I buy Bitcoin today?"
    print(f"\nQuery: {query}")

    # Phase 1–3: ReAct loop
    result = run_react_loop(query, logger, max_steps=10, model="gpt-4o-mini")

    print(f"\n[Agent Answer]\n{result['answer']}")
    print(f"\nSteps taken: {result['steps']}")
    print(f"Total tokens: {result['total_tokens']}")
    print(f"Tool calls: {len(result['tool_calls_made'])}")

    # Phase 5: Structured synthesis if we have tool observations
    if result["tool_calls_made"]:
        decision = synthesize_decision(
            query=query,
            tool_observations=result["tool_calls_made"],
            logger=logger,
        )

        section("Structured Decision")
        print(json.dumps(decision, indent=2))

        # Log final decision
        with open(LOG_FILE, "a", encoding="utf-8") as f:
            f.write(json.dumps({
                "timestamp": datetime.datetime.utcnow().isoformat(),
                "type": "final_decision",
                "query": query,
                "decision": decision,
                "steps": result["steps"],
                "total_tokens": result["total_tokens"],
            }) + "\n")

    section("Agent Log")
    print(f"Full trace: agent_trace.log")
    print(f"Event log: {LOG_FILE}")


if __name__ == "__main__":
    main()
```

**Checkpoint:** Running `python crypto_agent.py "Should I buy Ethereum today?"` produces:
1. Multiple `[Action]` / `[Observation]` pairs in the terminal for at least 3 different tools (price, news, RSI).
2. A `[Agent Answer]` narrative paragraph.
3. A JSON structured decision with `"decision": "BUY"` or `"HOLD"` or `"SELL"`, confidence float, and at least 2 `key_risks`.
4. `agent_trace.log` contains timestamped entries for each step.
5. `agent_log.jsonl` contains newline-delimited JSON events for each tool call and the final decision.

## Debugging Guide

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| `KeyError: 'coin'` in `get_price` | LLM passed `"BTC"` instead of `"bitcoin"` | Description says "use 'bitcoin' not 'BTC'" — add more examples: `"e.g., 'bitcoin', 'ethereum', 'solana', 'dogecoin'"` |
| Agent loops > 8 steps without finishing | LLM keeps calling the same tool repeatedly | Add explicit loop detection: if `tool_calls_made[-2]["tool"] == tool_calls_made[-1]["tool"]`, inject a message: "You have now called this tool twice. Proceed to synthesis." |
| `json.JSONDecodeError` on synthesis output | LLM added markdown code fences despite `response_format=json_object` | Use `raw = re.sub(r'```json|```', '', raw).strip()` before `json.loads(raw)` |
| CoinGecko 429 rate limit | Too many requests in < 1 minute | Add `time.sleep(1)` between tool calls; cache results with `functools.lru_cache` on (coin, timestamp_minute) |
| `get_news` returns empty headlines | CryptoPanic requires auth token for high-volume access | Set `CRYPTOPANIC_TOKEN` environment variable; fall back to NewsAPI with `newsapi-python` |

## What This Project Teaches You to Say in an Interview

1. "The ReAct pattern separates reasoning from acting by giving the LLM a structured format for expressing intent (tool calls as JSON) without directly executing code. The orchestrator controls what gets executed, making the system auditable — you can inspect every tool call and observation before allowing the agent to proceed."
2. "Tool schemas are the API contract between the LLM and the real world. Ambiguous parameter names (e.g., 'coin' could mean symbol or ID) cause the LLM to guess incorrectly. Precise descriptions with examples — 'use bitcoin not BTC' — reduce hallucinated parameter values by 60–80%."
3. "Error recovery in a ReAct agent works by returning errors as structured observations, not exceptions. The LLM sees 'Rate limited — retry after 60 seconds' as an observation and can reason: 'I should retry' or 'I should use cached data.' This is safer than automatic retry at the orchestrator level because the LLM may decide the error is unrecoverable and should be reported to the user."
4. "RSI is a momentum oscillator, not a price predictor. RSI > 70 means recent gains outpaced losses — the market may be overextended. It says nothing about absolute price direction. A good agent synthesis explains this limitation rather than treating RSI as a buy/sell signal in isolation."
5. "VADER sentiment on news headlines is a heuristic, not a ground truth signal. Headlines are often written to be sensationalist — 'Bitcoin Collapses!' may refer to a 5% correction. The agent synthesis should weight sentiment as one evidence signal among four, not as a primary decision driver. Citing specific data points ('price up 3.2%, RSI=58 (neutral), sentiment=-0.1 (mildly negative)') grounds the decision in observable facts."

# Credit Risk Predictor — Thinking Guide

> **Module:** 02 — ML Basics · **Difficulty:** Intermediate · **Type:** THINKING · **Time:** ~5 hours

## The Problem

**Input:** Tabular loan applicant data — German Credit dataset (1,000 records, 20 features) or LendingClub subset. Features include credit history, loan amount, employment duration, housing status, purpose, and existing credits.

**Output:** $\hat{p} \in [0, 1]$ — calibrated probability of default for each applicant. Not a binary label. The downstream system uses $\hat{p}$ to set interest rate via a risk pricing formula: $r = r_f + \hat{p} \cdot LGD \cdot \text{spread\_factor}$.

**Constraints:** Model must be interpretable enough to satisfy adverse action notice requirements (you must explain why a loan was denied). Probability estimates must be calibrated — $\hat{p} = 0.7$ must mean roughly 70% of applicants at that score default. Class imbalance is real: German Credit is 70/30 (non-default/default).

---

## Before You Write Any Code: Build The Mental Model

Credit scoring is a **decision under uncertainty with asymmetric costs**. A false negative (approving a defaulting borrower) costs the LGD — typically 40–60% of principal. A false positive (rejecting a creditworthy borrower) costs the foregone interest margin. These costs are not symmetric, and accuracy treats them as equal, which is why accuracy is useless here.

The output is not a label — it is a **risk estimate that enters a pricing formula**. A model that always outputs 0.6 for every borrower with a base rate of 60% has perfect calibration but zero discrimination. You need both calibration (reliability of the probability scale) and discrimination (separation between defaulters and non-defaulters, measured by AUC).

Class imbalance is not a problem with the data — it is a **fact about reality**. The question is not how to remove it but how to make the model's loss function reflect business costs. SMOTE, class weights, and threshold tuning are three different interventions at three different stages of the pipeline; understanding when each is appropriate requires knowing what breaks first.

Regulatory interpretability in credit (ECOA, FCRA in the US; equivalent in EU) means you cannot hide behind a black box. If you deny a loan, you must produce the top reasons. This is why logistic regression with binned features — not gradient boosting — is still the dominant model in regulated lending, and why SHAP on a tree model is a workaround, not a replacement for interpretability.

## Decompose It

### Phase 1 — Understand the Label and Its Leakage Risk

**The question this phase answers:** Which features in your training set are contaminated by knowledge of the loan outcome?

**Key decisions:**
- **Include or exclude post-origination features?** Features computed after the loan is booked (e.g., "months since last delinquency on this loan", "number of times 30+ days past due") are outcomes, not predictors. Including them inflates training AUC to ~0.99 and produces a model that cannot score new applicants — they have no post-origination history yet. The German Credit dataset is relatively clean on this, but LendingClub data is not: `last_pymnt_d`, `out_prncp`, `total_rec_int` all leak.
- **Temporal split vs random split?** Always split chronologically when a time axis exists. Random splitting allows future data to inform past predictions (data leakage at the split level), making CV results optimistic. The German Credit dataset has no explicit timestamp, so random split is acceptable there, but document the assumption.

**Trap:** The feature "number of existing credits at this bank" sounds like a pre-origination attribute, but it can include the loan you are scoring if the data was extracted post-booking. Check the data dictionary, not just the column name.

**Done when:** You have a feature list with each feature tagged: pre-origination (safe), post-origination (drop), or ambiguous (investigate). The target variable is binary: 1 = default, 0 = repaid.

---

### Phase 2 — Handle Class Imbalance: Pick the Right Tool

**The question this phase answers:** Should you resample, reweight, or tune the threshold — and when does each intervention belong?

**Key decisions:**

**SMOTE (Synthetic Minority Oversampling Technique):** Generates synthetic minority-class samples by interpolating between existing minority examples in feature space. Use when: your model is insensitive to class weights (e.g., some nearest-neighbor models), or you want to physically balance the training set for a model that treats classes symmetrically. Do not use SMOTE before a train/test split — synthetic points generated from the full dataset leak test information into training. SMOTE must be applied inside the cross-validation loop.

**class_weight='balanced':** Reweights the loss function so minority-class errors cost $\frac{n_{total}}{2 \cdot n_{minority}}$ times more. This is equivalent to resampling with replacement to balance but leaves the decision boundary intact. Use this first — it's simpler, doesn't change the data, and works well with logistic regression and tree models. sklearn's `LogisticRegression(class_weight='balanced')` and `RandomForestClassifier(class_weight='balanced')` support it natively.

**Threshold tuning:** After training, the default decision threshold is 0.5. In imbalanced settings this is wrong — the model's predicted probability distribution shifts, and the optimal threshold for your business metric is not 0.5. The correct threshold minimizes expected loss: $\text{threshold}^* = \arg\min_\tau \; C_{FN} \cdot \text{FNR}(\tau) + C_{FP} \cdot \text{FPR}(\tau)$. Compute this on the validation set, not the test set.

**The priority order:** Start with `class_weight='balanced'`. If calibration is poor, add Platt scaling. If recall on the minority class is still inadequate after threshold tuning, add SMOTE. Never do all three simultaneously without ablation.

**Trap:** Applying SMOTE to the entire dataset before splitting, then evaluating on a held-out set that contains synthetic neighbors of test points. The AUC will be 0.05–0.10 higher than reality. Always use `imblearn.pipeline.Pipeline` to sequence SMOTE inside CV folds.

**Done when:** You have selected one primary imbalance strategy, applied it correctly inside cross-validation, and confirmed that the validation AUC does not change more than ±0.02 when you switch from SMOTE to class weights (if it changes a lot, investigate why).

### Phase 3 — Why Calibration Matters and How to Achieve It

**The question this phase answers:** Is your model's $\hat{p} = 0.7$ actually 70% of the time correct, and how do you measure and fix that?

**Key decisions:**

**What calibration means mathematically:** A model is calibrated if $P(Y=1 \mid \hat{p} = p) = p$ for all $p \in [0,1]$. Measure this with a reliability diagram: bin predictions into deciles, plot mean predicted probability vs observed frequency per bin. Perfectly calibrated = diagonal line. A reliability diagram that curves toward the top-left means the model is overconfident (predicts high probabilities that aren't realized). Random forests and gradient boosting are systematically overconfident.

**Brier Score:** $BS = \frac{1}{N}\sum_i (\hat{p}_i - y_i)^2$. Decomposes into calibration and refinement (discrimination). Lower is better. A model that always predicts the base rate has $BS = p_0(1-p_0)$ — this is your baseline.

**Platt Scaling:** Fit a logistic regression $\sigma(a \cdot \hat{p} + b)$ on held-out calibration data, where $a$ and $b$ are learned parameters. Requires at least ~1,000 calibration samples to be reliable. Use when the uncalibrated model produces monotone but badly scaled probabilities (the ranking is correct, the scale is wrong).

**Isotonic Regression:** Non-parametric step function that maps raw scores to calibrated probabilities. More flexible than Platt — can fix non-monotone miscalibration. Requires more calibration data (~5,000+) to avoid overfitting. If you have enough data, isotonic generally outperforms Platt. Apply using `sklearn.calibration.CalibratedClassifierCV`.

**The split you need:** Calibration must be done on data the base model has never seen. Use a three-way split: train (60%), calibration (20%), test (20%). Or use cross-val calibration (`cv=5` in `CalibratedClassifierCV`), which is safer on small datasets.

**Trap:** Calibrating on the training set inflates calibration metrics to perfect — you're just overfitting a mapping. Always calibrate on held-out data.

**Done when:** Reliability diagram for calibrated model lies within ±0.05 of the diagonal across all deciles. Brier score improves relative to uncalibrated model.

### Phase 4 — Choose and Optimize the Right Business Metric

**The question this phase answers:** What are you actually minimizing, and does your CV metric match it?

**Key decisions:**

**Expected Loss as the objective:** Each prediction leads to a lending decision. If you approve a defaulter, the loss is $LGD \times \text{exposure}$. If you reject a good borrower, the opportunity cost is the interest margin $r \times \text{exposure} \times \text{tenor}$. Define expected loss per applicant:

$$\mathcal{L}(\tau) = \sum_{i: \hat{p}_i < \tau} \hat{p}_i \cdot LGD_i \cdot EAD_i + \sum_{i: \hat{p}_i \geq \tau} (1 - \hat{p}_i) \cdot \text{margin}_i$$

Sweep $\tau$ from 0.1 to 0.9. The optimal threshold $\tau^*$ minimizes $\mathcal{L}(\tau)$ on validation data. This is the threshold you deploy, not 0.5.

**Why not optimize AUC directly?** AUC measures ranking quality across all thresholds. It is useful for model selection (comparing two models) but not for deployment (you deploy at one threshold). A model with AUC 0.82 and optimal threshold 0.35 can outperform a model with AUC 0.85 at its optimal threshold 0.5, depending on the cost asymmetry.

**KS statistic:** Maximum separation between CDFs of default and non-default score distributions. Used by credit bureaus and regulators as a simple discrimination measure. KS > 0.40 is good for credit scoring; KS > 0.50 is excellent.

**Trap:** Optimizing threshold on the test set — you've now overfit to the test set and have no unbiased estimate of true performance. The threshold must be selected on the validation set only.

**Done when:** You have a confusion matrix at $\tau^*$, the expected loss at $\tau^*$ computed on the test set, and a comparison to baseline (always approve = 0 revenue for defaulters, all interest income for good borrowers; never approve = 0 loss but 0 revenue).

---

### Phase 5 — Interpretability for Regulatory Compliance

**The question this phase answers:** Can you explain every individual rejection decision with ranked, human-readable reasons?

**Key decisions:**

**Logistic regression with WoE encoding:** Weight of Evidence transforms categorical features: $WoE_i = \ln\left(\frac{P(\text{event}_i)}{P(\text{non-event}_i)}\right)$. Information Value (IV) ranks features by predictive power. This pipeline is fully interpretable: each coefficient has a direct interpretation as the log-odds contribution of that feature bin. Regulators accept this without question.

**SHAP on gradient boosting:** If you use XGBoost or LightGBM (better discrimination), SHAP TreeExplainer gives exact Shapley values in $O(T \cdot L \cdot D)$ time. For an adverse action notice, the top-3 negative SHAP features are the "reasons for denial." This is defensible but requires you to explain what SHAP is during an audit.

**Monotonic constraints:** Business domain knowledge says "higher credit utilization → higher default probability" monotonically. Violating this in a model produces predictions that will be rejected by a risk committee even if they're empirically accurate. LightGBM supports `monotone_constraints` parameter. Encode these constraints from the start.

**Trap:** Reporting global feature importance (e.g., mean |SHAP|) as if it answers "why was this specific applicant denied." Adverse action notices require instance-level explanation, not global attribution.

**Done when:** For any test applicant, you can print the top-5 features driving the prediction, their direction, and their magnitude. The explanation must be in plain English (not SHAP values — translate them).

```
        |
        v

        |
        v

        |
        v

```

## Edge Cases to Handle Before You Code

1. **Zero-variance features after WoE encoding:** If a feature has only one category in the training fold (after a stratified split on a small dataset), WoE is undefined (log of 0). Add Laplace smoothing: add 0.5 to event and non-event counts before computing WoE.

2. **Predicted probability at exactly 0 or 1:** Logistic regression with perfectly separating features can produce $\hat{p} \in \{0, 1\}$. Brier score will be 0 or 1 for those predictions, and log-loss blows up. Add `solver='lbfgs'` with `max_iter=1000` and check for perfect multicollinearity.

3. **Calibration fold too small for isotonic regression:** If calibration set has fewer than 1,000 samples, isotonic regression will overfit. Switch to Platt scaling or use `CalibratedClassifierCV(cv=5)` which pools calibration across folds.

4. **Test set class imbalance differs from training:** If the test period has a different default rate (e.g., recession), your calibration will be off. Expected calibration error (ECE) will spike. Flag this as distribution shift, not model error.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|----------|----------|----------------|
| Imbalance strategy | `class_weight='balanced'` | SMOTE + balanced sampling | Pick A unless model has no native class weight support or you need physical balance for oversampling-sensitive algorithm |
| Calibration method | Platt scaling | Isotonic regression | Pick A when calibration set < 2,000 samples; isotonic overfits on small data |
| Model family | Logistic regression + WoE | XGBoost + SHAP | Pick A in regulated context where adverse action notice must survive legal audit without black-box explanation |
| Threshold selection | Minimize expected loss at known LGD | Maximize F1 | Pick A always — F1 assumes equal cost of FP and FN, which is wrong for credit |

---

## What Makes This Non-Trivial

The difficulty is not algorithmic — logistic regression and XGBoost are standard. The difficulty is that every modeling choice has a financial consequence that's invisible to standard ML metrics. A model with AUC 0.80 can lose more money than a model with AUC 0.75 if the threshold is set incorrectly. Calibration error of 0.05 across the 0.6–0.8 probability band can mean mispricing hundreds of basis points of interest rate for a large loan book. The interaction between imbalance strategy, calibration method, and threshold selection is not modular — changing one changes the optimal value of the others, requiring a joint optimization that most tutorials skip.

Regulatory constraints also force you to make choices that sacrifice predictive performance for interpretability. Understanding where that trade-off is (and isn't) acceptable, and how to quantify it, is the actual skill this project develops.

## Interview Angles This Project Develops

1. **"Why not just use accuracy for credit scoring?"** Strong answer: accuracy treats FP and FN as equal-cost; in credit, a false negative (approving a defaulter) costs LGD × EAD while a false positive costs only the forgone margin. The optimal decision rule minimizes expected loss, not error rate. Demonstrate you can write the loss formula.

2. **"What is calibration and why does it matter here?"** Strong answer: calibration means $P(Y=1|\hat{p}=p)=p$. It matters because the probability output enters a pricing formula — uncalibrated scores produce systematically mispriced loans. Describe the reliability diagram and how you'd measure ECE.

3. **"Where would you put SMOTE in a cross-validation pipeline?"** Strong answer: inside the CV fold, after the train/validation split, applied only to the training fold. Applying before splitting leaks synthetic minority neighbors into the validation set and inflates AUC. Use `imblearn.pipeline.Pipeline` to enforce this.

4. **"How do you handle adverse action notice requirements?"** Strong answer: for logistic regression + WoE, the top-3 features by |coefficient × WoE value| are the denial reasons. For tree models, SHAP TreeExplainer gives instance-level attributions; the top-3 negative contributors are the reasons. The model choice affects what you can say legally, not just statistically.

5. **"What's the difference between Platt scaling and isotonic regression for calibration?"** Strong answer: Platt fits a two-parameter logistic function — parametric, low variance, works well with ~500+ calibration samples. Isotonic fits a non-parametric step function — more flexible, but needs ~2,000+ samples to avoid overfitting. For small datasets, Platt is safer; for large datasets with non-monotone miscalibration, isotonic wins.

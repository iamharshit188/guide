# Canary Deployment Monitor & Auto-Rollback — Thinking Guide
> **Module:** 11 · **Difficulty:** Advanced · **Type:** THINKING · **Time:** ~2.5 hours

## The Problem

**Input:** A running stable deployment (v_stable) and a new candidate build (v_canary). A traffic ramp schedule: 1% → 5% → 10% → 25% → 50% → 100%, each step held for a configurable dwell period (e.g., 10 minutes).

**Output:** Either a completed promotion (100% on canary) or a rollback event: traffic snaps back to 100% stable, an alert fires, and the canary is quarantined until the next deploy cycle.

**Constraints:** Rollback must happen within one monitoring interval (e.g., 30 seconds). False positive rollbacks (caused by transient spikes) must be minimised. The system must correctly handle the cold-start window when canary has too few requests to produce reliable metrics.

---

## Before You Write Any Code: Build The Mental Model

A canary deployment is a risk management strategy, not just a deployment strategy. The fundamental bet is: if the new version is broken, it is better to expose 1% of traffic to that breakage and detect it early than to expose 100% of traffic and trigger a full outage. The blast radius is proportional to traffic percentage — this is why the ramp is exponential in shape rather than linear: the early steps (1%, 5%) are cheap experiments, the later steps (25%, 50%) carry real risk and should only be reached after the cheap steps confirm stability.

The controller is a state machine: at each step it asks "are all health signals within tolerance?" If yes, advance. If no, rollback immediately. The challenge is that health signals are noisy, lagged, and sometimes misleading. Error rate is reliable but slow to accumulate at 1% traffic. Latency is fast to measure but sensitive to load patterns unrelated to the canary. Prediction distribution shift is the hardest signal — it catches model degradation that produces no errors and no latency change, but changes what the model outputs.

The cold-start problem is the most underappreciated constraint: at 1% traffic, if baseline is 10,000 req/min, the canary sees 100 req/min. After 30 seconds you have ~50 samples — not enough to compute a stable error rate. Your rollback logic must not fire based on 5 errors in 10 requests (50% error rate from noise). Minimum sample size before trusting any metric is not optional — it is a correctness requirement.

## Decompose It

### Phase 1 — Traffic Ramp Controller

**The question this phase answers:** Who decides when to advance to the next traffic percentage, and what state does that decision depend on?

**Key decisions:**
- The ramp schedule is best stored as a list of `(percentage, min_dwell_seconds)` tuples. Each tuple represents a checkpoint: you must hold that percentage for at least `min_dwell_seconds` and pass all health checks before advancing.
- Dwell time must be long enough for metrics to stabilise (at least 3–5× the request latency × the volume needed to trust the rate). At 1% of 10k req/min = 100 req/min; to get 300 samples takes 3 minutes minimum dwell.
- Advancing is irreversible within a deploy cycle. Retreating (rollback) sends you all the way to 0% canary, not to the previous step. This asymmetry reflects the risk profile: partial rollback (e.g., going from 25% back to 10%) does not guarantee safety because the canary may have already corrupted state.

**Trap:** Advancing because the dwell timer expired, without checking whether health signals actually passed. Always gate advancement on both `dwell_elapsed AND all_checks_passing`. Time alone is not a signal.

**Done when:** The controller can be in exactly one state at a time (one active traffic percentage), advances only when both conditions hold, and triggers rollback atomically by setting the traffic weight to 0% canary.

---

### Phase 2 — Metrics Collection and the Cold-Start Guard

**The question this phase answers:** What is the minimum data required before trusting any metric, and how should the monitor behave before that threshold is reached?

**Key decisions:**
- Define a `min_samples_before_evaluating` constant per metric. Error rate needs at least 100 requests (so a 1% error rate can manifest as at least 1 event). Latency percentiles (p99) need at least 200 requests to be meaningful — you need enough samples to populate the tail of the distribution.
- During the cold-start window, the monitor should log "insufficient data, holding position" — not advance and not rollback. This is the most conservative choice: if you don't know, don't change state.
- Track the canary's request count as its own first-class metric, updated per-request by the traffic router. The health evaluator checks this count before evaluating any other signal.

**Trap:** Using a percentage threshold as the cold-start guard ("wait until canary has handled at least 10% of requests in the window") rather than an absolute count. At very low traffic steps the percentage and absolute count can both be small simultaneously. Always use absolute count.

**Done when:** The evaluator has a `has_sufficient_data(variant_name) -> bool` function that the main check loop calls before any other evaluation. All checks short-circuit if data is insufficient.

---

### Phase 3 — Health Signal Evaluation

**The question this phase answers:** Which signals to monitor, at what thresholds, and how to weight them against each other.

**Key decisions:**

**Error rate:** The most actionable signal. `canary_error_rate > stable_error_rate * (1 + tolerance_multiplier)` where `tolerance_multiplier` is typically 1.5 (50% relative increase is a rollback trigger). Using a relative threshold rather than an absolute one prevents false positives when stable itself has a non-zero baseline error rate.

**P99 latency:** `canary_p99 > stable_p99 * 1.2` (20% relative degradation). P99 is more appropriate than mean because degradation typically manifests in the tail. A canary that adds a slow code path may have identical mean latency but double the p99.

**Prediction distribution shift:** Compare the distribution of model output scores (or class labels) between stable and canary using a KL divergence or a chi-squared test. This catches: canary always predicting class 0, canary outputting NaN scores silently masked by error handling, canary producing valid-looking numbers from a broken model. Threshold: KL divergence > 0.1 nats is a soft warning, > 0.3 is a rollback trigger.

**Business metrics (not in technical monitoring but should be):** Click-through rate, conversion rate, user session length. These lag by hours but are the ground truth for "does the new model produce better outcomes?" Technical metrics are proxies; business metrics are the objective.

**Trap:** Treating all three signals as equally urgent. Error rate rollback should be near-immediate (next monitoring cycle). Latency degradation can tolerate one cycle of "watch and wait" because it may be a transient load spike. Distribution shift should trigger a human alert before auto-rollback, because shift is sometimes intentional (the new model was trained to be less confident, for example).

**Done when:** Each signal has its own `check_*()` function returning `(passed: bool, detail: str)`, and the evaluator aggregates them with the right urgency mapping.

---

### Phase 4 — Auto-Rollback Execution and Alerting

**The question this phase answers:** What does rollback mean atomically, and how do you ensure the alert reaches a human even if the alerting subsystem is also degraded?

**Key decisions:**
- Rollback is: (1) set canary traffic weight to 0 atomically, (2) verify stable is handling 100% of new requests within one routing cycle, (3) fire alert, (4) mark canary as quarantined in the deploy state (so the ramp controller cannot inadvertently advance again).
- Alert destinations must be independent of the service being rolled back. If your app is down, paging via the same app fails. Use an external webhook (PagerDuty, Slack) called directly from the monitor process, with a timeout — if the webhook call fails, write the alert to a local file as a last-resort audit trail.
- Include in the alert: which metric triggered rollback, the metric value at trigger time, canary version, timestamp, and number of affected requests during the canary window. This is the forensic record for the post-mortem.

**Trap:** Performing rollback asynchronously ("fire a task to roll back in the background"). Rollback must be synchronous with the failing health check. Every request served after the decision to rollback but before traffic is actually shifted is a request to a known-bad version.

**Done when:** Rollback completes (traffic shifted + alert sent) within a single monitoring interval. The monitor's loop cannot re-enter a "check if healthy" state for the quarantined canary.

---

### Phase 5 — Stateful Session Handling During Rollback

**The question this phase answers:** What happens to users who were mid-session on the canary version when rollback fires?

**Key decisions:**
- Sticky routing (session affinity): some load balancers route a user to the same backend for their entire session using a cookie or a consistent hash of their user ID. When rollback fires, sessions previously pinned to canary must be released. Two options: (a) let them drain naturally (session ends → next request routes to stable), (b) force-clear affinity immediately (next request from any session goes to stable). Option (a) is safer for user experience but means some users stay on the broken canary longer. Option (b) is safer for correctness but may cause session state loss if canary and stable have different session state schemas.
- Stateless inference APIs (the common ML case) have no session affinity issue — each request is independent. The problem is only relevant for stateful APIs (e.g., multi-turn chat sessions, shopping carts, user authentication flows).
- For ML APIs specifically: the main concern is consistency. A user who sees a recommendation from canary and then gets a different recommendation from stable on the next request may notice the inconsistency. This is a product concern, not a correctness concern — weigh it against the cost of staying on a broken model.

**Trap:** Assuming all ML serving is stateless. Multi-turn LLM conversations maintain KV cache state on specific server instances. Rolling back mid-conversation evicts the cache and forces a cold start for the next turn.

**Done when:** The rollback procedure documents the session handling policy explicitly and the implementation matches that policy.

```
                                     │
                                     └──────→  canary (w%)

                                                    │

```

## Edge Cases to Handle Before You Code

1. **Stable itself degrades during the canary window.** Your health check compares canary to stable. If stable's error rate rises from 1% to 3% due to an unrelated issue, and canary is at 2%, canary will appear healthy by relative comparison. Monitor absolute thresholds as a floor, and track stable metrics independently.

2. **The canary has a slow memory leak.** Error rate and p99 latency are fine for hours, then the canary process starts OOMing. Monitoring intervals of 30 seconds will not detect a leak that manifests over 4 hours. Add a `process_memory_rss_mb` metric to the canary health payload and check for monotonic growth.

3. **Thundering herd at step transition.** When the router shifts from 10% to 25% canary, the canary's request rate triples instantly. If the canary has a connection pool sized for 10% load, it will immediately hit pool exhaustion. Ramp dwell periods should include time for the canary to stabilise after the traffic step, not just before.

4. **Clock skew between monitor and router.** If the monitor and the traffic router are on different machines with unsynchronised clocks, a rollback command issued at t=T by the monitor may be executed at t=T+skew by the router. For rollback latency guarantees, use logical timestamps (sequence numbers) rather than wall clock time for ordering commands.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|---|---|---|---|
| Rollback trigger: any metric fails | Immediate rollback on first failing signal | Rollback only if ≥2 signals fail simultaneously | Pick A for error rate (one bad signal is enough). Pick B for latency-only to avoid false positives from load spikes. |
| Dwell period | Fixed duration per step | Adaptive: extend dwell until metric variance drops below threshold | Pick fixed for simplicity and predictability. Adaptive is better when traffic volume is uneven throughout the day. |
| Rollback destination | Always roll back to 0% canary | Roll back to previous step | Always to 0%. Rolling back to the previous step (e.g., 10% after failing at 25%) still exposes some traffic to a known-bad version. |
| Canary promotion criteria | All health checks passing for entire dwell | Health checks passing for the last 80% of dwell | The last-80% rule tolerates a brief spike at step transition (thundering herd). The all-dwell rule is stricter and preferred for high-stakes promotions. |
| Alert on rollback | Page on-call immediately | Send Slack message, page only if rollback happens twice in 24h | Page immediately. A rollback is a production incident, not a scheduled event. Developers need to investigate before the next deploy attempt. |

## What Makes This Non-Trivial

The rollback decision function sounds simple — "check if metrics exceed thresholds" — but the threshold itself depends on sample size, and sample size depends on traffic percentage, and traffic percentage is the thing you're controlling. This circular dependency means you cannot define thresholds in advance without knowing traffic volume, which varies with time of day and user geography. A fixed threshold that works at 25% traffic may be far too tight at 1% (triggering false positives) or far too loose at 50% (requiring many failures before triggering).

The correct framing is Bayesian: treat error rate as a Beta-distributed random variable, update it with each observed request, and roll back when the posterior probability that `true_error_rate > threshold` exceeds some confidence level (e.g., 95%). This is more complex to implement but eliminates the cold-start problem naturally — at low sample counts the posterior is wide and won't cross the confidence threshold from noise alone.

## Interview Angles This Project Develops

1. **"How do you decide when a canary is safe to promote?"** Strong answer: dwell period + absolute sample count + relative metric comparison to stable baseline + passing all three signal classes (error, latency, distribution). Weak answer: just watch the error rate.
2. **"What's the blast radius of a bad canary at 1% vs 25%?"** Strong answer: blast radius scales linearly with traffic percentage, but risk of detection scales sublinearly with sample size — early steps have low blast radius but also low power to detect subtle bugs.
3. **"Why not just use blue-green instead?"** Strong answer: blue-green cuts over 100% of traffic instantly. A canary is strictly safer for detecting bugs that manifest at scale, cost more in infrastructure (two live environments sharing load), and require the new version to serve production traffic during validation. Blue-green is right when rollback speed is paramount and the new version has been extensively tested in staging.
4. **"How does your monitor avoid paging at 3am for a p99 spike caused by a cron job?"** Strong answer: compare canary p99 to stable p99 at the same wall-clock time (relative threshold), not to a static baseline set at deploy time. Both versions see the cron job load simultaneously, so the relative comparison cancels out the spike.
5. **"What happens to in-flight requests during rollback?"** Strong answer: in-flight requests to canary complete normally — rollback changes the routing for new requests only. The router stops sending new requests to canary immediately, existing connections drain. If the canary is crashing mid-request, those requests will fail regardless; the rollback prevents additional failures but does not recover already-failed requests.

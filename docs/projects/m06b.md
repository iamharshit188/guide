# Semantic Document Retrieval System — Thinking Guide

> **Module:** 06 — GenAI Core · **Difficulty:** Intermediate · **Type:** THINKING · **Time:** ~6 hours

---

## The Problem

**Input:** A collection of 50,000+ documents (academic abstracts or news articles). A free-form natural language query. An optional labeled test set: for each query, a list of relevant document IDs with relevance grades (binary or graded).

**Output:** For each query, the top-10 retrieved documents, ranked by semantic relevance. An evaluation report with MRR@10 and NDCG@10 on the labeled test set.

**Constraints:** Query latency must be under 100ms. The index is static (documents don't change frequently). Embeddings are computed offline; only query embedding is computed at request time. The system must handle queries about concepts not present verbatim in any document (vocabulary mismatch — the core motivation for semantic, not keyword, retrieval).

---

## Before You Write Any Code: Build The Mental Model

Semantic retrieval separates the problem into two independent steps: **encode everything into a metric space** where semantic similarity corresponds to geometric proximity, then **search that space efficiently**. The encoding step determines retrieval quality ceiling; the search step determines latency.

A bi-encoder produces embeddings independently for the query and each document. The similarity between query $q$ and document $d$ is $\cos(\mathbf{e}_q, \mathbf{e}_d) = \frac{\mathbf{e}_q^T \mathbf{e}_d}{||\mathbf{e}_q|| \cdot ||\mathbf{e}_d||}$. Because document embeddings are precomputed, query-time cost is one encoder forward pass (the query) plus a nearest-neighbor search in the document embedding space. This is $O(\dim)$ per document for exact search, or $O(\log N)$ for approximate search with FAISS.

A cross-encoder concatenates the query and document and passes them through a transformer together: `[CLS] query [SEP] document [SEP]`. The model produces a single relevance score via the [CLS] output. This is much more expressive (the model sees both at once and can compute fine-grained attention between them) but requires a separate forward pass for every (query, document) pair — $O(N)$ per query, infeasible at $N=50,000$.

The industry-standard solution is a **two-stage pipeline**: bi-encoder retrieves top-100 candidates in milliseconds; cross-encoder re-ranks those 100 in seconds. Quality approaches cross-encoder on the full collection; latency remains practical.

## Decompose It

### Phase 1 — Embedding Strategy: Mean Pooling vs [CLS] Token

**The question this phase answers:** How do you turn a variable-length document into a fixed-size vector, and which pooling strategy gives better retrieval quality?

**Key decisions:**

**[CLS] token embedding:** In BERT-style models, the [CLS] token is prepended to the input. Its final hidden state is designed to aggregate sequence information and is used as the sequence representation for classification fine-tuning. For retrieval, using [CLS] as the document embedding relies on this aggregation being semantically meaningful. For models fine-tuned on sentence similarity (like SBERT variants), [CLS] is explicitly trained to be a useful sentence representation.

**Mean pooling:** Average the final hidden states of all non-padding tokens:
$$\mathbf{e}_{doc} = \frac{\sum_{t=1}^{L} m_t \cdot \mathbf{h}_t}{\sum_{t=1}^{L} m_t}$$
where $m_t \in \{0,1\}$ is the attention mask (0 for padding tokens). This uses more of the model's contextual information — every token's representation contributes. Empirically, mean pooling outperforms [CLS] for semantic similarity and retrieval on most benchmarks when using base BERT-style models that were not fine-tuned specifically on sentence tasks.

**Why mean pooling wins for generic models:** The [CLS] token in BERT (not SBERT) is trained only for classification tasks during pretraining (NSP in original BERT). Its embedding is not necessarily a good sentence representation for arbitrary domains. Mean pooling averages all contextualized token representations, which encodes more of the semantic content even without task-specific fine-tuning.

**For sentence-transformer models (all-MiniLM-L6-v2, all-mpnet-base-v2):** These were explicitly fine-tuned using a mean-pooled sentence embedding as the loss target. For these models, mean pooling is the correct pooling strategy by design — the [CLS] embedding is not fine-tuned and will perform worse.

**Trap:** Using `model.encode()` from `sentence-transformers` without checking what pooling it applies internally. The `SentenceTransformer` class applies mean pooling by default for most models. If you extract embeddings manually from the transformer output and use `[:, 0, :]` (the [CLS] position), you get the wrong representation for these models.

**Done when:** You have a function `embed_batch(texts, model, tokenizer, pooling='mean') -> np.ndarray` that returns embeddings of shape `(batch_size, D)`, handles padding masks correctly, and runs on batches of 32–64 documents for throughput.

### Phase 2 — FAISS Index Selection: IVF vs HNSW vs Flat

**The question this phase answers:** How do you store 50,000 embeddings such that nearest-neighbor queries complete in <100ms and return high-quality results?

**Key decisions:**

**Flat (exact search):** Stores all embeddings in a contiguous array. Query performs exhaustive inner product or L2 computation against all vectors. Time complexity: $O(N \cdot D)$ per query. At $N=50,000$, $D=768$: $50,000 \times 768 = 38.4M$ multiply-adds per query — runs in ~5ms on modern CPU, ~0.5ms on GPU. For $N \leq 100K$, exact search is fast enough and produces perfect recall. Use `faiss.IndexFlatIP` for inner product (equivalent to cosine similarity on L2-normalized vectors) or `faiss.IndexFlatL2` for L2 distance.

**IVF (Inverted File Index):** Clusters the embedding space into $k$ Voronoi cells using k-means. Each vector is assigned to its nearest centroid. At query time, compute distance to all $k$ centroids, search the nearest `nprobe` cells only (typically `nprobe = sqrt(k)`). Time complexity: $O(k + \text{nprobe} \cdot N/k \cdot D)$. The recall-speed trade-off is controlled by `nprobe` — higher `nprobe` increases recall but increases latency. Recall is never 100% (approximate search). Requires a training step (k-means on the document embeddings) before adding vectors. `faiss.IndexIVFFlat(quantizer, D, nlist)`.

**HNSW (Hierarchical Navigable Small World):** Graph-based ANN. Build a multi-layer graph where each node connects to its `M` nearest neighbors at each layer. Query traverses the graph greedily from a random entry point, descending layers. Recall is high (>95% at standard parameters) with low latency. Memory cost is higher than IVF: each vector stores `M` graph edges in addition to the embedding. For $N=50K$, memory overhead is manageable. `faiss.IndexHNSWFlat(D, M)`. No training step required — add vectors directly.

**When to use which:**
- $N < 100K$: Flat exact search. Recall = 100%, latency acceptable, no tuning.
- $N$ = 100K–10M: HNSW for good recall + low latency, or IVF if memory is constrained.
- $N > 10M$: IVF with product quantization (IVFPQ) to compress vectors and fit in RAM.

**For this project at $N=50K$:** Use Flat for correctness during development; profile latency. Switch to HNSW if latency exceeds 100ms (unlikely at this scale on modern hardware).

**Trap:** Not L2-normalizing embeddings before using `IndexFlatIP`. Inner product is equivalent to cosine similarity only on unit-norm vectors. `faiss.normalize_L2(embeddings)` normalizes in-place. Skipping this step means you're ranking by the magnitude of the embedding times the cosine similarity — high-magnitude vectors rank higher regardless of direction.

**Done when:** Index built, vectors added, a test query returns top-10 document IDs in <10ms. Verify that the top result for a specific test query makes semantic sense.

### Phase 3 — Evaluation Metrics: MRR and NDCG

**The question this phase answers:** How do you quantify retrieval quality, and when is each metric the right choice?

**Key decisions:**

**MRR@K (Mean Reciprocal Rank):** For each query, find the rank of the first relevant document in the top-K results. MRR = $\frac{1}{|Q|} \sum_{q=1}^{|Q|} \frac{1}{r_q}$ where $r_q$ is the rank of the first relevant result. If no relevant result appears in the top-K, the reciprocal rank is 0. MRR penalizes systems that bury the first relevant result deep in the list. Good when the user just needs one correct answer (e.g., question answering retrieval).

**NDCG@K (Normalized Discounted Cumulative Gain):** Accounts for multiple relevant documents and graded relevance. DCG@K = $\sum_{i=1}^{K} \frac{rel_i}{\log_2(i+1)}$ where $rel_i$ is the relevance grade at rank $i$ (binary: 0 or 1; or graded: 0, 1, 2, 3). IDCG (Ideal DCG) = DCG of the perfect ranking. NDCG@K = DCG@K / IDCG. Ranges from 0 to 1.

**Why NDCG is better than MRR when multiple documents are relevant:** MRR ignores all relevant documents after the first. If your system retrieves the second-most-relevant document at rank 1 and the most-relevant at rank 2, MRR gives full credit (rank 1). NDCG penalizes this because the lower-grade document is ranked above the higher-grade one. For academic retrieval where multiple papers are relevant to a query, NDCG is the correct metric.

**Computing NDCG correctly:**
```
DCG = rel_1 / log2(2) + rel_2 / log2(3) + ... + rel_K / log2(K+1)
IDCG = sort retrieved relevances desc, compute DCG of that perfect list
NDCG = DCG / IDCG
```

**Trap:** Computing IDCG as the DCG of ALL relevant documents in the corpus (not just those in the top-K). The standard BEIR evaluation computes IDCG over the full relevance list, not capped at K. This gives a lower NDCG than capping at K, because IDCG can be achieved only if all relevant documents are in the top-K — check the benchmark's definition carefully and be consistent.

**Done when:** A function `evaluate(queries, retrieved_ids, qrels, K=10) -> dict` returns `{'mrr': float, 'ndcg': float}`. Verified: a perfect retrieval system (returns exactly the relevant documents at top positions) scores 1.0 on both metrics.

### Phase 4 — Two-Stage Pipeline: Bi-Encoder + Cross-Encoder Re-Ranker

**The question this phase answers:** When does adding a cross-encoder re-ranker improve results enough to justify the latency cost?

**Key decisions:**

**When re-ranking helps:** Bi-encoders compress the full document into a single vector — they lose fine-grained term overlap and multi-hop reasoning. Cross-encoders see the full (query, document) text pair and use attention to compute fine-grained relevance. For short queries and long documents, bi-encoders are weak: the query vector must match a document vector summarizing a 500-word abstract. Cross-encoders shine in this regime.

**Pipeline architecture:**
```
Query → Bi-encoder → top-100 candidates from FAISS
       → Cross-encoder scores all 100 pairs
       → Re-rank by cross-encoder score
       → Return top-10
```

**Latency budget:** Bi-encoder query embedding: ~10ms. FAISS search (N=50K, Flat): ~5ms. Cross-encoder on 100 pairs: ~200–500ms (CPU), ~20ms (GPU). Total with cross-encoder: ~230ms. Without cross-encoder: ~15ms. If <100ms is a hard requirement on CPU, skip re-ranking or limit to top-20 candidates with a smaller cross-encoder.

**When to skip re-ranking:** If your bi-encoder was fine-tuned on the same domain as your documents, bi-encoder quality may be close to cross-encoder quality. Measure: compute NDCG@10 for bi-encoder only and bi-encoder + cross-encoder on your eval set. If the improvement is <0.02 NDCG, skip re-ranking.

**Cross-encoder models:** `cross-encoder/ms-marco-MiniLM-L-6-v2` is a strong general-purpose re-ranker (6-layer, fast). `cross-encoder/ms-marco-electra-base` is more accurate but slower.

**Trap:** Using the cross-encoder to score all 50,000 documents instead of only the top-100 candidates. A cross-encoder forward pass is O(L²) in attention where L is the concatenated query+document length. 50,000 × 500ms = 25,000 seconds per query. The bi-encoder pre-filter is mandatory.

**Done when:** NDCG@10 measured for (a) bi-encoder only, (b) bi-encoder + cross-encoder. Report the improvement and the latency cost.

### Phase 5 — Handling Vocabulary Mismatch and Long Documents

**The question this phase answers:** What breaks when queries use different terminology than documents, and how do you handle documents longer than the model's token limit?

**Key decisions:**

**Vocabulary mismatch:** A query "cardiac arrest" may match a document about "myocardial infarction" if the embedding model was trained on medical text. Without semantic embeddings, BM25 would miss this. Evaluate mismatch severity: compute BM25 scores and semantic retrieval scores on the same test set; the delta is the semantic gain. If they're similar, your queries and documents share vocabulary.

**Token length limit:** Most sentence transformers have a 512-token maximum input. A 500-word abstract often exceeds this. Options:
- **Truncation:** Take first 512 tokens. Fast. Works if the abstract front-loads key information (it usually does — title + first sentences).
- **Sliding window:** Embed overlapping chunks (e.g., 256 tokens with 128-token stride). Average the chunk embeddings. Better coverage but 2–4× slower.
- **Title + abstract:** Concatenate `title [SEP] abstract` — the title often captures the key topic, and the model can use it to disambiguate the abstract.

**For 50-word to 300-word abstracts:** Truncation at 512 tokens is almost always sufficient — academic abstracts are under 300 words ≈ 400 tokens. Document this assumption; it breaks for full papers.

**Trap:** Not adding the document title when embedding abstracts. "Attention Is All You Need" as a standalone abstract embedding differs from "Attention Is All You Need [SEP] We propose a new... model" because the title adds highly discriminative signal. Always prepend the title.

**Done when:** A test with a deliberately long document (1,500 words) shows no IndexError or silent truncation without warning. Sliding window implemented as a separate function that can be switched to via a config flag.

        |

        |

        |
        v

## Edge Cases to Handle Before You Code

1. **Empty or very short documents:** A document with 3 words produces a valid but degenerate embedding. Mean pooling of 3 tokens is noisier than mean pooling of 100. Flag documents under 20 tokens and optionally exclude them from the index or weight them down in search results.

2. **Duplicate documents:** Exact or near-duplicate documents in the collection cause NDCG inflation — if the correct document has 10 near-duplicates, a system that retrieves any of them gets credit. Deduplicate with MinHash LSH before indexing. The `datasketch` library implements this efficiently.

3. **FAISS index integer overflow:** `faiss.IndexFlatIP` uses 32-bit integer IDs by default. If your document collection exceeds 2 billion documents... this is not your problem at 50K. But if you ever use `IndexIDMap`, pass `np.int64` document IDs, not Python ints, to avoid silent truncation.

4. **Model max sequence length vs tokenizer truncation:** The `SentenceTransformer.encode()` method truncates silently to the model's max length. If a document is 2,000 tokens, you silently embed only the first 512. Add a warning log when token length exceeds the model's limit so you can decide to switch to the sliding window approach.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|----------|----------|----------------|
| Pooling strategy | Mean pooling | [CLS] token | Pick A for sentence-transformer models (all-MiniLM, all-mpnet, BGE) — they're fine-tuned with mean pooling as the target |
| FAISS index type | Flat (exact) | HNSW (approximate) | Pick A for N < 100K — exact search is fast enough and recall is 100%; HNSW adds complexity without benefit at this scale |
| Re-ranking | Bi-encoder only | Bi-encoder + cross-encoder | Pick cross-encoder only if latency budget allows (>100ms acceptable) AND NDCG improvement > 0.03 on eval set |
| Long document handling | Truncation at 512 tokens | Sliding window + avg pooling | Pick truncation for abstracts (<300 words); sliding window for full papers or technical reports |

## What Makes This Non-Trivial

The hard part is not building the retrieval system — it is evaluating it correctly. NDCG requires a complete relevance judgments set (qrels), which is expensive to create for a custom document collection. The BEIR benchmark provides pre-built qrels for 18 retrieval tasks; using these forces you to match your evaluation setup to their definition exactly. A 0.01 NDCG difference between models can be the entire gap between a publishable result and a null result — and that difference can also come from an evaluation bug (wrong IDCG normalization, wrong padding of results to K when fewer than K documents are retrieved, or off-by-one in the log base of the DCG formula). The engineering is straightforward; the evaluation rigor is where this project develops real skill.

## Interview Angles This Project Develops

1. **"What's the difference between a bi-encoder and a cross-encoder, and when would you use each?"** Strong answer: bi-encoder encodes query and document independently — fast, precomputable, but loses cross-attention between them. Cross-encoder sees both together, enabling fine-grained relevance assessment, but requires a forward pass per (query, doc) pair — O(N) per query. Practical systems use bi-encoder to retrieve top-100, cross-encoder to re-rank. The trade-off is latency vs quality.

2. **"Why do you L2-normalize embeddings before using IndexFlatIP?"** Strong answer: inner product equals cosine similarity only on unit-norm vectors. Without normalization, a document with a high-magnitude embedding ranks higher than a semantically more relevant document with a lower-magnitude embedding — you're measuring magnitude × cosine, not cosine. Always normalize before index construction and before each query.

3. **"Walk me through the NDCG formula."** Strong answer: DCG@K sums relevance grades discounted by log2 of rank position, $\sum_{i=1}^K rel_i / \log_2(i+1)$. IDCG is the DCG of the ideal ranking (all relevant documents at top positions). NDCG = DCG / IDCG. It rewards retrieving highly relevant documents at top positions and penalizes placing less relevant documents above more relevant ones. MRR would give full credit to a system that finds a marginally relevant document at rank 1 but misses the highly relevant document; NDCG penalizes this.

4. **"Why does mean pooling outperform [CLS] for sentence similarity?"** Strong answer: [CLS] in base BERT is trained for classification (NSP), not for encoding semantic similarity. It is a pooled representation optimized for a binary classification loss, not for geometric similarity in embedding space. Mean pooling averages all contextualized token representations, distributing the semantic content across the full sequence. Sentence-transformer models mitigate this by fine-tuning [CLS] or by explicitly training with mean pooling as the sentence embedding target.

5. **"How would you scale this system to 10 million documents?"** Strong answer: switch from Flat to IVF-PQ (Inverted File with Product Quantization). PQ compresses each 768-dimensional vector from 3KB to ~64 bytes by splitting into sub-vectors and encoding each with a 256-code codebook. IVF partitions the space into nlist clusters and searches only nprobe cells per query. Combined, IVF-PQ reduces memory from 22GB (Flat) to ~640MB and query time from 400ms to ~5ms, at a recall cost of 3–8% relative to exact search.

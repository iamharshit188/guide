# Semantic Document Retrieval System — Thinking Guide
> **Module:** 06 — GenAI Core · **Difficulty:** Intermediate · **Type:** THINKING · **Time:** ~5 hours

## The Problem

**Input:** 50,000+ documents (academic abstracts or news articles). Free-form natural language queries. A labeled test set: list of `(query, relevant_doc_ids)` pairs.

**Output:** Top-10 documents per query, ranked by relevance. Evaluation metrics: MRR@10 and NDCG@10 on the labeled test set.

**Constraints:** Index built once offline, query latency < 500ms for 50K docs. Retrieval quality measurable and improvable. Architecture must support a two-stage pipeline for quality improvement without full re-indexing.

---

## Before You Write Any Code: Build the Mental Model

Semantic retrieval requires mapping both documents and queries into the same vector space where proximity means relevance — not keyword overlap. This is fundamentally different from BM25, which counts term frequencies. A sentence embedding model projects text into $\mathbb{R}^d$ where $d$ is fixed (typically 384 or 768), and cosine similarity between query and document vectors is the relevance proxy.

The bi-encoder architecture is the enabling insight: encode documents once during indexing, encode each query at query time, then retrieve by vector similarity. The alternative — cross-encoder — processes each (query, document) pair jointly through the transformer, capturing fine-grained token-level interaction but requiring $O(N)$ forward passes per query. With $N = 50,000$ documents, cross-encoding at query time is 25–50 seconds per query at batch inference speeds — completely impractical for interactive retrieval.

Mean pooling over the last hidden state outperforms [CLS] token for general semantic similarity. [CLS] is only meaningful for sentence-level tasks when the model was fine-tuned with a [CLS]-specific objective (e.g., NSP in vanilla BERT). For sentence-transformers fine-tuned with contrastive loss over sentence pairs, mean pooling with attention mask weighting is the standard and the target of the fine-tuning loss.

FAISS is a library for billion-scale approximate nearest neighbor search. The index choice determines the quality-speed-memory trade-off. You must understand this before choosing: Flat gives exact results but scans all 50K vectors ($O(N)$); IVF clusters vectors and searches only nearby clusters (configurable via `nprobe`); HNSW builds a navigable small-world graph enabling sub-linear search with high recall.

## Decompose It

### Phase 1 — Document Embedding and Index Construction

**The question this phase answers:** How do you encode 50K documents efficiently, and which FAISS index type is appropriate for this scale?

**Key decisions:**
- Batch encoding: `model.encode(texts, batch_size=64, show_progress_bar=True)` processes documents in mini-batches. GPU batch size should max out VRAM without OOM; CPU batch_size=32–64 is safe.
- Index type for 50K documents: `IndexFlatIP` (inner product, equivalent to cosine after L2 normalization) gives exact results with < 100ms query time at 50K scale. `IndexIVFFlat` only pays off beyond 500K documents. Use Flat here — premature ANN optimization at 50K is engineering theater.
- Normalize before indexing: `faiss.normalize_L2(embeddings)` converts dot product search to cosine similarity. Do this before `index.add(embeddings)`.
- Persistence: save index with `faiss.write_index(index, "docs.index")` and document IDs as a separate `np.save("doc_ids.npy", doc_ids)`. The index stores only vectors; you must separately maintain the mapping from vector position to document content.

**Trap:** Storing full document text inside FAISS is not possible — FAISS stores only float32 vectors. You need a parallel data structure: a list or dict mapping FAISS integer index to document text and metadata. If you sort documents before indexing, the FAISS position must match your mapping exactly. Off-by-one here means returning document $i+1$ for every query.

**Done when:** Index loads in < 5 seconds, `index.ntotal == n_documents`, and a sanity-check query returns the query document itself as rank-1 when you embed a document and search for it (self-retrieval cosine score ~1.0).

---

### Phase 2 — Query Encoding and FAISS Retrieval

**The question this phase answers:** How is a query embedded and matched against the index, and what does the output look like?

**Key decisions:**
- Query encoding: same model, same pooling, same normalization as document encoding. Any deviation (different tokenization, skipping normalization) produces systematic retrieval errors that don't surface until you compute MRR.
- `index.search(query_vec, k=100)` returns `(distances, indices)` both shape `(1, 100)`. Distances are dot products (cosine after L2-norm), ranging 0–1. Indices are integer positions into the index.
- Return top-100, not top-10: the bi-encoder is approximate in quality. Retrieve a candidate set of 100, then re-rank with a cross-encoder to get the final top-10. Retrieval recall@100 is what matters here, not precision@10.

**Trap:** `index.search` returns `-1` for missing results if $k > \text{index.ntotal}$. Always clip $k$ to `min(k, index.ntotal)`. Also, `faiss.normalize_L2` modifies the array in-place — if you reuse the query array across multiple calls, the normalization has already been applied. Re-normalize each time from a fresh copy.

**Done when:** Single query latency < 50ms on CPU for 50K documents with `IndexFlatIP`. Returned indices all map to valid documents.

---

### Phase 3 — Two-Stage Retrieval with Cross-Encoder Re-Ranking

**The question this phase answers:** How much does re-ranking improve precision@10, and what is the latency cost?

**Key decisions:**
- Cross-encoder model: `cross-encoder/ms-marco-MiniLM-L-6-v2` from sentence-transformers is fast (6 layers) with strong MRR on passage ranking. It takes `[(query, doc_text), ...]` pairs and returns logit scores.
- Re-rank top-100 from bi-encoder: `cross_encoder.predict([(query, docs[i]) for i in top100_indices])` returns `(100,)` scores. Sort descending, take top-10. This is 100 forward passes through a 6-layer transformer — roughly 200ms on CPU, 30ms on GPU.
- Why not cross-encode all 50K: O(N) cross-encoder passes = 50,000 × 2ms = 100 seconds per query on CPU. The bi-encoder pre-filter is mandatory for any interactive system.

**Trap:** Cross-encoder scores are logits, not probabilities — they can be negative. Do not filter by score > 0; rank by score descending regardless of sign. Also, cross-encoder re-ranking requires access to full document text at query time, not just vectors. Ensure your document text store is accessible during the re-ranking step.

**Done when:** MRR@10 improves by at least 5–10 percentage points vs bi-encoder alone on your labeled test set. Total query latency (bi-encoder retrieve + cross-encoder re-rank) < 500ms on CPU.

---

### Phase 4 — Evaluation: MRR@10 and NDCG@10

**The question this phase answers:** How do you quantify retrieval quality, and what is the difference between the two metrics?

**Key decisions:**
- MRR@10 (Mean Reciprocal Rank): for each query, find the rank of the first relevant document. $\text{MRR} = \frac{1}{|Q|} \sum_{q} \frac{1}{\text{rank}_q}$ where rank is 0 if no relevant doc is in top-10. MRR only rewards the first relevant hit — if your system returns 9 relevant docs but the first relevant one is at rank 3, MRR contributes $\frac{1}{3}$.
- NDCG@10 (Normalized Discounted Cumulative Gain): rewards finding multiple relevant documents and placing higher-relevance docs earlier. $\text{DCG@k} = \sum_{i=1}^{k} \frac{rel_i}{\log_2(i+1)}$. Normalize by ideal DCG (perfect ranking denominator). NDCG is appropriate when you have multiple relevant docs per query; with binary single-relevant-per-query, it degenerates to MRR.
- MRR is better for scenarios where users stop reading after the first relevant result (fact lookup). NDCG is better for research/exploration where completeness across top-10 matters.

**Trap:** Computing NDCG requires knowing how many total relevant documents exist per query (the ideal ranking denominator). If your test set only lists one relevant doc per query, your NDCG denominator is 1 and the metric degenerates. Verify your test set has complete relevance annotations before trusting NDCG numbers.

**Done when:** You can print a table: `{metric: {bi_encoder: X, bi_encoder+rerank: Y}}` showing measurable improvement from re-ranking. Target MRR@10 > 0.5 on a standard IR benchmark (MS-MARCO dev baseline is ~0.33 for BM25, ~0.39 for DPR-style bi-encoder).

---

### Phase 5 — Failure Analysis and Iteration

**The question this phase answers:** For queries where top-1 is wrong, is the failure in retrieval (wrong candidates) or re-ranking (right candidates, wrong order)?

**Key decisions:**
- Retrieval failure diagnosis: check if the correct document appears in bi-encoder top-100. If not, retrieval recall@100 is the bottleneck — try a larger or domain-specific bi-encoder model.
- Re-ranking failure diagnosis: if correct doc is in bi-encoder top-100 but not in cross-encoder top-10, the cross-encoder is the bottleneck — try a 12-layer cross-encoder or a domain-fine-tuned version.
- Embedding model selection: for scientific abstracts, `allenai-specter` outperforms general `all-MiniLM-L6-v2` because it was fine-tuned on citation pairs. Domain match matters more than model size for specialized corpora.

**Trap:** Overfitting your hyperparameters (batch size, `nprobe`, $k$) to the test set and reporting those numbers as validation performance. Use a dev set for tuning, test set only for final evaluation — otherwise your reported numbers are optimistic and not reproducible.

**Done when:** Written diagnosis for the 20 worst-performing queries. At least half should show a clear, actionable failure mode (bad retrieval candidate set vs bad re-ranking of correct candidates).

## Data Flow

```
Documents (50K texts)
    |
    | model.encode(batch_size=64)       [bi-encoder, one-time offline]
    v
Embeddings (50K × 384 float32)
    |
    | faiss.normalize_L2()
    | index.add()
    v
FAISS IndexFlatIP (on disk: docs.index)

Query (string) at request time
    |
    | model.encode([query])
    | faiss.normalize_L2()
    v
Query vector (1 × 384)
    |
    | index.search(k=100)
    v
Top-100 candidates (indices + cosine scores)
    |
    | cross_encoder.predict([(query, doc_text)] × 100)
    v
Re-ranked scores (100,)
    |
    | argsort descending, take top-10
    v
Final top-10 results (doc_id, text, score)
    |
    | MRR@10, NDCG@10 against labels
    v
Evaluation table
```

## Edge Cases to Handle Before You Code

1. **Documents longer than model's max token length (typically 512 tokens):** `sentence-transformers` silently truncates. For long documents, chunk into paragraphs and index each chunk separately, then aggregate chunk scores (max pooling) to the document level at retrieval time. Failing to handle this means long documents are represented by their first 512 tokens only.

2. **Duplicate documents in the index:** FAISS does not deduplicate. If your corpus has near-duplicate documents (news wire syndication), a query can return 5 variants of the same article in top-10, collapsing effective recall. Hash document text before indexing and skip duplicates.

3. **Test set queries overlap with training data of the embedding model:** This inflates metrics. If using MS-MARCO for evaluation, verify your bi-encoder wasn't trained on MS-MARCO or its derivatives. Use a held-out benchmark if possible.

4. **Query contains domain-specific terminology not in model vocabulary:** Unlike BM25, neural retrieval handles this via sub-word tokenization — "myocardial" is tokenized as sub-word pieces even if unseen during pre-training. The failure mode is not vocabulary mismatch but domain distribution mismatch: if the embedding space was not shaped by biomedical text, distances between biomedical concepts are unreliable.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|----------|----------|----------------|
| Pooling strategy | Mean pooling (attention-mask weighted) | [CLS] token | Always for sentence-transformers; use [CLS] only if model was explicitly fine-tuned with [CLS] as the sentence representation target |
| FAISS index at 50K docs | IndexFlatIP (exact, O(N)) | IndexIVFFlat (approximate, nprobe tuning) | Use Flat — at 50K, exact search is < 100ms; IVF only needed beyond 500K documents |
| Re-ranking budget | Cross-encode top-100 (~200ms) | Cross-encode top-20 (~40ms) | Top-100 when MRR is the priority; top-20 for latency-constrained APIs with hard < 100ms SLAs |
| Evaluation metric | MRR@10 (first relevant result) | NDCG@10 (graded, multiple relevant) | MRR for navigational/factual queries; NDCG for research where retrieving multiple relevant docs matters |

## What Makes This Non-Trivial

The gap between bi-encoder and cross-encoder quality is not constant across query types. Short factual queries show little improvement from re-ranking because the bi-encoder already clusters the answer semantically. Complex multi-clause queries ("What are the mechanisms by which mTOR inhibition affects autophagy in cancer cells?") see 15–25 MRR point improvements from re-ranking because the cross-encoder can model fine-grained interaction between all query terms and document terms simultaneously — something a single embedding vector cannot encode.

## Interview Angles This Project Develops

1. "Why can't you use a cross-encoder for retrieval over 50K documents?" — Demonstrates awareness of the $O(N)$ vs $O(1)$ distinction and can quantify: 50K docs × 2ms/pair = 100 seconds per query, vs bi-encoder's < 100ms total.

2. "When does mean pooling outperform [CLS] for sentence similarity?" — Strong answer: [CLS] is only semantically meaningful if the pre-training or fine-tuning objective used [CLS] as the aggregate representation; general sentence-transformers use mean pooling as the optimization target — [CLS] is not fine-tuned for proximity.

3. "How would you handle a corpus that grows by 1,000 documents per day?" — FAISS `IndexFlatIP` supports `index.add()` incrementally without re-indexing. Add new embeddings nightly. For HNSW, same — no retraining needed. Alternatively, use `IndexIDMap` to assign stable external IDs independent of insertion order.

4. "What's the difference between MRR and NDCG, and which is appropriate for a medical literature search engine?" — Medical literature search: NDCG is better because clinicians need multiple relevant papers (not just the first hit) and relevance is graded (directly answers clinical question > tangentially related background).

5. "Your retrieval system returns good documents but users aren't clicking. What do you check?" — UX + ranking distinction: check click-through logs vs position bias (users rarely read past result 3 regardless of quality), verify the snippet generation is extracting the relevant passage, and analyze query reformulation rate (users typing again = retrieval failure signal even when retrieved docs are technically correct).

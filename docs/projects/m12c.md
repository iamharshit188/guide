# DPO Dataset Curator — Thinking Guide

> **Module:** 12 — RLHF & Alignment · **Difficulty:** Expert · **Type:** THINKING · **Time:** ~5 hours

## The Problem

**Input:** A list of prompts $\mathcal{P}$ (e.g., 500 instruction-following prompts), a generative model $\pi$ for producing candidate responses, and a reward model $r_\theta$ for ranking them.

**Output:** A validated DPO training dataset in `trl.DPOTrainer`-compatible format: Parquet or JSON lines where each row has `{"prompt": str, "chosen": str, "rejected": str}`. The dataset must pass: format validation (correct keys, no nulls), quality validation (margin threshold filter), diversity validation (prompt dedup, response dedup), and statistical validation (chosen/rejected length distribution, score distribution).

**Constraints:** Pairs where $r_\theta(\text{chosen}) - r_\theta(\text{rejected}) < \tau$ (margin threshold) must be filtered out. The final dataset must have at least 200 pairs. No chosen and rejected response may be identical. Prompts must be deduplicated at a semantic level (near-duplicate prompts must not appear in both train and a held-out set).

## Before You Write Any Code: Build The Mental Model

DPO (Direct Preference Optimization) replaces the explicit RL loop of PPO-based RLHF with a supervised objective derived from the Bradley-Terry preference model. The DPO loss is:

$$\mathcal{L}_{\text{DPO}}(\theta) = -\mathbb{E}_{(x, y_w, y_l) \sim \mathcal{D}} \left[ \log \sigma\left( \beta \log \frac{\pi_\theta(y_w|x)}{\pi_{\text{ref}}(y_w|x)} - \beta \log \frac{\pi_\theta(y_l|x)}{\pi_{\text{ref}}(y_l|x)} \right) \right]$$

where $y_w$ is the chosen response, $y_l$ is the rejected response, and $\pi_{\text{ref}}$ is the frozen reference policy. The gradient signal is proportional to the difference in log-probability ratios between chosen and rejected. This has two critical implications for data quality: (1) if $y_w$ and $y_l$ are very similar, both log-ratio terms are near zero and the gradient is negligible; (2) if the data distribution does not match the model's own generation distribution (off-policy data), the log-ratio $\log \pi_\theta / \pi_{\text{ref}}$ can become arbitrarily large, causing training instability.

DPO data quality is more sensitive than PPO data because there is no online correction. In PPO, the policy generates new data each iteration and the reward model evaluates it — bad training data from previous iterations is diluted. In DPO, the dataset is fixed before training. If 20% of your pairs have noisy preference labels, the model trains on that noise for all epochs with no opportunity to correct. You cannot recover from bad DPO data during training.

The margin threshold $\tau$ controls the difficulty of the training signal. Very easy pairs (high margin: chosen is clearly better) teach the model quickly but don't generalize well. Very hard pairs (low margin: nearly indistinguishable) are often mislabeled noise. The sweet spot: pairs where the reward margin is in the 0.5–2.0 range (on a normalized reward scale). Below 0.3: noisy labels. Above 3.0: trivially easy, no learning signal.

## Decompose It

### Phase 1 — Prompt Curation and Deduplication

**The question this phase answers:** Which prompts should be in the dataset, and how do we prevent contamination between train and evaluation splits?

**Key decisions:**
- **Semantic deduplication:** Exact string deduplication catches copies; semantic deduplication catches paraphrases ("What is gradient descent?" and "Explain gradient descent" are functionally identical). Use TF-IDF similarity or sentence embeddings: compute pairwise cosine similarity; if two prompts have similarity > 0.85, keep one, discard the other. At 500 prompts, pairwise comparison is $O(n^2) = 125K$ pairs — feasible in seconds with matrix operations.
- **Prompt diversity requirements:** A DPO dataset for general instruction-following should span at least 5–7 task categories: factual Q&A, code generation, summarization, creative writing, reasoning, math, classification. If all prompts are factual Q&A, the model will learn to follow instructions only in that domain. Category distribution should be roughly uniform — check it explicitly.
- **Train/eval split strategy:** Dedup at the prompt level before splitting. Splitting after dedup risks near-duplicate prompts ending up in both train and eval. Split first, then verify no cross-split near-duplicates (similarity > 0.85 between any train prompt and eval prompt).
- **Prompt quality filtering:** Filter prompts that are: too short (< 5 words — ambiguous), too long (> 200 words — out-of-distribution for instruction-following), or contain personal information, harmful content, or formatting errors. A simple heuristic: reject prompts that are all-caps, contain URLs, or have more than 3 consecutive punctuation characters.

**Trap:** Deduplicating based on exact hash only. RLHF datasets often contain paraphrase clusters — the same question rephrased 5 different ways. Training on all 5 variants plus their preference pairs effectively upweights that question 5x, causing the model to overfit to that topic. Semantic deduplication at the prompt level is required, not optional.

**Done when:** From an initial set of 500 prompts, the pipeline produces a deduplicated set with similarity < 0.85 between any pair, covers ≥ 5 task categories with no category comprising more than 30% of the total, and is split into 80% train / 20% eval with no cross-split contamination.

### Phase 2 — Response Generation: On-Policy vs. Off-Policy

**The question this phase answers:** Should responses come from the model you'll fine-tune, from a stronger model, or from both?

**Key decisions:**
- **On-policy generation:** Generate $y_w$ and $y_l$ using the same base model you plan to fine-tune ($\pi_{\text{ref}}$). The DPO gradient involves $\log \pi_\theta / \pi_{\text{ref}}$ — when responses are generated by $\pi_{\text{ref}}$, this ratio is near 1.0 at initialization, making training numerically stable. The gradient is well-scaled and the learning signal is directly relevant to the model's current capabilities.
- **Off-policy generation:** Generate responses using a stronger model (e.g., GPT-4). Pros: higher overall quality, clearer preference signal. Cons: the chosen and rejected responses are far from the base model's output distribution. At fine-tuning start, $\log \pi_\theta(y_w|\text{x}) \ll \log \pi_{\text{ref}}(y_w|\text{x})$ for GPT-4 outputs — the model must simultaneously learn to produce the style and content of GPT-4 outputs AND the preference ranking. This distributional mismatch causes training instability and poor generalization.
- **Hybrid approach:** Use the base model to generate $N=8$ candidates, rank with the reward model, use rank-1 as chosen and rank-8 as rejected. This is on-policy (both responses are from $\pi_{\text{ref}}$) while still having a clear preference signal. The margin $r_\theta(\text{rank-1}) - r_\theta(\text{rank-8})$ is typically large enough to provide a clean signal.
- **Temperature for diversity:** Generating $N=8$ candidates at temperature 1.0 produces diverse candidates. At temperature 0.3, candidates are nearly identical — the chosen and rejected are indistinguishable and the training signal disappears. Always generate candidates at temperature ≥ 0.8.

**Trap:** Using `top_k=1` (greedy decoding) for candidate generation. Greedy decoding produces one deterministic output — there is no diversity, and all 8 "candidates" are identical. DPO requires distinct chosen and rejected responses. Always use sampling with temperature > 0.

**Done when:** For each prompt, the pipeline generates 8 candidate responses with measurable diversity (average pairwise BLEU-4 < 0.7 between candidates of the same prompt), scores them with the reward model, and extracts the top-1 (chosen) and bottom-1 (rejected) with their reward scores.

### Phase 3 — Margin Filtering

**The question this phase answers:** Which (chosen, rejected) pairs are high enough quality to train on?

**Key decisions:**
- **Raw margin vs. normalized margin:** The reward model's scalar output has an arbitrary scale. A margin of 0.5 from one RM might be "very confident" while 0.5 from another is "noisy." Normalize by the RM's score standard deviation across all responses in the dataset: $\hat{\tau} = \tau / \sigma_r$. This makes the threshold interpretable: $\hat{\tau} = 0.5$ means the chosen response scored 0.5 standard deviations higher than rejected.
- **Margin threshold selection:** The threshold $\tau$ is a hyperparameter with a precision/recall trade-off. Low $\tau$ (e.g., 0.2): keeps more pairs, includes noisy ones, training is noisy. High $\tau$ (e.g., 2.0): keeps only clear winners, very few pairs, training may not generalize. Optimal range: $\tau \in [0.5\sigma, 1.5\sigma]$ (0.5–1.5 standard deviations). Plot the margin distribution and choose a threshold in the middle of the distribution, not at the tail.
- **Both ends matter:** Filter out near-zero margins (noisy) AND very high margins (trivially easy). Trivially easy pairs (margin > $3\sigma$) teach the model nothing new — it already "knew" which was better. The DPO gradient $\sigma(\beta \Delta r) \approx 1$ for large $\Delta r$, making these pairs saturated with near-zero gradient. Include them if you need to reach minimum dataset size; otherwise filter them.
- **Category-stratified filtering:** Apply margin filtering within each prompt category. If math prompts have higher reward variance than factual prompts, a global threshold will filter more math pairs and bias the dataset toward factual questions. Stratified filtering maintains category balance.

**Trap:** Setting $\tau$ based on the full dataset distribution before splitting. If you compute $\sigma$ on all 500 prompts and then filter train-only pairs, the threshold is contaminated by eval data statistics. Compute $\sigma$ on training data only; apply that threshold to training data only.

**Done when:** The margin filter reduces the initial pairs to a final set where the margin distribution has: mean > $0.5\sigma$, std > $0.2\sigma$, no pairs with margin < $0.2\sigma$ or > $3\sigma$ (unless needed for minimum size). Prints: "Filtered to X pairs from Y (Z% retained)".

### Phase 4 — Response Deduplication and Similarity Analysis

**The question this phase answers:** Are any chosen/rejected pairs so similar that the DPO gradient will be negligible?

**Key decisions:**
- **Chosen-rejected similarity check:** For each pair, compute token-level overlap (BLEU-4 or Jaccard on unigrams). If similarity > 0.9, the pair is near-identical — DPO's log-ratio term will be near zero for both, and the gradient is negligible. Filter these out even if the margin is above threshold.
- **Why very similar pairs fail in DPO specifically:** The DPO loss gradient $\propto \sigma(-\beta(\log \pi_\theta(y_w|x) - \log \pi_{\text{ref}}(y_w|x)) + \beta(\log \pi_\theta(y_l|x) - \log \pi_{\text{ref}}(y_l|x)))$. If $y_w \approx y_l$, both log-prob terms are nearly equal, the argument to $\sigma$ is near zero, $\sigma(0) = 0.5$, and $d\sigma/dx|_{x=0} = 0.25$. Small gradient = slow learning. Unlike PPO which can compensate with reward scale, DPO cannot.
- **Prompt-level dedup across pairs:** If the same prompt appears twice in the training set (with different response pairs), the model sees that prompt 2x per epoch — effectively upweighting it. Deduplicate at the prompt level: one pair per unique prompt. If you have multiple high-quality pairs for the same prompt, keep only the one with the highest margin.
- **Cross-pair response dedup:** Ensure the same response text doesn't appear as "chosen" for one prompt and "rejected" for a different prompt. This is rare but creates contradictory training signal: the model is simultaneously rewarded and penalized for the same text.

**Trap:** Only checking chosen-vs-rejected similarity within a pair. Two different prompts can have nearly identical chosen responses — if one is from a topic the model knows well and one is paraphrased, training on both pairs essentially upweights that response style. Response-level clustering (embed all chosen responses, identify clusters with intra-cluster similarity > 0.85) catches this.

**Done when:** All pairs pass: chosen ≠ rejected (exact), BLEU-4(chosen, rejected) < 0.8, no prompt appears more than once, no response appears in contradictory roles. Prints similarity statistics: mean BLEU-4 between chosen and rejected, fraction of pairs filtered for similarity.

### Phase 5 — Format Validation and Dataset Statistics

**The question this phase answers:** Is the dataset ready for trl.DPOTrainer, and what are its statistical properties?

**Key decisions:**
- **Format requirements for trl.DPOTrainer:** Keys must be exactly `"prompt"`, `"chosen"`, `"rejected"`. All values must be non-null, non-empty strings. The prompt must not already include the response — DPOTrainer concatenates them internally. Leading/trailing whitespace should be stripped. UTF-8 encoding required.
- **Tokenization length check:** DPOTrainer has a `max_length` parameter (typically 512–1024 tokens). Pairs where `len(tokenize(prompt + chosen)) > max_length` will be silently dropped by the trainer. Pre-filter: for each pair, check tokenized length. Report: "X pairs (Y%) would be truncated at max_length=512 — consider increasing max_length or filtering long pairs."
- **Statistical report to print:**
  - Distribution of prompt lengths (words)
  - Distribution of chosen/rejected lengths (words), and the ratio
  - Reward score distribution: mean, std, min, max, p25, p75 for chosen and rejected scores
  - Margin distribution: mean, std, min, max
  - Category breakdown: N pairs per task category
  - Length bias warning: if mean(chosen_len) / mean(rejected_len) > 1.5
- **Validation function that mimics DPOTrainer's checks:** Load the dataset with `datasets.load_dataset("json", ...)`, verify column names, attempt to tokenize 5 random examples with the actual model tokenizer. If this succeeds without errors, the format is correct.

**Trap:** Assuming the dataset is valid because the JSON is syntactically correct. JSON validity ≠ DPOTrainer compatibility. The trainer has specific assumptions: response tokens are expected at specific positions in the concatenated sequence, the attention mask must correctly separate prompt from response, and the tokenizer's special tokens (BOS, EOS, padding) must be handled consistently with training expectations.

**Done when:** The pipeline produces a validated Parquet file, prints a full statistical report, and runs a smoke-test of loading 5 examples through the actual tokenizer. If any check fails, the pipeline exits with a descriptive error message rather than silently producing an unusable dataset.

## Data Flow

```
Raw Prompts (500)
     │
     ▼
Semantic Dedup (cosine sim < 0.85)
     │
Category Diversity Check (≥ 5 categories, ≤ 30% per category)
     │
     ▼
Train/Eval Split (80/20, contamination check)
     │
     ▼ [Train set only]
Response Generation
     │  For each prompt: sample N=8 at T=1.0 from π_ref
     │
     ▼
Reward Model Scoring
     │  Score all 8N candidates
     │
     ▼
Pair Extraction
     │  chosen = rank-1, rejected = rank-N per prompt
     │  Record: prompt, chosen, rejected, r_chosen, r_rejected, margin
     │
     ▼
Margin Filter
     │  Compute σ_r on training scores
     │  Filter: margin ∈ [0.5σ, 3σ]
     │
     ▼
Response Similarity Filter
     │  BLEU-4(chosen, rejected) < 0.8
     │  No chosen == rejected (exact)
     │
     ▼
Cross-Pair Dedup
     │  One pair per unique prompt
     │  No response in contradictory roles
     │
     ▼
Format Validation
     │  Keys: prompt, chosen, rejected
     │  Tokenization length check
     │  Smoke-test with actual tokenizer
     │
     ▼
Final Dataset (Parquet) + Statistics Report
```

## Edge Cases to Handle Before You Code

1. **Reward model assigns identical scores to all candidates for a prompt:** All 8 candidates get reward 0.43 (the RM didn't learn meaningful discrimination for this prompt type). The margin is 0.0, which fails the margin filter. Handle: flag this prompt as "reward model blind spot," skip it, and log it for reward model retraining analysis.

2. **A prompt generates only one unique response across 8 samples:** The model's output is nearly deterministic for this prompt (low entropy). All 8 samples are near-identical. Chosen and rejected are nearly the same. The similarity filter will catch this, but it wastes N=8 generation calls. Heuristic: after the first 2 samples, compute their BLEU-4; if > 0.95, increase temperature to 1.3 for remaining samples.

3. **Final dataset has fewer than 200 pairs after all filters:** The filter chain was too aggressive. Diagnosis: print how many pairs were filtered at each stage (margin filter, similarity filter, dedup). Relax the most aggressive filter first: margin threshold (lower $\tau$), then similarity threshold (raise to 0.85 BLEU-4). Never relax prompt dedup — contamination between train and eval is unrecoverable.

4. **The chosen response is shorter than the rejected for many pairs:** This happens when the reward model is length-biased toward shorter responses (unusual but possible if the RM was trained on data that penalizes verbosity). Length ratio < 0.8 (chosen shorter) in > 30% of pairs should trigger a warning: "Length distribution anomaly — check reward model for inverse length bias."

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|---|---|---|---|
| On-policy vs. off-policy generation | On-policy: generate from the model you'll fine-tune | Off-policy: generate from GPT-4 or stronger | On-policy for distribution stability; off-policy only if the base model is too weak to generate any useful chosen responses |
| Margin threshold: absolute vs. normalized | Absolute: filter if margin < 0.5 | Normalized: filter if margin < 0.5σ | Normalized — different reward models have different score scales; absolute threshold is not transferable across RMs |
| One pair per prompt vs. multiple pairs | One pair per prompt (strict dedup) | Allow up to 3 pairs per prompt with diverse responses | One pair per prompt for clean statistics; multiple pairs only if prompts are genuinely multi-answer (creative writing, code with multiple valid solutions) |
| Chosen-rejected similarity filter | BLEU-4 < 0.8 | Embedding cosine similarity < 0.9 | BLEU-4 for surface similarity (what DPO loss cares about — token-level log-probs); cosine similarity for semantic similarity (ensures conceptual distinction) |

## What Makes This Non-Trivial

DPO's sensitivity to data distribution is the defining challenge. PPO with an online reward model can recover from distributional mismatch because the policy generates new training data each iteration. DPO cannot — the training distribution is frozen at dataset creation time. This means every distributional flaw in the dataset (wrong on-policy distribution, biased margin filtering, similar chosen/rejected pairs) becomes a permanent bias in the fine-tuned model with no correction mechanism.

The margin threshold creates an information-theoretic constraint: you are selecting pairs that are both learnable (not too hard — noisy) and informative (not too easy — saturated gradient). The optimal threshold depends on both the reward model's calibration and the base model's capabilities. A weak base model needs easy pairs (large margin) to make progress; a strong base model needs hard pairs (small margin) to learn fine-grained distinctions. This means the threshold should be adaptive to model capability — a hyperparameter that is re-tuned per model, not a constant.

Prompt diversity is not just about topic coverage — it is about coverage of the model's failure modes. The highest-value prompts for DPO training are those where the model's chosen response is clearly better than its rejected response on dimensions that matter (reasoning, factual accuracy, instruction following) rather than superficial dimensions (length, formatting). Constructing such prompts requires understanding the model's specific weaknesses, which requires evaluation first — making dataset curation and model evaluation a tight feedback loop.

## Interview Angles This Project Develops

1. **"Why is DPO data quality more sensitive than PPO data quality?"** — Strong answer explains that DPO uses a fixed dataset with no online correction, while PPO generates new data each iteration. A bad DPO pair is trained on for all epochs; a bad PPO sample is diluted by subsequent on-policy data.

2. **"What is the margin threshold in DPO dataset curation, and how do you choose it?"** — Strong answer explains the noisy-pairs problem (low margin) and the saturated-gradient problem (high margin), frames it as a precision/recall trade-off, and explains normalization by reward standard deviation.

3. **"Why is on-policy generation better than off-policy for DPO?"** — Strong answer traces through the DPO loss: $\log \pi_\theta / \pi_{\text{ref}}$ is near 1.0 for on-policy data at initialization, making the gradient well-scaled. Off-policy data requires the model to first close the distribution gap before learning preferences.

4. **"How do very similar chosen and rejected responses affect DPO training?"** — Strong answer traces through the gradient: the DPO signal is proportional to $\sigma(-\beta \Delta \log r)$ where $\Delta \log r \approx 0$ for near-identical responses. The gradient is $\approx 0.25 \beta \cdot \text{small number}$ — near zero — meaning no learning.

5. **"What does Goodhart's Law mean for your DPO dataset?"** — Strong answer explains that the reward model is a proxy for human preference; pairs selected by the RM to be "high quality" may be high-quality only according to the RM's biases (length, confidence language). If the fine-tuned model's behavior exactly matches what the RM rewards, it has Goodharted the RM — not learned to be genuinely helpful.

# RAGAS-Driven RAG Optimizer — Thinking Guide

> **Module:** 08 — RAG & Vector Search · **Difficulty:** Expert · **Type:** THINKING · **Time:** ~6 hours

## The Problem

**Input:** An existing RAG pipeline with configurable hyperparameters (chunk_size, k, overlap, reranking_threshold, temperature). A test set of 100 questions, ideally with reference answers. A set of 4 RAGAS metrics to measure per question.

**Output:** A ranked comparison table showing how each hyperparameter configuration affects each RAGAS metric. A diagnosis of whether the weak link is retrieval or generation. A recommended configuration that maximizes faithfulness score, the highest-stakes metric for production deployments.

**Constraints:** The evaluation harness must re-run the full RAG pipeline for each configuration. With 100 questions × 5 configurations, that is 500 LLM calls — cost and latency management are non-trivial. Some RAGAS metrics require a reference answer (ground truth); building the test set without ground truth introduces circular evaluation risks. The hyperparameter search space is not convex — faithfulness and context recall can trade off against each other, requiring Pareto reasoning rather than single-objective optimization.

## Before You Write Any Code: Build The Mental Model

RAGAS is a framework for evaluating RAG pipelines along four orthogonal dimensions. Each metric measures a different part of the system, which is why you need all four: a high score on one metric with a low score on another reveals a specific failure mode. Optimizing for a single metric (e.g., context recall) without watching the others causes pathological configurations — large chunks give high context recall but low context precision (too much irrelevant text retrieved along with the relevant text).

The fundamental diagnostic question RAGAS answers is: is the problem in retrieval or generation? This matters because the fixes are completely different. If retrieval is failing (low context recall), no prompt engineering or temperature tuning will help — you need better chunking, a stronger embedding model, or more data. If generation is failing (high context recall, low faithfulness), the retrieval is working but the LLM is hallucinating or ignoring the context — you need a stronger LLM, tighter system prompt constraints, or reduced temperature.

The test set construction problem is the subtlest part of this project. A test set requires (question, reference_answer) pairs. If you generate reference answers using the same LLM you're evaluating, you introduce circular evaluation: the LLM will score best on the answers it would naturally generate, not on the answers that the documents actually support. The correct approach is to generate questions and reference answers directly from document text (where the reference answer is a verbatim excerpt), not from LLM-generated summaries.

## Decompose It

### Phase 1 — Test Set Construction

**The question this phase answers:** How do you build 100 evaluation questions with reference answers without making the evaluation circular?

The gold standard approach is human annotation: domain experts read documents and write (question, reference_answer) pairs. This is expensive ($5–50/question depending on domain complexity) but produces the only truly unbiased evaluation.

The practical approach for this project: generate questions and reference answers programmatically from document text, then manually filter to remove circular or ambiguous examples. The generation strategy:

1. For each chunk in the corpus, prompt an LLM: "Given the following text, generate one specific factual question that can be answered using ONLY this text. Then provide the exact answer from the text." This produces (question, reference_answer, source_chunk) triples where the reference answer is anchored to the source.
2. Filter: discard questions that are too general ("What is the document about?"), too specific (trivially unique), or whose reference answer contains pronouns without referents (e.g., "It increases by 15%" — what is "it"?).
3. For context recall computation, the reference answer must be a substring or paraphrase of the source chunk — verify this before adding to the test set.

The circular evaluation risk: if you use the LLM to both generate reference answers and evaluate faithfulness, the LLM's biases cancel in a way that inflates faithfulness scores. Mitigation: generate reference answers with GPT-4o and evaluate with a different model (e.g., Claude Haiku), or use NLI-based faithfulness rather than LLM-based faithfulness.

**Key decisions:**
- How many questions per document: aim for 5–10 per document (approximately one per major section). Too few questions per document gives high variance on per-document metrics; too many creates redundancy that inflates aggregate scores.
- Whether to include trick questions (questions whose answer is not in the documents): yes — include 10–15 unanswerable questions. A production RAG system must handle out-of-scope queries gracefully. These questions specifically test the system's ability to say "I don't know" rather than hallucinating.

**Trap:** Generating questions from the same chunks that will be retrieved. If your test question was generated from chunk X, and chunk X is always retrieved for that question (because it was generated from it), context recall will always be 1.0 — artificially inflated. Generate questions from a held-out subset of chunks, then test retrieval from the full corpus.

**Done when:** You have 100 (question, reference_answer) pairs in a JSON file. ~85 answerable from the corpus, ~15 deliberately out-of-scope. Every reference answer is verifiable against source text. The test set does not overlap with the chunks used to generate it.

### Phase 2 — RAGAS Metric Implementation

**The question this phase answers:** What exactly does each RAGAS metric measure, how is it computed, and what does a specific value tell you about system failure?

**Faithfulness** measures: are all claims in the generated answer supported by the retrieved context? It is an NLI-based precision metric on the (answer, context) pair. Implementation: decompose the answer into individual atomic claims using an LLM ("List every factual claim made in this answer, one per line"). For each claim, run NLI against each retrieved chunk. A claim is supported if any chunk entails it. Faithfulness = (number of supported claims) / (total claims in answer). A faithfulness score of 0.7 means 30% of the answer's claims are unsupported by the retrieved context — likely hallucinated.

$$\text{Faithfulness} = \frac{|\text{claims supported by context}|}{|\text{total claims in answer}|}$$

**Answer Relevancy** measures: does the answer address the question? It is a geometric proxy: embed both the original question and the generated answer, compute cosine similarity. High relevancy means the answer is topically on-target; low relevancy means the answer drifted (answered a related but different question, or refused when it should have answered). This metric does not check factual correctness — an answer can be highly relevant and completely wrong.

$$\text{AnswerRelevancy} = \text{cosine}(\text{embed}(\text{answer}), \text{embed}(\text{question}))$$

**Context Recall** measures: how much of the reference answer is covered by the retrieved context? It requires a reference answer. Implementation: decompose the reference answer into sentences. For each sentence, check if any retrieved chunk contains or entails it. Context recall = (sentences covered by context) / (total sentences in reference). Low context recall means the retrieval missed relevant chunks — the right information exists in the corpus but was not retrieved.

$$\text{ContextRecall} = \frac{|\text{reference sentences covered by context}|}{|\text{total reference sentences}|}$$

**Context Precision** measures: what fraction of the retrieved context is actually relevant? Among the top-$k$ retrieved chunks, how many were useful for answering the question? Implementation: for each retrieved chunk, classify it as relevant (contains information used in the reference answer or actual answer) or irrelevant. Context precision = (relevant chunks) / (total retrieved chunks). Low context precision means the retrieval is noisy — it retrieves many chunks, but most are off-topic.

$$\text{ContextPrecision} = \frac{|\text{relevant retrieved chunks}|}{k}$$

**Key decisions:**
- Whether to use the RAGAS Python library or implement metrics from scratch: the library is faster to deploy but harder to debug and customize. Implementing from scratch gives you insight into where metrics fail (e.g., when NLI is wrong) and allows you to swap the NLI model.
- Which NLI model for faithfulness and context recall: `cross-encoder/nli-deberta-v3-small` is a good default (fast, accurate on factual claims). `facebook/bart-large-mnli` is more accurate but slower.

**Trap:** Computing faithfulness on answers to out-of-scope questions. If the answer is "The documents do not contain this information," its atomic claims are near-zero — faithfulness is technically 1.0 (no false claims). This inflates faithfulness for the refuse-to-answer case. Exclude unanswerable questions from faithfulness computation; include them in a separate "appropriate refusal rate" metric.

**Done when:** Four functions `faithfulness(answer, context_chunks)`, `answer_relevancy(answer, question)`, `context_recall(context_chunks, reference)`, `context_precision(context_chunks, reference)` each return a float in [0, 1]. Running all four on a single (question, answer, context, reference) tuple completes in under 5 seconds.

### Phase 3 — Hyperparameter Search Space Design

**The question this phase answers:** Which hyperparameters control retrieval quality vs. generation quality, and how do you explore the space without running 500+ LLM calls?

Hyperparameters that affect **retrieval** (context recall and context precision):
- `chunk_size`: smaller chunks = higher precision (less irrelevant context per chunk), lower recall (relevant information may be split across chunks). Test range: [256, 512, 1024] tokens.
- `overlap`: higher overlap increases recall at chunk boundaries. Test range: [0, 50, 128] tokens.
- `k` (number of retrieved chunks): higher k increases recall, decreases precision. Test range: [3, 5, 10].
- `reranking_threshold`: cross-encoder score below which chunks are excluded. Higher threshold = higher precision, lower recall. Test range: [0.0, 0.3, 0.5].

Hyperparameters that affect **generation** (faithfulness and answer relevancy):
- `temperature`: lower = more faithful, potentially less fluent. Test range: [0.0, 0.1, 0.3].
- System prompt strictness: "Answer only from the provided context" vs. "Use the provided context as your primary source."

Search strategy: grid search over all combinations is $3^4 \times 3 = 243$ configurations × 100 questions × 4 metrics = prohibitively expensive. Use a two-phase strategy:
1. Fix generation hyperparameters (temperature=0.1, strict prompt) and sweep retrieval hyperparameters over 20 randomly sampled questions. Identify the retrieval configuration that maximizes context recall × context precision.
2. Fix the best retrieval configuration and sweep generation hyperparameters over the full 100 questions.

**Key decisions:**
- How to handle LLM cost in hyperparameter search: cache LLM calls. For the same (system_prompt, context_block, question), the LLM response is deterministic at temperature=0. Store (hash(inputs), response) in a SQLite cache — repeated configurations with identical context reuse cached responses, reducing LLM calls by 60–80%.
- Whether to use approximate metrics (bi-encoder cosine for faithfulness) or exact metrics (NLI-based): bi-encoder approximations run 10x faster and correlate ~0.85 with NLI-based metrics on most RAG use cases. Use bi-encoder approximations for the hyperparameter sweep; run exact NLI-based metrics only on the top-3 configurations.

**Trap:** Treating the hyperparameter search as independent optimizations per metric. Maximizing context recall independently selects large k (retrieve many chunks). Maximizing faithfulness independently selects small context (fewer, more precise chunks). The optimal configuration is on the Pareto frontier — you must jointly evaluate both and accept a trade-off.

**Done when:** You have a configuration space of at most 30 distinct configurations. Each configuration can be run on the full 100-question test set in under 30 minutes. Results are logged to a structured file (JSON or CSV) with all 4 metrics per question per configuration.

### Phase 4 — Diagnosis: Retrieval Failure vs. Generation Failure

**The question this phase answers:** Given the RAGAS scores for a configuration, how do you determine whether to improve retrieval or generation?

The diagnostic matrix:

| Context Recall | Faithfulness | Diagnosis | Fix |
|---------------|-------------|-----------|-----|
| High (> 0.8) | High (> 0.8) | System working well | Focus on edge cases |
| Low (< 0.5) | High | Retrieval failing — right info not retrieved | Better chunking, embedding model, or more data |
| High | Low (< 0.5) | Generation hallucinating despite good context | Stricter prompt, lower temperature, stronger LLM |
| Low | Low | Both failing | Fix retrieval first — generation cannot improve without relevant context |
| High | Medium (0.5–0.7) | LLM partially ignoring context | Investigate: are the retrieved chunks too long? Is the relevant sentence buried at the end of a 512-token chunk? |

Implement this diagnostic automatically: for each test question, classify it into one of the matrix cells. Report the fraction of questions in each cell — a histogram of failure modes is more actionable than aggregate metrics.

For questions in the "Low Recall, High Faithfulness" cell, perform manual inspection of retrieval results: what chunks were retrieved? What chunk contains the reference answer? Compute the bi-encoder similarity between the query and the reference chunk — if it's above 0.4, the chunk should have been retrieved (threshold issue). If it's below 0.3, the embedding model genuinely doesn't see the query and chunk as related (semantic gap issue).

**Key decisions:**
- Faithfulness threshold for "high" vs "low": 0.8 is appropriate for factual Q&A where hallucinations are costly (medical, legal); 0.6 may be acceptable for creative or open-ended domains.
- How to handle questions where the LLM correctly refuses (unanswerable questions): count these as successes, not failures. The system should score high on these questions if it returns an appropriate "I don't know" response.

**Done when:** The evaluation harness outputs a per-question diagnosis table and an aggregate failure mode distribution. The top-3 failure modes (by frequency) are identified with representative example questions for each.

### Phase 5 — Ranked Comparison Table and Recommendation

**The question this phase answers:** How do you aggregate metrics across 100 questions per configuration and produce a defensible recommendation?

Aggregate each metric as mean ± standard deviation over the 100 questions. Standard deviation matters: a configuration with mean faithfulness 0.85 ± 0.05 is preferable to one with mean 0.87 ± 0.20 (high variance means some questions get very poor answers, which is unacceptable in production).

Compute a composite score for each configuration: faithfulness is the primary metric (weight 0.4), context recall is secondary (weight 0.3), context precision is tertiary (weight 0.2), answer relevancy is supporting (weight 0.1). These weights encode the priority for a factual Q&A system where hallucination is the primary risk.

$$\text{Score} = 0.4 \times F + 0.3 \times CR + 0.2 \times CP + 0.1 \times AR$$

Present the Pareto frontier: show all configurations that are not dominated (no other configuration scores higher on all four metrics simultaneously). A dominated configuration is never optimal — the Pareto frontier contains all reasonable recommendations.

**Done when:** A ranked table shows all tested configurations sorted by composite score. The top-3 configurations are highlighted with their Pareto status (on frontier or dominated). A specific recommended configuration is stated with the rationale (e.g., "chunk_size=512, k=5, overlap=50, threshold=0.3, temperature=0.1 achieves the best faithfulness–recall trade-off with low variance").

## Edge Cases to Handle Before You Code

1. **LLM refuses to answer during test set generation:** When generating (question, reference_answer) from a chunk, the LLM may produce "I cannot generate a question from this text" for chunks that are navigation menus, bibliography entries, or very short snippets. Filter these out by checking if the generated question contains a `?` and the reference answer is at least 10 tokens. Robust generation requires a retry with a simpler prompt and a hard max of 3 retries per chunk.

2. **NLI model produces incorrect entailment for numeric facts:** Current NLI models are notoriously bad at number comparison. "The paper reports 85% accuracy" does not entail "The accuracy is above 80%" for many NLI models, despite it being logically true. This causes faithfulness scores for numerically-dense answers (tables, statistics) to be systematically underestimated. Supplement NLI with a regex-based numeric claim extractor that checks whether numbers in the answer appear verbatim in the context.

3. **Context block exceeds LLM input limit during evaluation:** With k=10 and chunk_size=1024, the context block alone is ~10,000 tokens — exceeding GPT-4o's effective context for a single API call (it accepts 128K tokens but quality degrades on long contexts). During evaluation, track whether any API call returned a truncation warning or an error. If so, flag that configuration as "context overflow" and exclude it from the final ranking rather than reporting artificially inflated metrics from a truncated context.

4. **Circular evaluation when using GPT-4 for both generation and RAGAS scoring:** If GPT-4o generates the RAG answer and you use GPT-4o to evaluate faithfulness (by checking if claims are entailed by context), the model may rationalize its own outputs as supported even when they are not. Use a different model family (Claude or a local NLI model) for faithfulness scoring. At minimum, use a lower-capacity model (GPT-4o-mini) for scoring to reduce this bias.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|---------|---------|----------------|
| Test set generation | LLM-generated Q&A from chunks | Human-annotated Q&A | A for rapid iteration (days not weeks); B when evaluation is the ground truth for a production system |
| Faithfulness computation | NLI model (accurate, slow, ~3s/question) | Bi-encoder cosine proxy (fast, ~0.3s/question, noisier) | NLI for final evaluation of top configurations; bi-encoder for the hyperparameter sweep over many configurations |
| Hyperparameter search | Grid search (exhaustive, expensive) | Sequential model-based optimization (Bayesian, efficient) | Grid search for < 30 configurations (simpler, reproducible); Bayesian for > 50 configurations where cost matters |
| LLM call caching | Cache by hash(inputs) in SQLite | No caching (always re-call) | Always cache during hyperparameter search — identical configurations on identical questions should never cost duplicate API calls |

## What Makes This Non-Trivial

RAGAS metrics interact with each other in ways that make single-metric optimization dangerous. Increasing k improves context recall (more chunks retrieved, more reference information covered) but decreases context precision (more irrelevant chunks dilute the relevant ones). Decreasing chunk_size improves context precision (each chunk is more focused) but may decrease context recall (relevant information split across chunk boundaries). The optimal chunk_size and k are jointly determined — you cannot tune them independently.

The retrieval vs. generation diagnostic is necessary because the two failure modes look identical from the user's perspective (wrong or incomplete answers) but require completely different fixes. A team that increases LLM capability (better model, better prompt) when the problem is actually retrieval will spend significant engineering effort with no improvement. The RAGAS diagnostic gives a principled way to allocate optimization effort before investing in model upgrades.

Building a valid test set without circular evaluation is harder than most practitioners assume. The temptation is to use the same LLM for everything (generation, reference answer creation, evaluation) because it's the simplest pipeline. But this creates a system that grades its own homework — faithfulness scores will be inflated because the evaluator (LLM) will tend to find its own outputs faithful. Production RAG evaluation requires either human annotation or at minimum a model diversity strategy (generate with one model family, evaluate with another).

## Interview Angles This Project Develops

1. **"How do you know if your RAG system is actually grounded vs. hallucinating?"** — Strong answer: faithfulness measures this directly. Decompose the answer into atomic claims, check each claim against retrieved context using NLI, report the fraction supported. A faithfulness score below 0.8 means the LLM is generating claims not supported by the context — these are by definition hallucinations (they may still be factually true from parametric memory, but they are not grounded in the retrieved documents).

2. **"What is the difference between context precision and context recall, and which do you optimize for?"** — Strong answer: precision is "of what I retrieved, how much was useful?" — it penalizes irrelevant retrieval. Recall is "of what I needed, how much did I retrieve?" — it penalizes missed retrieval. For high-stakes factual Q&A, recall is usually more important (missing a critical fact is worse than including a slightly irrelevant paragraph). For user-facing systems where long context degrades answer quality, precision matters more. The optimal trade-off depends on the cost function of your application.

3. **"How would you build a test set for a RAG system over proprietary documents you can't share with annotators?"** — Strong answer: use LLM-based synthetic generation (question + reference answer from chunks) with post-hoc filtering to remove circular or low-quality examples. The filtering criteria: questions must be specific (contain named entities or specific numbers), reference answers must be verbatim or close paraphrase of source text, and at least one domain expert must review a 10% random sample before the test set is considered valid.

4. **"Why can't you just maximize faithfulness and ignore the other metrics?"** — Strong answer: high faithfulness with low context recall means the LLM is very faithful to the limited context it receives, but the context is missing critical information. The answer is grounded but incomplete. A system optimized only for faithfulness will achieve this by retrieving very few, very specific chunks (high precision, low recall) and generating conservative answers that never stray from the retrieved text — but missing the actual answer because it wasn't retrieved.

5. **"At what point does hyperparameter tuning on RAGAS metrics lead to overfitting to the test set?"** — Strong answer: when you run more than 20–30 configurations on the same 100 questions, you are effectively optimizing hyperparameters to the test set. The selected configuration may perform well on those specific 100 questions but generalize poorly to production queries. Mitigation: hold out 20 questions from the hyperparameter search; use them only for final validation of the selected configuration. The gap between hyperparameter-search performance and holdout performance quantifies the degree of test-set overfit.

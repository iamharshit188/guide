# Research Assistant Agent — Thinking Guide

> **Module:** 10 — LLM Agents & Tool Use · **Difficulty:** Advanced · **Type:** THINKING · **Time:** ~5 hours

## The Problem

**Input:** A natural language research question (e.g., "What are the main failure modes of RLHF-trained models?"). A token budget $B$ (e.g., 8 000 tokens of context to spend on retrieved content). A maximum number of search iterations $N$ (e.g., 6).

**Output:** A structured research brief with: (1) a 3–5 sentence synthesis, (2) a claim list where each claim is annotated with source URLs, confidence level (High / Medium / Low), and supporting evidence, (3) a contradictions section listing claims where sources disagree and why, (4) a gaps section listing what the agent could not find.

**Constraints:** The agent must not fabricate citations. If a claim appears in the synthesis but cannot be traced to a retrieved source, it must be flagged as "Unverified." The agent must terminate within budget $B$ and attempt limit $N$ regardless of whether it has "enough" information.

## Before You Write Any Code: Build The Mental Model

The agent's core loop is plan → search → read → extract → integrate → decide. At each iteration the agent has partial knowledge and must decide whether to search for more, extract from what it has, or synthesize and stop. The stopping decision is not a single threshold — it is a combination of budget exhaustion, confidence convergence, and diminishing returns on new queries.

Multi-hop reasoning is qualitatively different from single-hop retrieval. If question A requires fact F1 (from source 1) and fact F2 (from source 2), and F2 is only findable once you know F1 to formulate the right query, the agent must chain queries. The agent needs a working memory of intermediate conclusions to seed subsequent searches. Without this, it retrieves documents that contain each fact in isolation and never connects them.

Contradictory sources are unavoidable in any real domain. The correct response is not to pick one and suppress the other — it is to surface the contradiction explicitly and classify it: temporal (one source is newer), methodological (they measured different things), or genuine disagreement (same question, opposite conclusions). The brief should report all three types differently.

Hallucination risk in synthesis is highest at the final step. The LLM has read 10–20 documents, internalized their content, and is asked to produce a synthesis. It will fill gaps with plausible-sounding content from training data, not from retrieved documents. The defense is claim-by-claim grounding: every sentence in the synthesis must cite the passage it came from, or be explicitly marked unverified.

## Decompose It

### Phase 1 — Query Planning

**The question this phase answers:** Given the user's question, what should the agent search for first, and how should the search plan adapt as evidence accumulates?

**Key decisions:**
- **Initial query decomposition:** The user's question is rarely a good search query. Decompose it into 2–4 sub-questions, each searchable independently. "What are RLHF failure modes?" → ["reward hacking examples language models", "RLHF sycophancy behavior", "specification gaming reinforcement learning"]. The LLM generates these decompositions from the original question before any search.
- **Query expansion vs. precision:** Broad queries return more results but noisier content. Narrow queries return precise results but miss adjacent evidence. Start broad; narrow on subsequent iterations based on what's missing from current evidence.
- **Adaptive replanning:** After each search-and-read cycle, the agent updates its query plan: which sub-questions are answered, which need better queries, which are newly discovered. The plan is a living document, not a fixed list.

**Trap:** Generating all queries upfront and executing them in batch. This prevents the agent from learning from early results to refine later queries. The query plan must be replanned after each retrieval cycle, not just executed linearly.

**Done when:** The agent generates an initial 3-query plan from any input question, executes the first query, retrieves and reads the top-3 results, and produces an updated query plan based on what those results revealed.

### Phase 2 — Retrieval and Content Extraction

**The question this phase answers:** How does the agent convert raw web content into structured, citable evidence?

**Key decisions:**
- **Search API vs. scraping:** A search API (SerpAPI, Brave Search) returns snippets and URLs without scraping. Scraping the full page gives more content but introduces noise: nav bars, ads, cookie banners, JavaScript-rendered content that may not parse. The practical compromise: use the search API for discovery, scrape only pages whose snippets suggest high relevance. Check `robots.txt` before scraping; some sites disallow it.
- **HTML to text:** `BeautifulSoup` with `get_text(separator="\n")` strips tags but leaves navigation text. Better: extract only `<article>`, `<main>`, or the largest `<div>` by character count. Remove `<nav>`, `<footer>`, `<header>`. Use `trafilatura` for production-quality extraction — it handles boilerplate removal better than hand-rolled parsers.
- **Claim extraction from text:** The agent sends each page's text to the LLM with a prompt: "Extract up to 5 factual claims relevant to the question. For each claim, quote the sentence from the source verbatim." Verbatim quoting is the anti-hallucination constraint — the LLM cannot fabricate quotes if it must copy them exactly.
- **Context budget per page:** If a page is 10 000 tokens but the total budget is 8 000, the agent cannot include it fully. Summarize the page first (LLM compression), then extract claims from the summary. This introduces one layer of lossiness — document this in the output's gaps section.

**Trap:** Treating the extracted claim list as ground truth. The LLM doing the extraction can still misread or subtly distort a source. The verbatim-quote constraint mitigates but does not eliminate this. Treat claims as "evidence supporting X" not "proof of X."

**Done when:** Given a URL, the agent can scrape the page, extract the main content, extract 3–5 verbatim-quoted claims with relevance scores, and return a structured `Evidence` object with fields: `url`, `title`, `claims: list[Claim]`, `tokens_used: int`.

---

### Phase 3 — Cross-Validation and Contradiction Detection

**The question this phase answers:** How does the agent decide whether a claim is reliable, and what does it do when sources disagree?

**Key decisions:**
- **Corroboration threshold:** A claim is "High confidence" if it appears in ≥3 independent sources with consistent phrasing. "Medium" if in 2 sources or 1 authoritative source (peer-reviewed, major institution). "Low" if in only 1 non-authoritative source. Independence matters: if Site B scraped Site A, they are not independent.
- **Contradiction classification:** The agent compares all claims about the same topic. Contradictions are classified as:
  - **Temporal:** Source A (2021) says X; Source B (2024) says not-X. Resolution: prefer the newer, but note the change.
  - **Methodological:** Source A measured task A; Source B measured task B. Resolution: both correct in their domain — the question was ambiguous.
  - **Genuine:** Same question, same domain, same methodology, opposite conclusions. Resolution: report the disagreement, cite both, note this is an open question.
- **Authority weighting:** Domain authority (arXiv > blog post > forum) should shift confidence but never override empirical corroboration. A single arXiv paper is still one source. A claim in 5 blog posts is still only Medium unless the blogs cite primary research.

**Trap:** Majority vote across sources. If 4 low-quality blogs repeat a myth originally from 1 incorrect source, majority vote gives "High confidence" to a false claim. Always check whether multiple sources are citing the same original source — trace the claim to its origin.

**Done when:** Given a list of `Evidence` objects, the agent produces a `ClaimRegistry` that groups claims by topic, assigns confidence levels, and flags contradictions with classification and both source citations.

---

### Phase 4 — Stopping Criterion

**The question this phase answers:** When does the agent have "enough" information to synthesize, and how does it detect diminishing returns?

**Key decisions:**
- **Token budget as hard limit:** When `tokens_used >= B * 0.85`, the agent must synthesize with what it has. The 0.85 factor reserves 15% for the synthesis prompt and output.
- **Confidence convergence as soft signal:** After each retrieval cycle, compute the fraction of claims in the registry that are High or Medium confidence. If this fraction has not changed by more than 0.05 over the last 2 cycles, retrieval has converged — additional searches are unlikely to add new high-confidence claims.
- **Diminishing returns detection:** If the last search cycle produced zero new claims not already in the registry (exact or near-paraphrase match), the agent should generate a different type of query (narrow rather than broad, specific rather than general) before giving up.
- **Sub-question coverage:** If any sub-question from Phase 1 has zero evidence, the agent must make one more targeted search for it before synthesizing. An unanswered sub-question must appear in the gaps section.

**Trap:** Using confidence thresholds as a stopping criterion without accounting for question difficulty. Some questions are genuinely unsettled — no threshold of search effort will produce High-confidence claims across all sub-questions. The stopping criterion must time-out gracefully on hard questions rather than looping forever.

**Done when:** The agent's main loop terminates reliably — either budget exhaustion, attempt limit, or confidence convergence — and prints which criterion triggered the stop and why.

---

### Phase 5 — Synthesis and Brief Generation

**The question this phase answers:** How does the agent produce a grounded synthesis that does not hallucinate?

**Key decisions:**
- **Synthesis prompt structure:** The prompt sends the LLM the full `ClaimRegistry` (claims, sources, confidence levels, contradictions) and asks it to write a synthesis using only those claims. The instruction is explicit: "Every sentence must cite at least one claim ID from the registry. Do not add claims not present in the registry."
- **Post-generation verification:** After the synthesis is generated, the agent does a second pass: for each sentence, check if the cited claim IDs exist in the registry and if the sentence is semantically consistent with the cited claim. Flag mismatches as "Unverified."
- **Contradictions section:** Pull all contradiction pairs from the ClaimRegistry. Format: "Claim A (Source 1, 2022) contradicts Claim B (Source 2, 2023). Classification: Temporal. Resolution: Claim B is more recent."
- **Gaps section:** For each sub-question with zero High/Medium confidence evidence, report it as a gap. For each claim flagged Unverified, list it.

**Trap:** Asking the LLM to write a synthesis and cite sources in one shot. The LLM will write fluently and fabricate plausible-looking citations. The correct pattern is: write synthesis first, then verify citations separately. Two-step is slower but grounded.

**Done when:** The agent produces a brief where every High/Medium confidence claim in the synthesis has a valid source citation and the contradictions + gaps sections are non-empty and accurate.

## Edge Cases to Handle Before You Code

1. **Page returns 403 or requires JavaScript rendering:** The scraper gets an empty or error page. The agent must detect this (empty text after extraction, HTTP error code) and mark that URL as "failed to retrieve" in the gaps section. Do not retry the same URL — move to the next result.

2. **Search API returns zero results:** This happens with highly technical or niche queries. The agent must broaden the query (remove domain-specific jargon, use synonyms) and retry once. If the second attempt also fails, log the sub-question as unreachable and continue.

3. **All sources are downstream of one original source:** The agent finds 6 blog posts that all cite the same arXiv paper. Confidence should not be upgraded to High — the corroboration is illusory. Detecting this requires checking: do all sources link to the same original? A citation graph check at depth-1 is sufficient.

4. **LLM refuses to extract claims from a page:** Some pages contain content the LLM's safety filters flag (e.g., a page about adversarial attacks on AI). The agent must catch the refusal response, skip that page, and note it in the gaps section.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|---|---|---|---|
| Verbatim quote vs. paraphrase in claim extraction | Verbatim: copy exact sentence from source | Paraphrase: LLM summarizes the claim | Always use verbatim — paraphrase breaks the citation chain and introduces LLM interpretation errors |
| Confidence aggregation | Corroboration count (how many sources) | Source authority weighted sum | Use corroboration count for unknown domains; add authority weighting only when source credibility is assessable |
| Contradiction resolution | Report and present both | Pick newer / higher authority | Report both — resolution requires human judgment; the agent's job is to surface, not to decide |
| Context budget allocation | Equal budget per page | Relevance-weighted: more budget to high-scoring pages | Relevance-weighted if you have a scoring signal; equal if scraping blindly from search rank |

## What Makes This Non-Trivial

The agent must maintain coherent state across multiple LLM calls — the query plan, the claim registry, and the working set of sub-questions all change between iterations. A stateless agent (one that re-reads the question fresh each iteration) will re-search the same queries and fail to accumulate progress. The state management pattern — what goes in the prompt, what is stored outside the LLM — is the central architectural decision.

Hallucination risk compounds across multi-hop chains. Each hop introduces a small probability that the LLM misread a source or subtly distorted a claim. By the time a 3-hop chain reaches synthesis, the cumulative drift from the original sources can produce a plausible-sounding but incorrect conclusion. Verbatim quoting at every extraction step is the only practical defense.

The stopping criterion is a policy decision disguised as an engineering problem. There is no objectively correct point to stop — only trade-offs between recall (finding more sources) and reliability (verifying what you have). The agent should expose this decision to the user: let them set the confidence threshold and budget, and report which limit was reached.

## Interview Angles This Project Develops

1. **"How do you prevent hallucination in an LLM-powered research tool?"** — Strong answer distinguishes synthesis-time hallucination (fill gaps in knowledge) from retrieval-time distortion (misread sources), and explains that verbatim quoting + post-hoc citation verification are necessary in combination, not separately.

2. **"When does an agent stop searching?"** — Strong answer identifies the three stopping signals (budget, convergence, attempt limit), explains diminishing returns detection, and notes that hard questions need graceful timeout rather than infinite retries.

3. **"How do you handle contradictory sources?"** — Strong answer gives the three-way taxonomy (temporal, methodological, genuine disagreement) and explains that surfacing contradictions is more valuable than resolving them unilaterally.

4. **"How does multi-hop reasoning work in an agent?"** — Strong answer explains that the agent uses intermediate conclusions as search seeds, maintains a working memory of partial answers, and that this is fundamentally different from a single-turn RAG retrieval.

5. **"What's the difference between a good and bad search query when planning agent tool use?"** — Strong answer explains query decomposition, the broad-to-narrow progression, and why adaptive replanning (learning from early results) outperforms upfront batch planning.

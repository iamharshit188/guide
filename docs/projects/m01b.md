# Gradient Descent Visualizer — Thinking Guide

> **Module:** 01 — Math for ML · **Difficulty:** Beginner · **Type:** THINKING · **Time:** ~2 hours

## The Problem

**Input:** A user-supplied loss function $f: \mathbb{R}^2 \to \mathbb{R}$ (e.g., a 2D bowl, a saddle, a Rosenbrock valley), an initial point $(w_1^{(0)}, w_2^{(0)})$, a learning rate $\eta$, and a maximum number of iterations $T$.

**Output:** A trajectory table printed to terminal — each row is one optimizer step: `[step, w1, w2, loss, grad_norm]` — for three optimizers running in parallel: vanilla gradient descent, gradient descent with momentum, and Adam.

**Constraints:** The system must work for *any* twice-differentiable $f$ supplied as a Python callable. No symbolic math, no closed-form gradients — numerical differentiation only. No matplotlib.

## Before You Write Any Code: Build The Mental Model

Every gradient-based optimizer is a rule for updating parameters given local gradient information. The gradient $\nabla f(w)$ points in the direction of steepest ascent; descent means moving opposite to it. The core question is how much to move (step size) and in what direction (raw gradient vs. history-adjusted gradient).

Vanilla GD is memoryless: $w^{(t+1)} = w^{(t)} - \eta \nabla f(w^{(t)})$. It has no knowledge of past gradients, so it oscillates in narrow valleys and stalls near saddle points.

Momentum introduces "velocity" — a running average of past gradients. The parameter "rolls" down the loss surface like a ball with inertia, smoothing oscillations and accelerating through flat regions.

Adam adapts the learning rate *per parameter* using estimates of the first moment (mean gradient) and second moment (uncentered variance). Parameters with consistently large gradients get smaller effective learning rates; rarely-updated parameters get larger ones.

The system you are building is an *optimizer simulator*, not a trainer. Its job is to faithfully implement update rules and record the trajectory so a user can reason about optimizer behavior from first principles.

## Decompose It

### Phase 1 — Numerical Gradient Computation

**The question this phase answers:** How do we get $\nabla f(w)$ without an analytical formula?

**Key decisions:**
- **Forward difference vs. central difference:** Forward: $\frac{f(w + \epsilon e_i) - f(w)}{\epsilon}$ — $O(\epsilon)$ error. Central: $\frac{f(w + \epsilon e_i) - f(w - \epsilon e_i)}{2\epsilon}$ — $O(\epsilon^2)$ error. Central difference costs one extra function evaluation per dimension but is dramatically more accurate near saddle points and at small $\epsilon$. For a 2D function, that's 4 evaluations vs. 3 — choose central difference.
- **Epsilon choice:** Too large ($\epsilon = 0.1$) and truncation error dominates; too small ($\epsilon = 10^{-12}$) and floating-point cancellation dominates. $\epsilon = \sqrt{\text{machine epsilon}} \approx 10^{-7}$ is the standard sweet spot for `float64`.
- **Return type:** Always return a `np.ndarray` of shape `(2,)`, never a Python list. Downstream optimizer code assumes vector arithmetic.

**Trap:** Using a single global $\epsilon$ for all functions. A function like $f(w) = w_1^4 + w_2^4$ has gradients that change scale drastically near zero; an adaptive $\epsilon$ scaled to $|w|$ is more robust: $\epsilon_i = \epsilon \cdot \max(1, |w_i|)$.

**Done when:** The numerical gradient of $f(w) = w_1^2 + w_2^2$ at $(1, 1)$ is $(2.0, 2.0)$ with error below $10^{-9}$.

### Phase 2 — Implement Vanilla Gradient Descent

**The question this phase answers:** What is the simplest correct optimizer, and what state does it need?

**Key decisions:**
- **State:** Only the current parameter vector $w \in \mathbb{R}^2$. No history.
- **Stopping criteria:** Three options — (a) fixed $T$ iterations, (b) gradient norm below threshold $\|\nabla f\| < \delta$, (c) loss change below $\delta$. Option (b) is most principled; option (a) is simplest to implement first. Use (a) + (b) in combination: stop at whichever comes first.
- **Divergence detection:** If `loss > 1e10` or `np.isnan(loss)`, the optimizer has diverged — likely because $\eta$ is too large. Detect this and print a warning rather than silently running to completion.

**Trap:** Mutating the parameter vector in place across optimizers. If you run three optimizers from the same starting point, each must have its own copy of $w$. `w_gd = w0.copy()` — not `w_gd = w0`.

**Done when:** Starting from $(2, 2)$ on $f = w_1^2 + w_2^2$ with $\eta = 0.1$, the trajectory converges to $(0, 0)$ in ~50 steps and the loss decreases monotonically.

---

### Phase 3 — Implement Momentum

**The question this phase answers:** How does velocity memory change the trajectory on curved loss surfaces?

**Key decisions:**
- **Update rule:** The standard "heavy ball" formulation:
  $$v^{(t+1)} = \beta v^{(t)} + \nabla f(w^{(t)})$$
  $$w^{(t+1)} = w^{(t)} - \eta v^{(t+1)}$$
  vs. Nesterov:
  $$v^{(t+1)} = \beta v^{(t)} + \nabla f(w^{(t)} - \eta \beta v^{(t)})$$
  Heavy ball is simpler; Nesterov has better convergence guarantees for convex functions. Implement heavy ball first; note in the output which variant is running.
- **Momentum coefficient $\beta$:** Standard value is 0.9. Larger $\beta$ (e.g., 0.99) means more "memory" — smoother trajectory but slower initial response. $\beta = 0$ reduces exactly to vanilla GD.
- **Initial velocity:** $v^{(0)} = \mathbf{0}$. Never initialize to the gradient — that would make the first step different from vanilla GD.

**Trap:** Interpreting momentum as "add a fraction of the previous gradient" instead of "maintain an exponential moving average of gradients." The velocity $v^{(t)}$ is a geometric sum: $v^{(t)} = \sum_{\tau=0}^{t} \beta^{t-\tau} g^{(\tau)}$. This means recent gradients dominate but old gradients never fully disappear. On a saddle point, opposing gradients from either side of the saddle will cancel in the momentum term — the optimizer can escape, which vanilla GD cannot.

**Done when:** On $f = w_1^2 + 10 w_2^2$ (an elongated ellipse), momentum converges faster than vanilla GD. The momentum trajectory overshoots the minimum and oscillates before converging.

---

### Phase 4 — Implement Adam

**The question this phase answers:** Why does per-parameter adaptive learning rate matter, and how do you correctly handle the bias-correction terms?

**Key decisions:**
- **Adam update rule in full:**
  $$m^{(t)} = \beta_1 m^{(t-1)} + (1 - \beta_1) g^{(t)}$$
  $$v^{(t)} = \beta_2 v^{(t-1)} + (1 - \beta_2) (g^{(t)})^2$$
  $$\hat{m}^{(t)} = m^{(t)} / (1 - \beta_1^t)$$
  $$\hat{v}^{(t)} = v^{(t)} / (1 - \beta_2^t)$$
  $$w^{(t+1)} = w^{(t)} - \eta \cdot \hat{m}^{(t)} / (\sqrt{\hat{v}^{(t)}} + \epsilon)$$
  Use $\beta_1 = 0.9$, $\beta_2 = 0.999$, $\epsilon = 10^{-8}$, $\eta = 0.01$.
- **Bias correction is not optional:** Without bias correction, the first step is tiny because $m^{(1)} = (1 - 0.9) g = 0.1g$ — ten times smaller than the actual gradient. This makes Adam appear to converge slowly when it is actually working correctly but under-stepping. The correction divides by $(1 - \beta^t)$, which is small for small $t$ and approaches 1 as $t$ grows.
- **Element-wise operations:** $v$ accumulates squared gradient elements, not the squared gradient norm. The division $\hat{m} / (\sqrt{\hat{v}} + \epsilon)$ is element-wise. In 2D: `w -= lr * m_hat / (np.sqrt(v_hat) + eps)`.

**Trap:** Using the same $\eta$ for Adam as for GD. Adam is scale-invariant in a way GD is not, so sensible Adam learning rates are much smaller (0.001–0.01). Running Adam with $\eta = 0.1$ on a convex problem will often overshoot worse than GD.

**Done when:** Adam on $f = (1 - w_1)^2 + 100(w_2 - w_1^2)^2$ (Rosenbrock) converges noticeably faster than vanilla GD to $(1, 1)$.

---

### Phase 5 — Trajectory Table and Comparison

**The question this phase answers:** How do you present three optimizer trajectories in a way that makes their differences immediately readable?

**Key decisions:**
- **Interleave vs. separate tables:** Three separate tables are easier to follow step-by-step. One interleaved table (GD row, Momentum row, Adam row per step) makes cross-optimizer comparison at the same step easier. The interleaved format is more information-dense.
- **Which columns to print:** `step`, `w1`, `w2`, `loss`, `grad_norm`. Also consider `step_size = eta * grad_norm` for GD — it tells you immediately if the step is too large.
- **Precision:** 6 decimal places for weights (small changes matter), 4 for loss (relative scale matters more than absolute), 2 for grad_norm.
- **Sampling:** Printing every step for $T = 1000$ produces an unreadable 1000-line table. Print every 10th step, plus step 0 and the final step. Let the user override the print interval via a CLI flag.

**Trap:** Recomputing gradients inside the printing loop. Gradients computed for display must match exactly the gradients used for the update — compute once, store, display later.

**Done when:** The output table for all three optimizers fits in a terminal window (80 columns), clearly shows each optimizer reaching or failing to reach the minimum, and grad_norm at the last step is below $10^{-4}$ for at least one optimizer on a convex function.

## Edge Cases to Handle Before You Code

1. **Saddle points:** $f(w_1, w_2) = w_1^2 - w_2^2$ has a saddle at $(0,0)$. Gradient is zero there. GD gets stuck; Adam with sufficient noise can escape. Your visualizer should detect `grad_norm < 1e-10` and label the stop as "saddle point or minimum" rather than "converged."

2. **Flat regions / plateaus:** Functions like $f = \tanh(w_1) + \tanh(w_2)$ are nearly flat far from origin. Numerical gradient is nonzero but tiny; the optimizer makes tiny steps. Without a maximum step budget, it runs forever. Always enforce `max_iter`.

3. **Non-convex with multiple local minima:** $f = \sin(w_1) + \sin(w_2)$. The optimizer converges to whichever local minimum is closest to the start — the visualizer should print the final $(w_1, w_2, \text{loss})$ to let the user verify convergence quality. Don't label any convergence as "global minimum found."

4. **NaN or Inf in loss:** Triggered by $\eta$ too large. Detect after each update: `if not np.isfinite(loss): break` and print `"DIVERGED at step t"`. This is more informative than Python raising `RuntimeWarning` from numpy and continuing with garbage values.

5. **Very small initial learning rate:** If $\eta = 10^{-6}$ and `max_iter = 100`, the optimizer makes negligible progress. The user may think the code is broken. Print `"Warning: grad_norm at T=100 is X — optimizer may not have converged."` if final grad_norm is above a threshold.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|----------|----------|----------------|
| Gradient computation | Numerical (central diff) | Analytical (user supplies) | When you want the system to work for *any* callable without user effort |
| Stopping criterion | Fixed `max_iter` | Gradient norm threshold | Fixed `max_iter` for reproducibility; combine both |
| Print interval | Every step | Every $k$th step (user-set) | Every step for short runs (<50 iter), every 10th for long runs |
| Optimizer coupling | Run sequentially | Run in parallel (interleaved) | Sequential is simpler to implement; interleaved is more comparable |

## What Makes This Non-Trivial

The implementation is simple. The insight requires understanding *why* each optimizer behaves differently on the same landscape. Adam's bias correction makes it take large steps early but controlled steps later — the opposite of what raw moment accumulation would do. Momentum's overshoot on convex functions is not a bug; it's why momentum converges faster in theory (Nesterov achieves $O(1/t^2)$ vs $O(1/t)$ for vanilla GD). The saddle point problem reveals why first-order methods can fail: at a saddle, $\nabla f = 0$ but it is not a minimum. GD is completely blind to this. Noise (from stochastic gradients) or second-moment adaptation (Adam's $v_t$) is what helps escape saddles in practice — and this visualizer, using full-batch numerical gradients on deterministic $f$, will show GD stuck at a saddle indefinitely.

## Interview Angles This Project Develops

1. "Why does Adam converge faster than SGD in practice?" — Strong answer: Adam adapts the learning rate per parameter using EMA of squared gradients; parameters with high-variance gradients (common in sparse embeddings or layers with varied activation magnitudes) get smaller effective steps, preventing oscillation. The bias correction terms ensure the first update has the right scale.

2. "What is momentum in gradient descent, and what problem does it solve?" — Strong answer: Momentum maintains an exponential moving average of past gradients (heavy ball) or evaluates the gradient at a look-ahead point (Nesterov). It solves the oscillation problem in elongated loss surfaces — vanilla GD zigzags across the narrow axis while making slow progress along the long axis; momentum accumulates signal along the long axis and dampens the cross-axis oscillations.

3. "How do you verify your gradient implementation is correct?" — Strong answer: Gradient check — compare numerical gradient (central difference) against analytical gradient and verify the relative error is below $10^{-5}$. Also verify that the gradient at a known minimum is zero within floating-point tolerance.

4. "What is the difference between a local minimum and a saddle point, and how do optimizers distinguish them?" — Strong answer: Both have $\nabla f = 0$, but a local minimum has a positive-definite Hessian ($H \succ 0$) while a saddle has an indefinite Hessian. First-order methods cannot distinguish them without evaluating $H$. Stochastic gradient descent escapes saddles via noise; deterministic methods require second-order information or trust-region methods.

5. "When would you choose vanilla SGD over Adam?" — Strong answer: SGD with momentum often generalizes better on image classification tasks (evidenced by ResNet papers using SGD). Adam's per-parameter adaptation can overfit to the gradient statistics of training data. When you have strong priors about learning rate schedules and a convex or well-conditioned loss, SGD is more predictable and tunable.

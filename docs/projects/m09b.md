# Instruction Dataset Builder & Quality Filter — Thinking Guide

> **Module:** 09 — Fine-Tuning LLMs · **Difficulty:** Advanced · **Type:** THINKING · **Time:** ~3 hours

## The Problem

**Input:** A domain (e.g., medical QA, legal drafting, Python code generation) and a budget of N fine-tuning examples.

**Output:** A `.jsonl` file (or HuggingFace `Dataset`) where every row is a `{instruction, input, output}` triple ready for SFTTrainer. Every example is deduplicated, length-filtered, format-consistent, and free of test-set contamination.

**Constraints:** The pipeline must be reproducible (fixed seed), auditable (each example has a provenance tag — scraped, synthetic, curated), and large enough to shift model behavior while small enough to avoid memorizing noise.

## Before You Write Any Code: Build The Mental Model

Instruction fine-tuning is not about volume — it is about distributional shift. You want the model to move from its pretrained distribution $P_{\text{base}}$ toward a target distribution $P_{\text{domain}}$ without losing the general capabilities that make it useful. Every low-quality example is not merely wasted compute: it actively degrades the signal because the model's gradient update is proportional to its prediction error, so noisy targets inject noisy gradients that partially overwrite good learned representations.

Think of quality filtering as a precision-recall trade-off on the data itself. High recall (keep everything) gives you volume but dilutes signal-to-noise ratio. High precision (keep only clean examples) gives cleaner gradients but may reduce coverage of rare domain patterns. For most domain fine-tuning tasks, the optimal operating point is high precision — 1,000 clean examples consistently outperforms 10,000 noisy ones.

The three axes of data quality are orthogonal and all necessary: **format consistency** (can the model predict where answers begin and end?), **semantic quality** (are the answers factually correct and useful?), and **distributional coverage** (do the examples span the full range of real user queries?). Failing on any single axis degrades the fine-tuned model in a distinct, diagnosable way.

Deduplication is not optional. Web-scraped Q&A exhibits Zipf-like repetition — a small number of question types appear hundreds of times. Without deduplication, the model overfits to the frequent patterns and loses coverage. The cost of deduplication is almost always worth paying.

## Decompose It

### Phase 1 — Source Collection

**The question this phase answers:** Where do the examples come from, and how is each source tagged for downstream auditing?

**Key decisions:**
- **Web scraped vs curated vs synthetic**: Web scraping (Stack Overflow, PubMed Q&A, legal databases) gives volume but noisy formatting and unknown license. Curated datasets (Dolly, OpenAssistant, FLAN) give cleaner examples but may not match your specific domain. Synthetic data via an LLM (self-instruct style) gives format control and domain coverage but inherits whatever biases the generating model has.
- **Provenance tags**: Tag every example with `source: {scraped|synthetic|curated}` and `source_url` or `dataset_name`. Without provenance, you cannot diagnose which source is contributing noise, and you cannot honor DMCA takedowns.
- **License audit**: CC-BY and MIT are safe for commercial use. Stack Overflow content is CC-BY-SA 4.0 — attribution required. Many medical databases prohibit redistribution. Check before you build.

**Trap:** Mixing sources without provenance tags feels harmless at collection time but makes debugging impossible. When you notice the model behaving oddly on medical questions two weeks later, you need to know which rows came from PubMed and which came from a synthetic generator.

**Done when:** You have a raw pool ≥3× your target N, every row has a `source` tag, and you've confirmed the license on each source.

### Phase 2 — Format Normalization

**The question this phase answers:** Are all examples in a consistent instruction-input-output schema before any filtering?

**Key decisions:**
- **Template choice matters less than template consistency**: Whether you use Alpaca format (`### Instruction:`, `### Response:`), ChatML (`<|im_start|>`, `<|im_end|>`), or a custom delimiter, use exactly one template throughout. The model learns to predict the response given the context of the instruction marker. If 30% of examples use one template and 70% use another, the model learns a confused mixture.
- **Input field optionality**: In Alpaca format, `input` is the "context" field (optional supporting text). If your domain has no additional context (e.g., coding: the instruction is fully self-contained), always pass an empty string to `input` rather than omitting the field — absent keys break template renderers that use `.format()`.
- **Normalization scope**: Strip leading/trailing whitespace. Collapse multiple consecutive newlines to one. Normalize Unicode (use `unicodedata.normalize("NFC", text)`). These are cheap operations that prevent tokenizer surprises.

**Trap:** Synthetic data from GPT-4 often arrives with markdown formatting (asterisks, code fences). If your target model was not pretrained on markdown-heavy corpora, leaving these in increases perplexity on output tokens without adding semantic content.

**Done when:** Every row serializes to the same template string with no missing fields and no non-ASCII artifacts.

### Phase 3 — Length Filtering

**The question this phase answers:** Which examples are too short to carry useful signal, and which are too long to fit in the training context window?

**Key decisions:**
- **Minimum length**: An output shorter than ~20 tokens is almost always a one-word answer, a "yes/no", or a formatting artifact. These examples train the model to produce terse outputs — harmful if your domain requires explanation. Filter out examples where `len(output_tokens) < 20`.
- **Maximum length**: Set a hard cap at 90% of `max_seq_len`. SFTTrainer truncates silently at `max_seq_len`; if an example's answer is truncated mid-sentence, the model receives a gradient signal that rewards stopping mid-thought. Better to remove the example entirely.
- **Length distribution**: After filtering, check that the length distribution of outputs is roughly log-normal. A heavy spike at very short lengths suggests many one-liners survived the minimum filter; a spike at the maximum suggests many examples are being truncated.

**Trap:** Filtering on word count instead of token count is unreliable. The tokenizer's subword vocabulary means "antidisestablishmentarianism" is 6 tokens but "a" is 1. Always tokenize and measure in tokens.

**Done when:** No example exceeds 90% of `max_seq_len` in tokens. Output length distribution has median ≥ 50 tokens.

### Phase 4 — Deduplication

**The question this phase answers:** Which examples are near-duplicates, and which should be removed?

**Key decisions:**
- **Character-level MinHash (exact/near-exact)**: Fast, scalable to millions of examples. Use 5-gram shingling on the instruction text, build a MinHash signature (128 hash functions), and cluster via Locality Sensitive Hashing (LSH). Two examples in the same LSH bucket with Jaccard similarity > 0.8 are near-duplicates — keep one. This catches copy-paste variants ("What is X?" vs "What is X?!") but misses paraphrases.
- **Semantic deduplication (embedding cosine)**: Embed each instruction with a small sentence encoder (e.g., `all-MiniLM-L6-v2`). Compute pairwise cosine similarity and remove one of any pair with similarity > 0.92. This catches paraphrases but is $O(n^2)$ — use approximate nearest neighbors (FAISS) for N > 10,000.
- **Which to use**: Apply MinHash first (fast, removes exact duplicates), then semantic deduplication on the survivors (catches paraphrases). Semantic deduplication alone on a million examples requires a GPU and takes hours.
- **Keep the higher-quality duplicate**: If two examples are near-duplicates, keep the one from the curated source over the scraped source. Within a source, keep the one with the longer, more structured output.

**Trap:** Deduplicating on full `text` (instruction + output) misses cases where two different questions have the same answer. The duplicate that matters is the instruction — the output can be similar if the questions are genuinely similar.

**Done when:** No two instructions in the filtered set have Jaccard similarity > 0.8 (MinHash) or cosine similarity > 0.92 (embedding). Dataset size is typically 60–85% of pre-dedup size.

### Phase 5 — Perplexity Filtering

**The question this phase answers:** Which examples are outliers in the model's learned distribution — either too predictable (low perplexity) or too noisy (high perplexity)?

**Key decisions:**
- **Perplexity as a quality signal**: Score each output with a small language model (KenLM trained on your domain, or a 1B-parameter causal LM). Low perplexity means the output is highly predictable — the model already knows this pattern, and training on it adds little. High perplexity means the output is surprising — could be genuinely novel domain content, or could be malformed text, transcription errors, or language-mixed content.
- **Filtering bands**: Remove the bottom 10th percentile (too easy — the model will barely update on these) and the top 5th percentile (too surprising — likely noise). The sweet spot is examples where the model is moderately uncertain, meaning it has something to learn but the target is not garbage.
- **Domain-specific calibration**: Perplexity thresholds are not universal. Medical text is inherently high-perplexity for a general LM. Score with a domain-specific model if possible, or calibrate the percentile cutoffs on a manually reviewed sample.

**Trap:** Using the same model you are about to fine-tune to score perplexity creates a bias — the model's pretraining distribution determines what it finds surprising. A better approach is using a separate reference model (ideally a different architecture) or n-gram perplexity via KenLM.

**Done when:** Output perplexity distribution has no heavy tails. You've manually reviewed a sample from each removed percentile band and confirmed the filtering decision makes sense.

## Data Flow

```
Raw Sources
    │
    ├── Web Scrape (Stack Overflow, PubMed, GitHub Issues)
    ├── Curated HF Datasets (Dolly, FLAN, OpenAssistant)
    └── Synthetic (Self-Instruct pipeline via LLM)
         │
         ▼
    [Tag with source + provenance]
         │
         ▼
    Format Normalization
    (template render → NFC unicode → whitespace strip)
         │
         ▼
    Length Filter
    (tokenize → discard < 20 output tokens, > 0.9 * max_seq_len)
         │
         ▼
    MinHash Dedup  →  remove near-exact duplicates (Jaccard > 0.8)
         │
         ▼
    Semantic Dedup  →  FAISS ANN + cosine threshold (> 0.92)
         │
         ▼
    Perplexity Filter  →  discard bottom 10% + top 5%
         │
         ▼
    Toxicity Filter  →  discard examples flagged by classifier
         │
         ▼
    Final Dataset
    (shuffle, split train/val 95/5, save as HF Dataset or .jsonl)
```

## Edge Cases to Handle Before You Code

1. **Multi-language contamination**: Web scraping often pulls non-English Q&A even when you query in English. Use `langdetect` or `fasttext` language identification and filter to target language. A 2% non-English contamination rate is enough to noticeably degrade output quality in English.
2. **Test set leakage**: If any of your target benchmark datasets (MMLU, MedQA, HumanEval) appear in the web scrape, your eval metrics are inflated. Run an exact-match and fuzzy-match check between your training examples and all benchmarks you plan to evaluate on. This is called data contamination, and it invalidates your results.
3. **Template injection in scraped outputs**: Some Q&A sites format answers with `### ` headers or `---` dividers — exactly the tokens your Alpaca template uses as delimiters. These confuse the tokenizer's understanding of where instructions end and responses begin. Escape or strip these patterns from scraped text.
4. **Empty instruction after normalization**: Occasionally a scraping regex extracts a question that consists entirely of a URL or an image placeholder. After normalization, the instruction field is empty. Add a hard check: `assert len(example["instruction"].strip()) > 10` before any example enters the pipeline.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|---|---|---|---|
| Dedup method | MinHash (char n-gram) | Embedding cosine (semantic) | N > 100K or no GPU available; always run MinHash first |
| Synthetic data ratio | ≤ 20% synthetic | Majority synthetic (Self-Instruct) | You have reliable curated/scraped data; synthetic > 50% risks distributional collapse |
| Perplexity scorer | Separate reference LM | Target model itself | Always prefer separate scorer to avoid circularity |
| Template | Alpaca (3-field) | ChatML (role-based) | Match the base model's chat template if using an instruction-tuned base |
| Quality vs coverage | Strict filters (fewer, cleaner) | Loose filters (more, noisier) | Domain requires precision (medical, legal); loose filters acceptable for coding |

## What Makes This Non-Trivial

Instruction data quality is not a solved problem. The empirical finding from LIMA (Zhou et al., 2023) that 1,000 carefully curated examples outperform 50,000 FLAN examples implies that quality filtering is the dominant factor — but "quality" was defined by human annotators in that study, not by any automated metric. Your automated pipeline approximates human quality judgment via proxy signals (perplexity, length, deduplication), and each proxy captures a different failure mode while missing others. An example can pass all automated filters and still be subtly wrong (factually incorrect but fluent, appropriate length, unique) — this is the residual noise that makes instruction data collection fundamentally hard.

The deduplication-diversity tension is also non-trivial. Aggressive deduplication maximizes example uniqueness but can remove all instances of a valid pattern just because it appears frequently in natural data. If "What is the dosage of X?" appears 500 times in a medical dataset, removing all but one means the model only sees that pattern once — if it appears in the real distribution more often, the model will be underfit. A better approach is stratified subsampling: keep at most K examples per semantic cluster rather than exactly 1.

## Interview Angles This Project Develops

1. "What makes a good instruction-following dataset?" — Strong answer names the three axes (format consistency, semantic quality, distributional coverage), gives a concrete example of each failing (inconsistent templates → confused loss surface, wrong answers → noisy gradients, narrow coverage → poor generalization).
2. "How do you deduplicate a dataset of 10 million text examples?" — Strong answer gives MinHash + LSH as the scalable solution, explains Jaccard similarity and shingling, mentions FAISS for semantic dedup, and notes that character-level dedup misses paraphrases.
3. "What is perplexity filtering and why does it help?" — Strong answer explains that low-PPL examples offer no learning signal (model already fits them), high-PPL examples are likely noise, and the optimal training distribution is in the middle — moderately uncertain for the model.
4. "How would you detect test set contamination in your training data?" — Strong answer: exact-match and n-gram overlap check between training examples and all benchmark test sets before training. Contamination inflates eval metrics and produces a model that is genuinely worse at generalization than it appears.
5. "LIMA showed 1K examples can match 50K — does that mean dataset curation doesn't scale?" — Strong answer distinguishes between the two claims: quality matters more than volume at small scale, but large scale high-quality datasets (FLAN, OpenHermes) do outperform small-scale — the key finding is that noise dilutes signal, not that volume is useless.

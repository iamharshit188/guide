# Self-Debugging Code Agent — Thinking Guide

> **Module:** 10 — LLM Agents & Tool Use · **Difficulty:** Expert · **Type:** THINKING · **Time:** ~6 hours

## The Problem

**Input:** A broken Python script (full source, string), an error traceback (string), an expected output (string — what stdout should print when the fix is correct), and two limits: `max_attempts: int` (e.g., 5) and `timeout_seconds: int` per execution (e.g., 10).

**Output:** Either `{status: "success", fixed_code: str, attempts: int, execution_output: str}` or `{status: "failed", reason: str, attempt_history: list[dict], last_code: str}`. On success, the fixed code must produce output that matches expected output under the correctness criterion (configurable: exact string match, normalized whitespace match, or functional equivalence check).

**Constraints:** Code is never executed with `exec()` or `eval()`. Every execution happens in a subprocess with a timeout and a restricted environment. The agent must not repeat a fix it has already tried. The agent must terminate within `max_attempts` regardless of state.

## Before You Write Any Code: Build The Mental Model

The agent operates in a loop: observe error → hypothesize fix → apply fix → execute → verify → decide. This is the standard debugging cycle, compressed into an automated loop where the LLM is the hypothesizer. The key insight is that the LLM's hypothesis quality is determined by the quality of the context it receives — not just the error message, but the structural understanding of what the code was supposed to do.

Safe code execution is the first and hardest constraint. Running untrusted code means accepting that the code could do anything: delete files, open network connections, import system modules, enter infinite loops. The security model must assume adversarial input at worst and buggy input at best. Subprocess with timeout, restricted filesystem access, and no network is the minimum viable sandbox.

Cycle detection separates a competent agent from one that thrashes. The LLM has a strong prior toward certain fix patterns — adding a missing import, wrapping in try/except, fixing off-by-one errors. If a fix fails, the LLM will sometimes regenerate the exact same fix with different variable names. Storing the hash of each attempted fix and blocking re-attempts is necessary, not optional.

The correctness criterion is not self-evident. "Output matches expected" sounds simple, but trailing newlines, floating-point formatting differences, and OS-specific line endings all cause spurious failures. Defining correctness as "normalized output equality" (strip, lower, collapse whitespace) catches most spurious mismatches without allowing false positives.

## Decompose It

### Phase 1 — Safe Execution Environment

**The question this phase answers:** How do we run untrusted Python code without risking the host system?

**Key decisions:**
- **subprocess vs. exec() vs. eval():** `exec()` runs code in the current process — any `sys.exit()`, `os.remove()`, or infinite loop affects the host. `eval()` is for expressions only and still runs in-process. `subprocess.run()` creates a separate OS process: the host survives any crash, timeout kills the process cleanly, and resource usage is isolated. Use `subprocess.run()`. Never `exec()` or `eval()`.
- **Timeout enforcement:** `subprocess.run(cmd, timeout=N)` raises `subprocess.TimeoutExpired` if the process doesn't finish in N seconds. On timeout, the subprocess is killed with SIGKILL (Unix) or TerminateProcess (Windows). Always set a timeout — infinite loops are the most common bug in student code. 10 seconds is sufficient for any code that doesn't do I/O.
- **Environment isolation:** Pass `env={"PATH": "/usr/bin:/bin"}` to strip the subprocess environment. This prevents the code from inheriting API keys, database URLs, or other environment variables from the parent. Pass only what the script legitimately needs.
- **Filesystem restriction:** On Linux, use `subprocess` inside a `bubblewrap` or `firejail` sandbox for filesystem isolation. On macOS/Windows without sandboxing tools, at minimum: run the subprocess from a temp directory, and pass `cwd=tempdir` so any file writes go to a throwaway location.
- **Capture stdout/stderr separately:** Always use `capture_output=True` or `stdout=PIPE, stderr=PIPE`. Print them separately in the agent's output — stderr contains the traceback even when the code runs (e.g., warnings), and conflating the two makes correctness checking unreliable.

**Trap:** Using `shell=True` in subprocess. This passes the command through the shell, which enables shell injection: if the code string contains `;rm -rf ~` and you concatenate it into a shell command, you have a critical vulnerability. Always pass a list of arguments: `subprocess.run(["python3", script_path], ...)`.

**Done when:** A function `run_code(source: str, timeout: int) -> ExecutionResult` exists that: (1) writes source to a temp file, (2) runs it in a subprocess with timeout, (3) returns stdout, stderr, returncode, and whether it timed out. Tested with an infinite loop (should timeout), a syntax error (should return stderr with traceback), and a correct script (should return stdout).

### Phase 2 — Context Assembly for the LLM

**The question this phase answers:** What information does the LLM need to diagnose and fix the bug?

**Key decisions:**
- **Full file vs. relevant sections:** The LLM context window is finite. For scripts under 200 lines, always send the full file — line numbers help the LLM reference specific locations. For scripts over 200 lines, extract: (a) the function containing the error (from the traceback line number), (b) all imports, (c) any global state referenced by that function. Sending only the error message without code produces low-quality fixes.
- **Traceback format:** Python tracebacks include file path, line number, function name, and the offending line. Strip the file path (it contains temp directory noise) but keep line numbers. Send the full traceback, not just the last line — the chain of calls matters for debugging `AttributeError` or `KeyError` that surface in a library call but originate in user code.
- **Attempt history in context:** Send the N most recent fix attempts (code diff + execution result) as context. The prompt structure: "Previous fix attempt [N]: changed line X to Y. Result: NameError on line Z." This prevents the LLM from repeating dead-end paths and nudges it toward different strategies.
- **Expected output in context:** Always send the expected output. The LLM can often infer the intended algorithm from what the output should look like — especially useful for logic bugs where the code runs but produces wrong results.

**Trap:** Sending only the error message without the surrounding code. The LLM will produce generic fixes ("add try/except", "check for None") that address the error type but not the specific bug. Context specificity determines fix quality — the LLM needs to see exactly which line, which variable, and what the value should be.

**Done when:** A function `build_prompt(source: str, traceback: str, expected: str, history: list[dict]) -> str` assembles a structured prompt that includes all required context, fits within a 4 000-token limit (truncating source if needed, preserving traceback and history), and follows a consistent format the LLM can parse.

### Phase 3 — Fix Generation and Application

**The question this phase answers:** How does the agent extract a clean, applicable fix from the LLM's response?

**Key decisions:**
- **Identify-first vs. patch-first:** Some bugs require understanding the algorithm to fix them. A sorting algorithm with wrong comparison logic cannot be fixed by patching one line — the whole comparison function needs rethinking. The LLM should first output a diagnosis ("the issue is that the comparison is inverted") before the fix. Prompt structure: "First, in one sentence, explain what is wrong. Then output the fixed code in a code block." This forces identify-first behavior.
- **Code extraction from LLM response:** The LLM will wrap the fixed code in ```python ... ``` blocks. Extract with a regex: `re.search(r'```python\n(.*?)```', response, re.DOTALL)`. If no code block is found, the LLM gave a textual explanation without code — handle this by prompting again with "Now provide the fixed code in a code block."
- **Applying the fix:** Replace the full source with the LLM's fixed version. Do not try to apply line-level diffs from the LLM — diff application requires exact matching and the LLM's diffs are often slightly off. Full file replacement is more robust even if it re-sends unchanged code.
- **Syntax validation before execution:** Before running, call `compile(source, '<string>', 'exec')` to catch syntax errors without execution. If the LLM's fix has a syntax error, don't waste a subprocess call — note the syntax error, add it to history, and regenerate.

**Trap:** Trusting that the LLM outputs only one code block. Sometimes it outputs two: "here is the problematic function" and "here is the full fixed file." Always take the last code block, which is typically the full file. Or, more reliably: instruct the LLM in the prompt to "output exactly one code block containing the complete fixed file."

**Done when:** Given an LLM response string, the agent reliably extracts exactly one code block, validates its syntax, and either returns the fixed source or re-prompts on extraction failure.

### Phase 4 — Cycle Detection

**The question this phase answers:** How does the agent detect when it is repeating itself and break out of the loop?

**Key decisions:**
- **Hash-based deduplication:** After each fix is extracted, compute `hashlib.sha256(fixed_source.encode()).hexdigest()`. Store all previous hashes in a set. If the new hash is already in the set, the LLM generated an identical fix — block execution, add a "duplicate fix detected" note to history, and force a different prompt.
- **Near-duplicate detection:** The LLM may generate a fix that changes only whitespace or variable names. A pure hash check misses this. Normalize the source before hashing: strip comments, normalize whitespace, sort imports. Near-duplicate normalization catches ~80% of semantically-identical fixes.
- **Breaking the cycle:** When a duplicate is detected, modify the prompt: "Your previous attempt used approach X. That approach failed. Try a completely different strategy. Consider: [list 2-3 alternative fix strategies for this error type]." Seeding alternative strategies from a fixed lookup table (e.g., for `AttributeError`: "check None, check attribute name spelling, check import path") breaks the LLM's fixation.
- **Escalation:** If 3 consecutive duplicates are detected, the agent should escalate the prompt to "explain what you think the fundamental problem is without trying to fix it" — sometimes the LLM needs to re-analyze before it can generate a new approach.

**Trap:** Only checking exact hash equality. The LLM sometimes adds a comment saying "attempt 2" to its code, producing a different hash for what is functionally identical code. Normalize before hashing.

**Done when:** A `FixHistory` class tracks all attempted hashes, detects duplicates, generates a modified prompt when a duplicate is detected, and can report: "Attempt N is semantically identical to Attempt M."

---

### Phase 5 — Correctness Verification and Termination

**The question this phase answers:** What does "correct" mean, and when does the agent give up?

**Key decisions:**
- **Exact match:** `actual_output.strip() == expected_output.strip()`. Appropriate when expected output is a well-defined string (test suite output, specific numeric result).
- **Normalized match:** Collapse all whitespace, normalize floats to 2 decimal places, lowercase. Appropriate for outputs with formatting variability.
- **Functional equivalence:** Parse the output as a structured object (JSON, list of numbers) and compare semantically. Appropriate when the output format is flexible but the values must be correct. Implement as a configurable strategy passed to the agent at construction time.
- **Partial credit reporting:** On failure, report the edit distance between actual and expected output, and the longest common prefix. This helps the LLM understand how close it is on the next attempt — especially useful for off-by-one errors where the output is almost right.
- **Termination conditions:** (1) `returncode == 0` and output matches expected → success. (2) Attempt count reaches `max_attempts` → fail with reason "max attempts reached". (3) Three consecutive duplicate fixes → fail with reason "stuck in fix loop". (4) Syntax error on final attempt → fail with the syntax error. The agent always returns a structured result, never raises an unhandled exception.

**Trap:** Treating "code runs without error" as success. A script that prints `None` instead of the expected value has `returncode == 0` but is wrong. Always verify output against expected, not just return code.

**Done when:** The agent's main loop terminates under all four conditions, returns a structured result in both success and failure cases, and the attempt history contains enough information to reconstruct exactly what each attempt tried and why it failed.

## Data Flow

     │
     ▼
attempt_count = 0

## Edge Cases to Handle Before You Code

1. **The fix makes the code worse:** The code had a `NameError` on line 10; the LLM's fix introduces a `TypeError` on line 5. This is regression — now the error is different, the traceback is different, and the next attempt context is misleading. The agent must always send the traceback from the most recent execution, not the original. Never assume the bug is in the same place after a failed fix.

2. **Expected output contains regex-sensitive characters:** If expected output is `[1, 2, 3]` and you do a regex match, the brackets are metacharacters. Always use string equality, not regex, for expected output comparison unless the caller explicitly provides a regex pattern.

3. **The script has external dependencies not installed:** If the script imports `pandas` and pandas isn't installed, the fix will always fail with `ModuleNotFoundError` regardless of what the LLM does. The agent must detect import errors specifically and report "dependency not satisfiable" rather than burning attempts on a fixable-looking error.

4. **The script reads from stdin or a file:** A script with `input()` will hang indefinitely in the subprocess. Detect this: if the script contains `input(` or `sys.stdin`, either inject a mock stdin (pipe a string to the subprocess's stdin) or flag the script as requiring interactive input and refuse to execute it.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|---|---|---|---|
| Execution isolation | Full subprocess (separate process) | `exec()` in restricted namespace | Always use subprocess — namespace restrictions can be bypassed and the host process is still at risk |
| Correctness criterion | Exact string match | Normalized / functional equivalence | Use exact match for well-defined outputs; normalized for formatting variability |
| Context window when source is long | Send full source (may exceed limit) | Extract relevant function + imports only | Extract when source > 150 lines; full file when shorter — traceback line numbers help more when full context is visible |
| Duplicate fix handling | Skip and re-prompt immediately | Count duplicates; escalate after 3 | Count and escalate — skipping immediately can oscillate between two equally-bad fixes |

## What Makes This Non-Trivial

The agent's correctness depends on two independent systems working in concert: the LLM's ability to reason about code, and the execution environment's ability to faithfully test the result. A failure in either system produces incorrect behavior — a bad fix passes if the correctness check is too lenient; a good fix fails if the subprocess environment lacks required packages. The agent must be transparent about which system caused each failure.

Cycle detection is not a simple optimization — it is a correctness requirement. An agent without cycle detection can produce an infinite loop of identical failing fixes, wasting all attempts and returning a useless error. The fix history is not just for debugging humans; it is the agent's memory that prevents thrashing, and it must be integrated into every LLM prompt from attempt 2 onward.

The escalation policy when max_attempts is reached is an open design question with no universally correct answer. The agent can return the last attempted fix (which may be close to correct), the attempt that had the lowest edit distance to expected output, or a structured diagnostic report. The choice depends on the downstream use case — a human debugging assistant benefits from the diagnostic; an automated CI pipeline benefits from the closest-to-correct code.

## Interview Angles This Project Develops

1. **"Why is exec() dangerous for running untrusted code?"** — Strong answer explains in-process execution: sys.exit() kills the host, os.remove() operates on the real filesystem, and no amount of namespace restriction prevents all attacks. Subprocess is the correct boundary.

2. **"How does the agent avoid getting stuck in an infinite loop of bad fixes?"** — Strong answer describes hash-based deduplication of fix attempts, normalization before hashing, and the escalation prompt strategy that seeds alternative approaches.

3. **"What context does the LLM need to fix a bug well?"** — Strong answer lists: full source (or relevant function + imports), the full traceback with line numbers, expected output, and the history of failed attempts with their specific error messages. Any missing element degrades fix quality in a predictable way.

4. **"How do you define 'correct output' in an automated testing agent?"** — Strong answer distinguishes exact match, normalized match, and functional equivalence, explains when each is appropriate, and notes that the correctness criterion should be configurable and documented — not hardcoded.

5. **"What happens when the LLM's fix introduces a new bug?"** — Strong answer explains that the agent always uses the traceback from the most recent execution, never the original; that regression is expected and the history prevents repeating the pre-regression state; and that attempt limit exists precisely because regression chains can be long.

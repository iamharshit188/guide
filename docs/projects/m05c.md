# Production Data Drift Detection System — Thinking Guide

> **Module:** 05 — Deep Learning & MLOps · **Difficulty:** Advanced · **Type:** THINKING · **Time:** ~5 hours

---

## The Problem

**Input:** A reference dataset $D_{ref}$ (training data distribution), a sequence of incoming live windows $D_1, D_2, \ldots, D_T$ (each a batch of recent inference requests), and a deployed model's feature schema (column names, dtypes, expected ranges).
**Output:** Per-window drift report: PSI score per feature, KS statistic and p-value per numerical feature, chi-squared statistic and p-value per categorical feature. A boolean `DRIFT_DETECTED` flag per window. All metrics logged to MLflow per window.
**Constraints:** No ground-truth labels available during monitoring (live inference context). Detection must run on the feature vector only — not model outputs. Windows must be processed incrementally without reloading the reference dataset each time.

---

## Before You Write Any Code: Build The Mental Model

Drift is not a single phenomenon. **Covariate drift** ($P(X)$ changes, $P(Y|X)$ stable) means the input distribution shifted — maybe your user base changed, or a data pipeline started dropping a field. **Concept drift** ($P(Y|X)$ changes) means the relationship between features and labels changed — your model's learned function is now stale. **Label drift** ($P(Y)$ shifts) means the class balance changed. Of these, only covariate drift is detectable without labels, because you only observe $X$ at inference time. Label and concept drift require ground truth, which arrives with a delay (if at all). This system detects covariate drift.

PSI (Population Stability Index) measures whether a distribution has shifted enough to invalidate a model trained on the reference. It is symmetric: $PSI = \sum_{i=1}^B (P_i - Q_i) \ln(P_i / Q_i)$ where $P_i$ is the reference proportion in bin $i$ and $Q_i$ is the current proportion. PSI < 0.1 is considered stable; 0.1–0.25 is a warning; > 0.25 signals significant drift requiring investigation. These thresholds are industry heuristics (from credit risk modelling), not statistical guarantees.

The KS test is a non-parametric two-sample test. Its statistic $D = \sup_x |F_{ref}(x) - F_{live}(x)|$ is the maximum vertical gap between the two empirical CDFs. The p-value under the null hypothesis that both samples come from the same distribution decreases as the gap grows. A p-value < 0.05 rejects the null at the 5% significance level — but with 100 features and 100 windows, expect ~5 false positives by chance (multiple testing problem). Bonferroni correction adjusts the threshold to $\alpha / n_{features}$.

PSI and KS are complementary. PSI aggregates drift across the entire distribution into one scalar — good for monitoring dashboards. KS is a formal test with a p-value and works with smaller samples. Use both: PSI for alerting, KS for statistical confirmation.

## Decompose It

### Phase 1 — Reference Profile Computation

**The question this phase answers:** What summary of the training distribution do you store so you can compare any future window against it?

**Key decisions:**
- For each numerical feature, compute: bin edges (using `np.histogram_bin_edges` with `bins=10` on reference data), reference bin proportions $P_i = \text{count}_i / N_{ref}$ with Laplace smoothing (add $\epsilon=0.001$ to each bin count before normalising to prevent $\ln(0)$ in PSI).
- For each categorical feature, compute: reference category proportions $P_c = \text{count}_c / N_{ref}$ with smoothing.
- Store bin edges alongside proportions — they define the discretisation scheme. Live windows must use the **same bin edges** as the reference, not recomputed bin edges. If you let each window define its own bins, you cannot compare proportions.
- Serialise the reference profile to JSON/dict so it loads instantly without recomputing from the raw training set.

**Trap:** Do not compute bin edges on the live window. If a live window has a different range than the reference (e.g., an outlier pushes max to 1000 when training max was 100), the bins shift and PSI becomes meaningless. Clamp live values to `[ref_min - δ, ref_max + δ]` or place all values outside the reference range into edge bins.

**Done when:** Reference profile dict has keys for each feature; each key maps to `{bins, ref_probs}` for numerical features or `{categories, ref_probs}` for categorical ones. Can be serialised to disk in < 1ms.

---

### Phase 2 — Window Drift Computation

**The question this phase answers:** Given a live window of $N_{live}$ samples, what is the PSI and KS statistic for each feature?

**Key decisions:**
- PSI computation:
  ```
  live_counts, _ = np.histogram(live_col, bins=ref_bins)
  live_probs = (live_counts + eps) / (live_counts.sum() + eps * n_bins)
  psi = np.sum((ref_probs - live_probs) * np.log(ref_probs / live_probs))
  ```
- KS test: `scipy.stats.ks_2samp(ref_sample, live_col)`. This requires keeping a reference sample (e.g., 10K rows from training data), not just the binned profile. Store this separately.
- Chi-squared for categoricals: `scipy.stats.chisquare(live_counts, f_exp=ref_probs * N_live)`. Expected counts < 5 in any bin will make the test invalid — merge rare categories before computing.
- Apply Bonferroni correction: alert threshold for KS p-value is $\alpha / n_{features}$ where $\alpha = 0.05$.

**Trap:** The KS test requires the raw reference sample, not a histogram. If your reference dataset is 1M rows, you can't store all of it — subsample 10K–50K rows at profile creation time and store that as the reference sample. This introduces slight approximation but is acceptable.

**Done when:** Function `compute_drift(window_df, reference_profile, ref_sample)` returns a dict of `{feature: {psi, ks_stat, ks_pval, chi2, chi2_pval, drift_flag}}`.

---

### Phase 3 — Window Streaming and Alert Logic

**The question this phase answers:** How do you process a sequence of windows and aggregate drift signals into an actionable alert?

**Key decisions:**
- Define window size $W$ and stride $S$. Tumbling windows (stride = window, no overlap) are simplest. Sliding windows (stride < window) smooth out transient spikes at the cost of correlation between adjacent windows.
- Alert conditions:
  - **Feature-level:** PSI > 0.25, OR KS p-value < Bonferroni threshold.
  - **Window-level:** DRIFT_DETECTED if > $k$ features trigger feature-level alert (e.g., $k = \max(1, 0.1 \times n_{features})$ — at least 10% of features drifting).
- Window-level aggregation prevents single-feature outliers from triggering a false alarm. A single feature with PSI = 0.26 is usually noise; 15 out of 20 features drifting simultaneously is a pipeline failure.

**Trap:** Don't treat every window independently. If window 3 shows borderline drift (PSI = 0.22) but windows 1, 2, 4, 5 are all stable, window 3 is likely a statistical fluctuation — especially if $N_{live}$ is small. Log the drift score as a time series and apply a smoothing rule: alert only if 2 consecutive windows exceed the threshold.

**Done when:** `process_stream(windows, reference_profile, ref_sample)` returns a list of `{window_id, drift_detected, features_drifted, max_psi, min_ks_pval}` dicts.

---

### Phase 4 — MLflow Logging

**The question this phase answers:** How do you make drift trends observable and queryable over time?

**Key decisions:**
- One MLflow run per monitoring session (e.g., one day or one deployment). Log each window as a step: `mlflow.log_metric("psi_feature_age", 0.15, step=window_id)`.
- Log summary metrics per window: `max_psi`, `n_drifted_features`, `drift_detected` (0 or 1).
- Log model version and deployment date as run params so you can correlate drift events with deployment history.
- Log the reference profile dict as a JSON artifact once — it's the anchor for all drift comparisons in this run.

**Trap:** Do not create a new MLflow run per window — you won't be able to plot per-feature PSI as a time series in the MLflow UI. Time series in MLflow requires consistent metric keys logged at sequential steps within the same run.

**Done when:** `mlflow ui` shows a run with metric history for each feature's PSI over all windows. Alert events are visible as spikes.

---

### Phase 5 — Alert Response and Report

**The question this phase answers:** When drift is detected, what should the system recommend?

**Key decisions:**
- Print a terminal report per window:
  ```
  Window 07 | 2 features drifted | Max PSI: 0.31 | DRIFT DETECTED
    feature: income    | PSI: 0.31 | KS p-val: 0.003 | ALERT
    feature: age       | PSI: 0.08 | KS p-val: 0.21  | ok
    feature: zip_code  | PSI: 0.18 | KS p-val: 0.04  | WARNING (chi2)
  ```
- Include a recommendation engine (simple rule table):
  - PSI > 0.25 on ≥ 3 features → "Investigate upstream data pipeline"
  - PSI > 0.25 on 1 feature, ks p < 0.001 → "Check sensor/API for that feature"
  - Drift detected but model accuracy unchanged (if labels available) → "Covariate drift without performance degradation — monitor only"

**Why retraining is not always the right response:** If a feature drifts but the model relies on it weakly (low importance), retraining on new data may not improve accuracy and risks losing generalisation to edge cases. The right response depends on: (a) whether the drifted feature matters to the model, (b) whether performance has actually degraded, and (c) whether enough new labeled data exists to retrain on. Drift is a signal to investigate, not an automatic retraining trigger.

**Done when:** Final summary report printed. Drift log saved as CSV. MLflow run complete.

## Edge Cases to Handle Before You Code

1. **New categories in live data:** A categorical feature in the live window has a value not seen in training (e.g., a new country code). Your chi-squared expected vector has no entry for it. Bin all unseen categories into an `<OTHER>` bucket in both reference and live profiles.
2. **Small windows:** If $N_{live} < 30$ per feature, the KS test has very low power — it can't reliably detect even moderate drift. Print a warning and rely on PSI only. Set a minimum window size threshold.
3. **Missing features in live data:** A feature is present in the reference profile but missing from the live window (upstream schema change). This is itself a form of drift — log it as `psi = inf`, flag immediately, print a schema mismatch alert.
4. **Constant features:** A feature has zero variance in the live window (e.g., all values = 0 — pipeline zeroed it out). KS test returns a statistic of 1.0 and PSI may be very high. The monitoring system should distinguish constant-feature drift from distributional drift; they have different root causes.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|----------|----------|----------|---------------|
| Window type | Tumbling (non-overlapping) | Sliding (overlapping) | Tumbling for lower compute; sliding when window size is large and you need finer time resolution |
| PSI bin count | 10 bins | 20 bins | 10 bins for features with sparse support; 20 bins for continuous features with large $N_{ref}$ |
| Alert aggregation | Any-feature threshold | Fraction-of-features threshold | Fraction-of-features is more robust — single-feature spikes are often noise |
| Reference KS sample | 10K subsample | Full training set | 10K is sufficient; KS convergence requires $O(\sqrt{n})$ — 10K gives ≈ 0.01 resolution |
| Retraining trigger | PSI > 0.25 on any feature | PSI > 0.25 + accuracy drop > 5% | Always condition on accuracy drop if labels are available — drift alone is not sufficient justification |

## What Makes This Non-Trivial

PSI's thresholds (0.1, 0.25) are empirical rules from credit risk, where PSI was developed. They are not universally calibrated. A PSI of 0.15 on a highly impactful feature may degrade model performance more than a PSI of 0.30 on an unimportant feature. The system can compute drift but cannot assess its impact without knowing feature importances.

Multiple testing is a genuine problem. With 50 features and $\alpha = 0.05$, you expect 2–3 features to appear drifted in every window by chance. Without Bonferroni (or FDR) correction, your alert system fires constantly. With aggressive Bonferroni, you lose power to detect real drift in individual features. There is no perfect answer — you are making an explicit trade-off between false positive rate and detection sensitivity.

Window size is a free parameter with a non-obvious optimum. Small windows detect drift faster but have high variance — many false alarms. Large windows reduce variance but introduce detection latency. In a production system processing 10K requests/hour, a 1-hour window of 10K samples gives reasonable statistical power for most KS tests. For lower-traffic models, you may need to aggregate across days.

## Interview Angles This Project Develops

1. **"What is the difference between covariate drift and concept drift, and which can you detect without labels?"** — Covariate drift changes $P(X)$; concept drift changes $P(Y|X)$. Only covariate drift is detectable without ground truth labels. Monitoring model output distribution (prediction drift) is a proxy for concept drift but is not equivalent.
2. **"Explain PSI and its thresholds."** — $PSI = \sum_i (P_i - Q_i) \ln(P_i/Q_i)$; symmetrised KL divergence scaled by bin differences. Thresholds (< 0.1 stable, 0.1–0.25 warning, > 0.25 significant) are heuristics from credit risk, not universal.
3. **"Why isn't retraining the automatic response to detected drift?"** — Because drift on a low-importance feature doesn't affect model performance; because the new data may not have labels yet; because retraining on drifted data may overfit to a transient pattern and hurt performance when the distribution reverts.
4. **"How do you handle the multiple testing problem in drift detection?"** — Bonferroni correction (divide $\alpha$ by number of features) is conservative but simple. Benjamini-Hochberg FDR control is less conservative and preferred when you have many features.
5. **"What's the difference between PSI and KS as drift metrics?"** — PSI is a single aggregated scalar good for dashboarding; no p-value, so no formal significance test. KS gives a test statistic and p-value for formal significance testing but requires storing a reference sample, not just a histogram.

## Extensions

1. **Feature importance weighting** — multiply each feature's PSI by its model feature importance score. Compute a weighted PSI to prioritise alerts on features the model actually relies on.
2. **Wasserstein distance** — replace KS with the earth mover's distance ($W_1$) for numerical features. `scipy.stats.wasserstein_distance(ref_sample, live_col)` is a more sensitive measure for distribution shift in heavy-tailed data.
3. **Auto-threshold calibration** — run the drift detector on the training data itself (splitting into reference and held-out halves, no actual drift). Record the PSI distribution under no drift. Set the alert threshold at the 95th percentile of that null PSI distribution.

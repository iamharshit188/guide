# Multi-Model Serving Gateway — Thinking Guide

> **Module:** 11 — Deployment & Production ML · **Difficulty:** Expert · **Type:** THINKING · **Time:** ~7 hours

## The Problem

**Input:** HTTP requests to a single API gateway at `POST /predict/{model_name}` where `model_name` ∈ {`text-classifier`, `image-embedder`, `summarizer`}. Each request body contains the input (text or image bytes) and an optional `model_version` field.

**Output:** Model predictions returned as JSON within an SLA of 200ms for the classifier, 500ms for the embedder, 1000ms for the summarizer. The gateway also exposes: `GET /health/{model_name}`, `POST /models/{model_name}/load`, `DELETE /models/{model_name}`, `GET /metrics` (per-model request count, latency p50/p95/p99, error rate, GPU memory usage).

**Constraints:** Total GPU memory is 16 GB shared across all models. No model should be able to cause another model to OOM. A model that is crashing repeatedly must be circuit-broken — its requests must fail fast rather than queueing and degrading the gateway. Model swaps (v1 → v2) must complete without dropping requests in flight.

## Before You Write Any Code: Build The Mental Model

A multi-model gateway is fundamentally a resource multiplexer. GPU memory, GPU compute, and CPU threads are all shared resources. The three failure modes you are designing against are: (1) one model consuming all GPU memory and starving others, (2) one model's slow requests blocking the queue and increasing latency for everyone, (3) a crashing model exhausting retry budget and degrading the entire gateway's throughput.

Batching is a GPU throughput optimization at the cost of individual request latency. A single image embedding request takes 5ms on the GPU; batching 32 requests takes 20ms total — a 8x throughput improvement. But the 32nd request waited up to 19ms in the queue before being processed. The queue timeout is the control knob: set it too low and you batch rarely; set it too high and latency SLA is violated. This is not a problem with a single correct answer — it is a latency/throughput trade-off that must be tuned empirically per model.

GPU memory fragmentation is subtler than RAM fragmentation. PyTorch's caching allocator holds freed memory in a cache rather than returning it to CUDA. When you load model A, use it, unload it, then load model B which needs slightly different block sizes, the cached blocks may not fit model B's allocations. `torch.cuda.empty_cache()` releases the PyTorch cache but does not defragment CUDA's internal allocator. The only reliable way to reclaim memory is to ensure the model's tensor references are deleted before loading a new model.

Circuit breakers prevent cascade failures. If the summarizer is OOMing on every request, without a circuit breaker, every incoming summarizer request queues, waits, OOMs, and returns a 500 — consuming GPU memory and CPU threads for the full duration of each attempt. With a circuit breaker in the open state, requests fail immediately with a 503 and no resources are consumed. The gateway remains healthy for the other two models.

## Decompose It

### Phase 1 — Model Registry and Lifecycle Management

**The question this phase answers:** How do you load, track, and unload models without leaving orphaned tensors on the GPU?

**Key decisions:**
- **Model registry as single source of truth:** A `ModelRegistry` class owns all loaded model objects. No module outside the registry holds a direct reference to a model tensor. This enforces that unloading goes through one code path: `del registry.models[name]` followed by `torch.cuda.empty_cache()`. If other modules hold references, `del` won't free the tensor.
- **Reference counting before unload:** Before unloading a model, check that no active requests are using it. The registry tracks `active_requests: dict[model_name, int]`. Unload is deferred until `active_requests[name] == 0`. A request handler increments on enter, decrements on exit (in a `finally` block — always, even on exception).
- **GPU memory accounting:** Before loading a new model, estimate its memory requirement (from a config file or measured on first load) and check against `torch.cuda.mem_get_info()[0]` (available memory). If the model won't fit, return an error rather than attempting to load and OOMing.
- **Model states:** `UNLOADED → LOADING → WARM → DEGRADED → UNLOADING → UNLOADED`. The `LOADING` and `UNLOADING` states are critical — requests arriving during these states should receive a 503, not queue indefinitely.

**Trap:** Calling `model.cpu()` and expecting GPU memory to free immediately. Moving a model to CPU frees GPU memory only after the garbage collector collects the GPU tensor — which is non-deterministic. Use `del model` followed by `torch.cuda.empty_cache()`, and verify with `torch.cuda.memory_allocated()` that memory dropped.

**Done when:** The registry can load a model, verify it appears in `torch.cuda.memory_allocated()`, unload it, verify memory returns to baseline, and reload it — without memory growing each cycle (no orphaned tensors).

### Phase 2 — Per-Request Batching with Queue Timeout

**The question this phase answers:** How do you batch concurrent requests without violating latency SLAs?

**Key decisions:**
- **Batching architecture:** A `BatchCollector` per model runs in a background thread. Incoming requests add their input to the collector's queue. The collector fires a batch when: (a) `batch_size` requests are queued, or (b) `batch_timeout_ms` has elapsed since the first request in the current batch arrived. Each request is a `Future` — the caller blocks on the future until the batch result is written back.
- **Batch timeout as the tuning knob:** The timeout controls the latency/throughput trade-off. For the text classifier (SLA: 200ms), batch_timeout should be ≤ 20ms — you need 180ms margin for model execution. For the summarizer (SLA: 1000ms), batch_timeout can be 100ms. The timeout is per-model and should be a configuration parameter, not a constant.
- **Max batch size:** Determined by GPU memory per inference, not by a magic number. Profile the model: how much GPU memory does a forward pass of batch size 32 consume? Set max_batch_size such that the forward pass + model weights fit in the allocated memory pool.
- **Result routing:** A batch of 8 requests produces 8 outputs. Each output must be routed back to the correct caller. Index the futures: `futures[i].set_result(outputs[i])`. If the batch fails (OOM, model error), set an exception on all futures in the batch: `futures[i].set_exception(e)`.

**Trap:** Using a single global batch queue. If the summarizer is slow (1-2s per batch) and the classifier is fast (20ms per batch), a global queue means classifier requests can be delayed waiting behind summarizer batches. Per-model queues are non-negotiable.

**Done when:** A load test sending 100 concurrent classifier requests completes with measured throughput 5–10x higher than serial execution, and p99 latency stays within SLA. The batch collector logs: batch sizes (should cluster near max_batch_size at high load), and timeout-triggered batch ratio (should approach 0 at high load and 1 at low load).

### Phase 3 — GPU Memory Pool Management

**The question this phase answers:** How do you prevent one model from OOMing the others?

**Key decisions:**
- **Memory budgets per model:** Assign each model a GPU memory budget at startup. Text classifier: 2 GB. Image embedder: 6 GB. Summarizer: 7 GB. Total: 15 GB of 16 GB. Reserve 1 GB as an unallocated buffer — CUDA itself needs ~500 MB for its own data structures.
- **CUDA memory limits:** PyTorch exposes `torch.cuda.set_per_process_memory_fraction(fraction)` — but this limits the whole process, not per-model. For true per-model isolation, run each model in a separate process with its own `memory_fraction`. The gateway communicates with model processes via IPC (shared memory or a fast queue). This adds latency (~0.5ms) but provides genuine isolation: a model OOM kills its worker process, not the gateway.
- **Fallback: soft limits with monitoring:** If separate processes are too complex, implement soft limits: before each forward pass, check `torch.cuda.memory_allocated()`. If the model's current allocation exceeds its budget, reject the request with a 507 (Insufficient Storage). This doesn't prevent OOM if the forward pass itself uses more than expected, but it prevents steady-state memory exhaustion.
- **GPU memory fragmentation:** After model load/unload cycles, memory can fragment: many small free blocks, no large contiguous block. Detected by: `torch.cuda.mem_get_info()[0]` (free) is large but `torch.cuda.memory_reserved()` - `torch.cuda.memory_allocated()` (fragmented cache) is also large. Resolution: schedule periodic `torch.cuda.empty_cache()` during low-traffic periods, or restart the worker process on a schedule.

**Trap:** Assuming `torch.cuda.empty_cache()` fixes fragmentation. It releases the PyTorch cache to CUDA but does not defragment CUDA's virtual memory. Fragmentation can only be truly fixed by releasing all allocations (process restart) or by using `torch.cuda.memory.set_allocator_settings({"expandable_segments": True})` (PyTorch ≥2.0), which allocates memory in expandable segments that avoid some fragmentation patterns.

**Done when:** Simultaneous load of all three models stays within 15 GB total. Loading and unloading the text classifier 10 times does not grow total memory allocation. The memory dashboard shows per-model allocation tracked separately.

### Phase 4 — Circuit Breaker per Model

**The question this phase answers:** How does the gateway fail fast on a broken model without degrading healthy models?

**Key decisions:**
- **Circuit breaker states:**
  - **Closed (normal):** Requests pass through. Failure rate tracked in a sliding window (last 60 seconds). If failure rate > threshold (e.g., 50%) and minimum request count > 10, transition to Open.
  - **Open (fail fast):** All requests immediately return 503. No model is invoked. A half-open timer starts (e.g., 30 seconds).
  - **Half-open (probe):** After the timer fires, allow one request through. If it succeeds, transition to Closed. If it fails, reset the timer and stay Open.
- **Failure classification:** Not all failures should count toward the circuit breaker. Client errors (400 — bad input format) should not. Infrastructure errors (500 — OOM, timeout, CUDA error) should. The failure counter tracks infrastructure errors only.
- **ML-specific triggers:** For ML models specifically, add: (a) latency circuit breaker — if p95 latency exceeds 3× SLA for 60 seconds, open the circuit; (b) memory circuit breaker — if the model's GPU memory allocation exceeds its budget, open the circuit immediately without waiting for the failure rate threshold.
- **Health check integration:** The `GET /health/{model_name}` endpoint returns: `{state: "closed|open|half-open", failure_rate: 0.12, last_failure: "2024-01-15T10:23:00Z", gpu_memory_mb: 2048}`. The orchestrator (Kubernetes readiness probe) uses this to decide whether to route traffic.

**Trap:** Using request count as the sole circuit breaker trigger. A model that processes 1 request per minute and fails 1 in 2 (50% failure rate) may not meet the "minimum 10 requests" threshold for minutes. Add a time-based trigger: "3 consecutive failures within 60 seconds → open circuit regardless of total request count."

**Done when:** Simulating a model that fails every request: the circuit opens after 10 failures, requests fail fast with 503 (no model invocation), the circuit transitions to half-open after 30 seconds, one probe request is sent, and if the model is still failing, the circuit reopens. Measured: latency of a 503 from open circuit < 5ms vs. ~2000ms when the circuit is closed and the model is failing.

### Phase 5 — Zero-Downtime Model Versioning

**The question this phase answers:** How do you swap model_v1 for model_v2 without dropping requests in flight?

**Key decisions:**
- **Blue-green model swap:** Load `model_v2` alongside `model_v1` in the registry. Gradually shift traffic: 10% to v2, monitor error rate and latency. If healthy after 5 minutes, shift 50%, then 100%. The traffic split is implemented in the gateway's routing layer: `route(request) → v1 or v2` based on configurable weights.
- **In-flight request protection:** When initiating a swap, mark v1 as `DRAINING`: new requests route to v2, but existing in-flight requests on v1 complete normally. Check `active_requests["model_v1"] == 0` before unloading v1. Draining timeout: if in-flight requests take more than 30 seconds, cancel them (they are likely stuck) and proceed with unload.
- **Memory constraint during swap:** Running v1 and v2 simultaneously requires enough GPU memory for both. For a model that uses 7 GB (summarizer), a simultaneous blue-green swap needs 14 GB — potentially impossible. Alternative: rolling swap — unload v1, load v2, no overlap. This has a brief unavailability window (< 1 second for model loading if weights are memory-mapped). The choice between blue-green and rolling depends on the model's memory footprint vs. available GPU budget.
- **Version routing header:** Clients can pin to a specific version with `X-Model-Version: 1` or `X-Model-Version: 2`. The gateway respects this header — useful for testing, A/B evaluation, or legacy clients that haven't been validated with v2.

**Trap:** Checking `active_requests == 0` without a timeout. If a client holds a connection open indefinitely (e.g., a stuck streaming response), the drain will never complete. Always combine drain check with a timeout: drain for at most 30 seconds, then forcibly unload.

**Done when:** A version swap on the text classifier (small model, fast to load) completes without any request returning a 5xx error. Load test running during swap shows no dropped requests, a brief latency spike during the overlap period, and stable latency after v2 is fully loaded.

## Data Flow

```
HTTP Request → Gateway Router
                    │
                    ├── Route by model_name
                    │
                    ▼
           Circuit Breaker Check
              │ open  → 503 immediately
              │ closed ↓
           Model Registry Lookup
              │ not loaded → 503
              │ loading    → 503
              │ warm ↓
           active_requests[model] += 1
              │
              ▼
           BatchCollector.enqueue(input) → Future
              │                               │
              │  background thread:           │
              │  wait (batch_size or timeout) │
              │  batch = drain_queue()        │
              │  output = model.forward(batch)│
              │  futures[i].set_result(out[i])│
              │                               │
              └───────────── await Future ────┘
              │
           active_requests[model] -= 1
              │
           Record metrics (latency, success/fail)
              │
           Circuit Breaker update (failure rate)
              │
           Return HTTP response
```

## Edge Cases to Handle Before You Code

1. **All three models are loading simultaneously at startup:** The total memory requirement exceeds 15 GB temporarily if all are in `LOADING` state at once. Implement sequential startup: load classifier first (smallest), then embedder, then summarizer. Monitor `torch.cuda.mem_get_info()` after each load.

2. **A request arrives for an unloaded model:** The gateway should not auto-load the model on demand — cold load takes 5–30 seconds and the request will time out. Return 503 with body `{"error": "model not loaded", "load_endpoint": "/models/summarizer/load"}`. Demand-loading is a separate explicit operation, not implicit per-request behavior.

3. **Two concurrent version swaps for different models:** Swap for text-classifier (draining v1, loading v2) while simultaneously swapping image-embedder. Memory pressure calculation must account for both ongoing swaps. The swap controller must serialize: only one swap active at a time, or check combined memory budget before starting each swap.

4. **The batch collector's background thread crashes:** An unhandled exception in the batch thread kills it silently. All futures in the queue will never resolve — callers hang indefinitely. Wrap the batch loop in a `try/except BaseException`, log the crash, set exceptions on all pending futures, and restart the thread. The circuit breaker will open if the crash causes request failures.

## The Hard Design Decisions

| Decision | Option A | Option B | When to pick A |
|---|---|---|---|
| Model process isolation | Separate OS process per model | Single process, soft memory limits | Separate processes when genuine memory isolation is required; single process for simpler deployment with less overhead |
| Blue-green vs. rolling swap | Blue-green: run both versions simultaneously | Rolling: unload v1, then load v2 | Blue-green when models are small enough to fit both; rolling when GPU memory is tight |
| Circuit breaker failure definition | Infrastructure errors only (5xx, timeout, OOM) | Any failure including client errors (4xx) | Infrastructure errors only — client errors are the caller's fault and should not degrade the model's health status |
| Batch timeout per model vs. global | Per-model timeout tuned to SLA | Single global timeout | Always per-model — summarizer can afford 100ms queue wait; classifier cannot |

## What Makes This Non-Trivial

GPU memory is not a simple pool. PyTorch's caching allocator, CUDA's virtual memory, and the OS's memory management interact in ways that make "available memory" a multi-layered concept. A gateway that treats GPU memory like CPU heap memory will experience fragmentation, unexpected OOMs, and memory that "doesn't come back" after unloading. The only reliable memory management strategy is: delete all references, call empty_cache, verify with memory_allocated.

The circuit breaker's half-open state requires careful timing. If the half-open probe request fails and the timer resets, but traffic volume is low, the circuit may never have enough successful probes to close. The circuit breaker needs a manual override endpoint (`POST /models/text-classifier/circuit-reset`) for operators to force-close after a fix is deployed.

Zero-downtime versioning becomes exponentially harder as model size grows. Swapping a 7 GB summarizer requires either 14 GB of free GPU memory (blue-green) or accepting a ~10-second unavailability window (rolling). The engineering trade-off is real and should be surfaced as a documented operational constraint, not buried in the implementation.

## Interview Angles This Project Develops

1. **"How do you prevent one model from OOMing others in a shared GPU environment?"** — Strong answer distinguishes soft limits (check before inference), hard limits (per-process memory_fraction), and the architectural solution (separate processes with IPC) — and explains the latency cost of each approach.

2. **"Explain circuit breaker states for an ML serving system."** — Strong answer gives all three states with precise transition conditions, explains why ML systems need latency-based and memory-based triggers in addition to standard failure rate triggers.

3. **"How does batching affect latency SLA?"** — Strong answer explains the latency/throughput trade-off, identifies batch_timeout as the tuning knob, explains how to set it per model based on SLA headroom, and knows that batch size is bounded by GPU memory per forward pass.

4. **"How do you swap a model to a new version without dropping requests?"** — Strong answer describes blue-green vs. rolling trade-offs, the draining pattern for in-flight requests, and the memory constraint that makes blue-green infeasible for large models.

5. **"Why isn't torch.cuda.empty_cache() sufficient for GPU memory management?"** — Strong answer distinguishes the PyTorch cache layer from CUDA's allocator, explains fragmentation, and identifies process restart as the only guaranteed defragmentation strategy.

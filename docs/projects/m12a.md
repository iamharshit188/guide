# Mini-RLHF Preference Tuner — Step-by-Step Guide

> **Module:** 12 — RLHF & Alignment · **Difficulty:** Expert · **Type:** GUIDED · **Time:** ~6 hours

## What You'll Build

A complete miniature RLHF pipeline: construct a preference dataset of 200 (prompt, chosen, rejected) triples, train a reward model using Bradley-Terry loss on top of sentence-transformer embeddings, evaluate the reward model for pairwise accuracy and length bias, use it to rank candidate responses, and simulate a REINFORCE policy gradient signal. Prints reward distributions, pairwise accuracy, and win rates — all terminal output, no plots.

## The Mental Model

RLHF teaches a model to produce outputs that humans prefer by first learning what "good" means (reward model), then optimizing the policy to score well under that learned criterion (RL fine-tuning). The reward model is trained on preference pairs: given two responses to the same prompt, humans label which is better. The Bradley-Terry model treats this as: the probability that response $a$ beats response $b$ is $P(a \succ b) = \sigma(r_\theta(a) - r_\theta(b))$, where $r_\theta$ is the scalar reward the model assigns.

The key insight is that you never need absolute reward values — only relative comparisons. This sidesteps the intractable problem of defining "what is a reward of 7.3?" and replaces it with the tractable problem of "which of these two responses is better?", which humans can answer reliably.

The REINFORCE signal connects reward model scores to policy gradient updates. If the policy generates response $a$ with log-probability $\log \pi_\theta(a|s)$ and the reward model assigns it reward $R$, the gradient is $\nabla_\theta \mathcal{L} = -\mathbb{E}[R \cdot \nabla_\theta \log \pi_\theta(a|s)]$. Responses with high reward get their probability increased; low reward responses get decreased. Baseline subtraction ($R - b$ where $b$ is the mean reward) reduces variance without introducing bias.

## Phase 1 — Preference Dataset Construction

**Why this first:** The dataset is the foundation of the entire pipeline. Data quality directly determines reward model quality — there is no algorithm that compensates for noisy or trivially-separated preference pairs.

```python
#!/usr/bin/env python3

from __future__ import annotations
import hashlib

from datasets import Dataset
from sentence_transformers import SentenceTransformer

def section(title: str) -> None:

# Extend to 200 pairs by paraphrasing with controlled variations
def build_preference_dataset(n: int = 200, seed: int = 42) -> Dataset:
    rng = np.random.default_rng(seed)
    prompts, chosens, rejecteds = [], [], []

    base_prompts = PROMPTS * (n // len(PROMPTS) + 1)
    base_chosen = (CHOSEN_TEMPLATES * (n // len(CHOSEN_TEMPLATES) + 1))
    base_rejected = (REJECTED_TEMPLATES * (n // len(REJECTED_TEMPLATES) + 1))

    for i in range(n):
        p_idx = i % len(PROMPTS)
        c_idx = i % len(CHOSEN_TEMPLATES)
        r_idx = i % len(REJECTED_TEMPLATES)

        # Add minor variation to avoid exact duplicates
        noise_words = ["Furthermore, ", "Notably, ", "In essence, ", "Specifically, ", ""]
        prefix = rng.choice(noise_words)

        prompt = base_prompts[p_idx]
        chosen = prefix + base_chosen[c_idx]
        rejected = base_rejected[r_idx]

        prompts.append(prompt)
        chosens.append(chosen)
        rejecteds.append(rejected)

    dataset = Dataset.from_dict({
        "prompt": prompts,
        "chosen": chosens,
        "rejected": rejecteds,
    })
    return dataset

    print(f"\nSample prompt:    {dataset[0]['prompt']}")
    print(f"Chosen length:    {len(dataset[0]['chosen'].split())} words")
    print(f"Rejected length:  {len(dataset[0]['rejected'].split())} words")

    print(f"\nAvg chosen len:  {avg_chosen:.1f} words")
    print(f"Avg rejected len: {avg_rejected:.1f} words")
    print(f"Length ratio:     {avg_chosen/avg_rejected:.2f}x  ← length bias risk if > 1.5")
```

**Checkpoint:** Dataset has 200 rows. `avg_chosen` is ~50–80 words; `avg_rejected` is ~10–15 words. Length ratio ~4–6x — note this for the length bias evaluation in Phase 3.

## Phase 2 — Reward Model Training (Bradley-Terry Loss)

**Why this second:** The reward model is the central component — everything downstream (ranking, REINFORCE signal) depends on it learning a useful reward signal from the preference pairs.

```python
    section("Phase 2 — Reward Model Training")

    # ── encoder backbone ─────────────────────────────────────────
    encoder = SentenceTransformer("all-MiniLM-L6-v2")  # 384-dim embeddings
    EMBED_DIM = 384

    class RewardModel(nn.Module):
        """Linear head on top of frozen sentence-transformer embeddings."""
        def __init__(self, embed_dim: int) -> None:
            super().__init__()
            self.head = nn.Linear(embed_dim, 1)  # scalar reward

        def forward(self, embeddings: torch.Tensor) -> torch.Tensor:
            # embeddings: (batch, embed_dim) → rewards: (batch, 1)
            return self.head(embeddings)

    # ── Bradley-Terry loss ────────────────────────────────────────
    def bradley_terry_loss(
        reward_chosen: torch.Tensor,   # (batch, 1)
        reward_rejected: torch.Tensor, # (batch, 1)
    ) -> torch.Tensor:
        """
        P(chosen > rejected) = σ(r_chosen - r_rejected)
        Loss = -log P = -log σ(r_chosen - r_rejected)
             = log(1 + exp(-(r_chosen - r_rejected)))
        Numerically stable via F.logsigmoid.
        """
        margin = reward_chosen - reward_rejected  # (batch, 1)
        loss = -torch.nn.functional.logsigmoid(margin).mean()
        return loss

    # ── embed all texts ───────────────────────────────────────────
    print("Encoding preference pairs with sentence-transformer...")
    # Encode prompt+response pairs concatenated (prompt context matters)
    chosen_texts = [
        f"{p} {c}"
        for p, c in zip(dataset["prompt"], dataset["chosen"])
    ]
    rejected_texts = [
        f"{p} {r}"
        for p, r in zip(dataset["prompt"], dataset["rejected"])
    ]

    chosen_embs = encoder.encode(
        chosen_texts, convert_to_tensor=True, show_progress_bar=False
    )   # (200, 384)
    rejected_embs = encoder.encode(
        rejected_texts, convert_to_tensor=True, show_progress_bar=False
    )   # (200, 384)

    print(f"Chosen embeddings shape:   {tuple(chosen_embs.shape)}")
    print(f"Rejected embeddings shape: {tuple(rejected_embs.shape)}")

    # ── train/val split ───────────────────────────────────────────
    n_train = 150
    chosen_train, chosen_val = chosen_embs[:n_train], chosen_embs[n_train:]
    rejected_train, rejected_val = rejected_embs[:n_train], rejected_embs[n_train:]

    # ── training loop ─────────────────────────────────────────────
    reward_model = RewardModel(EMBED_DIM)
    optimizer = torch.optim.AdamW(reward_model.parameters(), lr=3e-4, weight_decay=0.01)

    N_EPOCHS = 5
    BATCH_SIZE = 32

    n_batches = (n_train + BATCH_SIZE - 1) // BATCH_SIZE

    print(f"\nTraining for {N_EPOCHS} epochs, batch_size={BATCH_SIZE}")
    print(f"{'Epoch':>6} {'Train Loss':>12} {'Val Acc':>10}")
    print("-" * 30)

    for epoch in range(1, N_EPOCHS + 1):
        reward_model.train()
        epoch_loss = 0.0
        perm = torch.randperm(n_train)
        chosen_train_shuf = chosen_train[perm]
        rejected_train_shuf = rejected_train[perm]

        for b in range(n_batches):
            start = b * BATCH_SIZE
            end = min(start + BATCH_SIZE, n_train)
            c_batch = chosen_train_shuf[start:end]
            r_batch = rejected_train_shuf[start:end]

            r_chosen = reward_model(c_batch)
            r_rejected = reward_model(r_batch)
            loss = bradley_terry_loss(r_chosen, r_rejected)

            optimizer.zero_grad()
            loss.backward()
            optimizer.step()
            epoch_loss += loss.item()

        # Validation accuracy (pairwise)
        reward_model.eval()
        with torch.no_grad():
            r_c_val = reward_model(chosen_val)
            r_r_val = reward_model(rejected_val)
            correct = (r_c_val > r_r_val).float().mean().item()

        print(f"{epoch:>6} {epoch_loss/n_batches:>12.4f} {correct:>10.4f}")
```

**Checkpoint:** After 5 epochs, validation accuracy should be ≥0.80. Train loss decreases monotonically from ~0.69 (chance) to ~0.25–0.35. If validation accuracy is near 0.5, the reward model is not learning — check that embeddings are not being frozen when they should not be (they are pre-computed here, so the encoder is correctly decoupled from the training loop).

## Phase 3 — Reward Model Evaluation and Bias Analysis

**Why this third:** A reward model with 85% pairwise accuracy might still be useless if it achieves that accuracy purely by preferring longer responses. This phase separates genuine quality signal from surface-level confounds.

```python
    section("Phase 3 — RM Evaluation and Bias Analysis")

    reward_model.eval()

    with torch.no_grad():
        r_chosen_val = reward_model(chosen_val).squeeze()     # (50,)
        r_rejected_val = reward_model(rejected_val).squeeze() # (50,)

    # ── pairwise accuracy ─────────────────────────────────────────
    pairwise_acc = (r_chosen_val > r_rejected_val).float().mean().item()
    print(f"Pairwise accuracy (held-out 50 pairs): {pairwise_acc:.4f}")

    # ── reward distributions ─────────────────────────────────────
    print(f"\nChosen  rewards — mean: {r_chosen_val.mean():.4f}  "
          f"std: {r_chosen_val.std():.4f}  "
          f"min: {r_chosen_val.min():.4f}  max: {r_chosen_val.max():.4f}")
    print(f"Rejected rewards — mean: {r_rejected_val.mean():.4f}  "
          f"std: {r_rejected_val.std():.4f}  "
          f"min: {r_rejected_val.min():.4f}  max: {r_rejected_val.max():.4f}")
    print(f"Mean margin (chosen - rejected): "
          f"{(r_chosen_val - r_rejected_val).mean():.4f}")

    # ── length bias analysis ──────────────────────────────────────
    # Does the RM always prefer longer responses regardless of content?
    val_chosen_texts_list = chosen_texts[n_train:]
    val_rejected_texts_list = rejected_texts[n_train:]

    chosen_lens = np.array([len(t.split()) for t in val_chosen_texts_list])
    rejected_lens = np.array([len(t.split()) for t in val_rejected_texts_list])

    # Length-controlled: look at pairs where chosen is SHORTER than rejected
    shorter_chosen_mask = chosen_lens < rejected_lens
    n_shorter = shorter_chosen_mask.sum()

    if n_shorter > 0:
        shorter_acc = (
            r_chosen_val[shorter_chosen_mask] > r_rejected_val[shorter_chosen_mask]
        ).float().mean().item()
        print(f"\nLength bias check:")
        print(f"  Pairs where chosen is shorter: {n_shorter}/{len(chosen_lens)}")
        print(f"  RM accuracy on those pairs:    {shorter_acc:.4f}")
        print(f"  (If < 0.5, RM is length-biased and wrong when longer≠better)")
    else:
        print("\n  All chosen responses are longer — cannot test length bias separately.")
        print("  In a real RLHF pipeline: construct adversarial length-controlled pairs.")

    # ── score separation visualization ───────────────────────────
    margins = (r_chosen_val - r_rejected_val).numpy()
    bins = [-2, -1, 0, 0.5, 1, 2, float("inf")]
    labels = ["< -1", "-1–0", "0–0.5", "0.5–1", "1–2", "> 2"]
    counts = np.histogram(margins, bins=bins)[0]
    print(f"\nMargin distribution (chosen_score - rejected_score):")
    for label, count in zip(labels, counts):
        bar = "█" * count
        print(f"  {label:>8}: {bar} ({count})")
```

**Checkpoint:** Pairwise accuracy ≥ 0.80. Mean chosen reward > mean rejected reward with clear separation. If `shorter_acc < 0.5`, the model is length-biased — this is expected in this synthetic dataset since chosen responses are systematically longer. Note it explicitly: this is a known failure mode of real RLHF reward models.

## Phase 5 — REINFORCE Policy Gradient Signal Demo

**Why this fifth:** This phase connects the reward model to the policy gradient math, showing how reward scores translate into parameter update directions — the core of the RL fine-tuning step.

```python
    section("Phase 5 — REINFORCE Policy Gradient Signal")

    # Load GPT-2 as a toy policy
    print("Loading GPT-2 as toy policy...")
    gpt2_tokenizer = GPT2Tokenizer.from_pretrained("gpt2")
    gpt2_model = GPT2LMHeadModel.from_pretrained("gpt2")
    gpt2_tokenizer.pad_token = gpt2_tokenizer.eos_token
    gpt2_model.eval()

    RL_PROMPT = "Explain gradient descent."
    N_SAMPLES = 8
    MAX_NEW_TOKENS = 40

    print(f"Sampling {N_SAMPLES} responses from GPT-2...")

    # Sample responses from policy
    input_ids = gpt2_tokenizer.encode(RL_PROMPT, return_tensors="pt")
    responses, log_probs_list = [], []

    with torch.no_grad():
        for _ in range(N_SAMPLES):
            output = gpt2_model.generate(
                input_ids,
                max_new_tokens=MAX_NEW_TOKENS,
                do_sample=True,
                temperature=1.0,
                pad_token_id=gpt2_tokenizer.eos_token_id,
            )
            # Decode only the generated part
            generated_ids = output[0][input_ids.shape[1]:]
            response_text = gpt2_tokenizer.decode(generated_ids, skip_special_tokens=True)
            responses.append(response_text)

    # Compute log-probs for each response under the policy
    # (for a real REINFORCE, these would be tracked during generation)
    for response in responses:
        full_text = RL_PROMPT + " " + response
        tokens = gpt2_tokenizer(full_text, return_tensors="pt")
        with torch.no_grad():
            logits = gpt2_model(**tokens).logits  # (1, seq_len, vocab)
        # Shift: predict token t from token t-1
        log_probs = torch.nn.functional.log_softmax(logits[0, :-1], dim=-1)
        token_ids = tokens.input_ids[0, 1:]  # target tokens
        token_log_probs = log_probs[range(len(token_ids)), token_ids]
        total_log_prob = token_log_probs.sum().item()
        log_probs_list.append(total_log_prob)

    # Score responses with reward model
    response_texts = [f"{RL_PROMPT} {r}" for r in responses]
    response_embs = encoder.encode(
        response_texts, convert_to_tensor=True, show_progress_bar=False
    )
    with torch.no_grad():
        rewards = reward_model(response_embs).squeeze().numpy()

    log_probs_arr = np.array(log_probs_list)

    # REINFORCE with baseline (mean reward subtracted to reduce variance)
    # ∇θ L = -E[(R - b) · ∇θ log π_θ(a|s)]
    # Gradient magnitude proxy: |(R - baseline) * log_prob|
    baseline = rewards.mean()
    advantages = rewards - baseline

    # Policy gradient "signal" magnitude: how strongly each response
    # would push the policy. Positive advantage + high log_prob = large update.
    gradient_magnitudes = np.abs(advantages * log_probs_arr)

    print(f"\nPrompt: '{RL_PROMPT}'\n")
    print(f"{'#':>2}  {'Reward':>8}  {'Advantage':>10}  {'LogProb':>10}  "
          f"{'GradMag':>10}  Response[:50]")
    print("-" * 90)
    for i in range(N_SAMPLES):
        snippet = responses[i][:50].replace("\n", " ")
        print(f"{i+1:>2}  {rewards[i]:>8.4f}  {advantages[i]:>10.4f}  "
              f"{log_probs_arr[i]:>10.2f}  {gradient_magnitudes[i]:>10.4f}  {snippet}")

    print(f"\nBaseline reward (mean):   {baseline:.4f}")
    print(f"Max advantage:            {advantages.max():.4f}  ← this response gets boosted")
    print(f"Min advantage:            {advantages.min():.4f}  ← this response gets suppressed")
    print(f"Reward std:               {rewards.std():.4f}")

    # Win rate: how often would best-of-8 sampling beat a random sample?
    best_reward = rewards.max()
    win_rate = (rewards > 0).mean()
    print(f"\nBest-of-{N_SAMPLES} reward:      {best_reward:.4f}")
    print(f"Win rate vs. baseline=0:  {win_rate:.2%}")


if __name__ == "__main__":
    main()
```

**Checkpoint:** Rewards vary across responses — GPT-2 generates low-quality completions, so most rewards should be low. The advantage column shows positive values for above-average responses and negative for below-average. Gradient magnitudes should be largest for responses with extreme (very positive or very negative) advantages combined with high log-probability. Win rate vs baseline=0 should be 50–70%.

## Debugging Guide

| Symptom | Likely Cause | Fix |
|---|---|---|
| Pairwise accuracy stuck at 0.5 after 5 epochs | Embeddings are the same for all inputs | Check encoder is loaded; verify `chosen_embs` and `rejected_embs` differ with `torch.allclose(chosen_embs[0], rejected_embs[0])` → should be False |
| Train loss doesn't decrease from ~0.693 | Learning rate too small or reward head not receiving gradients | Print `list(reward_model.parameters())` to verify head weights are non-zero; increase lr to 1e-3 |
| All candidate scores are identical | Reward head weights are near-zero (not converged) | Check training loss at epoch 1 is < 0.5; if not, increase lr or check optimizer targets `reward_model.parameters()` not the frozen encoder |
| GPT-2 generation hangs | Infinite loop in generation (eos_token not set as pad) | Confirm `gpt2_tokenizer.pad_token = gpt2_tokenizer.eos_token` and `pad_token_id=gpt2_tokenizer.eos_token_id` in generate() call |
| All advantages are near zero | Rewards have near-zero variance | GPT-2 generations are nearly identical; increase temperature to 1.2 or increase N_SAMPLES to 16 |

## What This Project Teaches You to Say in an Interview

1. **"Bradley-Terry loss learns relative preferences, not absolute reward values."** The model learns $r_\theta(a) - r_\theta(b)$, not individual values — this is why any constant offset in the reward is irrelevant, and why scaling the reward by a constant doesn't change the ranking.

2. **"The biggest practical failure mode of reward models is length bias."** Real RLHF reward models systematically prefer longer responses because human raters tend to rate detailed responses higher, even when length adds no information. Length-controlled pairwise accuracy is the correct diagnostic.

3. **"REINFORCE has high variance, which is why we subtract a baseline."** The baseline $b = \mathbb{E}[R]$ reduces variance without changing the expected gradient because $\mathbb{E}[\nabla_\theta \log \pi_\theta \cdot b] = b \mathbb{E}[\nabla_\theta \log \pi_\theta] = 0$ (log-derivative identity).

4. **"Best-of-N sampling is a cheap alternative to RL fine-tuning."** Instead of gradient updates, generate N candidates, score with RM, return the best. At N=8, this achieves significant quality improvement without a training loop — useful when the policy cannot be fine-tuned.

5. **"Sentence-transformer embeddings are a valid proxy for preference learning at small scale, but they miss instruction-following and factual correctness signals that matter at production scale."** In production, the reward model backbone is the same architecture as the policy (e.g., a 7B LM), fine-tuned from preference data with the last layer replaced by a scalar head.
